### v0.7.2: Tagging System
**Goal**: Let users categorize tracks by what they ARE — mood, energy, context, vibe — so they can find and filter tracks by meaning, not just by how much they like them. Tagging happens alongside preference-setting during triage: the user yah's a track and tags it `mood:chill` in the same gesture.

**Context**: The curator organizes tracks into hundreds of themed Spotify playlists in folders — "Chill Vibes", "Workout", "Late Night", genre-specific lists. These playlists function as tags, but they're locked inside Spotify: you can't query across them ("all my chill tracks with 5+ plays"), use them in Mixd workflows, or own them independently. Tags in Mixd replace this system. The curator's ongoing triage flow also involves tagging: after deciding to yah a track, they immediately add it to one or two themed playlists. In Mixd, this becomes setting a preference AND adding tags in the same gesture on the Track Detail page.

**What this unlocks**:
- "What is this track?" — `mood:chill`, `energy:high`, `context:workout`, `vibe:late-night`, or just `banger`
- Cross-tag queries that Spotify playlists can't do — "all my chill tracks with 5+ plays"
- Tags as first-class workflow filters (v0.7.5)
- Bulk bootstrap from existing Spotify playlists (v0.7.4)

**Persona**: Weekly Curator — "I already categorize everything into themed playlists. I want that categorization in Mixd where I can actually query across it."

**Manual test**: Open Track Detail → type `mood:chill` in the tag input → autocomplete suggests it → tag appears as a chip → navigate to Library → filter by `mood:chill` → the track appears. Remove the tag → it disappears from the filter.

**Key Design Decisions**:
- **Follows shared patterns from v0.7.0** — source priority, event log, source timestamps. See [Shared Patterns](v0.7.0-1.md#shared-patterns-established-in-v070-reused-by-v072-tags-and-v074-playlist-bootstrap). Tags use `MetadataSource` and `should_override()` from `sourced_metadata.py`. `tagged_at` follows the same NOT NULL / caller-provided / timezone-aware pattern as `preferred_at`.
- **Normalization keeps data clean at scale** — the user will have hundreds of tags across thousands of tracks. Without normalization, `Mood:Chill`, `mood:chill`, and `mood: chill` would be three different tags. Tags are stored lowercase, trimmed, whitespace collapsed. Max 64 chars, allowed characters `[a-z0-9 _-:]`.
- **Namespace convention provides optional structure** — `mood:chill` splits into namespace `mood`, value `chill` on first colon. Tags without a colon (`banger`) have no namespace. Leading/trailing colons rejected.
- **No predefined taxonomy** — UI suggests common namespaces (`mood:`, `energy:`, `context:`, `genre:`) via autocomplete but doesn't enforce them.
- **Batch operations at 15k scale** — "tag all tracks in this playlist as mood:chill" must work as a single operation. Batches are atomic: if any tag fails validation, the whole batch is rejected. `ON CONFLICT DO NOTHING` handles duplicates silently within valid batches. Hard cap at 15,000 `track_ids` per request.

#### Tagging Foundation Epics

- [ ] **Tag Domain Model**

    **Story**: The tagging system needs normalization rules and parsing
    logic unique to tags — the user types "Mood:Chill" and it must
    resolve to `mood:chill`, split into namespace `mood` + value `chill`.
    Without centralized normalization, the tag list fills with
    near-duplicates that fragment queries and autocomplete. The source
    tracking and event log patterns are shared with preferences (see
    [Shared Patterns](v0.7.0-1.md#shared-patterns-established-in-v070-reused-by-v072-tags-and-v074-playlist-bootstrap)).

    **Decisions**:
    - `normalize_tag()` and `parse_tag()` are pure domain functions,
      co-located in the same module. The character set rule and the
      colon-split rule must stay in sync.
    - Leading/trailing colons rejected — `:value` and `namespace:`
      would break namespace queries and autocomplete.
    - `source` uses `MetadataSource` from `sourced_metadata.py` (shared).
    - `tagged_at` follows shared pattern #3 — required, not defaulted.

    **Spec**:
    - `src/domain/entities/tag.py`: `TrackTag`, `TagEvent`, `normalize_tag`, `parse_tag`
    - `TrackTag(user_id, track_id, tag, namespace, value, tagged_at, source, id)` — namespace/value derived from tag at creation
    - `TagEvent(user_id, track_id, tag, action, source, tagged_at, id)` — follows shared event log pattern, `action: Literal["add", "remove"]`
    - `normalize_tag(raw: str) -> str`: lowercase, strip, collapse whitespace, validate chars (`[a-z0-9 _-:]`), max 64, reject empty
    - `parse_tag(normalized: str) -> tuple[str | None, str]`: split on first colon, reject leading/trailing colons
    - Repository protocol: `get_tags`, `get_tags_batch`, `add_tag`, `remove_tag`, `add_tags_batch`, `list_all_tags`, `find_tracks_by_tag`, `find_tracks_by_namespace`, `count_by_tag`, `find_tags_by_date`
    - Add `get_tag_repository()` to `UnitOfWorkProtocol`

    **Tests**:
    - (unit) `normalize_tag("Mood:Chill")` → `"mood:chill"`
    - (unit) `normalize_tag("  hello   world  ")` → `"hello world"`
    - (unit) `normalize_tag(":value")` / `normalize_tag("namespace:")` / `normalize_tag(":")` → raises `ValueError`
    - (unit) `normalize_tag("a" * 65)` → raises `ValueError`
    - (unit) `normalize_tag("cafe!")` → raises `ValueError`
    - (unit) `parse_tag("mood:chill")` → `("mood", "chill")`
    - (unit) `parse_tag("mood:chill:vibes")` → `("mood", "chill:vibes")` (first colon only)
    - (unit) `parse_tag("banger")` → `(None, "banger")`

    Effort: S | Dependencies: v0.7.0 (sourced_metadata.py) | Status: Not Started

- [ ] **Tag Database Schema & Repository**

    **Story**: The user wants to tag thousands of tracks, search by tag,
    and get autocomplete suggestions as they type. This needs efficient
    indexing — especially trigram search for autocomplete ("moo" →
    "mood:chill"). The event log follows the shared pattern from v0.7.0.

    **Decisions**:
    - GIN trigram index on `tag` for autocomplete — requires
      `CREATE EXTENSION IF NOT EXISTS pg_trgm` in the migration.
      This is tag-specific; preferences don't need trigram search.
    - `ON CONFLICT DO NOTHING` for batch operations — silently skip
      duplicates. Validation at Pydantic/domain layer.
    - `add_tags_batch()` must also bulk-INSERT event rows — row-by-row
      at 15k scale defeats the batch optimization.
    - Other patterns (ON DELETE CASCADE, timezone-aware timestamps,
      multi-user UNIQUE, UoW wiring) follow shared patterns from v0.7.0.

    **Spec**:
    - `track_tags` table: `id`, `user_id` (FK→users), `track_id` (FK→tracks, ON DELETE CASCADE), `tag` (String(64)), `namespace` (String(32), nullable), `value` (String(64)), `source` (String(32)), `tagged_at` (DateTime(timezone=True), NOT NULL), `created_at`
    - `track_tag_events` table: follows shared event log pattern — `id`, `user_id`, `track_id`, `tag`, `action` (CHECK IN ('add','remove')), `source`, `tagged_at` (DateTime(timezone=True)), `created_at`
    - UNIQUE on `(user_id, track_id, tag)` — note: three-part key, unlike preferences' two-part `(user_id, track_id)`, because a track can have many tags
    - Index on `(user_id, tag)`, `(user_id, namespace)`, `(user_id, tagged_at)`
    - GIN trigram index on `tag` — migration includes `CREATE EXTENSION IF NOT EXISTS pg_trgm`
    - Wire `get_tag_repository()` into `DatabaseUnitOfWork`, extend `make_mock_uow()`

    **Tests**:
    - (integration) `ON CONFLICT DO NOTHING`: duplicate tag silently ignored
    - (integration) UNIQUE: different users can tag same track with same tag
    - (integration) ON DELETE CASCADE: deleting track removes all tags + events
    - (integration) Batch insert of 1000 tags succeeds with bulk event writes
    - (integration) Event row created on add and remove

    Effort: M | Dependencies: Tag Domain Model | Status: Not Started

- [ ] **Tag Use Cases & API**

    **Story**: The user wants to tag a track from the web UI or CLI,
    remove tags, browse their tag vocabulary (for autocomplete), and
    bulk-tag thousands of tracks at once (for playlist bootstrap in
    v0.7.4). The API is the shared interface for all of these — web UI,
    CLI, and the playlist import pipeline all call the same endpoints.

    **Decisions**:
    - Tag validation at the Pydantic layer (422 before hitting DB) —
      the user gets immediate feedback on invalid tags, not a database
      constraint error.
    - `DELETE /tracks/{id}/tags/{tag}` requires URL-encoding — colons
      are safe in paths, but spaces need `%20`. Server decodes and
      re-normalizes before lookup to avoid mismatches.
    - Batch endpoint capped at 15,000 `track_ids` — prevents accidental
      DoS from a massive payload. Returns 422 if exceeded.
    - Batches are atomic: one invalid tag rejects the whole batch.
      This prevents partial tagging where some tracks get tagged and
      others don't.
    - Batch endpoint deduplicates `track_ids` at the application layer
      before passing to the repository — prevents redundant DB writes
      and duplicate event log rows from repeated IDs in a single request.
    - `DELETE` path: server must apply `normalize_tag()` to the
      URL-decoded path segment before DB lookup — a client sending
      `mood%3Achill` vs `mood:chill` must match the same stored tag.

    **Spec**:
    - `POST /api/v1/tracks/{id}/tags` body `{tag: "mood:chill"}` — validates via `normalize_tag()`, 422 on invalid
    - `DELETE /api/v1/tracks/{id}/tags/{tag}` — URL-decoded then normalized before lookup
    - `POST /api/v1/tracks/tags/batch` body `{track_ids: [...], tag: "mood:chill"}` — capped at 15k, 422 if exceeded, `track_ids` deduplicated
    - `GET /api/v1/tags` — all tags with counts (for autocomplete)
    - `GET /api/v1/tags/{tag}/tracks` — tracks with a given tag
    - `GET /api/v1/tracks/{id}` response includes `tags: ["mood:chill", "banger"]`

    **Tests**:
    - (unit) `TagTrackUseCase` normalizes input before persisting
    - (integration) `POST` with invalid chars → 422
    - (integration) `POST` with 65-char tag → 422
    - (integration) `DELETE` with URL-encoded tag matches stored tag
    - (integration) Batch with 15,001 track_ids → 422
    - (integration) Batch with valid input → all tracks tagged
    - (unit) `BatchTagUseCase` with mix of valid/invalid tags → zero repo writes (atomicity enforced before DB)

    Effort: M | Dependencies: Tag Database Schema & Repository | Status: Not Started

- [ ] **Tag Web UI**

    **Story**: The user wants to tag tracks during triage — they yah a
    track and immediately tag it `mood:chill` in the same gesture on
    the Track Detail page. They also want to browse the Library by tag,
    select multiple tracks and bulk-tag them, and have autocomplete
    suggest existing tags as they type. For first-time users with no
    tags yet, the UI needs to teach the `namespace:value` convention
    and suggest starting points — otherwise staring at an empty tag
    field on track 1 of 15,000 won't teach the system.

    **Decisions**:
    - Tag input is positioned alongside the preference toggle on Track
      Detail — preference + tag is one triage gesture, not two separate
      pages. The user evaluates a track and records both their opinion
      and its category in one visit.
    - Cold-start "Suggested tags" panel with common namespaces
      (`mood:`, `energy:`, `context:`, `genre:`) and examples. Without
      this, adoption risk is high — the user's existing Spotify
      playlist workarounds would look easier than learning a new system.
    - Autocomplete searches existing tags via the trigram index — as the
      user builds their tag vocabulary, the system learns.

    **Spec**:
    - Track Detail: tag chips with add/remove, autocomplete from existing tags
    - Library: tag filter (multi-select dropdown), tag column (comma-separated chips)
    - Bulk tagging: select tracks in Library → "Tag selected" → autocomplete → confirm
    - Cold-start: suggested tags panel on first use with namespace examples
    - Tag management section: browse all tags, see track counts, rename (future)

    **Tests**:
    - (vitest) Tag chip add/remove interaction
    - (vitest) Autocomplete suggests existing tags
    - (vitest) Bulk tagging flow with selection

### v0.7.3: Playlist Browser
**Goal**: Let the user browse all their Spotify playlists from within Mixd, search by name, and pull individual playlists in — choosing whether Spotify or Mixd is the source of truth for each one.

**Context**: The user has hundreds of Spotify playlists but no way to see them from Mixd. Today, importing a playlist requires knowing its Spotify ID and using the CLI. The user needs a visual browser that shows their full Spotify library so they can pick playlists to import — either as managed playlists (for workflow output) or as sources for the tag/preference bootstrap in v0.7.4. This is reusable infrastructure: any future feature that needs the user to select a Spotify playlist builds on this.

**What this unlocks**:
- See all Spotify playlists from within Mixd — searchable, with track counts and thumbnails
- Pull any playlist into Mixd with one click, choosing the sync direction
- Prerequisite for v0.7.4: you can't map playlists to tags if you can't browse and select them first
- Reusable for any future feature that involves selecting a Spotify playlist

**Persona**: Weekly Curator — "I have hundreds of playlists on Spotify. I need to see them all in Mixd so I can pick the ones I want to bring in."

**Manual test**: Open the playlist browser → see all Spotify playlists with thumbnails and track counts → type "chill" → list filters to matching playlists → click "Import to Mixd" on one → choose "Spotify-managed" → playlist appears in Mixd's playlist list.

**Key Design Decisions**:
- **Spotify API returns a flat list, no folders** — `GET /me/playlists` has no search, no folder hierarchy (folders are a desktop-only concept). The UI provides client-side search/filter by name. At ~200 playlists, this is 4 API requests (50/page) which is fast enough to fetch all and filter locally.
- **`snapshot_id` for cheap change detection** — stored on import. On re-sync, if the snapshot hasn't changed, skip the playlist entirely. No need to re-fetch and diff the track list.
- **Source of truth per playlist** — the user chooses sync direction at import time via existing `DBPlaylistMapping.sync_direction`. Spotify-managed (PULL) for bootstrap playlists. Mixd-managed (PUSH) for workflow output playlists. Reuses existing infrastructure, no new concept.
- **Builds on existing infrastructure** — `DBConnectorPlaylist`, `DBPlaylistMapping`, `get_playlist_with_all_tracks()` already exist. The only missing piece is `get_current_user_playlists()` on the Spotify client.
- **Track resolution** — Spotify tracks not in the Mixd DB are skipped and logged. The user may not have imported their full library yet. Summary reports "X of Y tracks resolved."
- **Use `fields` query parameter** — when fetching playlist details for metadata-only operations, request only needed fields to avoid fetching full track objects unnecessarily.

#### Playlist Browser Epics

- [ ] **Spotify Playlist List API**

    **Story**: The user wants to see all their Spotify playlists from
    within Mixd — to browse, to pick one to import, or to find
    playlists to map to tags. Today Mixd can fetch a single playlist's
    tracks but can't list which playlists exist. This is the missing
    building block: a simple API call that returns the user's full
    Spotify playlist library.

    **Decisions**:
    - Auto-paginate through all results — the user has ~200 playlists,
      which is 4 requests at 50/page. Callers shouldn't have to handle
      pagination manually.
    - Cache in `DBConnectorPlaylist` — upsert on
      `(connector_name, connector_playlist_identifier)`. Avoids
      re-fetching the full list on every page load. Store `snapshot_id`
      for change detection in v0.7.4.
    - `force_refresh` flag on the use case — bypasses cache when the
      user explicitly wants fresh data. Otherwise, serve from cache
      if it exists.
    - Scope: `playlist-read-private` — already in OAuth config, no
      new scope needed.

    **Spec**:
    - Add `get_current_user_playlists(limit, offset)` to `SpotifyAPIClient` — calls `GET /me/playlists`
    - Fields to capture: `id`, `name`, `description`, `images`, `owner`, `snapshot_id`, `collaborative`, `public`, `tracks.total`
    - This is a metadata-only path — does NOT fetch full track objects. The full-track import (`get_playlist_with_all_tracks()`) is only called in the Playlist Import story when the user explicitly imports a playlist. Browsing 200 playlists must not trigger 200 full-track fetches.
    - `ListSpotifyPlaylistsUseCase` — fetches, upserts cache, returns list
    - Alembic migration to add `snapshot_id` column to `db_connector_playlists` table (if not already present)

    **Tests**:
    - (unit, mock client) Paginates through 150 playlists (3 pages of 50)
    - (unit, mock client) Caches results in `DBConnectorPlaylist`
    - (unit, mock client) Second call with cache populated → zero Spotify API calls (cache-hit path)
    - (integration) Returns playlist metadata with correct fields

    Effort: S | Dependencies: None | Status: Not Started

- [ ] **Playlist Browser Web UI**

    **Story**: The user wants to see their Spotify playlists in Mixd
    and act on them — import one, map it to a tag, or just browse.
    With hundreds of playlists and no server-side search in the
    Spotify API, the UI needs client-side type-ahead filtering so
    the user can quickly find "Chill Vibes" among 200 playlists.
    Each playlist shows its name, thumbnail, track count, and
    whether it's already been imported into Mixd.

    **Decisions**:
    - Client-side search by name — Spotify API has no server-side
      filter, and 200 playlists is small enough to filter locally
      with instant results.
    - Import status indicator per playlist (not imported / imported /
      mapped) — the user needs to see at a glance which playlists
      they've already brought in, especially during the bulk bootstrap
      in v0.7.4.
    - Batch select with checkboxes — the user will import many
      playlists at once during initial setup. One-at-a-time would
      be tedious with hundreds of playlists.
    - Lives as a standalone page or Settings section — reusable beyond
      v0.7.x, not buried in a modal.

    **Spec**:
    - Searchable list: type-ahead filter by name
    - Each row: playlist name, thumbnail, track count, owner, import status
    - Actions: "Import to Mixd", "Map to tag/preference" (v0.7.4 shortcut)
    - Batch select: checkbox multi-select for bulk operations
    - Sort by: name, track count, import status

    **Tests**:
    - (vitest) Search filters playlist list by name
    - (vitest) Batch select enables bulk actions
    - (vitest) Import status indicator reflects state

    Effort: M | Dependencies: Spotify Playlist List API | Status: Not Started

- [ ] **Playlist Import from Spotify**

    **Story**: The user wants to bring a specific Spotify playlist
    into Mixd — to manage it, to use it as a workflow source, or as
    a prerequisite for tagging its tracks in v0.7.4. They want to
    choose who owns the truth: Spotify-managed (Mixd reads from
    Spotify) or Mixd-managed (Mixd pushes to Spotify). This builds
    on existing playlist infrastructure, adding the inbound import
    direction.

    **Decisions**:
    - Sync direction chosen at import time — Spotify-managed
      (`PULL`) for bootstrap/read-only playlists, Mixd-managed
      (`PUSH`) for workflow output. Uses existing
      `DBPlaylistMapping.sync_direction`, no new concept.
    - Track resolution: match Spotify tracks to existing Mixd tracks.
      Unresolved tracks are logged, not errors — the user may not
      have imported their full library. Summary: "X of Y resolved."
    - `snapshot_id` stored on import — enables cheap "has this
      playlist changed?" detection for re-import without diffing
      the full track list.
    - Batch import from the playlist browser — select multiple
      playlists, import all with the same sync direction.

    **Spec**:
    - `ImportSpotifyPlaylistUseCase` — fetch metadata + tracks via `get_playlist_with_all_tracks()`, resolve tracks, create `DBPlaylist` + `DBConnectorPlaylist` + `DBPlaylistMapping`
    - Sync direction picker in UI: Spotify-managed / Mixd-managed
    - Batch import from playlist browser selection
    - Store `snapshot_id` for change detection

    **Tests**:
    - (unit, mock connector) Creates DBPlaylist + DBPlaylistMapping with correct sync_direction
    - (unit, mock connector) Unresolved tracks logged, not errored
    - (integration) Imported playlist appears in Mixd playlist list
    - (unit, mock connector) Re-import with unchanged `snapshot_id` → zero Spotify API calls (not just same end state)
    - (integration) Re-import with unchanged snapshot_id is a no-op

    Effort: M | Dependencies: Spotify Playlist List API | Status: Not Started

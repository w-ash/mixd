### v0.9.0: Workflow Assistant (Right-Panel Chat)

**Goal**: A persistent right-panel chat assistant that turns "I want a playlist of upbeat tracks I haven't played in 6 months" into a generated `WorkflowDef`, rendered in the existing visual editor as a confirmation preview, and saved on user approval. Same UX pattern as Cursor / Linear / Notion AI: contextual, always-available, no modal interruption.

**Context**: mixd today is genuinely useful by Curators and Tinkerers but has no front door for the Casual Enthusiast — building a workflow currently requires understanding the node catalog and connector model. v0.9.0 collapses that learning curve into a sentence of natural language. Per the README's persona-alignment table this is "**THE adoption enabler** — changes who can use mixd."

The architectural decisions for this milestone are not novel research — our sister project **couplefins** has a complete, production-tested right-panel chatbot system built deliberately on the same stack (Vite + React + TypeScript + FastAPI + Python + SQLAlchemy + Tanstack Query) so we could port it. v0.9.0 is **substantially a port**, not greenfield. The mixd-specific work is the domain primer, the node-catalog → JSON Schema bridge, and a workflow-preview render of generated DAGs. The infrastructure (SSE streaming, Anthropic adapter, tool dispatch, two-phase confirmation, prompt caching, error mapping, Zustand store, edge-tab affordance) is already-solved.

The 2026 chatbot-UX research aligns with couplefins's existing choices: POST+SSE streaming with sub-300ms TTFT, schema-constrained tool use via `output_config.format` and `strict: true`, ephemeral prompt caching (90% cost reduction at scale), server-side context injection rather than prop-passing, persistent right-panel rather than floating modal. We don't need to invent any of this — couplefins already shipped it.

**What this unlocks**:
- One-shot natural-language workflow generation with schema-constrained output (no few-shot prior needed — the WorkflowDef JSON Schema is the prior)
- Persistent right-panel chat (Cmd+K toggle, edge-tab when collapsed) that follows the user across all routes
- Workflow preview rendered in the existing v0.4.3 visual editor as a read-only DAG before save
- Two-phase confirmation: LLM proposes → user inspects in the editor → user approves → save (mutations never silent)
- Foundation for any future mixd-side AI features (track picking, tag suggestions, listening insights) — the same pattern, different tools

**Persona**: Casual Enthusiast — primary. The natural-language entry point is the front door for users who'd never have built a workflow from scratch. Tinkerer — secondary. Generates a starting point in seconds, then tweaks in the editor they already know. Weekly Curator — tertiary. Faster ad-hoc workflow creation when a specific need arises ("I want my Friday-night dinner-party playlist refreshed weekly with songs I've loved this month").

**Key Design Decisions**:
- **Port couplefins, don't rebuild**: The right-panel chat infrastructure exists in `/Users/wash/Projects/couplefins/`. We copy components, adapt names/types to mixd's domain, and reuse the underlying patterns. The mixd-specific surface area is small: domain primer, tools list, workflow preview card.
- **Schema-constrained tool use, not few-shot priors**: The LLM doesn't need template workflows as training examples. The `WorkflowDef` JSON Schema (machine-derived from the existing node registry) IS the prior. Modern frontier models produce correct DAGs in one shot when given the schema as a `tool_use` definition with `strict: true`. This **drops the v0.8.1 → v0.9.0 dependency the previous file claimed** — they're independent.
- **Workflow Templates from v0.8.1 are user-facing, not LLM-facing**: Templates serve as empty-state inspiration on "new workflow" pages and as starting points for users who prefer copy-and-modify over from-scratch. They are NOT the LLM's training data. v0.9.0 can ship before, after, or alongside v0.8.1 with no architectural coupling.
- **Reuse the v0.4.3 visual editor for previews**: The "Visualization Confirmation UI" isn't a new component — it's the existing visual editor rendered in read-only mode inside the chat panel's `ToolResultCard`. Same component, different prop. No parallel viz codebase to maintain.
- **Refinement is first-class in both chat and editor**: A user can iterate on an unsaved or saved workflow in chat ("make it 30 tracks instead of 20", "add a recency filter that excludes anything I played this month", "sort by play count descending") OR in the visual editor after save. Both pathways are first-class — chat is great for high-level intent shifts and quick parametric tweaks; the editor is great for fine-grained graph surgery. Each refinement turn that changes the workflow re-emits a fresh `generate_workflow_def` tool call with the updated **complete** DAG; the Workflow Preview Card replaces its prior preview rather than appending. The chat history is the LLM's diff context — no separate "update_workflow_draft" tool needed.
- **Server-side context injection**: The chat panel doesn't pass `<ChatPanel context={...}>` props. The FastAPI route resolves "who is this user, what's in their library, what workflow are they currently editing" from authenticated session + the route they came from, and bakes it into the system prompt. Couplefins proves this pattern; we follow it.
- **Two-phase confirmation for saves**: Following couplefins's mutation pattern. The LLM's `save_workflow` tool returns a "pending action" rather than persisting directly. Frontend renders ConfirmationCard with the proposed WorkflowDef preview. User approves → second SSE request executes the save. Prevents silent writes from a single hallucination.
- **Ephemeral conversations, no DB persistence**: Couplefins has no `conversations` table — chat state lives in the Zustand store, hard-capped at 50 messages, cleared on "New Conversation." v0.9.0 follows suit. If conversation history becomes a real user need post-launch, add it as a Post-Deploy Revision.
- **Anthropic Claude as the provider**: Couplefins uses `AsyncAnthropic` via `AnthropicAdapter`. We do the same. `output_config.effort="medium"` matches couplefins's choice. System prompt cached via `cache_control: {"type": "ephemeral"}` to hit the 2048-token activation threshold (which means the domain primer needs to be substantive — not a problem given mixd's node catalog).
- **In-memory rate limiter**: 20 req/60s per user, matching couplefins. Right-sized for mixd's "dozens to low hundreds of users" deployment shape. No Redis needed.

    - Layout integration in `AppLayout.tsx`: persistent right sidebar on md+ breakpoints, edge tab on small screens, Cmd+K binding, lazy-loaded with Suspense.
    - Width 384px (`w-96`) — matches couplefins; revisit if mixd's content density differs in practice.
    - 50-message hard limit, "New Conversation" clears, no DB persistence.

    **Spec**:

    **Tests**:

- [ ] **Port SSE Streaming Infrastructure**

    **Story**: When the assistant is generating a response or running a tool, the user wants to see incremental output (tokens streaming in, tool call progress badges) rather than waiting for a complete reply — TTFT under 300ms is the industry baseline for "feels responsive."

    **Decisions**:
    - Port `web/src/api/chat-sse.ts` directly — fetch + ReadableStream pattern (not EventSource because body is POST).
    - Port `src/interface/api/sse.py` — `_run_with_sentinel`, error code map, exception → SSE event mapping.
    - Five SSE event types: `token`, `tool_start`, `tool_result`, `done`, `error`. Same shape as couplefins.
    - Abort handling via `AbortSignal` — user-initiated cancel returns silently; mid-stream errors emit `STREAM_ERROR` event.
    - Buffered incomplete-markdown rendering via Streamdown (port the dep + the wrapper component).

    **Spec**:
    - `web/src/api/chat-sse.ts` — `sendChatMessage(messages, callbacks, signal, confirmation?)` matching couplefins's signature
    - `src/interface/api/sse.py` — `SSEEventQueue`, `_run_with_sentinel`, `_ERROR_CODE_MAP`
    - Frontend `Streamdown` wrapper for partial-markdown rendering (port from couplefins)
    - Tests cover: token streaming, tool events, abort mid-stream, HTTP error response, parse errors, network drop

    **Tests**:
    - (frontend integration) Token events render incrementally
    - (frontend integration) AbortSignal cancels stream silently
    - (frontend integration) Mid-stream error emits typed error code
    - (backend integration) FastAPI route streams SSE with correct content-type and event format

    Effort: S | Dependencies: Port Right-Panel Chat Shell | Status: Not Started

- [ ] **Port FastAPI Chat Route + Anthropic Adapter**

    **Story**: The backend wants a single `POST /api/v1/chat` endpoint that takes the user's message history (plus optional confirmation), opens an Anthropic streaming connection, dispatches any tool calls Claude makes, and emits SSE events back to the frontend — the same shape couplefins ships and that 2026 best practices recommend.

    **Decisions**:
    - Port `src/interface/api/routes/chat.py`, `src/interface/api/schemas/chat.py`, `src/infrastructure/chat/anthropic_adapter.py`, `src/application/chat/tool_executor.py`, `src/application/chat/voices/` from couplefins.
    - Adapter implements `LLMClientProtocol` (port from couplefins). `AsyncAnthropic.messages.stream()` with `output_config.effort="medium"`.
    - Two-phase mutation pattern: tools that mutate state return a "pending action" stored in `pending_action_store` (in-memory, TTL-bounded). Frontend confirmation triggers `_handle_confirmation` flow.
    - In-memory rate limiter: 20 req/60s per user (port).
    - Error mapping: typed exceptions → SSE error codes via `_ERROR_CODE_MAP` (port + extend with mixd-specific codes if needed).

    **Spec**:
    - New module `src/application/chat/` (mirrors couplefins's structure)
    - `src/infrastructure/chat/anthropic_adapter.py` (port)
    - `POST /api/v1/chat` route handler (port, adapt to mixd auth/session pattern)
    - Pending action store with 5-minute TTL (port)
    - `ANTHROPIC_API_KEY` env var via existing settings pattern

    **Tests**:
    - (integration) `/api/v1/chat` returns SSE stream with text/event-stream content type
    - (integration) Tool calls are dispatched and results streamed
    - (integration) Rate limit returns `RATE_LIMIT_EXCEEDED` error event after 20 req/60s
    - (integration) Pending action expiry returns `ACTION_EXPIRED` after TTL elapses
    - (integration) Confirmation request executes the pending action and broadcasts result

    Effort: S | Dependencies: Port SSE Streaming Infrastructure | Status: Not Started

- [ ] **Compose Mixd Domain Primer + System Prompt**

    **Story**: The LLM needs to understand mixd's domain — what a track is, what a workflow does, what nodes are available, what the user's library looks like, what makes a "good" workflow — so it can generate sensible WorkflowDefs without hallucinating node names or asking the user for information already available server-side.

    **Decisions**:
    - Port couplefins's `system_prompt.py` structure: XML-tagged sections (identity, scope, domain_model, response_format, mutation_rules), composed per-request, cached via `cache_control: {"type": "ephemeral"}`.
    - Domain primer must hit ≥2048 tokens to activate Sonnet 4.6 prompt caching — not a problem given mixd's node catalog and connector model.
    - Section content is mixd-specific: workflow node taxonomy (sources, filters, transforms, sorters, destinations), connector model, preference/tag system, library statistics for the current user.
    - Voices system: keep couplefins's pluggable voice pattern (default + opt-in personas) — could surface as a settings toggle post-launch.
    - System prompt rebuilt per request so it always reflects current user state (library size, recent imports, current workflow being edited if any).

    **Spec**:
    - `src/application/chat/system_prompt.py` — `build_system_prompt(user, library_stats, current_workflow, today)` → list of system blocks
    - Domain primer sections: workflow architecture, node catalog summary (full schema delivered separately as tool definition), connector identity model, common workflow patterns
    - First voice: "Default" — straightforward, helpful, no persona affectation. Voice system stays pluggable for later.
    - Single test that the assembled prompt contains expected sections + crosses 2048-token threshold

    **Tests**:
    - (unit) `build_system_prompt` produces valid XML-tagged structure with all required sections
    - (unit) Token count ≥2048 (cache activation threshold)
    - (integration) System prompt reflects current user's library stats and active workflow context

    Effort: M | Dependencies: Port FastAPI Chat Route + Anthropic Adapter | Status: Not Started

- [ ] **Workflow Generation Tool — Node Registry → JSON Schema**

    **Story**: The LLM wants to call `generate_workflow_def` with a structured argument matching mixd's actual `WorkflowDef` shape — so it produces valid DAGs that reference real nodes with correct parameters every time, without needing the user to teach it what's available.

    **Decisions**:
    - Define a `generate_workflow_def` tool whose `input_schema` is the WorkflowDef JSON Schema, derived mechanically from mixd's existing node registry.
    - Use `strict: true` on tool parameters so Anthropic enforces schema conformance at the API layer — invalid DAGs never reach our code.
    - Tool returns the generated `WorkflowDef` as a `tool_result`. Frontend receives via SSE and renders it in the Workflow Preview Card.
    - Sibling tools: `validate_workflow_def` (dry-run validation pass), `save_workflow` (mutation, two-phase confirmation), `get_workflow` (load existing for refinement).
    - Tool description is 3–4 sentences per Anthropic's tool-use guidance — clear contract about when the LLM should use each.

    **Spec**:
    - New file `src/application/chat/tools.py` — list of tool definitions following couplefins's pattern
    - New helper `src/application/chat/workflow_schema.py` — `build_workflow_def_schema()` walks the node registry and produces a JSON Schema with `oneOf` per node type
    - Initial tool list: `generate_workflow_def`, `validate_workflow_def`, `save_workflow`, `get_workflow`, `list_user_workflows`, `describe_node` (for the rare case the LLM wants to confirm a node's parameter shape)
    - `save_workflow` is a mutation: returns pending action, requires confirmation
    - All tools dispatched via `tool_executor.py` (ported from couplefins, adapted to mixd's UoW pattern)

    **Tests**:
    - (unit) `build_workflow_def_schema` produces a JSON Schema with all current node types
    - (unit) Schema rejects unknown node types
    - (unit) Schema enforces required parameters per node
    - (integration) End-to-end: send "Build me a chill weekend playlist" → assistant calls `generate_workflow_def` → response contains valid WorkflowDef
    - (integration) `save_workflow` returns pending action, not silent persist

    Effort: M | Dependencies: Compose Mixd Domain Primer + System Prompt | Status: Not Started

- [ ] **Workflow Preview ToolResultCard (Reuses Visual Editor)**

    **Story**: When the assistant proposes a `WorkflowDef`, the user wants to see it as the actual graph they'd see in the editor — same node shapes, same connections, same colors — so they can read it at a glance and decide whether to save without learning a new visualization.

    **Decisions**:
    - Render the proposed `WorkflowDef` using the existing v0.4.3 visual editor in **read-only mode**, embedded inside the chat panel's `ToolResultCard`.
    - Editor must already support a read-only prop or be trivially extended to one — verify during implementation.
    - Below the embedded preview: "Save" and "Discard" buttons. "Save" emits the confirmation request; "Discard" tells the assistant to try again with feedback.
    - Card layout adapts to the chat panel's 384px width — likely shows a compressed view + "Open in editor" link for full-size inspection.
    - **Refinement-aware**: when a subsequent chat turn produces another `generate_workflow_def` tool call, the existing preview card **replaces its prior contents** rather than the chat appending a new card. The Zustand store tracks `currentWorkflowDraft` keyed by an action_id; new tool calls update the same draft. Save acts on whatever the latest preview shows. This keeps the panel readable across multi-turn refinement (no scroll-back to find the right card).
    - **Save vs Update**: if the chat session began with `get_workflow(id)` (i.e. the user is editing a saved workflow), the Save button maps to `update_workflow(id, new_def)` instead of `save_workflow(new_def)`. The card's button label changes accordingly ("Save changes" vs "Save workflow").

    **Spec**:
    - New component `web/src/components/chat/cards/WorkflowPreviewCard.tsx`
    - Reuses `WorkflowVisualEditor` (or whatever the v0.4.3 component is named) with `readOnly={true}`
    - Tool result handler in `ToolResultCard.tsx` dispatches on `tool_name === "generate_workflow_def"` to this card
    - Zustand store gains `currentWorkflowDraft: { action_id, def, source: "new" | { workflow_id } } | null`
    - Subsequent `generate_workflow_def` tool results with the same chat session replace the active draft, not append a new card
    - "Save" / "Save changes" button triggers SSE request with `confirmation: {action_id, approved: true}`
    - "Discard" appends a synthetic user message ("Try a different approach") and re-streams

    **Tests**:
    - (component) Renders proposed WorkflowDef as a graph
    - (component) Subsequent regeneration replaces the preview in place, not appended
    - (component) Save button label flips between "Save workflow" (new) and "Save changes" (editing existing)
    - (component) Discard button triggers regeneration
    - (E2E) Generate → refine in chat → preview updates → save → workflow matches latest preview
    - (E2E) Load existing → refine in chat → save changes → workflow updated, not duplicated

    Effort: M | Dependencies: Workflow Generation Tool, Port Right-Panel Chat Shell | Status: Not Started

- [ ] **LLM Feedback Collection (Thumbs Up/Down)**

    **Story**: The product wants to learn which prompt patterns produce accepted vs rejected workflows so the system prompt and tool descriptions can be iteratively refined — without building an A/B testing framework or analytics dashboard.

    **Decisions**:
    - Port couplefins's feedback pattern: thumbs-up / thumbs-down per generation, optional free-text note.
    - Store per-generation: user prompt, generated WorkflowDef, feedback signal, optional note.
    - **Not building**: A/B testing harness, feedback insights dashboard, automated model improvement framework. Manual prompt iteration is the right scale.
    - Feedback button appears on the workflow preview card. Save = implicit positive signal; explicit thumbs-up = strong signal; thumbs-down with note = the most actionable.

    **Spec**:
    - DB table `chat_feedback`: `id, user_id, prompt, generated_workflow_def (jsonb), signal (Literal["positive", "negative"]), note (nullable), created_at`
    - `POST /api/v1/chat/feedback` route
    - Frontend wiring on the workflow preview card

    **Tests**:
    - (integration) Feedback persists with full context
    - (component) Thumbs buttons appear on preview card and submit cleanly

    Effort: S | Dependencies: Workflow Preview ToolResultCard | Status: Not Started

# v0.9.x: LLM-Assisted Workflow Creation

For strategic overview, see the [planning overview](README.md).

**User Workflow Context**: Building a workflow today means learning the node catalog ŌĆö sources, filters, sorters, connector identities ŌĆö before the first playlist ever materializes. Curators and Tinkerers climb that curve; the Casual Enthusiast bounces off it. Meanwhile power users increasingly run LLM agents (Claude Desktop, Cursor, local toolchains) and expect their tools to be reachable from them. v0.9.x addresses both: natural-language workflow creation in the web UI (v0.9.0), and mixd as a tool surface any MCP-aware agent can drive (v0.9.1) ŌĆö built on one shared tool registry so the two never drift.

### Pre-Work: 2026 MCP + Chatbot Best-Practices Pass & Sequencing Review

> **Status: Ō£ģ Completed 2026-07-02 (Fable window).** The brief below replaces the original pre-work assignment. The v0.9.0 / v0.9.1 sections that follow were re-sequenced and corrected against it and are now the committed plan (no longer "pre-pre-work draft"). Primary sources, all read 2026-07-02: MCP spec 2025-11-25 (transports, authorization, tools, `schema.ts`), the MCP 2026-07-28 release-candidate announcement, Anthropic docs (prompt caching, structured outputs, MCP connector), and the couplefins source at HEAD `da407c5`.

#### State of the art, July 2026

**MCP servers.** Current spec revision is **2025-11-25**. Two standard transports: **stdio** (client launches the server as a subprocess; "clients SHOULD support stdio whenever possible") and **Streamable HTTP** (single `/mcp` endpoint, optional SSE streaming; the old HTTP+SSE transport is deprecated). Authorization is optional and transport-scoped: HTTP servers act as OAuth 2.1 resource servers (RFC 9728 protected-resource metadata, RFC 8707 resource indicators, PKCE), while **stdio servers explicitly SHOULD NOT use OAuth ŌĆö they "retrieve credentials from the environment"**. Tools carry optional behavior annotations ŌĆö `readOnlyHint` (default false), `destructiveHint` (default true), `idempotentHint` (default false), `openWorldHint` (default true) ŌĆö but the spec is blunt that **clients MUST treat annotations as untrusted**; safety gates belong on the server. Tools may declare an `outputSchema` and return `structuredContent`; input-validation failures are returned as `isError: true` tool results phrased as actionable feedback the model can self-correct from. One sequencing fact matters: the **2026-07-28 revision ships ~4 weeks after this brief** (RC locked 2026-05-21) ŌĆö stateless core (the `initialize` handshake and protocol sessions are removed), tasks restructured, `sampling` deprecated, full JSON Schema 2020-12 in tool schemas. Practical consequence: build on the official `mcp` Python SDK and treat protocol mechanics as SDK-owned; hand-rolled session or lifecycle logic would be obsolete on arrival.

**In-app LLM chat.** Browser-side, POST + SSE over `fetch` + `ReadableStream` remains the standard streaming transport ŌĆö nothing SDK-native replaced it. Anthropic-side, several things moved since couplefins shipped (April 2026): `output_config` (effort levels, JSON-schema output format) is GA and a native SDK parameter; **`strict: true` tool schemas are GA** but with hard complexity limits (max 20 strict tools per request; 24 optional parameters and 16 union-typed parameters *cumulative across all strict schemas*; `additionalProperties: false` mandatory on every object; no `oneOf`, no recursion, no numeric `minimum`/`maximum`, no `minLength`/`maxLength`); the current Sonnet tier is **`claude-sonnet-5`** ($3/$15 per MTok, intro $2/$10 through 2026-08-31, near-Opus agentic quality, adaptive thinking on by default, new tokenizer ~30% more tokens than Sonnet 4.6); the **prompt-cache minimum on Sonnet 5 is 1,024 tokens** (max 4 breakpoints; a top-level auto-cache parameter now exists). The SDKs ship a tool-runner that automates the tool loop, but it auto-executes tools ŌĆö Anthropic's own docs route human-in-the-loop flows to the manual loop, which is exactly the shape couplefins already has.

**The relationship (MCP Ōåö in-app chat).** The established 2026 pattern is *"expose your app's capabilities once via an MCP server so external agents get them"* ŌĆö there is no credible pattern of routing your own first-party chat through your own MCP server. Three primary-source facts close the question for mixd: (1) Anthropic's MCP connector (beta `mcp-client-2025-11-20`, tool calls only) **requires a publicly exposed HTTPS server ŌĆö local stdio servers cannot connect**, which breaks mixd's local-dev/self-host primary path; (2) annotations being untrusted means the mutation-confirmation gate must live inside mixd's process regardless ŌĆö an MCP hop between chat route and tools adds latency and a beta dependency without adding safety; (3) `sampling`, the mechanism an "MCP-as-internal-bus" design would lean on for server-initiated LLM calls, is deprecated in the 2026-07-28 RC. The durable convergence point is not the protocol ŌĆö it's a **shared in-process tool registry** that both the chat executor and the MCP server read.

#### Decisions

- **D1 ŌĆö The couplefins port stands, with a named modernization delta.** The architecture layer (POST+SSE transport, sentinel queue + error-code map, Zustand store, two-phase confirmation, per-request system-prompt composition, in-memory rate limiter, ephemeral conversations) verified clean against current docs ŌĆö port the patterns as-is. What must NOT be ported verbatim: the model ID, the `extra_body` workaround, the 2048-token cache claim, and the unconfigured-thinking assumptions (verification table below). Implementers port patterns, not constants.
- **D2 ŌĆö Chat-first retained; MCP is the second consumer (v0.9.1), not the foundation.** Evidence above. Flip condition checked and absent: no mature loopback/in-process MCP-client pattern with confirmation control comparable to direct dispatch exists as of 2026-07.
- **D3 ŌĆö The shared tool registry is a v0.9.0 deliverable.** The draft's "tolerated v0.9.1 refactor" bought nothing: the registry is one frozen dataclass plus a dispatch loop, and its `effect: read | mutate` classification is load-bearing for v0.9.0's own confirmation flow. Building tools as registry entries from day one costs ~an epic of size S and deletes v0.9.1's riskiest epic (retrofitting a second consumer onto organically-grown tool code). The former "Extract Shared Tool Registry" epic moves into v0.9.0 as "Build Shared Tool Registry"; v0.9.1 slims to transport + coverage + audit.

### v0.9.0: Workflow Assistant (Right-Panel Chat)

**Goal**: Today the Casual Enthusiast has no front door ŌĆö building a workflow means learning the node catalog. v0.9.0 gives them a persistent right-panel chat assistant: describe a playlist in plain English ("upbeat tracks I haven't played in 6 months") and get a real, editable `WorkflowDef` ŌĆö previewed in the existing graph renderer, saved on approval. Same always-available, no-modal pattern as Cursor / Linear / Notion AI.

**Context**: Per the README's persona-alignment table this is "THE adoption enabler ŌĆö changes who can use mixd." The architecture is not novel research: sister project **couplefins** has a production-tested right-panel chatbot on the same stack (Vite + React + TS + FastAPI + SQLAlchemy + Tanstack Query), built deliberately so mixd could port it. v0.9.0 is **substantially a port plus a tool registry**; the mixd-specific surface is the domain primer, the node-catalog ŌåÆ JSON Schema bridge, and the workflow-preview card. The Pre-Work brief above verified each ported choice against July-2026 primary docs ŌĆö implement against the verification table's "Action" column, not couplefins's pinned constants.

**What this unlocks**:
- One-shot natural-language workflow generation ŌĆö the machine-derived WorkflowDef JSON Schema is the prior, no few-shot templates needed
- Persistent right-panel chat (Cmd+K toggle, edge-tab when collapsed) across all routes
- Workflow preview rendered as the same read-only DAG users see elsewhere in the app, before save
- Two-phase confirmation: LLM proposes ŌåÆ user inspects ŌåÆ user approves ŌåÆ save (mutations never silent)
- The shared tool registry that v0.9.1's MCP server (and any future AI feature) consumes ŌĆö write once, expose everywhere

**Persona**: Casual Enthusiast ŌĆö primary. The natural-language entry point is the front door for users who'd never build a workflow from scratch. Tinkerer ŌĆö secondary. Generates a starting point in seconds, then tweaks in the editor they already know. Weekly Curator ŌĆö tertiary. Faster ad-hoc creation when a specific need arises.

**Key Design Decisions**:
- **Port couplefins, don't rebuild** ŌĆö the infrastructure (SSE streaming, adapter, tool dispatch, two-phase confirmation, prompt caching, error mapping, Zustand store, edge-tab affordance) is already-solved; apply the Pre-Work verification table's modernization delta while porting.
- **Tools are registry entries from day one (D3)** ŌĆö every capability is a `ToolSpec` (thin adapter over an application-layer use case) in `src/application/tools/`; the chat executor is a loop over the registry. This is what makes v0.9.1 mostly transport work.
- **Schema-guided generation, server-validated (revised from the draft)** ŌĆö the WorkflowDef JSON Schema is derived mechanically from `config_fields.py` and shipped as the generation tool's `input_schema`, but **without `strict: true`**: a full per-node schema exceeds Anthropic's documented strict-mode limits, and strict mode can't express free-form config objects or `min`/`max` bounds anyway. Conformance is enforced where it already lives ŌĆö `validate_workflow_def_detailed()` ŌĆö with failures returned to the model as error tool results so it self-corrects in-turn. `strict: true` is applied only to the small fixed-shape tools.
- **Workflow Templates (v0.8.9) are user-facing, not LLM-facing** ŌĆö no coupling in either direction.
- **Reuse `WorkflowGraph.tsx` for previews (corrected from the draft)** ŌĆö the read-only DAG renderer already exists (`web/src/components/shared/WorkflowGraph.tsx`: non-interactive React Flow, ELK layout, status/diff overlays). The preview card embeds it. There is no `readOnly` prop on the editable editor and none is needed.
- **Refinement is first-class in both chat and editor** ŌĆö each refinement turn re-emits a fresh `generate_workflow_def` call with the complete updated DAG; the preview card replaces its contents in place; chat history is the diff context. Fine-grained graph surgery stays in the visual editor.
- **Server-side context injection** ŌĆö the FastAPI route resolves user, library stats, and current-workflow context from the authenticated session and bakes it into the system prompt; the panel passes no context props.
- **Two-phase confirmation for saves** ŌĆö `save_workflow` returns a pending action; frontend renders a ConfirmationCard; approval triggers a second request that executes. Driven by the registry's `effect` flag.
- **Ephemeral conversations, no DB persistence** ŌĆö chat state lives in the Zustand store, capped at 50 messages; add persistence only as a post-deploy revision if real demand appears.
- **Anthropic Claude via `AsyncAnthropic`** ŌĆö model `claude-sonnet-5` and `output_config={"effort": "medium"}` as starting points (revisit at implementation); native parameters, no `extra_body`. Sonnet 5 runs adaptive thinking by default ŌĆö configure it deliberately and give `max_tokens` headroom for thinking spend. System prompt + tools cached via `cache_control: {"type": "ephemeral"}` (Sonnet 5 activation minimum: 1,024 tokens).
- **In-memory rate limiter** ŌĆö 20 req/60s per user, matching couplefins; right-sized for mixd's deployment shape.
- **New surfaces start from the Wave-3 primitives (v0.8.15)** ŌĆö every data view in the panel renders through `QueryStates` (+ `skeletons.tsx` primitives) and any picker composes `CommandSearchList`; hand-rolling a loading/error/empty ladder in ported couplefins code is a porting bug, not a style choice.

- [ ] **Build Shared Tool Registry**

    **Story**: A maintainer wants every assistant capability defined once ŌĆö name, description, input schema, side-effect class, dispatcher ŌĆö so the v0.9.0 chat assistant and the v0.9.1 MCP server expose identical tools, and adding a capability or fixing a description updates every consumer at once (D3: built here, not extracted later).

    **Decisions**:
    - Each tool is a frozen attrs `ToolSpec`: `name`, `description`, `input_schema` (JSON Schema), `effect: Literal["read", "mutate"]`, `strict: bool`, `dispatch` (async callable taking validated args + UoW, returning a JSON-serializable result).
    - **Tools are thin adapters over `src/application/` use cases** ŌĆö no business logic in dispatchers; where a use case is missing or awkwardly shaped, fix it at the source.
    - **`effect` is load-bearing, not documentation**: `mutate` routes the call through the pending-action confirmation flow in chat, and maps to MCP annotations in v0.9.1.
    - Registry entries carry LLM-facing descriptions (3ŌĆō4 sentences, concrete examples, prescriptive about *when* to call) ŌĆö the same writing standard both consumers inherit.

    **Spec**:
    - New module `src/application/tools/registry.py` ŌĆö `ToolSpec` + `TOOLS: tuple[ToolSpec, ...]`
    - `src/application/chat/tool_executor.py` dispatches by looping the registry ŌĆö no per-tool branching
    - MCP-ready metadata derivable per entry: `readOnlyHint = (effect == "read")`, `destructiveHint = (effect == "mutate")`, `openWorldHint = False` (mixd tools act on the user's own library, a closed world)

    **Tests**:
    - (unit) names unique; every description Ōēź50 chars; every `input_schema` is valid JSON Schema
    - (unit) every `mutate` entry refuses direct execution without a claimed confirmation
    - (unit) dispatcher signature consistent across entries

    Effort: S | Dependencies: ŌĆö | Status: Not Started

- [ ] **Workflow Generation Tools ŌĆö Node Catalog ŌåÆ JSON Schema**

    **Story**: The LLM wants to call `generate_workflow_def` with a structured argument matching mixd's actual `WorkflowDef` shape ŌĆö producing valid DAGs that reference real nodes with correct parameters, without the user teaching it what's available.

    **Decisions**:

    - All tools are registry entries; `save_workflow` is `effect: "mutate"` (two-phase confirmation), the rest are `read`.

### v0.9.1: MCP Server (Mixd as a Tool Surface)

**Goal**: Expose mixd's capabilities as an MCP server so any MCP-aware client ŌĆö Claude Desktop, Cursor, Claude Code, future agentic tools ŌĆö can read from and act on a user's mixd library. The surface is broad by design: most things doable via CLI or web UI are doable via MCP. The same registry that powers v0.9.0's chat powers this ŌĆö v0.9.1 is **transport, coverage, and audit**, not new tool logic.

**Context**: v0.9.0 built the tool registry (D3), so the riskiest work of the original draft ŌĆö retrofitting a shared registry onto organically-grown chat tools ŌĆö no longer exists. What remains: the stdio server scaffold on the official `mcp` Python SDK, filling out read/mutation tool coverage, and the cross-interface audit. The MCP 2026-07-28 spec revision (stateless core) will be final before this milestone starts ŌĆö build on the SDK's current release and let it own protocol mechanics (sessions, lifecycle, transport framing); any hand-rolled protocol logic would be building against a moving target.

**What this unlocks**:
- Local-first agentic workflows: "Claude, find all my starred tracks from last year that aren't on my Friday playlist, and add them" ŌĆö from Claude Desktop or Cursor, no code.
- A forcing function for use case completeness: any CLI/web action inexpressible as a tool reveals an application-layer gap.
- Foundation for future remote MCP (hosted mixd as a multi-user endpoint), gated on auth maturity.

**Persona**: Tinkerer ŌĆö primary. Self-hosters and power users who already pipe LLMs into their toolchains. Curator ŌĆö secondary. Ad-hoc scripted maintenance ("deduplicate my 'Loved' playlist against my starred preferences"). Casual Enthusiast ŌĆö tertiary, via whatever agents others build on top.

**Key Design Decisions**:
- **Stdio-only for v0.9.1; HTTP/remote deferred.** Stdio matches the spec's own recommendation ("clients SHOULD support stdio whenever possible") and mixd's local-first deployment. **Identity comes from the environment, per spec** ("implementations using STDIO transport SHOULD NOT [use OAuth]... retrieve credentials from the environment"): the server resolves the user exactly as every CLI command does ŌĆö `get_cli_user_id()` (`MIXD_USER_ID` env ŌåÆ `DEFAULT_USER_ID`). No new credential surface. *(Corrects the earlier draft, which assumed a `mixd auth login` keyring session that has never existed.)*
- **Remote/Streamable-HTTP MCP is a successor milestone, not a stretch goal**: it requires OAuth 2.1 resource-server work (RFC 9728 protected-resource metadata, RFC 8707 audience validation) ŌĆö mixd's existing Neon Auth JWT infrastructure is the natural authorization server when real demand materializes.
- **One registry, second consumer**: `list_tools`/`call_tool` iterate `src/application/tools/registry.py`. No tool defined twice, no drift possible.
- **Side-effect annotations derived from `effect`**: `readOnlyHint = (effect == "read")`, `destructiveHint = (effect == "mutate")`, `idempotentHint: true` on read tools, `openWorldHint: false` (mixd tools act on the user's own library). **The spec marks annotations as untrusted client-side hints** ŌĆö so mutation safety never relies on them.
- **Mutations require explicit confirmation in-band**: any `mutate` tool called without `confirm=true` returns a structured preview + confirm token instead of executing (the registry behavior v0.9.0 already enforces). Well-behaved clients will also prompt on `destructiveHint`; mixd doesn't depend on it.
- **Coverage audit is an explicit deliverable**: a maintained table mapping every CLI command and meaningful web action to its MCP tool or a written exclusion rationale.
- **Structured output where it helps**: read tools may declare `outputSchema` and return `structuredContent` (2025-11-25 feature) reusing the existing Pydantic API schemas ŌĆö decide per tool during implementation, not a blanket requirement.

#### Epics

- [ ] **MCP Server Scaffold (Stdio Transport)**

    **Story**: A Tinkerer wants to point Claude Desktop at mixd and immediately see mixd's tools in the client's palette ŌĆö the same way they'd add the filesystem server. No HTTP, no tokens, just a `mixd mcp serve` subprocess.

    **Decisions**:
    - Official Python MCP SDK (`mcp` package), FastMCP-style server registered as a `mixd mcp serve` Typer subcommand. The SDK owns protocol mechanics; mixd owns tool metadata + dispatch.
    - User resolved via `get_cli_user_id()`; unauthenticated/misconfigured state returns a clear per-call error with remediation, not a crash.
    - One UoW per tool call (`execute_use_case` with the resolved user id ŌĆö RLS scoping comes free). MCP calls are independent; no long-lived transactions.
    - **stdout is the protocol channel**: structlog routes to stderr/file (spec allows stderr for logging; stdout MUST carry only MCP messages).

    **Spec**:
    - New module `src/interface/mcp/server.py` ŌĆö builds the server from the shared registry: name, description, `input_schema`, annotations derived from `effect` (mapping in Key Design Decisions)
    - New CLI subcommand `mixd mcp serve` in `src/interface/cli/`
    - Log path configurable via env var

    **Tests**:
    - (integration) server boots, responds to `tools/list` over stdio; tool metadata matches registry entries
    - (integration) `tools/call` on a read tool returns a valid result scoped to the resolved user
    - (unit) annotations correctly derived from `effect` for every registry entry

    Effort: M | Dependencies: v0.9.0 (Build Shared Tool Registry) | Status: Not Started

- [ ] **Read-Tool Coverage (Library, Workflows, Tags, Preferences, Run History)**

    **Story**: An agent acting for the user wants to read everything queryable in the web UI or CLI ŌĆö tracks, playlists, tags, preferences, workflows, runs, connector identities ŌĆö so any question ("which starred tracks haven't I played in 6 months?") is answerable without leaving the chat.

    **Decisions**:
    - Read tools dispatch to existing repository/use-case query methods; no new persistence work expected.
    - Cross-tool consistency: every list tool takes the same `limit`/`cursor` shape; every get tool takes the canonical ID type; agents handle pagination loops natively.
    - Results serialize via existing Pydantic API schemas where possible (DRY with the REST layer); gaps logged for the audit. Tools may additionally declare `outputSchema`/`structuredContent` where the schema is stable.

    **Spec**:
    - Registry entries (initial): `list_tracks`, `search_tracks`, `get_track`, `list_playlists`, `get_playlist`, `list_tags`, `list_tagged_tracks`, `get_track_tags`, `list_preferences`, `get_track_preference`, `list_workflows`, `get_workflow`, `list_workflow_runs`, `get_workflow_run`, `list_connector_identities`, `get_library_stats`
    - Each: registry entry + dispatcher delegating to the existing query path

    **Tests**:
    - (integration) each read tool returns the expected shape against a seeded fixture user
    - (integration) pagination cursors round-trip; user scoping holds (no cross-user leakage)
    - (integration) search tools respect existing trigram behavior

    Effort: M | Dependencies: MCP Server Scaffold | Status: Not Started

- [ ] **Mutation Tools with Two-Phase Confirmation**

    **Story**: An agent wants to *do things* ŌĆö run a workflow, import a playlist, tag tracks, edit preferences ŌĆö but never silently. Each mutating call returns a preview first; the agent (or the user via the agent) explicitly re-issues with `confirm=true` to commit.

    **Decisions**:
    - Two-phase in-band: first call ŌåÆ `{ status: "needs_confirmation", preview, confirm_token }`; second call with `confirm=true` commits. Expired tokens return a fresh preview, never a stale commit.
    - Confirm tokens scoped to `(user_id, tool_name, args_hash)`, 5-minute TTL ŌĆö same store pattern as v0.9.0's pending actions (possibly the same store).
    - Args drift between preview and confirm is rejected with a clear error.
    - Mutations that already have preview surfaces (workflow dry-run) reuse them for the preview payload.

    **Spec**:
    - Registry entries: `run_workflow`, `save_workflow`, `update_workflow`, `delete_workflow`, `import_playlist`, `set_track_preference`, `add_track_tags`, `remove_track_tags`, `create_tag`, `delete_tag`, `relink_track`, `unlink_track`, `merge_tracks`
    - All execution lands in existing use cases; no new persistence logic

    **Tests**:
    - (integration) first call previews without mutating; `confirm=true` commits and returns post-mutation state
    - (integration) expired token ŌåÆ fresh preview; args drift ŌåÆ rejection
    - (integration) every mutation scoped to the resolved user

    Effort: M | Dependencies: Read-Tool Coverage | Status: Not Started

- [ ] **Cross-Interface Coverage Audit + `mixd mcp install`**

    **Story**: A user wants confidence that "anything I can do in the web UI or CLI, I can do via MCP" ŌĆö or a clear documented reason where not ŌĆö and wants to install the server in their MCP client with one command, not by hand-editing JSON.

    **Decisions**:
    - Audit table: capability (CLI command or web action) | covering MCP tool | exclusion rationale if uncovered. Reviewed whenever a feature ship adds a CLI command or web action.
    - `mixd mcp install` writes the config snippet for the detected client; `--print` emits JSON for manual paste.
    - Docs: `docs/guides/mcp.md` (install, tool list, 3ŌĆō4 worked walkthroughs); README's "What Mixd Does" gains the tool-surface sentence.

    **Spec**:
    - New doc `docs/web-ui/mcp-coverage.md`, populated during implementation
    - New CLI subcommand `mixd mcp install [--client claude-desktop|cursor|claude-code] [--print]`
    - New guide `docs/guides/mcp.md`; README headline updated

    **Tests**:
    - (unit) install emits valid JSON per supported client; `--print` writes nothing
    - (integration) coverage doc parses as a well-formed table with no blank rationale cells

    Effort: S | Dependencies: Mutation Tools with Two-Phase Confirmation | Status: Not Started

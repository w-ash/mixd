# v0.9.x: Agentic Workspace

For strategic overview, see the [planning overview](README.md).

**User Workflow Context**: Every mixd action today is manual — a web click-path or a CLI invocation the user drives step by step. Curators repeat the same Sunday maintenance ritual by hand; Casual Enthusiasts bounce off the node catalog before their first playlist materializes; Tinkerers who already run LLM agents (Claude Desktop, Cursor, local toolchains) can't point them at mixd at all. v0.9.x makes the same surface drivable by agents under one product contract — **bidirectional parity**: anything a human can do through the app, an agent can do (minus an explicit human-only blacklist: connector OAuth, account management, `mixd admin reset`), and nothing an agent does is invisible to the human (agent work lands in ordinary mixd entities and shows up in the same UI, run logs, and audit surfaces). The in-app assistant lands first (v0.9.0–v0.9.2), external agents via MCP second (v0.9.3) — both consuming one shared tool registry so the two surfaces never drift.

#### State of the art, July 2026

**MCP servers.** Current spec revision is **2025-11-25**; the **2026-07-28 revision publishes final on that date** (RC locked 2026-05-21; Tier-1 SDKs ship support within the ten-week validation window). What changes: the core goes **stateless** — the `initialize` handshake and protocol sessions are removed; any request can hit any server instance; state crosses calls via the **explicit-handle pattern** (tools return identifiers the model passes back). New opt-in **extensions**: **Tasks** (a server may answer `tools/call` with a task handle; the client drives `tasks/get`/`tasks/update`/`tasks/cancel`; task-vs-blocking is the *server's* decision) and **MCP Apps** (servers ship pre-declared HTML UI templates that hosts render in a sandboxed iframe, with UI actions flowing through the same JSON-RPC consent path as tool calls). A formal deprecation policy (≥12 months deprecated before removal) makes building on the spec safer. Two things did not change: **stdio** remains the recommended local transport ("clients SHOULD support stdio whenever possible"; identity from the environment, not OAuth), and tool behavior annotations (`readOnlyHint`, `destructiveHint`, …) remain **untrusted client-side hints** — mutation safety must live server-side. Practical consequence: build on the official `mcp` Python SDK *after* the 2026-07-28 release settles, treat protocol mechanics as SDK-owned, and design tools stateless-first.

**In-app LLM agents.** POST + SSE over `fetch` + `ReadableStream` remains the browser streaming standard. On the Anthropic side, the mid-2026 agentic surface is substantially richer than what the 2026-07-02 pass described: **tool search** (`tool_search_tool_bm25_*`; tools marked `defer_loading: true` are discovered on demand, keeping dozens of tools out of the upfront context and preserving the prompt cache) — Anthropic's guidance puts the accuracy cliff at roughly ten-plus tools; **programmatic tool calling** (the model writes code in the `code_execution` sandbox that calls tools marked with `allowed_callers`; intermediate results stay in the sandbox, only the final output enters model context — built for exactly the many-sequential-calls / large-intermediate-results shape); **context editing** (`clear_tool_uses_*` clears stale tool results server-side as a conversation grows); server-side **compaction** (beta) for conversations that outgrow the window; **task budgets** (beta) as a token-ceiling lever the model paces itself against. Two corrections to the 2026-07-02 brief's pinned constants: **`strict: true` is the wrong tool for a broad registry** — couplefins tried it and abandoned it (20-strict-tools-per-request cap, compiled-grammar-too-large at 16 write tools, numeric/array constraints rejected); handler-side validation is the boundary, schemas keep `additionalProperties: false` plus prose ranges. And **cache-activation minimums are model-specific and have shifted** (Opus 4.8 is 4096 tokens) — don't pin a number; verify via `usage.cache_read_input_tokens` at implementation.

**The relationship (MCP ↔ in-app chat).** Unchanged, and now double-confirmed by couplefins in production: expose capabilities once via an in-process tool registry; the first-party chat executor and the MCP server are two consumers of it. No credible pattern routes first-party chat through one's own MCP server, and annotations-as-untrusted means the confirmation gate lives inside mixd's process regardless.

#### Addendum (2026-07-10): couplefins went agentic — the port target moved

The 2026-07-02 brief verified couplefins at `da407c5`, when its chat was a 9-tool Q&A bot. Since then couplefins shipped its **v1.8.0–v1.8.3 "Agentic Chat" series** (74 commits, completed 2026-07-10, HEAD `105496f`) — the same chatbot-to-agentic-workspace leap this milestone plans, under the product principle *"anything a human can do through the app, the chatbot can do — and nothing more,"* enforced as a CI test. What exists there now, live-verified, with documented pitfalls:

- **Capability-parity registry** — `ToolSpec(name, schema, handler, use_cases, kind: read|write|agentic, defer_loading, …)`; 30 tools (15 read / 12 write / 3 agentic — the three agentic being `code_execution`, `delegate_analysis`, `tool_search_tool_bm25`); classification sets (`BLACKLISTED_USE_CASES`, `MECHANICALLY_EXCLUDED_USE_CASES`, `INTERNAL_USE_CASES`) with a CI parity test that fails when any app use case is unclassified.
- **Agentic loop** — flat tool loop with injected executor (so the subagent reuses it), `pause_turn`/`max_tokens` stop-reason handling, dual turn budgets (model turns + a ×5 sandbox-round backstop).
- **Agentic depth** — `code_execution` sandbox + programmatic tool calling (read tools carry `allowed_callers`; write tools never), a `delegate_analysis` research subagent (fresh context, read-only tools, low effort, bounded summary), tool search with only ~6 hot tools non-deferred, context editing with tuned thresholds. Server-side compaction was **evaluated and deliberately skipped** (conversations stay bounded by the 50-message cap + tool clearing).
- **Security layers** — two-phase confirmation (unchanged) *plus* systematic untrusted-content labeling: user-originated strings are wrapped in `<user_data>` tags for the model and stripped at the event boundary — a deterministic prompt-injection layer on top of mutations-as-proposals.
- **Config reality** — model `claude-opus-4-8` with explicit `thinking={"type": "adaptive"}` (Opus runs thinking-off when the field is omitted, and under-reaches for tools unless descriptions lead with a when-to-call trigger sentence); effort became a **per-request user control** (Quick/Standard/Thorough → low/high/xhigh); the SSE protocol grew to **7 events** (`code_start`/`code_result` added; `tool_start` carries `kind`); three cache breakpoints (last non-deferred tool, system prompt, last stampable message block).
- **Deliberately out of scope there** (mixd treats these as candidates, not commitments): memory tool (flagged as a prompt-injection *reinfection* vector), MCP tools inside chat, parallel subagent orchestration, conversation persistence.

Implementation sources to mine when each epic starts: couplefins `docs/chat-assistant.md` (architecture + rationale), `docs/backlog/v1.8.x-agentic-chat.md` (product principle, design-grounding citations), and `v1.8.0`–`v1.8.3` (especially v1.8.3's "Implementation learnings" — four programmatic-tool-calling bugs unit tests couldn't catch).

#### Decisions

- **D1 — The couplefins port stands; the port target is now the v1.8.x series.** Architecture layer (POST+SSE, sentinel queue + error-code map, Zustand store, two-phase confirmation, per-request system-prompt composition, in-memory rate limiter, ephemeral conversations) verified clean across both passes — port the patterns. Port the *v1.8.x* versions of everything (7-event SSE, `ToolContext` injection, untrusted-content wrapping, kind-aware registry), not the `da407c5` snapshots. As before: port patterns, not constants.
- **D2 — Chat-first retained; MCP is the second consumer, not the foundation.** Unchanged from 2026-07-02, reconfirmed 2026-07-10.
- **D3 — The shared tool registry is a v0.9.0 deliverable.** Unchanged — and strengthened: the registry's classification machinery (D4) must exist from the first tool.
- **D4 — Bidirectional parity is the product contract, enforced in CI.** Agent→human: every application use case is classified in the registry — covered by a tool, blacklisted (human-only: connector OAuth/token flows, account management, `mixd admin reset`), mechanically excluded (interactive-only UI affordances), or internal — and a CI test fails on unclassified use cases, so the contract can't silently rot as features ship. Human→agent: agents get no private capabilities — tools are thin adapters over the same use cases the UI and CLI call, agent mutations flow through the same two-phase-confirmation and operation-run surfaces, and everything an agent does is inspectable in the ordinary UI.
- **D5 — Agentic depth is committed scope, not research.** Programmatic tool calling, the research subagent, tool search, context editing, and per-request effort are all live-verified in couplefins v1.8.2/v1.8.3 with documented constants and failure modes. Mixd commits them as v0.9.2 — ports with mixd-shaped stories (batch-first library operations are the payoff), not experiments.
- **D6 — Four milestones, MCP last.** v0.9.0 foundation → v0.9.1 full parity → v0.9.2 agentic depth → v0.9.3 MCP, mirroring couplefins' shipped sequence (foundations → read parity → write parity → agentic). Each is a complete, manually testable slice; MCP inherits a mature, parity-complete registry so it reduces to transport + audit exposure. *(This re-purposes the v0.9.2 number the 2026-07-02 plan reserved for remote MCP; remote/Streamable-HTTP MCP remains a demand-gated successor — see v0.9.3's decisions.)*

### v0.9.0: Workflow Assistant (Agentic Foundation)

**Goal**: Today the Casual Enthusiast has no front door — building a workflow means learning the node catalog. v0.9.0 gives them a persistent right-panel assistant: describe a playlist in plain English ("upbeat tracks I haven't played in 6 months") and get a real, editable `WorkflowDef` — previewed in the existing graph renderer, saved on approval. Underneath the flagship story, this milestone lays the agentic-workspace foundation: the chat shell, the streaming loop, and the parity-classified tool registry that every later milestone (and the MCP server) consumes.

**Context**: Per the README's persona-alignment table this is "THE adoption enabler — changes who can use mixd." The architecture is a port of couplefins' production-tested v1.8.x agentic chat on the same stack (Vite + React + TS + FastAPI + SQLAlchemy + Tanstack Query), built deliberately so mixd could port it. The mixd-specific surface is the domain primer, the node-catalog → JSON Schema bridge, and the workflow-preview card. Implement against the Pre-Work verification table's "Action" column, not pinned constants.

**What this unlocks**:
- One-shot natural-language workflow generation — the machine-derived WorkflowDef JSON Schema is the prior, no few-shot templates needed
- Persistent right-panel chat (Cmd+K toggle, edge-tab when collapsed) across all routes
- Workflow preview rendered as the same read-only DAG users see elsewhere in the app, before save
- Two-phase confirmation: LLM proposes → user inspects → user approves → save (mutations never silent)
- The parity-classified tool registry (D3/D4) that v0.9.1 fills, v0.9.2 deepens, and v0.9.3 exposes over MCP

**Persona**: Casual Enthusiast — primary. The natural-language entry point is the front door for users who'd never build a workflow from scratch. Tinkerer — secondary. Generates a starting point in seconds, then tweaks in the editor they already know. Weekly Curator — tertiary. Faster ad-hoc creation when a specific need arises.

**Key Design Decisions**:
- **Port couplefins v1.8.x, don't rebuild** — the infrastructure (7-event SSE streaming, adapter, agentic tool loop, two-phase confirmation, prompt caching, error mapping, Zustand store, edge-tab affordance, untrusted-content wrapping) is already-solved; apply the Pre-Work verification table's delta while porting.
- **Tools are parity-classified registry entries from day one (D3/D4)** — every capability is a `ToolSpec` (thin adapter over an application-layer use case) in `src/application/tools/`; the chat executor loops over the registry; the CI parity test ships with the registry, initially with most of mixd's 68 use cases classified "not yet covered" (coverage is v0.9.1's job — classification discipline starts here).
- **Schema-guided generation, server-validated** — the WorkflowDef JSON Schema is derived mechanically from `config_fields.py` and shipped as the generation tool's `input_schema`; conformance is enforced where it already lives — `validate_workflow_def_detailed()` — with failures returned as error tool results so the model self-corrects in-turn. No `strict: true` anywhere (Pre-Work table).
- **Untrusted content is labeled, deterministically** — track titles, playlist names, and other user-originated strings are wrapped in `<user_data>` tags in model-facing JSON and stripped at the event boundary; combined with mutations-as-proposals this is the layered prompt-injection defense (music metadata is attacker-controllable text — a track title can contain instructions).
- **Workflow Templates (v0.8.9) are user-facing, not LLM-facing** — no coupling in either direction.
- **Reuse `WorkflowGraph.tsx` for previews** — the read-only DAG renderer already exists (`web/src/components/shared/WorkflowGraph.tsx`: non-interactive React Flow, ELK layout, status/diff overlays). The preview card embeds it.
- **Refinement is first-class in both chat and editor** — each refinement turn re-emits a fresh `generate_workflow_def` call with the complete updated DAG; the preview card replaces its contents in place; chat history is the diff context. Fine-grained graph surgery stays in the visual editor.
- **Server-side context injection** — the FastAPI route resolves user, library stats, and current-workflow context from the authenticated session and bakes it into the system prompt; the panel passes no context props.
- **Two-phase confirmation for saves** — `save_workflow` returns a pending action; frontend renders a ConfirmationCard showing what will be created/changed; approval triggers a second request that executes. Driven by the registry's `kind` flag.
- **Ephemeral conversations, no DB persistence** — chat state lives in the Zustand store, capped at 50 messages; add persistence only as a post-deploy revision if real demand appears.
- **Anthropic Claude via `AsyncAnthropic`** — model decided at implementation between `claude-sonnet-5` and `claude-opus-4-8` (Pre-Work table); explicit `thinking={"type": "adaptive"}`, `output_config={"effort": ...}` default `high`, `max_tokens=16384` as starting points. System prompt + tools cached via `cache_control: {"type": "ephemeral"}`; verify activation via `usage.cache_read_input_tokens`, don't assume a token minimum.
- **In-memory rate limiter** — 20 req/60s per user, matching couplefins; right-sized for mixd's deployment shape.
- **New surfaces start from the Wave-3 primitives (v0.8.15)** — every data view in the panel renders through `QueryStates` (+ `skeletons.tsx` primitives) and any picker composes `CommandSearchList`; hand-rolling a loading/error/empty ladder in ported couplefins code is a porting bug, not a style choice.

**Learnings for whoever finishes Phases 2–4** — each cost real debugging; don't rediscover:

- **SDK parity retired the biggest risk.** couplefins targets `anthropic>=0.97`; mixd installed **0.116**. Every symbol and `beta.messages.stream` param couplefins uses (`BetaAsyncMessageStream`, `output_config`, `context_management`, `thinking`, `betas`, `container`) exists unchanged — the adapter ported near-verbatim. Re-verify only on an SDK bump.
- **Parity is classified by `*UseCase` class (81), not module (58)** — a module like `sync_likes.py` holds several use cases of different read/write character. `BLACKLISTED_USE_CASES` is legitimately **empty**: mixd's human-only capabilities (OAuth, account mgmt, admin reset) aren't use-case classes; they live in the connector/interface layer. **Phase 3 must move `GetWorkflowUseCase`/`ListWorkflowsUseCase`/`CreateWorkflowUseCase`/`UpdateWorkflowUseCase` out of `NOT_YET_COVERED` into their `ToolSpec.use_cases`** as those tools land.
- **ruff B008 fires on the *second* `Depends()` in a route signature, never the first** (every existing mixd route has exactly one). Resolve extra dependencies (the LLM client) **in the route body**, not as a second `Depends`. Consequence: **tests inject the fake LLM via `monkeypatch.setattr(chat_route, "get_llm_client", …)`, not `dependency_overrides`** — it isn't a Depends.
- **`ASYNC119` is inherent to the streaming loop** (you must yield inside the LLM stream context). It's a justified per-file ignore *because* the SSE bridge's `try/finally` cancels the background task on disconnect, closing the context via GeneratorExit. Don't "fix" it by buffering.
- **`wrap_for_model` + the `UserData` str-subclass marker are deferred to v0.9.1.** No v0.9.0 tool returns user-library free text (`describe_node` is static catalog; workflow-gen returns model-generated defs), so results serialize directly while the event side still strips tags. When the first library-read tool lands, the `UserData(str)` subclass trips ruff **FURB189** ("use `collections.UserString`") — but `UserString` breaks JSON-serializability and str semantics, so it needs a `# noqa` **with user sign-off** (per `.claude/rules/python-conventions.md`) or a deliberate restructure. Decide before writing it.
- **Model is env-configurable**: `CHAT__MODEL_ID` (default `claude-opus-4-8`; `claude-sonnet-5` ≈ half cost). Adaptive thinking is always set **explicitly** — both models run thinking-off if the field is omitted.
- **The node→JSON-Schema bridge (Phase 3) must use `anyOf` per node type (not `oneOf`) and encode min/max as prose in `description` (not `minimum`/`maximum`)** — Anthropic structured outputs reject both `oneOf` and numeric constraints; no `strict:true`. Source: `config_fields.py` (`ConfigFieldDef`) + `list_nodes()`; round-trip validation via `validate_workflow_def_detailed()` returning failures as an error tool result.
- **Frontend seams**: the panel mounts as the 3rd flex child of `PageLayout.tsx` (outside `RouteOutlet`, so it persists across routes); SSE auth uses `getAuthToken()` from `#/api/auth`; **`CommandSearchList` is track-specific** (hardcodes the tracks query) — a Phase 3 entity picker over playlists/workflows needs generalization or a parallel component. The `codeExecutions[]` / `CodeExecutionCard` / `currentWorkflowDraft` store slots ship **dormant** (Phase 3 / v0.9.2 activate them).
- **`.claude/rules/web-frontend-patterns.md` already sanctions Zustand** — the backlog's earlier "No Zustand" flag was stale; the chat store is compliant. Scratch that follow-up.

**Phases 2–4 learnings (2026-07-11):**

- **`ToolResultCard` must lazy-load `WorkflowPreviewCard`** — the preview pulls React Flow + ELK, and the chat panel mounts on every page, so a static import lands the whole graph stack in the initial bundle. It also breaks vitest outright (`@jalez/react-flow-smart-edge` is CJS-in-ESM under jsdom). Type guard + `SaveProposal` live in `cards/workflow-preview-types.ts` so the dispatcher never evaluates the heavy module.
- **Save choreography**: the primer's `<mutation_rules>` has the model propose `save_workflow` in the same turn as a successful generate; the preview card confirms that sibling proposal (Save/"Save changes" label from `details.mode`) and suppresses the sibling `ConfirmationCard`; with no live proposal (expired TTL / model skipped it) Save degrades to a synthetic "Save this workflow." message. `currentWorkflowDraft` absorbs proposals from `details.definition` and covers the cross-turn case.
- **The date moved OUT of the cached primer block** — Phase 1 had it inside, invalidating the cache daily; volatile values (date, stats, current workflow) now trail as uncached blocks. A unit test guards the layout; primer sized ~17k chars to clear the 4096-token floor with margin.
- **Primer text mentioning `<current_workflow>` broke naive `"<current_workflow>" in system` test probes** — probe block *structure* (count, `startswith`) or a sentinel sentence, not tag substrings.
- **Feedback records explicit thumbs only** — save is its own durable acceptance signal. `RecordChatFeedbackUseCase` is `BLACKLISTED_USE_CASES`' first member (the agent must never file feedback on itself); the parity test caught it as unclassified within minutes of the class landing, exactly as designed.
- **Mobile route was nearly free**: `ChatPanel` already had `fullScreen`; `ChatPage` is the couplefins `AskPage` pattern (desktop → `setPanelOpen` + redirect) + a lazy route + a bottom-nav entry. Eyeball the `h-[calc(100svh-11rem)]` offset and 5-item nav density on a real phone viewport before ship.

- [x] **v0.9.0.1** (2026-07-11) — per-user, opt-in AI assistant (BYO Anthropic key)

    **Shipped:** delivered as specced — per-user key stored encrypted in the
    `oauth_tokens` store (`credentials.py`, RLS-scoped, write-only), a single
    `resolve_chat_credential()` precedence rule (user key → server fallback →
    none) behind `get_llm_client` / `/assistant/status`, and a `useChatAvailable()`
    gate that hides the whole surface (Cmd+K, edge tab, panel, mobile `/chat` +
    `MobileBottomNav`) until a key is connected. CLI: `mixd assistant
    connect|test|disconnect`. Keys are validated with a minimal live completion
    so a no-billing key is rejected at connect, not on first message. Post-review
    hardening: undecryptable stored key fails closed (no silent server-key
    billing); cached Anthropic clients close on shutdown; desktop `/chat`
    deep-link waits for the availability gate before redirecting.

    **Why:** v0.9.0 shipped the assistant behind a single deployment-wide
    `ANTHROPIC_API_KEY` and mounts the chat surface (Cmd+K, edge tab, panel,
    mobile `/chat`) for everyone unconditionally. That breaks mixd's multi-user
    principle two ways: one person's key (and spend) would serve every user, and
    a user who hasn't set anything up sees a chat affordance that only 503s
    (`CHAT_UNAVAILABLE`) when they try it. The assistant must be **per-user and
    opt-in**: a user brings their own credential, and until they do, the AI
    assistant simply isn't visible — no broken affordance, no shared key.

    **What:**
    - **Per-user credential**, stored encrypted at rest keyed by `user_id`,
      managed from settings (add / test / remove). The client never gets the key
      back after saving.
    - **Availability is per-user**: `get_llm_client` / the adapter resolves the
      acting user's key via `ctx.user_id` (the server-wide key stays only as an
      optional single-tenant fallback, if kept at all — decide at implementation).
      The health `chat_available` flag becomes a per-user capability signal.
    - **Frontend gates the whole surface** on that signal: the Cmd+K binding,
      `ChatEdgeTab`, the `ChatPanel` mount in `PageLayout.tsx`, and the mobile
      `/chat` route + `MobileBottomNav` "Ask" entry all disappear when the
      current user has no key. No key = no assistant, not an error card.

    **Decisions:**
    - **Opt-in, hidden by default** — surfacing an assistant a user can't use
      trains them to distrust it; absence is cleaner than a disabled-state card.
    - **Encrypted at rest, write-only from the client** — the same discipline as
      connector tokens; a leaked settings response must never expose the key.
    - **Resolved (2026-07-11) — OAuth is off the table; BYO key paste is the only
      compliant path.** Web research settled the open question: as of Feb 2026
      (server-side enforced since Jan 2026, still in force July 2026) Anthropic
      restricts OAuth tokens (`sk-ant-oat01-`) to Claude Code and Claude.ai only,
      offers **no third-party client-ID registration**, and treats consumer-plan
      OAuth in any other product as a ToS violation. The single supported route
      for a multi-tenant app is a user-supplied API key (`sk-ant-api03-…`) from
      console.anthropic.com. So the plan **drops OAuth entirely** (not "treat as
      an upgrade") and standardizes on encrypted key paste in both UI and CLI,
      with a Console deep-link + billing hint to cut setup friction. Sources:
      WinBuzzer (2026-02-19), claude-code issue #28091, Claude Code auth docs.

    **Spec (starting points, revisit at implementation):**
    - Credential storage: encrypted per-user secret (new column/table keyed by
      `user_id`), reusing the connector-token encryption pattern.
    - Resolution: adapter/`get_llm_client` takes `user_id`; per-user capability
      endpoint (or extend `/health`'s `chat_available` to a per-user check).
    - Frontend: a `useChatAvailable()` gate consumed at every assistant mount
      point above; settings panel to manage the key with a live "test key" call.

    **Tests:**
    - (integration) no per-user key → capability reports unavailable; every
      assistant affordance is absent from the rendered tree.
    - (integration) a user's key resolves only for that user (RLS); user A's key
      never serves user B's chat turn.
    - (integration) key is never echoed back in any settings/GET response.
    - (component) settings add/test/remove flow; invalid key surfaces cleanly.

    **Why a revision, not v0.9.1:** it hardens the *shipped* v0.9.0 assistant for
    multi-tenant reality rather than adding new agent capability (that's v0.9.1).
    Ships as `feat:`. Effort: M.

### v0.9.1: Full Capability Parity (In-App)

**Goal**: After v0.9.0 the assistant can build workflows but little else — ask it to tag the tracks it just found and it has no hands. v0.9.1 makes the D4 contract real: every read a user can perform in the web UI or CLI becomes a registry tool, every mutation becomes a two-phase-confirmed tool, long-running operations stream their progress into the chat, and everything the agent does is visible in the same UI a human uses. When this ships, "ask the assistant" is a legitimate alternative to every screen in the app.

**Context**: Mixd's surface is already crisply enumerated — 68 use cases in `src/application/use_cases/` mirrored by ~60 CLI subcommands and 15 web routes, all funneling through `execute_use_case()` with the `user_id` RLS seam. This milestone is coverage work over that enumeration: consolidated tools (couplefins learning: one well-described tool wrapping several related use cases beats 1:1 wrapping — their 68-use-case equivalent surface collapsed to ~31 tools), plus the plumbing to bridge mixd's existing long-operation machinery into chat.

**What this unlocks**:
- "Which starred tracks haven't I played in 6 months? Tag them `context:revival` and add them to my Revival playlist" — a three-surface task, end-to-end in one conversation
- Library maintenance by conversation: merge duplicates, fix mappings, resolve match reviews, re-sync playlists
- Imports and syncs launched from chat with live progress, not fire-and-forget
- The parity matrix flips from "classified" to "covered or justified" — the audit v0.9.3's MCP server inherits for free

**Persona**: Weekly Curator — primary. The Sunday ritual (review imports, fix mismatches, refresh playlists) becomes delegable sentence by sentence. Casual Enthusiast — secondary. Everyday actions ("add this to my dinner playlist") without learning where they live in the UI. Tinkerer — tertiary. The registry breadth is what makes v0.9.3's MCP surface worth pointing external agents at.

**Key Design Decisions**:
- **Consolidate, don't 1:1-wrap** — tools are shaped around user intent ("manage tags on tracks") rather than use-case granularity; each `ToolSpec.use_cases` tuple records which use cases it covers so the parity test stays mechanical.
- **Every mutation is a proposal** — `kind: "write"` tools return a pending action with a human-readable before/after preview; the ConfirmationCard must show enough detail to decide without leaving chat (couplefins approval-fatigue finding: vague confirmations train users to click yes).
- **Long-running mutations reuse the operations machinery** — tools that start syncs/imports/runs return the existing `{"operation_id"}` contract; the chat panel subscribes to the existing per-operation SSE progress endpoint (`routes/operations.py`, Last-Event-ID reconnect included) rather than inventing a chat-specific progress channel.
- **Agent actions are auditable in the human UI** — assistant-initiated mutations record through the same operation-run/audit surfaces humans see (`/settings/imports` log; entity timestamps), attributed to the assistant, so the human side of D4 holds without a new activity-feed subsystem.
- **Blacklist is explicit and tested** — connector OAuth/token flows, account management, and `mixd admin reset` live in `BLACKLISTED_USE_CASES` with rationale strings; the parity test fails if a blacklisted capability grows a tool.

**Key learnings — don't rediscover:**

- **Dispatchers are a package, not one file** — `tool_executor.py` was already 388 lines for 6 tools; the 20 new tools live in `src/application/chat/dispatchers/` by domain, each exposing a `SPECS: list[dict]` the registry collects (dicts, not `ToolSpec`, to avoid a `registry → dispatchers → registry` import cycle). `registry._spec_from_mapping` is the one typed conversion point.
- **The `<user_data>` marker became eager-wrap, not a `UserText` object.** A `str`-subclass marker trips ruff FURB189, and a wrapper object cascaded a custom recursive type through `ToolDispatch`/`execute_tool`/`use_case.py`. Instead dispatchers call `user_text(v)` → `wrap(v)` eagerly (plain `str`), reusing the *shipped* v0.9.0 boundaries: `json.dumps(summary)` carries the tags to the model, `strip_user_data(summary)` strips them for the frontend. No new type, no `noqa`. `resolve_user_data`/`UserText` were removed.
- **Parity is `what has a human surface`, not `wrap every use case`.** Six use cases with no direct route/CLI are `INTERNAL`, reachable by the agent the same way a human reaches them — via a workflow: `EnrichTracksUseCase`, `MatchAndIdentifyTracksUseCase` (needs a live connector instance), `Create/UpdateConnectorPlaylistUseCase` (destination nodes), plus `ExecuteWorkflowRunUseCase` and the SSE-watchdog `GetOperationSnapshotUseCase`. `MatchAndIdentify` also can't be driven standalone (infra connector). No `oneOf`/numeric bounds (Anthropic rejects both); per-op required-field gating lives in prose + dispatcher validation raising actionable `ToolExecutionError`.
- **Long ops can only launch from the interface layer.** `launch_sse_operation` is FastAPI-bound, so the 5 long tools are `kind:"write"` with `launches_operation=True` and no application `executor`; the registry runs them through an injected `OperationLauncher` (`chat_operations.py`) at confirm time. The write-tool invariant is now executor-XOR-launches_operation, CI-checked.

### v0.9.2: Agentic Depth

**Goal**: With full parity the assistant can do everything — one tool call at a time, with every intermediate result flowing through the model's context. v0.9.2 makes it *efficient and long-horizon capable*: programmatic tool calling composes batch library operations in a sandbox (only the answer reaches the model), a research subagent absorbs deep analysis without polluting the conversation, tool search keeps the now-large registry out of the upfront context, and context management + a user-facing effort control keep long sessions fast and affordable. This is a port of couplefins v1.8.3 with mixd-shaped stories.

**Context**: All four capabilities are live-verified in couplefins with tuned constants and documented failure modes (their v1.8.3 "Implementation learnings" pin four programmatic-tool-calling bugs unit tests couldn't catch — port those as spec notes, not rediscoveries). Mixd's batch-first principle makes this milestone disproportionately valuable here: libraries have tens of thousands of tracks, and "re-tag everything matching X" is the canonical mixd ask.

**What this unlocks**:
- "Retag every track that's `mood:melancholy` but has >8 plays this month as `mood:bittersweet`" — one sandbox script over batch tools instead of hundreds of model round trips
- "Which of my starred tracks drifted out of rotation this year, and why might that be?" — delegated to a subagent that reads widely and returns one dense summary
- A registry of 30–40 tools that costs ~500 upfront tokens instead of the full schema set, with the prompt cache preserved
- Long curation conversations that don't degrade or blow the context window

**Persona**: Weekly Curator — primary. Batch maintenance and analysis are their core loop. Tinkerer — secondary. The sandbox + subagent behavior is the "this is a real agent" moment. Casual Enthusiast — tertiary. Faster, cheaper responses via effort control they never have to think about.

**Key Design Decisions**:
- **Read tools become sandbox-callable; write tools never** — `allowed_callers` includes the code-execution sandbox on `kind: "read"` entries only. Mutations always surface as individual proposals to the user; a sandbox script can *compute* a batch mutation but the commit still flows through one visible, confirmable tool call with the full change list in the preview.
- **The subagent is read-only by construction** — it gets `build_subagent_tools()` (read slice of the registry), a fresh context, low effort, and a bounded turn budget; its entire output is one summary string in the parent conversation.
- **Tool search is the default posture at this registry size** — a handful of hot tools (workflow generation, library search, tag/preference basics) stay non-deferred; everything else is `defer_loading: true` with a system-prompt "search before you claim you can't" nudge (couplefins found the nudge necessary).
- **Context editing over compaction** — `clear_tool_uses` with couplefins' thresholds as starting points (trigger 30k input tokens, keep 3 tool uses, clear ≥5k); server-side compaction stays evaluated-and-skipped while conversations are bounded by the 50-message cap. Revisit only if persistence ships.
- **Effort is the user's knob** — Quick/Standard/Thorough in the panel (low/high/xhigh), persisted locally, default Standard; the subagent is always low.
- **Tool loading follows the user's screen** — the web client sends the coarse UI section and its tools promote into the loaded set (rule-based routing; ≤3/page to hold the ~10-tool ceiling). Promoted tools ride the *uncached tail* so the cached core prefix stays page-invariant — navigating between pages never re-pays for the tool schemas. Strictly additive: an unrouted page degrades to the static core + tool search.

**Key learnings — don't rediscover:**

- **`allowed_callers` is stamped in `build_tools`, not stored per-spec.** couplefins computes it uniformly for every read tool at build time (`if enable_code_execution and kind == "read"`), so mixd has **no** `ToolSpec.allowed_callers` field — the "reads only" rule is a property tested on `build_tools()` output. Simpler than a per-spec field and matches the reference.
- **Server tools (`dispatch is None` agentic) need a distinct `build_tools` branch.** Their `input_schema` holds the raw `{type, name}` block the API wants; the `{name, description, input_schema}` wrapper is emitted only for dispatched tools. Three registry quality tests (`test_descriptions…`, `test_input_schemas…`, the cache-breakpoint test) had to skip `dispatch is None` entries.
- **SDK 0.116 ships both `code_execution_20260120` and `_20260521`.** couplefins (SDK 0.97) said `20260521` didn't exist; on 0.116 both do. Use **`20260120`** — the REPL-persistence + programmatic-tool-calling variant that `allowed_callers` references. `BetaToolUseBlock.caller`, `BetaMessage.container`, and `BetaToolSearchToolBm25_20251119Param` all resolve in 0.116, so the ported adapter needed **no changes** — the live sandbox smoke is the authoritative block-shape check.
- **Flipping `defer_loading` default to True forces `defer_loading=False` on every agentic tool** — the construction invariant (`agentic must not be deferred`) fires at import otherwise. Hot dispatcher reads opt in via `"defer_loading": False` in their `SPECS` dict; `_spec_from_mapping` now threads the flag (default True).
- **The subagent needs no per-user injection.** couplefins injects `persons` into the subagent prompt; mixd's music-domain prompt is static + today's date (the subagent discovers library shape via read tools), so `ToolContext` only gained the optional `llm` handle — dispatcher tests still construct `ToolContext(user_id=…)` without one.
- **The frontend was already wired, not dormant** — the v0.9.0 "DORMANT" comments in `CodeExecutionCard.tsx` and `lib/effort.ts` were stale; the effort selector, code-execution card, and `code_start`/`code_result` SSE parsing all shipped and were committed in v0.9.1. The only net-new UI is the `kind === "agentic"` branch in `ToolCallIndicator` (`delegate_analysis` now reads "Running deep analysis…" / "Ran deep analysis" instead of a generic lookup).
- **Page routing is free on the cache axis *only* if the cached prefix is page-invariant.** Toggling `defer_loading` in place would put a promoted tool before the cache breakpoint and bust the tools cache on every navigation. `build_tools` instead segregates a page-invariant cached **prefix** (curated core + dispatched agentic) from an uncached **tail** (deferred/promoted pool + raw server-tool blocks); per-page variation only mutates bytes *after* the breakpoint. The same restructure fixed a latent bug — the breakpoint had been landing on the trailing `tool_search` server block (a `{type,name}` block that rejects `cache_control`); it now sits on `delegate_analysis`, asserted by `test_build_tools_stamps_one_cache_breakpoint`.

- [x] **Page-Contextual Tool Routing**

    **Story**: A curator working on the Playlists page asks "which of these have gaps this month?" and wants the assistant to reach for playlist tools straight away — no "let me search for the right tool" detour — while the same question on the Library page leans on library tools.     **Decisions**:
    - **Route by the UI section, rule-based not semantic** — a route is the cleanest domain signal there is, so a static hint map beats a classifier (deterministic, zero added latency). Matches the 2026 progressive-disclosure consensus: classify and load the relevant tools *before* the request enters the window.
    - **Promoted tools ride the uncached tail so navigation never busts the tools cache** — the cached core prefix is page-invariant by construction; per-page variation costs a few re-sent schemas, not a cache miss on every page change (the naive in-place `defer_loading` toggle would have).
    - **Capped at ≤3 promotions per page** — keeps the loaded set under the ~10-tool ceiling past which selection accuracy degrades (OpenAI's own guidance; corroborated across the progressive-disclosure literature).
    - **Strictly additive** — an unknown or absent section promotes nothing and degrades to the static core + tool-search discovery, so a new route never breaks the assistant.
    - **Same discipline extended to the research subagent** — its flat 14-read toolset became a curated ≤10 hot set + `tool_search`; low effort is exactly where a bloated tool list most degrades selection.

    **Spec**:
    - Registry: `build_tools(*, enable_code_execution, page)` splits tools into a page-invariant cached **prefix** (curated core + dispatched agentic) and an uncached **tail** (server-tool blocks + deferred/promoted pool); the single cache breakpoint sits on the prefix's last (dispatched) tool. `_PAGE_TOOL_HINTS: Mapping[str, tuple[str, ...]]` + `_promoted_tool_names(page)`.

    - API: `ChatRequest.page: str | None` (≤64 chars) → `routes/chat.py` → `build_tools(page=…)`.
    - Frontend: `pageSection(pathname)` in `ChatPanel.tsx` (index→`dashboard`; `SECTION_BY_SEGMENT` kept in sync with `_PAGE_TOOL_HINTS`) threaded through `startStream` → `sendChatMessage` body `page`.
    - Hint map (starting point, revisit as sections stabilize): `playlists`, `library`, `workflows`, `dashboard`, `imports`.

    **Tests**:
    - (unit) every page's loaded set stays ≤10; the cached prefix is byte-identical across all pages (navigation never invalidates the cache); a page's hinted tools load while off-page they stay deferred.
    - (unit) subagent hot set ≤10 with `tool_search` present and one cache breakpoint on a read tool.
    - (component) the current UI section rides the request; an unrouted page (`/settings/account`) sends none.

### v0.9.3: MCP Server (Mixd as a Tool Surface)

**Goal**: Expose mixd's capabilities as an MCP server so any MCP-aware client — Claude Desktop, Cursor, Claude Code, future agentic tools — can read from and act on a user's mixd library. By now the registry is parity-complete and battle-tested by the in-app assistant, so this milestone is **transport, long-op mapping, and exposure** — not new tool logic.

**Context**: The MCP 2026-07-28 revision is final before this milestone starts: stateless core (no handshake/sessions — the SDK owns protocol mechanics), a Tasks extension for long-running work, and MCP Apps for server-shipped UI. Build on the official `mcp` Python SDK at its post-2026-07-28 release; design tools stateless-first (explicit handles, no server-held conversation state).

**What this unlocks**:
- Local-first agentic workflows: "Claude, find all my starred tracks from last year that aren't on my Friday playlist, and add them" — from Claude Desktop or Cursor, no code.
- Long operations (workflow runs, imports) as first-class MCP Tasks with progress and cancellation, not blocked calls.
- The parity matrix doubles as the MCP coverage audit — generated, not hand-audited.

**Persona**: Tinkerer — primary. Self-hosters and power users who already pipe LLMs into their toolchains. Curator — secondary. Ad-hoc scripted maintenance ("deduplicate my 'Loved' playlist against my starred preferences"). Casual Enthusiast — tertiary, via whatever agents others build on top.

**Key Design Decisions**:
- **Stdio-only; identity from the environment.** Stdio matches the spec's recommendation and mixd's local-first deployment; the server resolves the user exactly as every CLI command does — `get_cli_user_id()` (`MIXD_USER_ID` env → `DEFAULT_USER_ID`). No new credential surface.
- **Remote/Streamable-HTTP MCP is a demand-gated successor**, not scope here: it requires OAuth 2.1 resource-server work under the 2026-07-28-hardened authorization spec — mixd's Neon Auth JWT infrastructure is the natural authorization server when real demand materializes.
- **One registry, second consumer**: `list_tools`/`call_tool` iterate `src/application/tools/registry.py`. `agentic` entries (sandbox, subagent, tool search) are chat-executor concerns and are **not exposed** over MCP — the MCP client brings its own agentic loop.
- **Annotations derived from `kind`**: `readOnlyHint = (kind == "read")`, `destructiveHint = (kind == "write")`, `idempotentHint: true` on read tools, `openWorldHint: false`. The spec marks annotations as untrusted — mutation safety never relies on them.
- **Mutations require explicit confirmation in-band**: any `write` tool called without `confirm=true` returns a structured preview + confirm token instead of executing. Well-behaved clients will also prompt on `destructiveHint`; mixd doesn't depend on it.
- **Long-running tools use the Tasks extension**: `run_workflow` and imports answer `tools/call` with a task handle backed by mixd's existing operation records; `tasks/get` maps to operation status, `tasks/cancel` to operation cancellation. Clients without the extension get the blocking fallback with a documented timeout caveat.
- **Structured output where it helps**: read tools may declare `outputSchema` and return `structuredContent` reusing existing Pydantic API schemas — decide per tool during implementation, not a blanket requirement.

**Key learnings — don't rediscover:**

- **SDK is `mcp==2.0.0b1` + `mcp-types==2.0.0b1` (both pre-release).** uv won't pull
  the transitive `mcp-types` pre-release unless it's *also* directly pinned — pin
  both (their specifiers match only pre-releases, so pre-releases aren't enabled
  globally). RE-VERIFY CHECKPOINT baked into `pyproject.toml`: bump to stable v2
  on/after 2026-07-28 and re-run `tests/unit/interface/mcp/` before the Tasks epic.
- **v2 low-level API differs from v1 and from the README (which only shows the
  high-level `@mcp.tool()` decorator).** Types moved to the separate `mcp_types`
  package (no more `mcp.types`); the low-level `Server` dropped the
  `@server.list_tools()`/`@server.call_tool()` decorators for
  `add_request_handler(method, params_type, handler)` (`"tools/list"` →
  `PaginatedRequestParams`, `"tools/call"` → `CallToolRequestParams`; handler
  returns a `BaseModel`). `Tool(input_schema=…)` serialises to `inputSchema`;
  annotation fields to `readOnlyHint` etc. The high-level `MCPServer.add_tool(fn)`
  infers schemas from Python functions, so it's the *wrong* tool for a
  registry-of-hand-built-schemas — the low-level `Server` is the faithful fit.
  All facts pinned in `server.py`'s module docstring.
- **stdout cleanliness is the killer gotcha, and it's an import-time problem.**
  structlog defaults to a **stdout** `PrintLogger` until `setup_logging()` runs, so
  import-time debug logs (the Rich console singletons in `console.py`/`cli_helpers`)
  land on stdout *before* any command body — fatal for a JSON-RPC channel. Fixed in
  two places: an early argv guard at the top of `app.py` (routes logging to stderr
  before the first import-time log for any `mixd mcp …` command) and the
  `init_cli` callback (stderr + skip the console-printing startup checks for the
  `mcp` group). `setup_logging` gained a keyword-only `console_stream`. Same
  discipline keeps `mcp install --print` stdout pure JSON.
- **The exposure filter is one shared classifier.** `src/interface/mcp/exposure.py`
  `mcp_exposure(spec)` (`exposed`/`agentic`/`pending_tasks`) is imported by both
  `server.exposed_specs()` and the capability-matrix generator — SDK-free so the
  script needn't import the MCP SDK, and so the doc can't drift from what the
  server actually exposes. 26 exposed / 5 pending-Tasks / 3 agentic today.
- **RUF029 per-file-ignore for `server.py`** follows the established FastAPI/
  `ToolDispatch` precedent: the SDK's request-handler protocol requires `async`
  handlers even where one (`tools/list`) does no I/O.
- **Long-op tools can't be exposed yet by construction.** The 5 `launches_operation`
  writes need an injected `OperationLauncher`, and the only implementation
  (`launch_chat_operation`) is FastAPI-coupled — so the Tasks epic bundles the
  launcher-decoupling refactor. Cleanly excluded now via `mcp_exposure == pending_tasks`.

#### Epics

- [x] **MCP Server Scaffold (Stdio, Stateless)**

    **Story**: A Tinkerer wants to point Claude Desktop at mixd and immediately see mixd's tools in the client's palette — the same way they'd add the filesystem server. No HTTP, no tokens, just a `mixd mcp serve` subprocess.

    **Decisions**:
    - Official Python MCP SDK at its post-2026-07-28 release, registered as a `mixd mcp serve` Typer subcommand. The SDK owns protocol mechanics (now stateless — no session logic to write, and none should be written); mixd owns tool metadata + dispatch.
    - User resolved via `get_cli_user_id()`; misconfigured state returns a clear per-call error with remediation, not a crash.
    - One UoW per tool call (`execute_use_case` with the resolved user id — RLS scoping comes free). Calls are independent; state crosses calls only via explicit handles (IDs in tool results).
    - **stdout is the protocol channel**: structlog routes to stderr/file (spec allows stderr for logging; stdout MUST carry only MCP messages).

    **Spec**:
    - New module `src/interface/mcp/server.py` — builds the server from the shared registry: name, description, `input_schema`, annotations derived from `kind`; `agentic` entries filtered out
    - New CLI subcommand `mixd mcp serve` in `src/interface/cli/`
    - Log path configurable via env var

    **Tests**:
    - (integration) server boots, responds to `tools/list` over stdio; tool metadata matches registry entries; no `agentic` tools exposed
    - (integration) `tools/call` on a read tool returns a valid result scoped to the resolved user
    - (unit) annotations correctly derived from `kind` for every exposed entry

    Effort: M | Dependencies: v0.9.1 (Parity Contract Closure) | Status: Completed (2026-07-12) — built against the locked RC + beta SDK (`mcp==2.0.0b1`); re-verify at the 2026-07-28 stable bump

- [x] **In-Band Mutation Confirmation over MCP**

    **Story**: An agent acting for the user wants to run a workflow or edit a playlist from Claude Desktop — but the user wants the same never-silent guarantee the in-app assistant gives: a preview first, and nothing committed until explicitly confirmed.

    **Decisions**:
    - Two-phase in-band: first call → `{ status: "needs_confirmation", preview, confirm_token }`; second call with `confirm=true` + token commits. Expired tokens return a fresh preview, never a stale commit.
    - Confirm tokens scoped to `(user_id, tool_name, args_hash)`, 5-minute TTL — same store pattern as the chat pending actions (possibly the same store). Args drift between preview and confirm is rejected with a clear error.
    - Previews reuse the same human-readable summaries the chat ConfirmationCard shows — one preview renderer, two consumers.

    **Spec**:
    - Confirmation wrapper in `src/interface/mcp/` fronting `kind: "write"` dispatch; token store shared with `src/application/chat/pending_actions.py`
    - Preview payload includes the structured change summary + prose rendering

    **Tests**:
    - (integration) first call previews without mutating; `confirm=true` commits and returns post-mutation state
    - (integration) expired token → fresh preview; args drift → rejection
    - (integration) every mutation scoped to the resolved user

    Effort: M | Dependencies: MCP Server Scaffold | Status: Completed (2026-07-12) — synchronous writes; args-drift rejected, expired/unknown token re-previews. Implemented in `confirmation.py`

- [ ] **Long-Running Tools via the Tasks Extension**

    **Story**: An agent kicks off a 50k-scrobble import or a workflow run from Claude Desktop; the user wants it to behave like real background work — observable progress, cancellable, no client timeout roulette on a blocked call.

    **Decisions**:
    - Advertise the Tasks extension; operation-returning tools answer `tools/call` with a task handle when the client supports it (server-directed, per spec). Handles map 1:1 to mixd operation ids.
    - `tasks/get` reads operation status/progress from the existing operation records; `tasks/cancel` routes to operation cancellation.
    - Clients without the extension get a blocking call with a documented duration caveat — no bespoke polling protocol.

    **Spec**:
    - Task adapter in `src/interface/mcp/tasks.py` mapping task handles ↔ operation ids
    - Applies to the same operation-returning tools v0.9.1 wired into chat

    **Tests**:
    - (integration) task-capable flow: call → handle → progress via `tasks/get` → completion state matches the run log
    - (integration) `tasks/cancel` cancels the underlying operation
    - (integration) non-task client falls back to blocking completion

    Effort: M | Dependencies: In-Band Mutation Confirmation over MCP | Status: Not Started — **gated to post-2026-07-28** (stable SDK + published Tasks extension). Bundles the `OperationLauncher` FastAPI-decoupling refactor, since the 5 `launches_operation` tools need an interface-layer launcher the current `launch_chat_operation` doesn't provide outside FastAPI. Until then they're excluded from MCP via `mcp_exposure == "pending_tasks"`.

- [ ] **Exposure Polish: `mixd mcp install`, Docs, Coverage**

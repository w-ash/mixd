### v0.9.0: First-Class Artists
**Goal**: Promote artists from embedded JSON strings to first-class entities with identity resolution, favorites, web UI, and workflow integration.

**Context**: Artists are currently stored as `{"names": ["..."]}` JSON on Track and ConnectorTrack records — "Radiohead" on Spotify and "Radiohead" on Last.fm are unrelated text. This milestone gives artists their own identity, cross-service mapping, favorites, and dedicated pages, following the same three-layer pattern established for tracks.

**Key Design Decisions**:
- **Favorites are Narada-only** — users favorite artists in the UI. Service sync (e.g., Spotify followed artists) deferred to a future milestone.
- **`track_artists` join table** replaces JSON artist embedding on tracks. Enables efficient "all tracks by artist X" queries and proper relational integrity.

#### Backend Data Model Epics

- [ ] **Artist Entity & Domain Model**
    - Effort: M
    - What: `Artist` as a proper domain entity with `id`, `name`, and relationships. `ArtistFavorite` entity for Narada-only favorites.
    - Why: Artists need identity to support cross-service mapping, favorites, and workflow filtering
    - Dependencies: None
    - Status: Not Started
    - Notes:
        - Domain: `Artist(id, name)` in `src/domain/entities/` — attrs frozen entity like `Track`
        - Domain: `ArtistFavorite(artist_id, is_favorited, favorited_at)` — mirrors `TrackLike` but without `service` field (Narada-only)
        - `ConnectorArtist(connector_name, connector_artist_identifier, name, raw_metadata)` — per-service artist record
        - Artist matching: name normalization, MusicBrainz MBID when available

- [ ] **Artist Database Schema**
    - Effort: M
    - What: `artists`, `connector_artists`, `artist_mappings`, `artist_favorites`, `track_artists` tables
    - Why: Relational model enables efficient queries, cross-service mapping, and favorites
    - Dependencies: Artist Entity & Domain Model
    - Status: Not Started
    - Notes:
        - Replicates three-layer pattern from `db_models.py`: `DBArtist` ↔ `DBArtistMapping` ↔ `DBConnectorArtist`
        - `artists`: `id`, `name` (unique), `created_at`, `updated_at`
        - `connector_artists`: `id`, `connector_name`, `connector_artist_identifier`, `name`, `raw_metadata` (JSON)
        - `artist_mappings`: `artist_id` → `artists`, `connector_artist_id` → `connector_artists`, `match_method`, `confidence`, `is_primary`
        - `artist_favorites`: `artist_id` → `artists`, `is_favorited`, `favorited_at` (no `service` column — Narada-only)
        - `track_artists`: `track_id` → `tracks`, `artist_id` → `artists`, `position` (ordering). Replaces JSON embedding.
        - Alembic migration: create tables, backfill `track_artists` from existing JSON, retain JSON column as fallback during transition

- [ ] **Artist Repository Layer**
    - Effort: M
    - What: `ArtistRepository`, `ArtistConnectorRepository`, `ArtistFavoriteRepository`
    - Why: Clean data access following existing repository patterns
    - Dependencies: Artist Database Schema
    - Status: Not Started
    - Notes:
        - Mirrors structure in `src/infrastructure/persistence/repositories/track/`
        - `ArtistRepository`: `get_by_id`, `get_by_name`, `list_artists` (paginated), `search_by_name`
        - `ArtistConnectorRepository`: `find_artists_by_connectors`, `bulk_upsert_connector_artists`, `set_primary_mapping`
        - `ArtistFavoriteRepository`: `get_favorites`, `save_favorite`, `is_favorited`
        - Mapper classes: `ArtistMapper`, `ConnectorArtistMapper`, `ArtistMappingMapper` following `mapper.py` patterns

- [ ] **Artist Identity Resolution**
    - Effort: L
    - What: Cross-service artist matching (Spotify artist ID ↔ Last.fm artist ↔ MusicBrainz artist MBID)
    - Why: Canonical artist identity enables deduplication, enrichment, and cross-service queries
    - Dependencies: Artist Repository Layer
    - Status: Not Started
    - Notes:
        - Leverages existing `MatchProvider` protocol from `domain/matching/protocols.py`
        - Spotify: artist ID from track metadata (already fetched)
        - Last.fm: artist name matching (case-insensitive, normalized)
        - MusicBrainz: artist MBID lookup via existing `MusicBrainzAPIClient`
        - Confidence scoring mirrors track matching: direct ID match (100), MBID match (95), name match (80)
        - Populate during track import — when tracks arrive with artist info, upsert canonical artists
        - **Artist alias/abbreviation table**: MusicBrainz artist data includes aliases — import as a lookup table for identity resolution. Addresses confirmed smoke test gap: "TEED" (abbreviation) vs "Totally Enormous Extinct Dinosaurs" (full name) are the same artist but text normalization alone cannot map abbreviations to full names. MusicBrainz API already integrated (`MusicBrainzAPIClient` at `musicbrainz/client.py`).

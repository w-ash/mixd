### v0.9.0: Workflow Assistant (Right-Panel Chat)

**Goal**: Today the Casual Enthusiast has no front door — building a workflow means learning the node catalog. v0.9.0 gives them a persistent right-panel chat assistant: describe a playlist in plain English ("upbeat tracks I haven't played in 6 months") and get a real, editable `WorkflowDef` — previewed in the existing visual editor, saved on approval. Same always-available, no-modal pattern as Cursor / Linear / Notion AI.

**Context**: mixd today is genuinely useful by Curators and Tinkerers but has no front door for the Casual Enthusiast — building a workflow currently requires understanding the node catalog and connector model. v0.9.0 collapses that learning curve into a sentence of natural language. Per the README's persona-alignment table this is "**THE adoption enabler** — changes who can use mixd."

The architectural decisions for this milestone are not novel research — our sister project **couplefins** has a complete, production-tested right-panel chatbot system built deliberately on the same stack (Vite + React + TypeScript + FastAPI + Python + SQLAlchemy + Tanstack Query) so we could port it. v0.9.0 is **substantially a port**, not greenfield. The mixd-specific work is the domain primer, the node-catalog → JSON Schema bridge, and a workflow-preview render of generated DAGs. The infrastructure (SSE streaming, Anthropic adapter, tool dispatch, two-phase confirmation, prompt caching, error mapping, Zustand store, edge-tab affordance) is already-solved.

The 2026 chatbot-UX research aligns with couplefins's existing choices: POST+SSE streaming with sub-300ms TTFT, schema-constrained tool use via `output_config.format` and `strict: true`, ephemeral prompt caching (90% cost reduction at scale), server-side context injection rather than prop-passing, persistent right-panel rather than floating modal. We don't need to invent any of this — couplefins already shipped it.

**What this unlocks**:
- One-shot natural-language workflow generation with schema-constrained output (no few-shot prior needed — the WorkflowDef JSON Schema is the prior)
- Persistent right-panel chat (Cmd+K toggle, edge-tab when collapsed) that follows the user across all routes
- Workflow preview rendered in the existing v0.4.3 visual editor as a read-only DAG before save
- Two-phase confirmation: LLM proposes → user inspects in the editor → user approves → save (mutations never silent)
- Foundation for any future mixd-side AI features (track picking, tag suggestions, listening insights) — the same pattern, different tools

**Persona**: Casual Enthusiast — primary. The natural-language entry point is the front door for users who'd never have built a workflow from scratch. Tinkerer — secondary. Generates a starting point in seconds, then tweaks in the editor they already know. Weekly Curator — tertiary. Faster ad-hoc workflow creation when a specific need arises ("I want my Friday-night dinner-party playlist refreshed weekly with songs I've loved this month").

**Key Design Decisions**:
- **Port couplefins, don't rebuild**: The right-panel chat infrastructure exists in `/Users/wash/Projects/couplefins/`. We copy components, adapt names/types to mixd's domain, and reuse the underlying patterns. The mixd-specific surface area is small: domain primer, tools list, workflow preview card.
- **Tools are thin adapters over the application layer, not their own logic** *(subject to the Pre-Work pass)*: Every chat tool in `src/application/chat/tool_executor.py` dispatches to an existing or newly-extracted use case in `src/application/` — no business logic lives in the tool itself. This discipline is load-bearing for any future MCP surface (v0.9.1 below) that reuses the same use case layer. **Whether the registry is built here or extracted later is exactly what the Pre-Work brief should resolve** — current draft tolerates a v0.9.1 refactor, but a registry-first sequence may be cleaner.
- **Schema-constrained tool use, not few-shot priors**: The LLM doesn't need template workflows as training examples. The `WorkflowDef` JSON Schema (machine-derived from the existing node registry) IS the prior. Modern frontier models produce correct DAGs in one shot when given the schema as a `tool_use` definition with `strict: true`. This **drops the v0.8.5 → v0.9.0 dependency an earlier draft claimed** — they're independent.
- **Workflow Templates (v0.8.5) are user-facing, not LLM-facing**: Templates serve as empty-state inspiration on "new workflow" pages and as starting points for users who prefer copy-and-modify over from-scratch. They are NOT the LLM's training data. v0.9.0 can ship before, after, or alongside v0.8.5 with no architectural coupling. _(v0.8.5 has been corrected to match this — templates no longer claim to feed LLM prompt engineering.)_
- **Reuse the v0.4.3 visual editor for previews**: The "Visualization Confirmation UI" isn't a new component — it's the existing visual editor rendered in read-only mode inside the chat panel's `ToolResultCard`. Same component, different prop. No parallel viz codebase to maintain.
- **Refinement is first-class in both chat and editor**: A user can iterate on an unsaved or saved workflow in chat ("make it 30 tracks instead of 20", "add a recency filter that excludes anything I played this month", "sort by play count descending") OR in the visual editor after save. Both pathways are first-class — chat is great for high-level intent shifts and quick parametric tweaks; the editor is great for fine-grained graph surgery. Each refinement turn that changes the workflow re-emits a fresh `generate_workflow_def` tool call with the updated **complete** DAG; the Workflow Preview Card replaces its prior preview rather than appending. The chat history is the LLM's diff context — no separate "update_workflow_draft" tool needed.
- **Server-side context injection**: The chat panel doesn't pass `<ChatPanel context={...}>` props. The FastAPI route resolves "who is this user, what's in their library, what workflow are they currently editing" from authenticated session + the route they came from, and bakes it into the system prompt. Couplefins proves this pattern; we follow it.
- **Two-phase confirmation for saves**: Following couplefins's mutation pattern. The LLM's `save_workflow` tool returns a "pending action" rather than persisting directly. Frontend renders ConfirmationCard with the proposed WorkflowDef preview. User approves → second SSE request executes the save. Prevents silent writes from a single hallucination.
- **Ephemeral conversations, no DB persistence**: Couplefins has no `conversations` table — chat state lives in the Zustand store, hard-capped at 50 messages, cleared on "New Conversation." v0.9.0 follows suit. If conversation history becomes a real user need post-launch, add it as a Post-Deploy Revision.
- **Anthropic Claude as the provider**: Couplefins uses `AsyncAnthropic` via `AnthropicAdapter`. We do the same. `output_config.effort="medium"` matches couplefins's choice. System prompt cached via `cache_control: {"type": "ephemeral"}` to hit the 2048-token activation threshold (which means the domain primer needs to be substantive — not a problem given mixd's node catalog).
- **In-memory rate limiter**: 20 req/60s per user, matching couplefins. Right-sized for mixd's "dozens to low hundreds of users" deployment shape. No Redis needed.

### v0.9.1: MCP Server (Mixd as a Tool Surface)

**Goal**: Expose mixd's capabilities as an MCP (Model Context Protocol) server so any MCP-aware client — Claude Desktop, Cursor, Claude Code, future agentic tools — can read from and act on a user's mixd library. The surface is broad by design: most things doable via CLI or web-UI are doable via MCP. The same tool registry powers v0.9.0's in-process chat assistant and v0.9.1's out-of-process MCP server.

**Context**: v0.9.0 established a tool layer for the workflow assistant. That layer was built as thin adapters over `src/application/` use cases precisely so a second consumer could plug in without duplication. v0.9.1 is that second consumer. The work is mostly **registry, transport, and audit** — not new business logic. Where a use case is missing or awkwardly shaped for tool consumption (e.g. takes a SQLAlchemy session, returns ORM objects), we fix it at the source rather than wrap it. This benefits the chat assistant too.

The Tinkerer persona is the most direct beneficiary — many already run local LLM agents and will reach for mixd as a tool surface. The Casual Enthusiast benefits indirectly when third-party agents grow workflows that touch mixd. The Curator benefits when scripted/automated maintenance becomes possible without writing CLI invocations.

**What this unlocks**:
- Local-first agentic workflows: "Claude, find all my starred tracks from last year that aren't on my Friday playlist, and add them" — works in Claude Desktop or Cursor without writing code.
- A forcing function for use case completeness: any CLI command or web-UI action that can't be expressed as an MCP tool reveals a gap in the application layer.
- Foundation for future remote-MCP support (hosted mixd as a multi-user MCP endpoint), gated on auth maturity.
- Shared tool registry means new mixd capabilities automatically become available to both the in-app assistant and external agents — write once, expose twice.

**Persona**: Tinkerer — primary. Self-hosters and power users who already pipe LLMs into their toolchain. Curator — secondary. Enables ad-hoc scripted maintenance ("agent, deduplicate my 'Loved' playlist against my starred preferences"). Casual Enthusiast — tertiary. Indirect benefit through whatever agents others build on top.

**Key Design Decisions**:
- **Stdio-only for v0.9.1; HTTP/remote deferred**: First ship is local stdio transport, launched by an MCP client as a subprocess (`mixd mcp serve`). Inherits the authenticated CLI user via the existing session mechanism — no new credential surface. Remote/HTTP MCP requires multi-user auth scoping and bearer token issuance; that lands as a Post-Deploy Revision or a v0.9.x successor only if real demand materializes.
- **Single tool registry, two adapters**: The registry of tool definitions (name, description, JSON schema, side-effect class, dispatcher) lives in `src/application/tools/` and is consumed by *both* `src/application/chat/tool_executor.py` (v0.9.0, in-process) and `src/interface/mcp/server.py` (v0.9.1, MCP-protocol-facing). No tool is defined in two places. If v0.9.0 already named tools well, v0.9.1 mostly re-exports.
- **Side-effect classification is explicit**: Every tool declares `effect: "read" | "mutate"`. The MCP server uses this to mark mutating tools in the protocol's `annotations.destructiveHint` / `readOnlyHint` fields so clients can prompt before running them. Chat assistant's two-phase confirmation reads the same flag.
- **Mutations require explicit confirmation argument**: Mirroring couplefins's two-phase pattern in spirit: any tool with `effect: "mutate"` rejects calls without `confirm=true` in args, returning a structured "preview + ask for confirmation" result. MCP clients display the preview, the user re-issues with `confirm=true`. Prevents silent destruction from an over-eager agent.
- **Coverage audit is explicit deliverable, not aspiration**: Spec includes an audit table mapping every CLI command and every meaningful web-UI action to either (a) the MCP tool that covers it, or (b) a written rationale for exclusion (e.g. OAuth login redirect can't be done out-of-band). Audit lives in `docs/web-ui/mcp-coverage.md` so it stays maintained.
- **JSON schemas are LLM-readable, not just machine-valid**: Tool descriptions are written for an LLM consumer — 3–4 sentences, concrete examples in `description` fields, no implementation jargon. Same standard v0.9.0 sets for chat tools. The MCP audit is also a writing pass.
- **Discovery & install**: `mixd mcp install` writes the Claude Desktop / Cursor config snippet automatically. Documentation in `docs/guides/mcp.md`. README's "What Mixd Does" gains a sentence — mixd is now a tool surface, not just a CLI/web app.

#### Epics

- [ ] **Extract Shared Tool Registry**

    **Story**: A maintainer wants tool definitions to live in one place — name, description, parameter schema, side-effect class, dispatcher function — so adding a capability or fixing a description updates every interface (in-app chat, MCP) at once. Without this, the chat assistant and MCP server drift apart over time, and the use case layer accrues thin wrapper functions duplicated in two adapters.

    **Decisions**:
    - New module `src/application/tools/` (or co-located with use cases — confirm during implementation). Each tool is a small dataclass: `name`, `description`, `input_schema` (JSON Schema), `effect`, `dispatch` (async callable taking validated args + a UoW or session, returning a result).
    - v0.9.0's `tool_executor.py` becomes a thin loop over registry entries — no per-tool branching.
    - v0.9.1's MCP server's `list_tools` / `call_tool` handlers iterate the same registry.
    - Where v0.9.0 chat tools accreted logic during its build, this epic refactors them into use case calls. Done as part of v0.9.1 work, not retroactively in a v0.9.0 revision, because the second consumer is what makes the seam matter.

- [ ] **Mutation Tools with Two-Phase Confirmation**

    **Story**: An agent wants to *do things*, not just look — run a workflow, import a playlist, tag tracks, edit preferences, save a generated workflow — but never silently. Each mutating call returns a preview first; the agent (or the user via the agent) explicitly re-issues with `confirm=true` to commit.

    **Decisions**:
    - Two-phase pattern: first call returns `{ status: "needs_confirmation", preview: {...}, confirm_token: "..." }`. Second call with `confirm=true` (and optionally `confirm_token`) commits.
    - Idempotency: confirm tokens scoped to the tool + args hash, valid for 5 minutes. Re-confirming an expired token returns a fresh preview rather than committing stale state.
    - MCP `annotations.destructiveHint=true` set on every mutation tool so well-behaved clients prompt the user even before the preview round-trip.
    - Tools list: `run_workflow`, `save_workflow`, `update_workflow`, `delete_workflow`, `import_playlist`, `set_track_preference`, `add_track_tags`, `remove_track_tags`, `create_tag`, `delete_tag`, `relink_track`, `unlink_track`, `merge_tracks`
    - Mutations that are already preview-friendly in the web UI (e.g. workflow run with dry-run mode) reuse that preview surface.

    **Spec**:
    - Confirm-token store: in-memory dict keyed by `(user_id, tool_name, args_hash)`, TTL 300s. Same pattern as v0.9.0's pending-action store — possibly the same store.
    - Per-mutation dispatcher: build preview → store confirm token → return; on confirm=true, validate token + re-validate args → execute via existing use case.
    - All execution paths land in existing CLI/web use cases — no new persistence logic.

    **Tests**:
    - (integration) First call returns preview without mutating
    - (integration) Confirm=true commits and returns post-mutation state
    - (integration) Expired confirm token returns fresh preview, not commit
    - (integration) Args drift between preview and confirm rejected with clear error
    - (integration) Each mutation tool scoped to authenticated user

    Effort: M | Dependencies: Read-Tool Coverage | Status: Not Started

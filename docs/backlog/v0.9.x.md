### v0.9.0: First-Class Artists
**Goal**: Promote artists from embedded JSON strings to first-class entities with identity resolution, favorites, web UI, and workflow integration.

**Context**: Artists are currently stored as `{"names": ["..."]}` JSON on Track and ConnectorTrack records ‚ÄĒ "Radiohead" on Spotify and "Radiohead" on Last.fm are unrelated text. This milestone gives artists their own identity, cross-service mapping, favorites, and dedicated pages, following the same three-layer pattern established for tracks.

**Key Design Decisions**:
- **Favorites are Narada-only** ‚ÄĒ users favorite artists in the UI. Service sync (e.g., Spotify followed artists) deferred to a future milestone.
- **`track_artists` join table** replaces JSON artist embedding on tracks. Enables efficient "all tracks by artist X" queries and proper relational integrity.

### v0.9.1: First-Class Albums
**Goal**: Promote `album` from a string field on Track to a proper entity with identity resolution, cross-service mapping, and dedicated web UI.

**Context**: Albums are currently a bare `VARCHAR(255)` on the tracks table ‚ÄĒ "OK Computer" on Spotify and "OK Computer" on Last.fm are unrelated strings. This milestone gives albums their own identity, following the same three-layer pattern (canonical ‚Üí mapping ‚Üí connector) established for tracks and extended for artists in v0.9.0.

**Key Design Decisions**:
- **Album ‚Č† Release**: An "album" is the abstract work (MusicBrainz calls this a "release group"). A "release" is a specific pressing/edition (UK vinyl 1997, Japanese CD with bonus tracks). This milestone covers albums; v0.9.2 adds the release/pressing layer.
- **MusicBrainz release groups as canonical**: Recording lookups already return release data ‚ÄĒ we promote it from ignored to first-class.
- **Album artists vs track artists**: An album has its own artist(s) (e.g., "Various Artists" for compilations). `album_artists` join table parallels `track_artists` from v0.9.0.

### v0.9.2: Physical Media & Discogs
**Goal**: Track which albums the user owns and on which media. Discogs as the primary source for physical release metadata ‚ÄĒ pressings, formats, labels, catalog numbers.

**Context**: Your music identity isn't just streaming ‚ÄĒ it's the vinyl on your shelf, the CDs in your car, the albums you've purchased digitally. Discogs has 15M+ releases with rich physical media metadata (format, label, catalog number, country, pressing notes). Importing a user's Discogs collection instantly populates "what I own and in what format."

**Key Design Decisions**:
- **Release ‚Č† Album**: An album (release group) can have many releases (pressings). "OK Computer" is one album; the 1997 UK vinyl, 2008 Japanese CD, and 2017 OKNOTOK remaster are three releases.
- **Discogs as primary physical source**: MusicBrainz has releases too, but Discogs has stronger collector community data (condition, notes, marketplace). Start with Discogs; MusicBrainz release data can supplement later.
- **Read-only initially**: Import from Discogs, don't write back. User manages their Discogs collection on Discogs.

### v0.9.3: Manual Scrobbling
**Goal**: Record physical album listens ‚ÄĒ "I put this record on at 3pm" ‚ÄĒ scrobble to Last.fm and store as canonical plays in narada.

**Context**: When you listen to a vinyl record, that listening event is invisible to your digital history. Narada should let you log it: pick the album, enter when you started, and the system figures out the per-track timestamps from track order and durations. The plays become part of your canonical history ‚ÄĒ indistinguishable from streaming plays in workflows and metrics.

**Key Design Decisions**:
- **Album-level input, track-level output**: User says "I listened to this album starting at 3pm." System calculates per-track timestamps and creates individual `TrackPlay` records. Each track gets its own play entry because that's what Last.fm expects and what narada's play history model requires.
- **Optional Last.fm scrobble**: User chooses whether to also scrobble to Last.fm or keep it narada-only. Some people want to keep physical and digital scrobbles separate.
- **Retroactive and real-time**: Support both "I listened yesterday at 3pm" and "I'm listening right now."

#### Scrobble Infrastructure Epics

- [ ] **Last.fm Scrobble API**
    - Effort: S
    - What: Implement `track.scrobble` on `LastFMAPIClient` ‚ÄĒ authenticated POST, batch up to 50 tracks per request
    - Why: Foundation for sending play events to Last.fm
    - Dependencies: None (builds on existing Last.fm auth ‚ÄĒ `track.love` write already works)
    - Status: Not Started
    - Notes:
        - Last.fm `track.scrobble` API: `method=track.scrobble`, params: `artist`, `track`, `timestamp` (Unix epoch), optional `album`, `duration`
        - Batch: up to 50 scrobbles per request via indexed params (`artist[0]`, `track[0]`, `timestamp[0]`, etc.)
        - Requires same session key auth as `track.love` (already implemented)
        - Scrobble rules: track must be >30s duration, listen must be >50% of duration or >4min

- [ ] **Album Listen Calculator**
    - Effort: S
    - What: Pure domain function that calculates per-track timestamps from album track order, durations, and a start time
    - Why: Core logic for converting "album listen at 3pm" ‚Üí individual track plays
    - Dependencies: v0.9.1 (first-class albums with ordered track listings)
    - Status: Not Started
    - Notes:
        - Domain: `calculate_listen_timestamps(tracks: list[Track], start_time: datetime) -> list[tuple[Track, datetime]]`
        - Track N+1 starts at: start_time + sum(duration of tracks 1..N)
        - Handle missing durations: skip track (log warning), or use average duration as estimate
        - Handle multi-disc albums: disc order ‚Üí track order (already modeled in `album_tracks` via disc_number + track_number)
        - Pure function, no side effects ‚ÄĒ testable in isolation

- [ ] **Record Album Listen Use Case**
    - Effort: M
    - What: `RecordAlbumListenUseCase` ‚ÄĒ takes album ID + start time, creates canonical plays, optionally scrobbles to Last.fm
    - Why: Orchestrates the full flow: resolve tracks ‚Üí calculate timestamps ‚Üí persist plays ‚Üí scrobble
    - Dependencies: Album Listen Calculator, Last.fm Scrobble API
    - Status: Not Started
    - Notes:
        - `RecordAlbumListenCommand(album_id, started_at, scrobble_to_lastfm: bool = True)`
        - Creates `TrackPlay` records in narada for each track (same table as streaming plays ‚ÄĒ canonical history is unified)
        - If `scrobble_to_lastfm`: batch-scrobble to Last.fm via `track.scrobble`
        - Result includes: tracks played, timestamps, Last.fm scrobble success/failure per track
        - Play source tracking: `service="manual"` or `service="physical"` to distinguish from streaming plays in queries

# Identity Resolution — Design-Space Memo

**Date**: 2026-07-02 · **Commission**: [identity-resolution-research-handoff.md](completed/identity-resolution-research-handoff.md) · **Status**: Active — research deliverable (no code changed); feeds v0.8.18, v0.10.2, v1.0.x

**Mission recap**: map how mixd resolves track identity across Spotify / Last.fm / MusicBrainz today, measure it against 2026 entity-resolution practice and fresh provider documentation, assess Apple Music readiness plus an 8–10-service future, and lay out *directions with trade-offs* — not a chosen design. Scope extensions (2026-07-02): artist-level identity (§7) and the omni-integration breadth ring (§6).

**Method**: every internal claim was verified by main-thread file reads at HEAD `5cba4588` (file:line anchors throughout); git archaeology dated each design decision; provider claims come from primary docs fetched 2026-07-02 (the three most load-bearing — Spotify's March 2026 `external_ids` reversion, MusicBrainz rate-limit/polling policy, and the backlog sketches — re-verified in the main session). Research-agent reports cross-checked ≥2 sources for anything that became a recommendation. Data quantification is pending a prod query run (§8) — the local dev DB is empty; every quantitative claim below is tagged `[Qn — pending prod]` where data would confirm scale.

**One-paragraph verdict** (non-prescriptive lean, evidence follows): the *matching engine* is genuinely sound — a correctly-shaped Fellegi–Sunter implementation that 2026 practice still endorses. The fragility lives **around** it: confidence numbers are corrupted after the fact by persistence-layer code that predates the model (§1 FM1), ISRC-as-identity leaks into schema constraints and silent upsert-merges that no suspect check guards (§1 FM2), Last.fm identity fragments across three identifier schemes (§1 FM3), healing is read-path-only and partially masked by a suspected stale-sync bug (§1 FM4), and nothing would tell anyone if matching silently degraded (§1 FM5). The engine being good doesn't help when four separate write paths assert identity around it.

## 1. Failure-mode taxonomy

Each mode: **evidence** (file:line, commit, live probe, or provider doc) and **handled?** status. Prod-data confirmation slots reference §8 query IDs.

### FM1 — Confidence semantics: the number stops meaning anything after persistence

The domain engine computes a calibrated probability (log-odds → sigmoid, `probabilistic.py:204-211`). Everything after that erodes it:

- **FM1a · Bump-to-100 on re-encounter** — `_bump_confidence_if_needed` (`repositories/track/connector.py:750-759`): any playlist import that re-encounters a mapped connector track raises the mapping to confidence 100 (manual overrides exempt). A 70-confidence search-fallback guess becomes a permanent 100; the stored `confidence_evidence` still records the real score, so confidence and evidence disagree in place. Re-encountering an ID confirms *the connector track exists*, not that the canonical match was right. Born 2025-08-21, predates the FS model. **Unhandled.** `[Q6 — pending prod]`
- **FM1b · Fast path discards provenance** — `get_existing_identity_mappings` (`track_identity_service_impl.py:110-140`) loads only primary-mapping ID strings, then synthesizes `MatchResult(confidence=90, method="existing_mapping", service_data={})`. Original method/confidence/evidence are never read; a review-accepted 65 and a direct 100 both re-assert as 90; any existing mapping — healthy or dead — short-circuits re-resolution forever. (Handoff tension #10: it is not that 90 is wrong; it is that *no* single number can stand in for discarded provenance.) **Unhandled.**
- **FM1c · Last.fm self-scoring** — `_create_lastfm_mapping` (`lastfm/inward_resolver.py:246-269`) evaluates the just-created skeletal track against the same artist/title strings it was created from: similarity 1.0 by construction, confidence ≈100 measures string identity with itself. **Unhandled.**
- **FM1d · Stale-MBID confidence inflation** — `LastFMProvider` labels a match `method="mbid"` whenever the response merely *contains* an MBID (`lastfm/matching_provider.py:203`), and scoring treats `method in ("isrc","mbid")` as ISRC-grade evidence, +9.2 log-odds (`algorithms.py:234`). C3's live probes show Last.fm MBIDs are frequently MusicBrainz *Track* MBIDs (not Recording MBIDs), merge-stale, and unlookupable in reverse; MetaBrainz's own guidance: *"Never trust any MBIDs from the Last.fm API"* ([LB-431](https://community.metabrainz.org/t/lb-431-last-fm-api-returns-track-mbid-instead-of-recording-mbid-for-new-scrobbles/431016)). The least trustworthy identifier in the stack gets the strongest evidence weight. **Unhandled.**
- **FM1e · Hand-tuned m/u, starved gray zone** — the m/u probabilities (`probabilistic.py:64-119`) are hand-set constants, never estimated from data, with no term-frequency adjustment (a title shared by 40 tracks scores like a unique one — the exact gap Splink's guidance names). Because comparison levels are discrete and the sigmoid squashes, reachable scores are effectively trimodal: most sums land >90 or <20; the 50–84 review zone corresponds to raw log-odds (0, 1.73], a sliver. The review queue is likely starved by construction, not by matching quality. **Unhandled.** `[Q1, Q3 — pending prod]`
- **FM1f · Confidence-constant sprawl** — EXISTING_MAPPING_CONFIDENCE=90 (`domain/matching/types.py:22`), ISRC_MATCH_CONFIDENCE=95 and LISTENBRAINZ_REUSE_CONFIDENCE=90 (`config/constants.py:237-238`), FULL_CONFIDENCE_SCORE=100 (`constants.py:65`), DIRECT_IMPORT hardcoded 100 (`spotify/inward_resolver.py:290`), ingest hardcoded 100 (`connector.py:797`). Five sources of truth, two layers, none derived from the probabilistic model — and business-rule constants in `config/` violate the repo's own domain-purity rule. **Unhandled.**
- **FM1g · Missing metadata scored as mismatch** — duration has a neutral MISSING level; title/artist don't. MusicBrainz ISRC matches are created with empty title/artist/duration (`musicbrainz/matching_provider.py:186-200`), score ≈98 (ISRC +9.2 overwhelms two false "mismatch" penalties), auto-accept — with the duration-based suspect check *structurally unreachable* because duration is None. **Unhandled.**

### FM2 — ISRC-as-identity leaks outside the engine

The engine treats ISRC as strong-but-fallible (suspect check, `isrc_validation.py`). Industry reality agrees: one ISRC ↔ many platform tracks and vice versa (B1/B2 sources; C3 live probe: one ISRC → 2 MB recordings; C6 live probe: Deezer collapses ISRC→one arbitrary representative). Four sites outside the engine treat it as truth, with **no suspect check at any of them**:

- **FM2a · `save_track` upsert-merge silently clobbers metadata** — for new tracks, `save_track` upserts keyed on `(user_id, isrc)`, then `mbid`, then `spotify_id` (`repositories/track/core.py:388-408`), *replacing* the existing row's title/artists/album/duration with incoming values. Importing a remaster that reuses an ISRC rewrites the original canonical's metadata. Note the hierarchy here (ISRC > MBID > Spotify ID) also contradicts the documented ID > MBID > ISRC hierarchy (v1.0.x.md:202). **Unhandled.**
- **FM2b · Schema-level identity assertion** — `uq_tracks_user_isrc` / `_mbid` / `_spotify_id` (`db_models.py:242-244`): one canonical per external ID per user is a hard constraint, so "same ISRC, different recording" *cannot* be represented. **Unhandled** (and load-bearing for FM2a: the upsert exists because the constraint forces merge-or-crash).
- **FM2c · Reuse-by-ISRC paths** — Spotify inward dedup (`spotify/inward_resolver.py:264-283`) and cross-discovery collision redirect (`cross_discovery.py:140-166`) both attach IDs to whichever canonical owns the ISRC, no duration/title cross-check. The cross-discovery variant is worse: it maps the Spotify ID onto the *other* canonical while the just-created skeletal Last.fm canonical keeps its own mappings — the same recording ends up **split across two canonicals** (also FM3b). **Unhandled.**
- **FM2d · `Track.has_same_identity_as`** — bare ISRC equality or any connector-ID overlap ⇒ same identity (`domain/entities/track.py:107-133`). One production caller (CLI merge gate, `track_commands.py:120`), but note FM3: an unverified Last.fm-supplied MBID lands in `connector_track_identifiers["musicbrainz"]` (`lastfm/inward_resolver.py:211`) and thereby counts as identity here and keys upserts in FM2a. **Unhandled.**

### FM3 — Identity fragmentation (one recording, many rows)

- **FM3a · Three Last.fm identifier schemes in one namespace** — (1) `mbid or url or "lastfm:{title}"` (`lastfm/conversions.py:144`, matching/enrichment path — a MusicBrainz MBID stored as a *Last.fm* identifier); (2) URL primary + lowercased `artist::title` secondary (`lastfm/inward_resolver.py:129-139`); (3) case-preserving `artist::title` (`cross_discovery.py:102` — `.strip()` without `.lower()`, unlike `make_lastfm_identifier`). The same Last.fm track can materialize as 3–4 distinct `connector_tracks` rows with mappings scattered across them. **Unhandled.** `[Q8, Q8b — pending prod]`
- **FM3b · Cross-discovery reuse leaves a dangling duplicate** — both reuse paths (ListenBrainz hit `cross_discovery.py:84-115`, ISRC collision `:140-166`) fire *after* the inward resolver has already created and mapped the skeletal canonical (`lastfm/inward_resolver.py:117-147` step order); nothing merges or removes it. Plays keyed on the identifier attach to the skeletal duplicate. **Unhandled.**
- **FM3c · Rename fragmentation is structural for Last.fm** — its URLs encode names; C3's live probe: MusicBrainz renamed "Totally Enormous Extinct Dinosaurs"→"TEED" and Last.fm now has two coexisting artist pages, one with no MBID. `artist::title` keys inherit every upstream rename. Partially mitigated (getCorrection exists, unused — §7). **Unhandled.**

### FM4 — Healing and staleness

- **FM4a · Read-path-only healing** — promotion of a fallback mapping happens only when a track is loaded with a session (`repositories/track/mapper.py:133-175`). What is never read never heals; there is no scheduled re-validation of anything. **Acknowledged in roadmap (v1.0.3) but unbuilt.**
- **FM4b · Healing masked by denormalized columns** — the mapper pre-populates `connector_track_identifiers["spotify"|"musicbrainz"]` from `DBTrack.spotify_id/mbid` *before* walking mappings (`mapper.py:111-114`), so the no-primary fallback/promotion pass never fires for those connectors while the column is non-empty. **Unhandled, undocumented interaction.**
- **FM4c · Two promotion policies** — the mapper promotes the *first* fallback in iteration order (docstring says "best", `mapper.py:134`); `ensure_primary_for_connector` promotes highest-confidence (`connector.py:953-980`). Same repair, two different outcomes depending on trigger. **Unhandled.**
- **FM4d · Stale denormalized-ID sync (suspected live bug)** — `map_track_to_connector` unconditionally syncs `DBTrack.spotify_id` to whatever mapping it just wrote (`connector.py:603-605`); the redirect/fallback flow writes primary (current ID) *then* secondary (dead ID) (`spotify/inward_resolver.py:176-196`), leaving **the canonical's `spotify_id` column holding the dead ID** after every REDIRECT/SEARCH_FALLBACK resolution. Downstream: `save_track` dedups against the stale ID (FM2a key), and FM4b then masks healing behind it. **Unhandled — highest-priority verification target.** `[Q7 — pending prod]`
- **FM4e · No supersession, no history** — mapping upsert is keyed `(user_id, connector_track_id, connector_name)` and overwrites method/confidence/evidence in place (`connector.py:432-437`); flip-flops are invisible; "we used to believe X" is unrepresentable. 2026 practice (Wikidata deprecated-rank, MusicBrainz merge-redirect, OpenSanctions statements — B1/B3) treats supersession as the core fault-tolerance primitive. **Unhandled.**
- **FM4f · No dead-ID debouncing** — one failed lookup routes an ID to fallback search (`spotify/inward_resolver.py:247-252`); Spotify communities document intermittent spurious 404s on valid IDs (C2). Contrast InternetArchiveBot: three consecutive failures over ≥9 days before declaring death (B3). **Unhandled.**

### FM5 — Observability: silent degradation is silent

- **FM5a · No resolution-quality metrics** — no counters/trends for fallback-resolution rate, read-path promotions, dead-ID rate on touch, confidence-distribution shift, review inflow. The five-signal minimal set from production ER systems (MoJ/Splink run deltas + distribution watch; Amperity benchmarks — B3) has no mixd equivalent. `get_match_method_stats` (`connector.py:1340`) and the v0.4.9 integrity checks are the only footholds. **Unhandled.**
- **FM5b · Failures degrade to empty values at debug level** — enrichment failure → skeletal track persists silently (`lastfm/inward_resolver.py:182-186`); whole cross-discovery pipeline → `except Exception → logger.debug → False` (`cross_discovery.py:62-72`); getInfo errors → `.empty()` (DUP-02, `lastfm/operations.py:77-113`). **Unhandled.**
- **FM5c · Auto-rejects leave no trace** — below-50 matches are logged and discarded (`evaluation_service.py:190-199`); every run re-fetches and re-rejects the same candidates; no negative-evidence memory (review-zone rejects *do* persist — `match_reviews` status + unique constraint). Wikidata's deprecation rationale — record the rejection so nothing re-adds it — applies directly (B1/B3). **Unhandled below the review threshold.**
- **FM5d · Review-queue rot risk** — roadmap-acknowledged UI gap; combined with FM1e starvation, the queue is both under-fed and under-drained; no aging/expiry policy exists (`match_reviews.status` has no TTL semantics). MusicBrainz's 7-day default-outcome expiry is the reference pattern with known trade-offs (B3). **Acknowledged, unbuilt.** `[Q3 — pending prod]`

### FM6 — Architecture drift around the engine

- **FM6a · Four identity-write entry points, not one** — the "SOLE ORCHESTRATOR" (`match_and_identify_tracks.py:3`) is one of four: (1) that pipeline; (2) inward resolvers during play import; (3) cross-discovery; (4) `ingest_external_tracks_bulk` during playlist import (`connector.py:610`). Each has its own confidence conventions (FM1f) and its own ISRC handling (FM2). **Design-intent drift.**
- **FM6b · `match_method` vocabulary fragmentation** — domain validator allows 4 values (`track.py:188-195`, excludes even `existing_mapping`); `config.constants.MatchMethod` defines 11 (+ a `mbid_match` category entry nothing writes); the DB column is a free string. **Unhandled.**
- **FM6c · Manual-override filter not user-scoped** — `_filter_manual_overrides` queries by `connector_track_id + origin` only (`connector.py:538-543`); unless RLS is active on every session that reaches it, user A's manual pin suppresses user B's automatic mapping. **Verify RLS coverage before multi-user.**
- **FM6d · Warm leads confirmed** — DUP-01 error-envelope ×3 verbatim (`lastfm/client.py:222,301,367`); DUP-02 as FM5b; DUP-04 partially resolved already (per-track scaffold now centralized in `_match_each`, `matching_provider.py:205` — the remaining altitude question is the ISRC/artist-title partition assumption itself, §6); DUP-05 two paging envelopes (`spotify/models.py:111-123,148-162`).

### FM7 — Provider-truth drift

- **FM7a · Spotify Feb/Mar 2026 whiplash** — Feb 2026 removed `external_ids`, `linked_from`, `popularity`, batch GETs *for dev-mode apps*; **March 2026 changelog officially reverted `external_ids`** (track+album) — ISRC remains available to all tiers (verified in main thread against the changelog). `linked_from`/`popularity` stay removed for dev mode; redirect detection via ID comparison survives by design (commit `8697b535`). `spotify/models.py:10-13` comment is stale ("still returned empirically" → officially retained). Dev-mode program limits (5 users/app, 1 client ID/developer, owner Premium) make **BYO-credentials-per-user the only viable self-hosted multi-user model**. *Handled in code; stale in comments; program limits unhandled in product design.*
- **FM7b · Spotify ISRC search is market-sensitive** — `search?q=isrc:` can return 0 for tracks that exist without a `market=` param, and 2026 semantics define market-less content as unavailable (C2, practitioner-corroborated). `search_by_isrc` (`spotify/client.py:184`) passes no market. **Unhandled.**
- **FM7c · Last.fm MBIDs unreliable in kind** — Track-vs-Recording type confusion, merge-stale, reverse-lookup broken (C3 live probes). Feeds FM1d and FM2d. **Unhandled.**
- **FM7d · MusicBrainz merges return 301 + new MBID** — old MBIDs redirect permanently (verified doc + C3 live probe); a client should detect `returned_id != requested_id` and rewrite. Whether mixd's MB client does this is untested; the mbid stored on tracks is never re-validated (FM4a). **Likely unhandled — verify.**

## 2. Design space

Three coherent directions, deliberately at different altitudes. They are **sequenceable, not exclusive** — D2 assumes D1's fixes; D3 assumes both. Per direction: what it fixes (taxonomy IDs), costs, risks, migration shape. Rule-of-three / AHA markers are explicit.

### D1 — Harden in place (no schema change)

Fix the semantics of what already exists; touch no tables.

**Components** (each independently shippable):
1. Kill or scope FM1a: stop bumping confidence on re-encounter, or record it as a separate signal (`last_seen_at` bump) instead of overwriting the score.
2. Fast-path provenance (FM1b): load the real mapping row (confidence/method already in `track_mappings`) instead of synthesizing 90 — one extra column set in an existing query, not a redesign.
3. Demote Last.fm "mbid" method (FM1d): score Last.fm matches as `artist_title` unless the MBID is *verified against MusicBrainz* (which C3 shows must handle Track-vs-Recording and 301s anyway).
4. Add MISSING comparison levels for title/artist (FM1g) so empty provider metadata is neutral, not mismatch — and require the MB provider to fetch minimal recording metadata before an ISRC match can auto-accept (closes the unreachable-suspect-check hole).
5. Guard the ISRC reuse sites (FM2a/2c): a duration/title sanity check (the existing `assess_isrc_match_reliability`) before `save_track` upsert-merge and before both cross-discovery redirects; on suspect, create a *new* canonical (see D2's schema note for the constraint conflict) or route to review.
6. Unify Last.fm identifiers (FM3a/3c): one scheme (`make_lastfm_identifier` after `getCorrection` normalization — C3), applied at all three sites; a one-shot data migration folds case-variants.
7. Fix FM4d (stale denorm sync — write primary last or gate `_sync_denormalized_id` on `auto_set_primary`) and FM4c (one promotion policy: highest-confidence).
8. Minimal drift metrics (FM5a): counters for fallback-resolutions, read-path promotions, dead-ID hits, review inflow; a `mixd stats --matching` view. The v0.4.9 integrity-check scaffold is the natural host.

**Fixes**: FM1a-d, FM1g, FM2a/2c (guarded), FM3a-c, FM4b-d, FM5a (minimal), FM7b (add market param). **Doesn't fix**: FM2b (schema), FM4e/4f (needs schema), FM5c, FM6, FM1e (needs data), anything cross-user.
**Cost**: S–M per component; no migrations except the FM3a fold. **Risk**: behavior changes in matching paths that the regression net only partially covers — the §4 characterization tests are prerequisites, not accompaniments. **Migration**: none (plus one data-fold).

### D2 — Boundary validation + supersession ledger (targeted schema change)

Adopt the two primitives 2026 fault-tolerant identity systems converge on (B1/B3: Wikidata ranks + reasons, MusicBrainz merge-redirects, OpenSanctions statements, IABot debouncing) — without building a global layer.

**Components**:
1. **Supersession columns** on `track_mappings`: `superseded_by_id` (self-FK), `superseded_at`, `supersede_reason` (typed: `id_dead`, `id_redirected`, `rematch_better`, `manual`, `mismatch` — the Wikidata "withdrawn vs deprecated" distinction). Upserts become insert-new + supersede-old for *changed* assertions; unchanged re-asserts just touch `last_verified_at`. Fixes FM4e; gives FM5 metrics real substrate (flip-flop count per track = supersession chain length).
2. **Resolution event log** (append-only, JSONB payload): every accept/review/reject/heal decision with matcher version + evidence. Fixes FM5c (rejects become memory: don't re-propose a superseded-as-`mismatch` candidate), enables FM1e calibration later (labeled outcomes to estimate m/u from — the Splink-style path B1 describes). ListenBrainz's production mapper is the reference for the *no-match* half: it persists `no_match` rows with an exponential-backoff `check_again` (1 day, doubling, 32-day cap) so absent matches retry without hammering providers — a shipped design, not a proposal.
3. **Dead-ID debouncing + hysteresis** (FM4f): a mapping is `suspect` after failure 1, dead after N consecutive failures over ≥T days (IABot: 3 over 9); a challenger must beat the incumbent by a margin to displace it (RFC 2439 pattern — mechanism durable, parameters tunable).
4. **`Isrc` value object at the Pydantic boundary** (the floated-and-deferred idea, handoff §2.4): assessment — *it moves normalization, it does not remove failure modes by itself*. All 11 `normalize_isrc` call sites are already in infrastructure conversions; the real win is pairing boundary validation with FM2 guards (component D1.5) so "validated ISRC" ≠ "identity-bearing ISRC". Worth doing as part of this direction, not standalone.
5. **Schema decision on FM2b** (the hard one): either (a) keep `uq_tracks_user_isrc` and accept merge-on-collision as policy *with* the D1.5 guard + supersession trail, or (b) drop the unique constraints and let identity live only in mappings (denormalized columns become hints). (b) is honest but touches `save_track` dedup, `_TRACK_ID_TYPES` lookups, and the fast path — a genuine migration project. This memo does not pick; §8 data (Q5/Q12) should — if real ISRC collisions are rare, (a)+guard is proportionate.

**Fixes**: FM4e/4f, FM5c, FM2b (decided either way), plus the D1 set it builds on. **Doesn't fix**: cross-user trust, scheduled re-validation at scale (though the event log is its prerequisite), FM6a.
**Cost**: M–L; 2 migrations (columns + event table); every mapping-write site touched (the four FM6a entry points — which is also the moment to consolidate them behind one application-service seam, addressing FM6a *by consequence* rather than by grand redesign).
**Risk**: supersession-aware reads — every query that assumes "one live mapping" must filter `superseded_by_id IS NULL` (Wikidata's constraint-tooling lesson: half-aware tooling generates false alarms forever). Mitigation: a partial index + a repository-level default filter.

### D3 — Global identity layer (the v1.0.3 `ConnectorIdentity`/`ObservationLedger` sketch, evaluated)

> **2026-07-02 follow-up**: the governance dimension of D3 — *who arbitrates shared identity, internal-vs-MusicBrainz, multi-user gameability* — is mapped in depth in the companion memo [identity-governance-design-space.md](identity-governance-design-space.md). Its headline findings: no shipped product uses user-curated shared music identity (the Songlink/Letterboxd architectures run on external anchors with zero curation staff); community voting/trust-weighting demonstrably fails below ~100 active users (v1.0.3's `strong` tier is a 3-sock-account quorum under open registration); the evidence-backed composition is resolve-at-view for public surfaces + external-anchor-only cross-user sharing, with TrustScore deferred. The evaluation below stands for the mechanical half of the sketch.

The existing sketch (v1.0.x.md:198-315) proposes: global no-RLS `connector_identities` with confidence tiers, append-only `observations` with trust weights, computed `TrustScore`, MusicBrainz re-validation on a schedule, and a backfill from `track_mappings`.

**Where the sketch is validated by this research**:
- Observation-ledger identity is exactly the shipped 2026 pattern (OpenSanctions statements, Senzing observations — B1), not speculative architecture.
- "External APIs are arbiter, users are observers" matches the survivorship consensus (source-trust rankings above crowd counts — B1 MDM sources).
- Append-only + `superseded_by_id` + retraction mirrors Wikidata/MusicBrainz practice (B3).
- Scheduled MB re-validation is **arithmetically feasible at mixd's scale** — C3's numbers: 10k mappings ≈ 2.8h at 1 req/s (trivial as a monthly trickle); 100k ≈ 27.8h (must be a continuous trickle, and it shares the per-IP budget with interactive enrichment — critical for hosted multi-tenant); beyond that, the CC0 database dumps are the intended path, and MusicBrainz explicitly asks not to poll (verified in main thread). The sketch's "batch ISRC lookups" language should be corrected: **no batch endpoint exists**; batch = sequential trickle.

**Where the sketch needs revision before build**:
- **Garbage-in tier seeding**: the backfill derives initial tiers from `match_method`/confidence — but FM1a/1b have corrupted stored confidences (bumped 100s, synthetic 90s). D1 fixes plus a re-derivation from `confidence_evidence.final_score` must precede any backfill, or the ledger launders today's inflation into "verified/strong" tiers.
- **Last.fm MBIDs must not produce `verified`**: the sketch's "MBID → verified (after MusicBrainz cross-check)" is safe *only* because of the parenthetical — C3 shows the cross-check must resolve Track-vs-Recording MBIDs and follow 301 merges, or it will verify wrong identities.
- **TrustScore machinery is premature**: multi-user trust weighting defends against adversarial *sharing* surfaces (v1.1/v1.2). Until those ship, it is anticipated duplication of effort — AHA discipline says build ConnectorIdentity + Observation first, add trust when a second real consumer (public sharing) exists.
- **Per-user vs global tension (handoff tension #1)** is real but narrower than framed: `connector_tracks` are already global; what's per-user is the *assertion*. The ledger's contribution is separating "what the world observes" (global) from "what this user has adopted" (per-user mappings stay, per the sketch's own dual-write plan). That separation is the actual fix for the drift scenario; the gaming scenario only matters when sharing ships. ListenBrainz's MSID/MBID split is the production precedent for exactly this separation — raw submitted identity ("messy") never mutates, the mapping row is what gets recomputed — which is why its re-mapping is non-destructive and reversible.
- **Re-validation substrate at scale**: MusicBrainz's *Canonical MusicBrainz Data* (CC0, published 2×/month, with a `combined_lookup` fuzzy key) plus full CC0 dumps are the intended high-volume path — the sketch's scheduled API polling should be the small-scale mode only (MusicBrainz explicitly asks not to poll; verified in main session).

**Fixes**: FM4a (scheduled re-validation), the cross-user class, audit/self-heal at the layer D2 only enables. **Cost**: L–XL (three tables, workers, backfill, dual-write transition). **Risk**: building trust machinery for users who don't exist yet; the dual-write transition period has two sources of truth. **Migration**: the sketch's own batched backfill plan is sound *after* the confidence-repair prerequisite.

### What this memo deliberately does not propose

- **No new matching algorithm** — FS is the right frame (B1: MoJ assessment, Splink standing); the fix is calibration inputs (D2's event log) not model replacement. LLM-based matching: only defensible niche is unseen-entity robustness; cost/drift profile wrong for a self-hosted app (B1, flagged HYPE for this context).
- **No connector-abstraction extraction yet** — Apple Music is the third real implementor; §6 names the forcing points and says wait for contact (AHA).
- **No embedding/vector blocking** — at mixd's scale, key-based candidate generation is orders of magnitude below where blocking gets hard (B1: Splink guidance, OpenSanctions precedent).

## 3. Apple Music readiness assessment

Sources: Apple Music API documentation fetched 2026-07-02; the two most load-bearing claims (`filter[isrc]` semantics, storefront/equivalents model) re-verified verbatim in the main session against Apple's doc backend. Today's connector is a UI stub only: `apple_music/connector.py` returns `auth_method="coming_soon"` with a bare-object factory; `error_classifier.py` is a skeleton.

### 3.1 What the current pattern absorbs as-is

- **`BaseMatchingProvider` fits Apple better than it fits Last.fm.** Apple has a true batch ISRC lookup — `GET /v1/catalog/{sf}/songs?filter[isrc]=…`, max 25 per request, documented "one ISRC value may return more than one song" — and artist/title search as fallback. The ISRC/artist-title partition assumption holds; `_match_by_isrc` gets a genuine batch implementation (MusicBrainz-style), `_match_by_artist_title` uses `_match_each`. The documented multi-song-per-ISRC behavior means candidate disambiguation (album/duration/`playParams` presence) is required — which the FM1g fix (fetch real metadata before accepting an ISRC match) already demands for MusicBrainz.
- **`InwardTrackResolver` fits.** Catalog song IDs are ID-shaped; batch hydration via `ids` (≤300/request); crucially, Apple's batch semantics — *missing IDs are silently omitted from a 200 response, no tombstones, no redirects* — is exactly the "requested ID absent from result ⇒ dead" contract the Spotify resolver already implements. The dual-mapping strategy (current primary + stale secondary) transfers unchanged, and is independently validated as industry practice (Lidarr keeps `OldRecordingIds`/`OldForeignReleaseIds` for the same purpose — B2).
- **Play-import pattern fits, but through the weak path** — see 3.3.

### 3.2 What Apple's identity model strains

1. **Storefront-scoped resolvability vs one `connector_track_identifier`.** Catalog IDs live in one global keyspace, but every request is storefront-scoped and Apple documents that "an album could have different identifiers between two separate storefronts… different versions of the same album" (WWDC21 shows the concrete 404-in-the-other-storefront failure). A bare stored ID is only guaranteed meaningful in the storefront that resolved it. Options: encode `{storefront}:{catalogId}` in the identifier, or store the resolution storefront in `connector_tracks.raw_metadata` and re-project per user via `filter[equivalents]` (≤300 IDs/request; response `meta` maps requested→equivalent ID). **This is the first connector that bends the implicit "globally shared connector_tracks cache" assumption** — an Apple connector track is storefront-relative the moment two users sit in different storefronts (handoff tension #1 becomes concrete here, not just at the sharing layer).
2. **Equivalents are a new, platform-asserted mapping species.** Storefront equivalents and clean/explicit substitution (`restrict=explicit`: "replaces content with an explicit rating with the best available clean equivalent version") are Apple-asserted identity links — the same trust class as Spotify relinking. Per the B3 policy synthesis (auto-heal is safe when the *source* asserts the successor), these deserve REDIRECT-style dual mappings with their own `match_method` values, not fuzzy re-matching.
3. **Auth is a materially higher barrier than any existing connector.** Each self-hosted instance owner needs a paid Apple Developer Program membership, a media identifier + ES256 `.p8` key (ES256-only; developer-token `exp` ≤ 6 months), and the Music User Token can only be minted by MusicKit in a client — for mixd that means **MusicKit JS in the web UI** (server-side use of the resulting token is documented and sanctioned; CLI-only onboarding is impossible — a first). MUT lifetime is undocumented; practitioner reports range from days to ~6 months; there is no refresh token — the connector must treat 403 as "needs re-auth" and surface it. The v0.6.5 shared-OAuth-utilities decision needs an extension for a non-OAuth token pair (developer JWT + MUT) — the `auth_method="coming_soon"` gate is exactly where that lands.
4. **Library IDs must never become identifiers.** `i.`-prefixed library IDs are per-user, regenerate on remove/re-add, and library songs carry **no ISRC**. Library import must resolve to catalog IDs at import time via `include=catalog` — documented as available "when known"; uploads/unavailable purchases have none and drop to a text-matching quarantine (size unknown until real libraries are imported).
5. **Play history enters through the weakest identity path.** `GET /v1/me/recent/played/tracks` returns max 30 items with **no timestamps and no counts** — scrobble-grade live polling is impossible. Full history exists only in the privacy.apple.com export, whose `Apple Music Play Activity.csv` has timestamps but **no song IDs and no ISRCs** (multiply corroborated by practitioner parsers) — so Apple history import is architecturally the *Last.fm shape* (title/artist/timestamp + text matching), not the Spotify shape (ID-native), despite Apple being an ID-native catalog. Rate limits are officially unpublished (429 per developer token; numbers frequently cited online belong to the App Store Connect API — a documented conflation trap).

### 3.3 Genuinely unknown until implementation

- Actual rate-limit numbers/windows per developer token.
- Real-world MUT lifetime distribution (docs silent; reports conflict).
- Whether the export's "Apple Music Track Identifier" (in `Apple Music Library Tracks.json`) resolves via the catalog API — would upgrade export import from pure text matching to partial ID matching.
- Fraction of a real library lacking the `catalog` relationship (sizes the quarantine).
- Retention depth/dedup behavior of recently-played for best-effort polling.
- Catalog ID churn rate — no changelog or tombstone API exists; the connector should ship with longitudinal 404 auditing from day one (the `diagnose_stale_spotify_ids.py` pattern, generalized).

## 5. Connector-contract scalability (omni-integration assessment)

Judged against the full §6 matrix: what would 8–10 services demand of the three abstractions — and where does DUP-04's template-altitude question resolve? AHA discipline throughout: forcing points are named; nothing is pre-extracted.

**5.1 `BaseMatchingProvider` (ISRC-partition template).** The template's fixed two-hook shape (`_match_by_isrc` → `_match_by_artist_title`) fits every ISRC-capable catalog: Spotify, MusicBrainz today; Apple (`filter[isrc]`, 25/batch), Tidal (`filter[isrc]`, first-class), Deezer (`/track/isrc:` — live-probed working) tomorrow. It structurally misfits three: Last.fm (already opted out with the documented LSP argument — batch API, no ISRC), YouTube Music (no ISRC anywhere public; search + type-filtering is the only path), SoundCloud (ISRC readable but *never queryable*). So at 8–10 services the split is ~6 template / ~3 protocol-only — and the system already handles that gracefully because **the real contract is the `MatchProvider` protocol, not the base class**. **DUP-04 verdict**: the per-track scaffold half was already extracted (`_match_each`, `matching_provider.py:205`); the remaining altitude question — should the fixed partition become an ordered strategy list? — **stays deferred**. The forcing point is the Apple implementation: if storefront/equivalents handling fits inside `_match_by_isrc` cleanly, the template survives; if Apple needs a third phase (equivalents projection before ISRC lookup), that is the rule-of-three moment (MusicBrainz batch-ISRC and Apple would then both deviate) to extract a strategy pipeline. Decide *at contact*, not before.

**5.2 `InwardTrackResolver`.** The 3-step contract (mapping lookup → canonical reuse → create) holds for every ID-shaped service (Spotify; Apple catalog IDs with a storefront encoding decision, §3.2.1; Tidal/Deezer numeric; YTM videoIds) and for string-keyed Last.fm. What 8–10 services actually demand is not a new abstraction but **an invariant the current one never states: one identifier scheme per connector, minted in exactly one place**. FM3a is the proof of what happens without it. `_normalize_id` already exists as the hook; the discipline (all identifier construction routes through the resolver's mint function) is a convention to enforce at review time — no code to build ahead of need.

**5.3 Platform-asserted successor resolution — the one concept where rule-of-three is already met in the wild.** Four services ship an official "this ID has been replaced" affordance: Spotify relinking (returned-ID ≠ requested-ID), **Tidal's `replacement` relationship** (explicit endpoint; the 2024 MQA purge proves mass ID substitution happens), **Deezer's `readable=false` + `alternative` pointer**, **Apple's `filter[equivalents]`**. mixd implements exactly one (Spotify REDIRECT dual-mapping). B3's policy synthesis says these are precisely the safe-to-auto-heal cases (source asserts the successor). This is a *named design seam* — a per-connector "resolve successor" capability the inward resolver consults for dead IDs — but with only one in-code instance today, extraction still waits for the second implementor (Apple or Tidal). What should NOT wait: the supersession schema (D2) must be able to *represent* platform-asserted succession distinctly from inferred re-matching, or the second implementor will force a migration.

**5.4 Registries over hardcoded dicts.** `TrackIdentityServiceImpl._provider_factories` is a 3-entry literal dict (`track_identity_service_impl.py:61-67`); connector configs are similar. At 4 connectors this is fine; at 8–10 it's N scattered registration points per connector. The forcing point is simply the Apple PR — count the files it must touch to register itself; if >3, consolidate registration then.

**5.5 Play-import is not a universal connector feature — capability data should drive sequencing.** Only three services offer scrobble-grade history (Spotify via export, Last.fm natively, YTM via Takeout — all import-file or unofficial paths); Apple's export is text-shaped; **Tidal has no history surface at all** (durable — the endpoint doesn't exist server-side, even for the unofficial libraries); Deezer's history endpoint is locked behind closed registration; SoundCloud retains a deduplicated last-25. The universal surface is **likes + playlists** (7 of 9). Consequence: the play-importer pattern stays a 2–3-connector pattern (no extraction pressure), and connector road-mapping should stop assuming history parity — the §6 matrix is the sequencing evidence.

**5.6 Program-risk classes belong in connector metadata.** The matrix shows three distinct risk classes: official-and-stable (MusicBrainz, Last.fm-by-inertia), official-but-perishable (Tidal program health, Deezer registration, SoundCloud paid gate, Spotify dev-mode caps, Apple paid membership), and unofficial (ytmusicapi). A future connector UI that badges these ("experimental", "BYO credentials", "enrichment-only") needs a place to declare them — the existing `get_connector_config` capability surface is the natural home when the need becomes concrete.

## 7. Artist identity (scope extension 1)

**The problem** (v0.10.x.md:58-71): canonical artist identity across Spotify artist IDs ↔ Last.fm artist names ↔ MusicBrainz artist MBIDs, including the alias/abbreviation gap ("TEED" vs "Totally Enormous Extinct Dinosaurs") that text normalization cannot bridge.

**What the research established, per service** (C2/C3 primary docs + live probes):
- **MusicBrainz is the only service with real artist identity machinery**: permanent MBIDs with merge-redirects; typed aliases (Artist name / Legal name / Search hint, with locales) fetched via `inc=aliases`; search matches aliases (live probe: querying the old full name returns TEED at score 100). Same-name artists are separate entities with disambiguation comments — the exact thing no other service offers.
- **Live-probe caveat that reshapes the alias-table sketch**: MusicBrainz has since made "TEED" the *primary* name with the full name as an alias — primary/alias direction inverts over time, so the lookup table must anchor on **artist MBID**, store all names/aliases as equal-rank keys, and refresh (aliases are editor-maintained, not frozen).
- **Spotify**: artist IDs treated as permanent in industry practice (no documented guarantee); renames happen *in place* via distributor request — same ID, new display name, old name unexposed; **no alias surface in the API**; API search is literal (documented divergence from consumer-app ML search) — do not assume `search?q=artist:TEED` bridges abbreviations (C2, flagged for an empirical probe at implementation).
- **Last.fm**: one page per name *string*; two artists sharing a name share scrobbles (support threads, 2024–2026); renames create parallel pages (TEED probe: two pages, one MBID-less). `artist.getCorrection` is the only normalization primitive (misspellings/variants → canonical Last.fm name) — currently unused by mixd.
- Breadth-ring artist identity is uniformly weaker: YTM artists have *multiple* channel IDs each (topic + official + Vevo; ytmusicapi warns returned channelId ≠ requested); Deezer has no aliases/disambiguation; SoundCloud's artist *is* a user account (n accounts per human, credit strings freeform).

**Design space for the v0.10.2 epic** (options, not a pick):
- **A. MB-alias lookup table (the current sketch), revised**: import aliases keyed by artist MBID (not by name), all alias types, with periodic refresh and 301-merge handling. Validated as the only mechanism that actually bridges abbreviations. Prerequisite: tracks must resolve to an MB artist MBID at all — which today rides on FM7c-tainted Last.fm MBIDs or MB search; the epic's "MusicBrainz artist MBID lookup via existing client" line is doing more work than it looks.
- **B. Canonical-artist entity with per-connector artist mappings** (mirrors the track pattern; the epic's data-model direction): sound, but inherits every FM1 lesson — the sketch's "direct ID match (100), MBID match (95), name match (80)" hardcodes a new copy of the scattered-constants problem (FM1f) before the track-side one is fixed. If built, artist confidence should come from the same evaluation service pattern, not fresh literals.
- **C. Normalization-first minimalism**: run `artist.getCorrection` + MB alias expansion at *comparison* time (matching-layer equivalence, no new tables). Cheapest; doesn't give canonical artist pages (which v0.10.2's UI epics need), so likely insufficient alone — but it is the right *matching* substrate under either A or B.
- **Sequencing note**: artist identity errors compound track-identity errors (artist string feeds `calculate_confidence` at `algorithms.py:203-215` — first-artist-only, FM1 §B.8). Alias-aware artist comparison (C) would measurably improve *track* matching before any artist entity ships.

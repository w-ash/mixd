# v0.10.x: Artists, Albums, Physical Media & Play History

For strategic overview, see the [planning overview](README.md).

## User Workflow Context

The user's music identity is bigger than their track list, but mixd models only tracks today. Artists are `{"names": ["..."]}` JSON strings — "Radiohead" on Spotify and "Radiohead" on Last.fm are unrelated text, so there is no artist page, no "all tracks by X", no favorites. Albums are a bare `VARCHAR(255)` on tracks. And the physical side of a collector's identity — the vinyl on the shelf, the CDs in the car — is invisible: owning a record and listening to it leave no trace in the canonical history. This cycle opens on the play history (v0.10.0–v0.10.1), then promotes artists (v0.10.2) and albums (v0.10.3) to first-class entities on the same three-layer identity pattern as tracks, then builds physical ownership (v0.10.4, Discogs) and physical listening (v0.10.5, manual listens) on top.

Play history goes first because the window is open: production holds **zero** play rows today, so the source-merge defects — the merge depends on arrival order, and one import order double-counts 58% of overlapping plays that the opposite order merges correctly ([findings](play-import-convergence-findings.md)) — can be fixed *before* the first at-scale import instead of repaired after it, with no collapse scripts or rebuilds ever touching prod data. v0.10.0 makes canonical plays a deterministic projection of an immutable observation ledger (order-free, re-import-safe, richest-field-wins); v0.10.1 adds continuous polling of Spotify's recently-played API — live play counts for workflows within the hour, with GDPR uploads later deepening the same plays instead of duplicating them. The first at-scale import (full Last.fm history + a fresh GDPR export) is those two milestones' joint acceptance, and every later milestone ships into an already-honest history.

Persona view (from the [alignment table](README.md#persona-alignment)): the **Weekly Curator** gets deeper library modeling, favorites, and the Discogs/vinyl story; the **Tinkerer** gets a rich relational data model to explore and new workflow nodes; the **Casual** listener gets the intuitive browse-by-artist/album navigation every music app is expected to have.

Cross-cutting contract (stated once — per-epic Specs reference it): every new use case this cycle must be classified in the chat-tool parity registry (`src/application/tools/registry.py`) — CI (`tests/unit/application/tools/test_registry_parity.py`) fails on any unclassified use case. Reads get a `ToolSpec` or an explicit classification; mutations additionally get two-phase confirmation wiring like the existing `*_write` dispatchers.

### v0.10.0: Convergent Play History

**Goal**: Make canonical play history a deterministic projection of imported source data — re-importing changes nothing, import order changes nothing, and each play carries the richest field values any source has offered.

**Context**: The sources report the same listening event differently — Spotify's export stamps the *end* of a play with `ms_played` and behavioral detail; Last.fm stamps the *start* with second precision and name-guessed track identity; manual listens are authored. Cross-source dedup (v0.4.10) merges them, but the outcome depends on arrival order: measured against real overlap data, Spotify-first merged 519 pairs correctly while Last.fm-first double-counted 267 (58% of that window's Last.fm plays) — same data, opposite orders, opposite results. All evidence, tolerances, and per-field trust rates are quantified in the [convergence findings](play-import-convergence-findings.md); this section's decisions cite it by section.

**What this unlocks**: trustworthy play counts and history-derived features (metrics, "loved but forgotten", the v0.10.5 physical-listen story) that stop drifting as sources are added or re-imported; a rebuild command that re-derives history from source records at any time; the substrate any future source (v0.10.1 polling, Apple Music, ListenBrainz) joins with zero new dedup code. Critically, per Findings §1 **no environment holds play data at scale yet** — the first big import is the real migration, which is why this milestone opens the cycle.

**Persona**: Curator-primary — their history *is* the product, and "own my listening data" only means something if the data is honest. Tinkerer secondary (rebuild/inspect commands, ledger to explore).

**Key Design Decisions**:
- **Canonical plays are a projection of the `connector_plays` observation ledger, not mutated-in-place records** — every source record is an immutable, bitemporal observation (event time + ingestion time); canonical state is a pure function of the observation *set*. This makes the merge associative, commutative, and idempotent by construction (a join-semilattice), which is what guarantees re-imports, arrival order, and batch boundaries can't change the result. In-place merging cannot be retrofitted to order-independence: re-merging on a richer late arrival needs the original per-source values, which only the ledger retains.
- **Grouping keys on observation channels — `(service, import_source)` — not services** — `spotify/spotify_export` and `spotify/spotify_api` are two observers of the same listen (v0.10.1), exactly like `lastfm` and `mixd` are; one grouping rule covers cross-service dedup, export↔API dedup, and the scrobble round-trip, so new channels need no new code.
- **Per-field survivorship, channel priority `spotify_export > spotify_api > mixd > lastfm`** — richest data wins per *attribute*, not per record (MDM golden-record practice): `ms_played` from export, timestamps from the highest-precision member, `loved` from Last.fm. `mixd` outranks `lastfm` so a round-tripped scrobble enriches the manual listen it echoes rather than displacing it.
- **`played_at` is stored as normalized start time** — Findings §3 validates the model (end − `ms_played` aligns sources to ±5s for 80% of true pairs; unnormalized, 92% would miss); converging at write kills the per-winner timestamp drift.
- **Tolerances: 30s tight, 180s fallback, greedy nearest one-to-one** — Findings §3/§4: 30s catches 82.7% of true pairs, the 30–180s band is real (pause-skewed, negative-heavy), ambiguity within windows is rare — nearest-first one-to-one assignment keeps skip-restarts distinct without probabilistic machinery.
- **Divergence bridging is exact-normalized only; the cure is upstream** — Findings §5b: all observed divergence (191 pairs) is casing/punctuation ("striptease" vs "Striptease") from Last.fm-created canonicals; an exact `normalize_for_comparison` bridge catches it at the play layer, and the casing fix epic removes the source. Fellegi–Sunter escalation explicitly not needed (§8.3).
- **Same-channel collapse for identical `(ts, identifier)` with differing `ms_played`** — Findings §7: 266 such pairs exist inside the GDPR export itself and would defeat the ON CONFLICT key; they are one observation (keep max `ms_played`).
- **Harden constraints before the first at-scale import** — Findings §5c: the NULL-`ms_played` hole in both dedup constraints has simply never been triggered; with no precious data anywhere (§1), fixing schema now is free.

#### Foundation Epics

- [ ] **Ledger Idempotency Hardening**

    **Story**: The user re-runs an import — or uploads the same GDPR file twice — and nothing changes. Today a `mode=full` Last.fm re-import would duplicate every scrobble: Last.fm rows always have `ms_played = NULL`, both dedup constraints include `ms_played`, and NULL ≠ NULL means ON CONFLICT never fires ([Findings §5c](play-import-convergence-findings.md)).

    **Decisions**:
    - **`NULLS NOT DISTINCT` over a sentinel value** (e.g. `ms_played = -1`) — a sentinel leaks into every consumer that averages or sums listening time; the constraint option confines the fix to the schema.
    - **Defensive duplicate-collapse pre-pass ships in the migration even though all environments are currently clean** (Findings §1/§5c) — migrations must be correct wherever they run, including a database that re-imported before upgrading.

    **Spec**:
    - Migration: recreate `uq_track_plays_deduplication` and `uq_connector_plays_deduplication` with `NULLS NOT DISTINCT` (PG15+; prod is PG17)
    - Pre-pass (batched, outside the DDL transaction): collapse exact-duplicate groups keeping lowest `id`
    - `db_models.py`: constraint args updated to match

    **Tests**:
    - (integration) NULL-`ms_played` exact re-insert now conflicts on both tables; collapse pre-pass idempotent
    - (manual) `mixd` Last.fm full import twice → identical row counts

    Effort: M | Dependencies: None | Status: Not Started

- [ ] **Resolution Write-Back**

    **Story**: The ledger can only be projected if each observation knows which canonical track it resolved to — `connector_plays.resolved_track_id` exists, is indexed, and is NULL on 100% of rows because no write-back was ever implemented (Findings §2).

    **Spec**:
    - `ConnectorTrackPlayRepository.bulk_update_resolution(updates: Sequence[tuple[UUID, UUID, datetime]])` — sets `resolved_track_id` + `resolved_at`
    - Resolvers return connector-play↔track pairs (they already hold both sides in memory); orchestrator persists after Phase 2
    - Unresolvable rows keep NULL and stay outside the projection (surfaced in import stats, as today)
    - `ix_connector_plays_unresolved` becomes a partial index (`WHERE resolved_track_id IS NULL`) in the same change — today it's a plain composite on `(connector_name, resolved_track_id)`, and the unresolved query is its only purpose
    - No historical-backfill problem exists: no environment has ledger rows at scale (Findings §8.6)

    **Tests**:
    - (integration) write-back round-trip; unresolved-rows query via `ix_connector_plays_unresolved`
    - (use case, mock UoW) orchestrator persists resolution after Phase 2

    Effort: M | Dependencies: None (parallel with Ledger Idempotency Hardening) | Status: Not Started

- [ ] **Last.fm Canonical Casing Fix**

    **Story**: The same song plays once and becomes two canonical tracks — "Striptease" (Spotify-created) and "striptease" (Last.fm-created) — so play dedup, keyed on `track_id`, is structurally defeated: 191 divergent play pairs and 143 duplicate canonical-track pairs trace to this (Findings §5b). The projection's bridge is the safety net; this is the cure.

    **Decisions**:
    - **Fix creation, don't repair data** — with no at-scale data imported anywhere (Findings §1), correcting the resolver before the first big import beats any cleanup tooling.

    **Spec**:
    - Verify the hypothesis in `lastfm/inward_resolver.py`: canonical `Track` rows created from the lowercased `artist::title` identifier parts instead of the API's display metadata, and/or created without first matching existing canonicals via the normalized lookup columns
    - Create canonicals from Last.fm display names; match-before-create against `title_normalized`/`artist_normalized`
    - Cross-reference [identity-resolution-design-space.md](identity-resolution-design-space.md) (FM3c rename fragility) — the fix must not regress the rename handling documented there
    - Already in place (v0.8.18): Last.fm *identifier* unification (migration 035 folded all rows to the normalized composite scheme; `make_lastfm_identifier` at all four mint sites) — this epic targets canonical `Track` creation one layer up, and must not regress that fold or the `LASTFM_RAW_ALIAS` quarantine (v0.8.18.1)

    **Tests**:
    - (unit) resolver preserves display casing on canonical creation
    - (integration) Last.fm resolution of a track already canonical via Spotify maps to the existing canonical instead of creating a twin

    Effort: S | Dependencies: None | Status: Not Started

#### Projection Epics

- [ ] **Deterministic Merge & Grouping Domain Module**

    **Story**: One pure function decides what a canonical play is, so the answer never depends on which import ran first — today opposite arrival orders produce 519 correct merges vs 267 double-counts on the same overlap data (Findings §5a).

    **Decisions**:
    - **The module is channel-generic from day one** — v0.10.1's `spotify_api` and Apple Music channels must join with zero new dedup code; that claim is v0.10.1's acceptance test.
    - **Property tests are the ACI laws** — permutation invariance, idempotence, and batch-boundary invariance are the convergence guarantee stated as executable properties; a merge that passes them cannot be order-dependent.

    **Spec**:
    - New `src/domain/matching/play_projection.py`, evolving `play_dedup.py` (keep `_normalize_to_start_time`, `_get_tolerance`, tolerance constants). Per-channel behavior consolidates into one frozen `ChannelSpec` registry — channel → priority rank, time semantics (start/end), tolerance override — replacing service-keyed `END_TIME_SERVICES` and the loose priority tuple; priority order `("spotify_export", "spotify_api", "mixd", "lastfm")`. A future channel (Apple, ListenBrainz) is one registry entry — this registry *is* the multi-service seam
    - `group_ledger_entries(entries) -> list[PlayGroup]`: bucket by `resolved_track_id`; bridge buckets on exact `normalize_for_comparison(artist)::title` equality within tolerance (bridged groups emit a `resolution_divergence` stat); same-channel collapse for identical `(played_at, identifier)` differing `ms_played` (keep max); greedy nearest one-to-one cross-channel assignment (sort by |Δstart|, then channel priority, then `id`) + union-find
    - `merge_group(group) -> CanonicalPlayFields` survivorship: `played_at` = normalized start of highest-precision member; `ms_played` = first non-null by priority; `context` = winner flat + `merged_from_<channel>` sub-objects (preserves today's persisted shape); `source_services` = priority-sorted union of services; ties broken by lowest `id` (uuid7, time-ordered)
    - Domain-pure: entries passed in, nothing fetched; constants are module-level `Final` per domain rules

    **Tests**:
    - (unit, hypothesis) permutation invariance: any partition of entries into batches, any order → identical groups and fields; idempotence: re-projecting projected input is a no-op; two skips 20s apart never merge; bridge unions only exact-normalized keys

    Effort: L | Dependencies: None | Status: Not Started

- [ ] **Projection Pipeline in the Import Path**

    **Story**: The user imports Last.fm then uploads GDPR files — or the reverse, or both twice — and gets byte-identical canonical plays. This is the property the user can personally verify, and the milestone's manual acceptance test.

    **Decisions**:
    - **`track_plays` keeps its exact shape** — only writers change; every reader (aggregations, recent plays, metrics, workflows) is untouched.
    - **Membership is materialized in a `play_sources` join table** — idempotence becomes a mechanical no-op diff (unchanged group → no writes), and provenance survives without parsing `context`.
    - **`play_dedup.py` and `bulk_update_play_source_services` are deleted after cutover** — clean break; two merge paths would reintroduce the disagreement this milestone removes.

    **Spec**:
    - `play_sources` table: `user_id`, `track_play_id` (FK→track_plays ON DELETE CASCADE), `connector_play_id` (FK→connector_plays), UNIQUE(`user_id`, `connector_play_id`); RLS per house pattern
    - Orchestrator Phase 3 replaces the dedup block (`play_import_orchestrator.py:184-236`): fetch affected ledger window (± fallback tolerance, closure-extended) → `group_ledger_entries` → `merge_group` → diff-apply
    - Diff-apply: unchanged → no-op; changed → UPDATE keeping existing `id`; two rows merging → keep lowest `id`, delete the other, repoint `play_sources`; orphaned → DELETE
    - Import result reports groups created/updated/merged + `resolution_divergence` count

    **Tests**:
    - (integration) diff-apply insert/update/merge-keeping-lowest-id/delete paths; `play_sources` uniqueness
    - (use case, mock UoW) Phase 3 wiring + stats
    - (manual — milestone acceptance) two fresh DBs, opposite import orders over the same fixtures → identical canonical dumps (ids aside); re-import → zero changes

    Effort: L | Dependencies: Ledger Idempotency Hardening, Resolution Write-Back, Deterministic Merge & Grouping Domain Module | Status: Not Started

- [ ] **History Rebuild Command**

    **Story**: The user (or a future algorithm change) can re-derive the entire canonical history from the observation ledger with one command — the projection is replayable, so "repair" is never bespoke surgery again.

    **Decisions**:
    - **Rebuild is the migration path, not a data migration** — Findings §1: no environment holds precious play data, so converging the SQLite-era testbed (and any future defect) is "replay the ledger," free of preservation constraints.

    **Spec**:
    - `mixd plays rebuild [--dry-run]` — batched (day-chunked like the Last.fm importer), full-history projection over the ledger
    - `--dry-run` prints a diff report: rows to merge/update/delete, divergences found
    - Long-op progress via existing SSE/progress emitters; parity (file-header contract): mutation → two-phase confirm
    - `src/application/use_cases/rebuild_play_history.py` + CLI command

    **Tests**:
    - (integration) rebuild over a fixture seeded with order-dependence doubles converges; second run is a no-op
    - (unit) `test_registry_parity.py` stays green

    Effort: M | Dependencies: Projection Pipeline in the Import Path | Status: Not Started

### v0.10.1: Continuous Play Polling

**Goal**: Play history flows in continuously — Spotify's recently-played API is polled on the existing scheduler before plays age out of its ~50-play window, with Apple Music designed-for as the next channel.

**Context**: The GDPR export delivers depth but arrives weeks late and ends the day it was requested (the user's current export stops 2025-02-28 — everything since exists only in Last.fm). Last.fm is live but identity-weak. Polling the source APIs adds a fresh, strong-identity observer: play counts land within the hour, and the convergence projection (v0.10.0) guarantees the later GDPR upload deepens those same plays (`ms_played`, skip/shuffle behavior) instead of duplicating them. This is the always-on-ingestion component [PDR-001](../decisions/PDR-001-data-ownership-model.md) identified as structural: missed polls are permanent data loss.

**What this unlocks**: live play-count workflows — "filter out tracks I already listened to today", "skip this week's heavy rotation" — plus redundancy where Last.fm misses and coverage after the export window closes.

**Persona**: Curator-primary (their workflows finally see today's listening); Tinkerer secondary (schedule + checkpoint surfaces to script against).

**Key Design Decisions**:
- **Polling rides the existing scheduler and sync-dispatch infrastructure** — `run_scheduler_loop` (v0.8.2) already dispatches per-user sync targets under their RLS + OAuth identity; a poller is a `SYNC_DISPATCH` entry, not new infrastructure. The `("spotify", "plays")` checkpoint combination is already declared in `CHECKPOINT_COMBINATIONS`, never produced.
- **Freshness is demand-pulled; the timer only guarantees coverage** — play counts must be current at the moments they're *consumed* (the user in the app, an agent reading over MCP, a workflow about to run), and those are all hookable events. The background heartbeat exists solely so plays don't age out of the ~50-play API window between sessions, and it adapts to the user's actual listening rhythm instead of firing hourly into silence.
- **API plays are a new observation channel (`spotify_api`), same service** — no `ms_played`, but per-play `played_at` and strong URI identity; the v0.10.0 channel-keyed grouping merges them with export and Last.fm observations with zero new dedup code (this is that module's acceptance claim, tested here).
- **Spotify API `played_at` start-vs-end semantics are calibrated, not assumed** — the export's end-time semantics were only confirmed empirically (Findings §3); the first real poll is compared against overlapping Last.fm scrobbles before its tight tolerance is trusted.
- **Apple Music is designed-for, not built** — its recently-played API returns no timestamps at all, so the channel contract (approximated `played_at`, wide tolerance, presence-only precedence) is specified now and implementation blocks on Apple Developer Program membership.

- [ ] **Spotify Recently-Played Importer**

    **Story**: The user's Spotify listening lands in mixd within the polling interval — the missing live observer between scrobbles and the eventual export.

    **Decisions**:
    - **Cursor-based incremental, never windowed re-reads** — the API's `after` cursor + checkpoint means each poll fetches only new plays; the ~50-play window makes missed data unrecoverable, so the checkpoint is the source of truth for "where we left off."
    - **First-poll calibration against Last.fm** — pin the API's `played_at` start-vs-end semantics empirically before the projection trusts the 30s tolerance for `spotify_api` pairs. API rows carry no `ms_played`, so if end-time semantics are confirmed, start-normalization subtracts the payload's `track.duration_ms` (full plays only); until calibrated, `spotify_api` pairs use the wide fallback tolerance via their `ChannelSpec`.
    - **The API's blind spots are expected divergence, not defects** — recently-played records only plays >30s and omits private sessions, so export-only short plays (Findings §7: 28.2% of GDPR rows are <30s) will never have an API twin; the calibration report buckets these out rather than counting them as misses.

    **Spec**:
    - Client: `GET /me/player/recently-played` (limit 50, `after` cursor, ms-epoch cursors) via `SpotifyBearerAuth` (silent refresh already unattended-capable) *(2026-07 check: endpoint explicitly retained in Spotify's Feb 2026 dev-mode endpoint purge; 50-item hard window — cursors cannot page beyond it. Dev mode now requires a Premium app owner and caps 5 authorized users per client — fine for mixd today, a constraint for any future multi-user rollout)*
    - Importer per `BasePlayImporter` pattern: ledger rows `import_source="spotify_api"`, `ms_played=None`, context = API `context` object (playlist/album source)
    - Checkpoint `("spotify", "plays")`: `cursor` = `after` value, `last_timestamp` = newest `played_at`
    - Lift the spotify=file-only restriction in `ImportTracksCommand` (`import_play_history.py`) → `mode=recent/incremental` for Spotify
    - Calibration: one-off comparison report of first-poll plays vs overlapping Last.fm scrobbles (delta distribution, mirroring Findings §3), with the >30s/private-session bucket excluded per the blind-spot decision
    - Expectation-setting: the first poll captures only the trailing ~50 plays — earlier history arrives via Last.fm and the next GDPR export; the connect-flow copy says so, so the user doesn't read a short first import as a failure

    **Tests**:
    - (unit) cursor pagination + checkpoint math; (integration, mocked transport) incremental resume; (use case) `SpotifyAuthRequiredError` surfaces cleanly when scope missing
    - (manual) listen on Spotify → run import → play visible with correct timestamp

    Effort: L | Dependencies: v0.10.0 Projection Pipeline, Recently-Played Scope & Re-Consent | Status: Not Started

- [ ] **Demand-Driven & Adaptive Polling**

    **Story**: The user's play counts are fresh at the moments they matter — opening mixd after a listening session, an agent reading plays over MCP, a 6am scheduled workflow filtering on "played today" — and the poller is quiet the rest of the time. A fixed hourly timer gets this exactly backwards: it wastes most polls on silence and can still be an hour stale at the moment of use.

    **Decisions**:
    - **Three demand triggers share one staleness-gated helper** — play-surface reads (web API + MCP tools) fire it *fire-and-forget* so reads never wait; the workflow executor *awaits* it (bounded timeout, run proceeds on expiry) when the graph contains play-history nodes, because a scheduled workflow is precisely where stale counts silently corrupt output and nobody is watching.
    - **Freshness gates on last-*polled*, never last-*play*** — the checkpoint's `last_timestamp` records the newest play seen, which goes stale whenever the user simply isn't listening; gating on it would fire a pointless poll on every page view of an idle account. A new `last_polled_at` is the gate: "when did we last check," not "when did they last listen."
    - **The coverage heartbeat decides per tick, not per schedule** — the schedule stays at a fixed base cadence and the sync target early-exits from backoff state persisted on the checkpoint (consecutive empty polls stretch the effective interval; any new plays or demand trigger snaps it back). Adaptivity lives in the target, so scheduler infrastructure is untouched.
    - **The backoff cap is redundancy-aware** — if Last.fm is connected and scrobbling, a missed window is recoverable (another live observer + the next GDPR export), so background polling may relax toward its slow cap; if `spotify_api` is the user's *only* live observer, the cap stays inside the window-turnover bound (~2h of dense listening), because then a missed poll really is permanent loss (PDR-001).
    - **Single-flight per user** — concurrent triggers (page focus + workflow start + heartbeat) collapse into one poll via a non-blocking pg advisory lock keyed `(user_id, "spotify:plays")`, the schedule repo's `try_acquire_poll_lock` pattern — DB-level because every replica serves demand triggers; losers no-op, and the cursor-based importer makes the winning poll cheap regardless.
    - **One execution path, three trigger sources (DRY)** — demand triggers and the scheduler heartbeat invoke the *same* shared sync-target runner, extracted from the scheduler's `_dispatch_sync` (where the `OperationRun` audit wrapping lives today), so every poll — scheduled or demand — produces the same audit row, updates the same checkpoint, and shows in the same run log and sync-status table. Without the extraction, demand polls would silently bypass the audit trail and mixd would grow a second, invisible class of background work. The early-exit veto runs *before* the audit row opens — a 30-min heartbeat vetoing inside the target would otherwise write ~48 no-op `OperationRun` rows per user per day; vetoed ticks log at debug and leave no audit row.
    - **Demand polls run in-process, by design** — fire-and-forget rides the existing `launch_background` primitive on the API event loop (where the scheduler loop already lives); a cursor poll is one ≤50-item API call, so no queue is warranted — anything heavier belongs on the scheduler path.

    **Spec**:
    - Extract `_dispatch_sync`'s audit wrapper (`scheduler.py`) into a shared application-service runner `run_sync_target(user_id, target, ...)` that evaluates the target's decision/veto **before** `start_run`; the scheduler loop and `ensure_fresh_plays` both call it. Provenance lands on `operation_runs` as `initiated_by="demand"` plus a new `trigger_detail` string (`web` / `mcp` / `workflow:<run_id>`); scheduled runs keep `initiated_by="schedule"` + `triggered_by_schedule_id`, so existing queries are untouched and the run log shows *why* every poll happened
    - Checkpoint schema (migration): `sync_checkpoints` gains `last_polled_at` (timestamptz — the freshness gate) and `poll_state` (JSONB: consecutive empty polls, effective interval) — today the table holds only `last_timestamp`/`cursor`/`remote_total`
    - `ensure_fresh_plays(user_id, max_age)` application helper: `last_polled_at` within `max_age` → no-op; else `run_sync_target(...)` for the incremental recently-played import; single-flight via the advisory lock. House precedent for the staleness gate: metrics freshness (`metrics_application_service.py` + `FreshnessConfig`)
    - Trigger wiring (verified seams): web = shared FastAPI dependency on the play-reading routes (`stats.py` dashboard, `tracks.py` detail — there is no single REST chokepoint) firing via `launch_background`; MCP + chat = one hook in `execute_tool` (`src/application/tools/registry.py`) gated by a play-reading tool set (`query_library`, `query_stats`) — one chokepoint covers both surfaces; workflow = pre-run check in `ExecuteWorkflowRunUseCase._run_to_completion` walking task types for `{source.played_tracks, enricher.play_history}` (same pattern as `extract_required_connectors`) → await with timeout
    - Heartbeat: `SYNC_DISPATCH` entry `"spotify:plays"` (`sync_targets.py`) whose decision function reads (staleness, `poll_state`, last-user-activity, Last.fm-connected) and polls or early-exits. The per-user schedule row is created/enabled when re-consent lands the `user-read-recently-played` scope, disabled on disconnect; while the scope is missing the decision early-exits silently — the re-connect prompt is the visible surface, not a stream of error runs
    - Starting-point constants (revisit against real telemetry): demand staleness threshold 10 min; heartbeat base tick 30 min; effective interval 30 min → cap; cap 2h sole-observer / 24h (daily floor) with Last.fm redundancy — the redundant-case poll only protects Spotify-side detail Last.fm already covered, so once a day is enough to keep the cursor warm and catch a mixd-less listening day
    - Activity signal: last authenticated-request timestamp per user, written only when the stored value is >5 min old (starting point, revisit) — one write per active burst, not per request

### v0.10.2: First-Class Artists

**Goal**: Promote artists from embedded JSON strings to first-class entities with identity resolution, favorites, web UI, and workflow integration.

**Context**: Artists are currently stored as `{"names": ["..."]}` JSON on Track and ConnectorTrack records — "Radiohead" on Spotify and "Radiohead" on Last.fm are unrelated text. This milestone gives artists their own identity, cross-service mapping, favorites, and dedicated pages, following the same three-layer pattern established for tracks.

**What this unlocks**: artist browsing and favorites in the web UI, "tracks by my favorite artists" workflows, and the artist-identity substrate that v0.10.3 albums and the v1.1.x social layer build on.

**Persona**: all three — the Curator favorites artists and builds artist-driven playlists; the Tinkerer explores the mapping model; the Casual listener finally gets browse-by-artist.

**Key Design Decisions**:
- **Favorites are Mixd-only** — users favorite artists in the UI, because the favorite is *their* curation signal, not a mirror of platform state. Service sync (e.g., Spotify followed artists) deferred to a future milestone.
- **`track_artists` join table** replaces JSON artist embedding on tracks — enables efficient "all tracks by artist X" queries and proper relational integrity.
- **Canonical artists are per-user, not global** — `tracks` is per-user today and cross-user identity is deliberately deferred to v1.0.3; a global artists table would smuggle in cross-user identity semantics (merge rights, shared edits) this cycle hasn't earned. `connector_artists` stays global like `connector_tracks` — service records are facts about the service, not the user.
- **The JSON→`track_artists` move is expand/contract, not a cutover** — the matching layer reads `artist_normalized`/`artists_text` precomputed from the JSONB today; a one-shot swap would put match confidence at risk mid-cycle. Expand (dual-write) → repoint readers → contract (drop the JSONB) as a separate final step, only after matching characterization tests pass against the new source.

### v0.10.4: Physical Media & Discogs

**Goal**: Track which albums the user owns and on which media. Discogs as the primary source for physical release metadata — pressings, formats, labels, catalog numbers.

**Context**: Your music identity isn't just streaming — it's the vinyl on your shelf, the CDs in your car, the albums you've purchased digitally. Discogs has 15M+ releases with rich physical media metadata (format, label, catalog number, country, pressing notes). Importing a user's Discogs collection instantly populates "what I own and in what format."

**What this unlocks**: the physical half of the library — ownership records that v0.10.5's manual listens and future collection features build on.

**Persona**: Curator-primary (the record collector's data reclamation); Tinkerer secondary (a new connector to explore). Casual: not served — physical collections are a Curator concern.

**Key Design Decisions**:
- **Release ≠ Album**: An album (release group) can have many releases (pressings). "OK Computer" is one album; the 1997 UK vinyl, 2008 Japanese CD, and 2017 OKNOTOK remaster are three releases.
- **Discogs as primary physical source**: MusicBrainz has releases too, but Discogs has stronger collector community data (condition, notes, marketplace). Start with Discogs; MusicBrainz release data can supplement later.
- **Read-only initially**: Import from Discogs, don't write back. User manages their Discogs collection on Discogs.

The client takes an injected auth strategy so OAuth 1.0a can slot in later if write/marketplace scopes are ever needed — mixd has zero RFC 5849 code today.

    **Spec**:

### v0.10.5: Manual Listens

**Goal**: Record physical album listens — "I put this record on at 3pm" — scrobble to Last.fm and store as canonical plays in mixd.

**Context**: When you listen to a vinyl record, that listening event is invisible to your digital history. Mixd should let you log it: pick the album, enter when you started, and the system figures out the per-track timestamps from track order and durations. The plays become part of your canonical history — indistinguishable from streaming plays in workflows and metrics.

**What this unlocks**: complete listening history — physical plays join the canonical record, so "loved but forgotten" workflows and play metrics stop under-counting vinyl listeners.

**Persona**: Curator-primary (the vinyl listener whose history has holes); Tinkerer secondary (CLI + scripting surface). Casual: not served by this milestone.

**Key Design Decisions**:
- **Album-level input, track-level output**: User says "I listened to this album starting at 3pm." System calculates per-track timestamps and creates individual `TrackPlay` records. Each track gets its own play entry because that's what Last.fm expects and what mixd's play history model requires.
- **Optional Last.fm scrobble**: User chooses whether to also scrobble to Last.fm or keep it mixd-only. Some people want to keep physical and digital scrobbles separate.
- **Retroactive and real-time**: Support both "I listened yesterday at 3pm" and "I'm listening right now."

#### Listen Infrastructure Epics

- [ ] **Last.fm Scrobble API**

    **Story**: Sending play events to Last.fm is the foundation for physical listens reaching the user's scrobble history.

    **Decisions**:
    - **Scrobble failure never blocks mixd persistence** — Last.fm is maintenance-mode and observably flaky *(2026-07 check: API operational, `track.scrobble` batch-50 current, no deprecations — but integrations report intermittent auth errors)*; the canonical play lands regardless, the scrobble retries via `_shared/retry_policies.py` and degrades to a per-track warning, because the user's own history must not depend on someone else's uptime.

    **Spec**:
    - Implement `track.scrobble` on `LastFMAPIClient` — authenticated POST, batch up to 50 tracks per request
    - Last.fm `track.scrobble` API: `method=track.scrobble`, params: `artist`, `track`, `timestamp` (Unix epoch), optional `album`, `duration`
    - Batch: up to 50 scrobbles per request via indexed params (`artist[0]`, `track[0]`, `timestamp[0]`, etc.)
    - Requires same session key auth as `track.love` (already implemented)
    - Scrobble rules: track must be >30s duration, listen must be >50% of duration or >4min

    **Tests**:
    - (unit) batch param encoding; (integration, mocked transport) partial-failure reporting

    Effort: S | Dependencies: None (builds on existing Last.fm auth — `track.love` write already works) | Status: Not Started

- [ ] **Album Listen Calculator**

    **Story**: Converting "album listen at 3pm" into individual track plays with correct timestamps is the core logic of manual listens — and it must be right, because it writes permanent history.

    **Decisions**:
    - **Missing durations: use any connector-known duration; otherwise skip that track's scrobble but keep the mixd play** (timestamp = previous track's end) — fabricating durations would corrupt permanent history, and dropping the play entirely would under-count, the exact defect this milestone fixes.
    - **"Listening right now" means "just finished", not "starting now"** — the real-time input is listen *end* = now, timestamps computed backward; computing forward from a now-start would write future-dated plays that Last.fm rejects and mixd can't honestly claim happened.

    **Spec**:
    - Pure domain function: `calculate_listen_timestamps(tracks: list[Track], start_time: datetime) -> list[tuple[Track, datetime]]`
    - Track N+1 starts at: start_time + sum(duration of tracks 1..N)
    - Handle multi-disc albums: disc order → track order (already modeled in `album_tracks` via disc_number + track_number)
    - Pure function, no side effects — testable in isolation

    **Tests**:
    - (unit) timestamp arithmetic incl. multi-disc ordering and missing-duration handling

    Effort: S | Dependencies: v0.10.3 (first-class albums with ordered track listings) | Status: Not Started

- [ ] **Record Album Listen Use Case**

    **Story**: The user logs one album listen; the system resolves tracks, calculates timestamps, persists canonical plays, and (optionally) scrobbles — one command, full flow.

    **Decisions**:
    - Plays land in the same table as streaming plays — canonical history is unified.
    - **Manual plays are `service='mixd'` + `import_source='manual'`, physical detail in `context`** — `service` records the observing system and mixd is the observer; inventing a 'physical' service value would leak provenance into the service axis and break existing `service` queries (the dedup constraint keys on it).
    - **Scrobble targets sit behind a `ScrobbleSink` seam** — Last.fm is the first sink; a ListenBrainz sink (token auth, batch-1000 submits — filed in [unscheduled](unscheduled.md#enrichment-sources)) slots in later without reopening this use case.

    **Spec**:
    - `RecordAlbumListenCommand(album_id, started_at, scrobble_to_lastfm: bool = True)`
    - Appends `connector_plays` observations per track (see **Manual Plays Join the Ledger** below — no direct canonical writes); the v0.10.0 projection creates the `track_plays` rows: `service='mixd'`, `import_source='manual'`, `source_services=['mixd']`, `context` JSONB `{"listen_type": "physical", "album_release_id": ...}` carried onto the projected row (winner-flat context rule; `mixd` outranks `lastfm`), so the album page's "Recent listens" query reads canonical history unchanged
    - Parity (file-header contract): mutation → `ToolSpec` + two-phase confirm
    - If `scrobble_to_lastfm`: batch-scrobble to Last.fm via `track.scrobble`
    - Result includes: tracks played, timestamps, Last.fm scrobble success/failure per track

    **Tests**:
    - (unit + mock UoW) plays persisted with calculated timestamps; scrobble skipped when toggled off; per-track scrobble failure reported, not swallowed
    - (unit) `test_registry_parity.py` stays green

    Effort: M | Dependencies: Album Listen Calculator, Last.fm Scrobble API | Status: Not Started

- [ ] **Manual Plays Join the Ledger**

    **Story**: The user logs a vinyl listen and scrobbles it to Last.fm; the next Last.fm import returns that same scrobble — one event, seen twice. The projection must know they're the same play without a special-case merge path, or every manual listen eventually double-counts, silently inflating the history this milestone exists to make honest.

    **Decisions**:
    - **Manual plays are ledger observations, not direct canonical writes** — `RecordAlbumListenUseCase` appends to `connector_plays` and lets the v0.10.0 projection create the canonical rows, because the round-trip merge then falls out of channel-keyed grouping (`mixd` and `lastfm` are just two observers of one event) instead of needing bespoke dedup code.
    - **The manual listen's timestamp and context survive the round-trip** — `mixd` outranks `lastfm` in channel priority, so the returning scrobble enriches (`source_services` gains `'lastfm'`) rather than displaces.

    **Spec**:
    - `RecordAlbumListenUseCase` writes `connector_plays` observations: `connector_name='mixd'`, `import_source='manual'`, identifier = `artist::title`, `resolved_track_id` set immediately, physical detail in `raw_metadata`
    - Migration backfills ledger observations for any pre-existing `service='mixd'` plays

    **Tests**:
    - (integration) log listen → scrobble → simulated Last.fm import of the same scrobble → play count unchanged, `source_services == ['mixd', 'lastfm']`, `played_at` stays the mixd timestamp

    Effort: M | Dependencies: Record Album Listen Use Case, Projection Pipeline in the Import Path (v0.10.0) | Status: Not Started

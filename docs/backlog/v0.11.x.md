# v0.11.x: Connector Foundations

For strategic overview, see the [planning overview](README.md).

## User Workflow Context

The user's musical life spans more services than mixd can see: Apple Music listening is invisible to the play history v0.10.x made honest, and the physical collection catalogued on Discogs has no presence in the library. This cycle lands the *bones* of both connectors — auth, API clients, conservative track resolution, and the Apple play channel — deliberately **before** first-class artists and albums (v0.12.x), for two reasons: entity modeling should be designed against real catalog data from five services (the v0.12.0 spike consumes these clients), and Apple plays should start accruing into the observation ledger now rather than after the entity work. Full feature surfaces — Apple library/likes, Discogs collection import — intentionally wait for v0.13.x, once the entities they attach to exist.

Persona view (from the [alignment table](README.md#persona-alignment)): the **Weekly Curator** gets Apple listening flowing into their canonical history; the **Tinkerer** gets two new connectors' raw surfaces to poke at; the **Casual** listener gets the groundwork for "connect Apple Music" being a first-class button.

Cross-cutting contract (stated once — per-epic Specs reference it): every new use case this cycle must be classified in the chat-tool parity registry (`src/application/tools/registry.py`) — CI (`tests/unit/application/tools/test_registry_parity.py`) fails on any unclassified use case. Reads get a `ToolSpec` or an explicit classification; mutations additionally get two-phase confirmation wiring like the existing `*_write` dispatchers.

### v0.11.0: Apple Music Foundation

**Goal**: mixd can authenticate to Apple Music, resolve its tracks conservatively, and capture Apple plays as a live observation channel — the bones that let the v0.12.0 spike see Apple's real catalog shapes and let play history start accruing immediately.

**Context**: An existing stub at `src/infrastructure/connectors/apple_music/` (`connector.py`, `error_classifier.py`) provides the starting point; shared infrastructure (`_shared/http_client.py`, retry policies, error classification) means the client is mostly wiring. Auth differs from Spotify: a developer token (JWT/ES256, requires the Apple Developer Program membership — in place) plus a per-user MusicKit user token, not OAuth 2.0. Apple has no "love" API (Add to Library is the like analogue — deferred with the rest of the library surface to v0.13.0). Play data accrues only once the user actually listens on Apple Music; the channel ships regardless so nothing is lost when they do.

**What this unlocks**: Apple catalog access for the v0.12.0 entity-representation spike; Apple plays flowing through the v0.10.0 projection; the auth substrate v0.13.0's library/likes integration reuses.

**Persona**: Curator-primary (Apple listening joins the canonical history); Tinkerer secondary (a fourth connector's auth model to inspect).

**Key Design Decisions**:
- **Ingest early, resolve conservatively** — Apple observations land in `connector_plays` immediately, but resolution is **ISRC-only** (excellent coverage in Apple's catalog); fallback candidates are recorded *unresolved* rather than matched, because the pre-alias comparator (alias-aware artist comparison lands in v0.12.1) would mint permanent weak mappings. v0.13.0's re-resolution pass + `mixd plays rebuild` promotes the deferred observations once the full matcher exists. No permanent weak mappings, no lost plays.
- **Supersession-native from birth** — landing after v0.10.2 means Apple's platform-asserted succession (`catalogId` vs `playParams.catalogId` equivalence, parallel to Spotify relinking) is representable in the mapping schema from the first row; the migration the old sequencing risked never happens.
- **The play channel is presence-weak by design** — Apple's recently-played API returns **no timestamps** (≤30 items, offset paging), so `played_at` is approximated from the poll window with a channel-specific wide tolerance, and the `apple_api` channel ranks below every timestamped channel in survivorship: it may create a play, but never wins timestamp fields against an observer that knows when.

#### Apple Foundation Epics

- [ ] **Apple Music Auth & API Client**

    **Story**: Every Apple Music operation — catalog lookups for the spike, play polling, the eventual library sync — needs an authenticated client; this is the connector's foundation and the external-prerequisite resolver for everything Apple.

    **Decisions**:
    - **Developer token minted server-side (JWT/ES256), user token via MusicKit flow** — Apple's model, not OAuth 2.0; the key/team/media IDs are instance configuration, the user token is per-user encrypted storage like every other connector credential.

    **Spec**:
    - `AppleMusicAPIClient(BaseAPIClient)` with httpx; Pydantic models for JSON:API response shapes
    - Developer-token minting (ES256, capped TTL, cached until expiry); MusicKit user-token capture flow; storage via the `TokenStorage` pattern (encrypted at rest)
    - Refactor existing `AppleMusicErrorClassifier` to the modern hook pattern
    - Reuse: `_shared/http_client.py`, `_shared/retry_policies.py`, `_shared/error_classifier.py`; stub at `src/infrastructure/connectors/apple_music/`

    **Tests**:
    - (unit) developer-token mint/expiry/cache; (integration, mocked transport) 401 → actionable reconnect error; retry policy on 5xx

    Effort: M | Dependencies: None (Apple Developer Program membership in place) | Status: Not Started

- [ ] **Conservative Track Resolution**

    **Story**: Apple tracks must connect to canonical tracks for plays and the spike to mean anything — but mappings are permanent until v0.10.2's supersession machinery is exercised, so the resolver must only assert what it can anchor.

    **Decisions**:
    - **ISRC-first, ISRC-only** — no artist+title fallback in this milestone: the alias-aware comparator doesn't exist until v0.12.1, and a weak mapping minted now persists. Fallback candidates are recorded as unresolved observations and queued via the v0.10.2 event log's no-match backoff — v0.13.0's "Deferred Play Re-Resolution" epic is their designed exit.
    - **Content equivalence via dual mapping** — `catalogId` vs `playParams.catalogId` reuses the Spotify-relinking dual-mapping pattern, recorded as platform-asserted succession (v0.10.2 typed reason).

    **Spec**:
    - `AppleMusicInwardResolver` + `AppleMusicMatchingProvider` with only the ISRC path active
    - Reuse: `_shared/inward_track_resolver.py`, `_shared/matching_provider.py`, `_shared/isrc.py` (including the ISRC-suspect duration cross-check)
    - Unresolved-candidate recording: ledger rows keep `resolved_track_id = NULL`; no-match rows enter the v0.10.2 backoff queue

    **Tests**:
    - (unit) ISRC hit → mapping with `match_method="isrc"`; no ISRC → **no** `artist_title` mapping minted, observation stays unresolved
    - (integration) unresolved Apple ledger rows excluded from the projection and queryable for the v0.13.0 re-resolution pass

    Effort: M | Dependencies: Apple Music Auth & API Client, v0.10.2 (supersession schema) | Status: Not Started

- [ ] **Apple Play Channel**

    **Story**: The Apple Music listener's plays should land in the same canonical history as everything else — this activates the channel the v0.10.0 projection was designed to accept without new dedup code (that claim is this epic's acceptance test).

    **Decisions**:
    - **Poll-window-midpoint timestamps, presence-only precedence** — Apple returns no `played_at`, so the channel contributes *that* a play happened, never *when* precisely; the projection's channel-keyed survivorship handles the rest.
    - **Rides the v0.10.1 polling policy wholesale** — demand triggers, adaptive heartbeat, redundancy-aware caps, and the shared `run_sync_target` runner all apply; Apple is one more `SYNC_DISPATCH` entry and one more checkpoint row, not new infrastructure.

    **Spec**:
    - Client: `GET /v1/me/recent/played/tracks` (`types=songs`, limit ≤30, offset paging)
    - Channel `("apple", "apple_api")`: identifier = Apple catalog song id; `played_at` = poll-window midpoint; channel-specific wide tolerance constant in `play_projection.py`; lowest timestamp-precedence rank
    - `SYNC_DISPATCH` entry `"apple:plays"` via `run_sync_target`; checkpoint `("apple", "plays")` appears in the sync-status table with effective-interval-aware staleness
    - Import stats + `OperationRun` audit come free from the shared runner

    **Tests**:
    - (unit) wide-tolerance grouping + presence-only survivorship for `apple_api` observations
    - (integration) simulated overlapping Apple + Last.fm observations of one listen → one canonical play, Last.fm's timestamp wins
    - (manual — milestone acceptance) connect Apple → listen → play appears in history and the sync table shows the checkpoint row

    Effort: M | Dependencies: Conservative Track Resolution, v0.10.1 (polling infrastructure) | Status: Not Started

### v0.11.1: Discogs Foundation

**Goal**: An authenticated Discogs client and a first look at the user's collection — the bones the v0.12.0 spike queries and v0.13.1 physical media builds on.

**Context**: Discogs has 15M+ releases with the richest physical-media metadata anywhere (formats, labels, catalog numbers, pressing notes) and a distinctive artist model (name variations, ANVs, join phrases) that the v0.12.0 spike must see before artist entities are designed. The user's own collection may still be small or empty — the snapshot degrades gracefully, and the *catalog* endpoints (release/master/artist) are what the spike consumes regardless.

**What this unlocks**: Discogs catalog access for the spike; the client and token plumbing v0.13.1's collection import reuses; the connector-card presence that invites the user to start cataloguing.

**Persona**: Curator-primary (the record collector's on-ramp); Tinkerer secondary.

**Key Design Decisions**:
- **Foundations only** — no canonical writes this milestone: release matching and collection import need albums (v0.12.2) and land in v0.13.1; shipping the client early is what makes the spike's Discogs leg real instead of speculative.

#### Discogs Foundation Epics

- [ ] **Discogs API Client & Auth**

    **Story**: Every Discogs operation — collection import, release detail — needs an authenticated client; this is the connector foundation.

    **Decisions**:
    - **Per-user personal access token (BYO-token), not OAuth 1.0a** *(2026-07 check: Discogs API healthy, no deprecation; tokens officially recommended for single-user apps — same 60 req/min, never expire)* — collection reads don't need the OAuth dance, and BYO-token matches the existing BYO-Anthropic-key pattern (v0.9.0.1): validated live, stored encrypted, write-only in the UI. The client takes an injected auth strategy so OAuth 1.0a can slot in later if write/marketplace scopes are ever needed — mixd has zero RFC 5849 code today.

    **Spec**:
    - `DiscogsAPIClient(BaseAPIClient)` with httpx, Pydantic models; auth-strategy seam (token header now, OAuth 1.0a slot open)
    - Token storage + settings surface follow the Anthropic-key pattern: encrypted at rest via `oauth_tokens`, connect-gated UI
    - Rate limit: 60 requests/minute (authenticated)
    - Key endpoints: `/users/{username}/collection/folders/0/releases` (full collection), `/releases/{id}` (release detail), `/masters/{id}` (master release)
    - Pydantic models for release, master, artist, label, format responses
    - Reuse: `_shared/http_client.py`, `_shared/retry_policies.py`

    **Tests**:
    - (unit) auth header injection + 401 → actionable reconnect error; (integration, mocked transport) pagination + rate-limit retry

    Effort: M | Dependencies: None | Status: Not Started

- [ ] **Collection Snapshot**

    **Story**: Connecting a service should show something real immediately — the collector connects Discogs and sees their collection acknowledged, before any of the matching machinery exists.

    **Decisions**:
    - **Raw display, zero canonical writes** — the snapshot lists what Discogs reports, unmatched; import-with-matching is v0.13.1's job, and pretending otherwise here would mint the exact premature mappings v0.11.0 is designed to avoid.

    **Spec**:
    - Fetch collection count + recent items via the collection endpoint (paginated)
    - CLI: `mixd discogs snapshot` — count + table of recent items; connector card shows "connected · N releases"
    - Empty collection renders an inviting zero-state, not an error
    - Parity (file-header contract): read → `ToolSpec`

    **Tests**:
    - (unit) empty-collection zero-state; (integration, mocked transport) pagination aggregation to a correct count

    Effort: S | Dependencies: Discogs API Client & Auth | Status: Not Started

### v0.7.0: Preference System & Likes Migration
**Goal**: Replace the binary like/unlike system with a 4-state preference model (null/dislike/like/love) and migrate all existing like data into the new system.

**Context**: The Weekly Curator already uses Spotify playlists as preference buckets — separate playlists for "loved" and "disliked" tracks, fed into workflows as sources. This is a workaround for the lack of a first-class preference signal beyond binary like. The new system promotes these workarounds into a canonical Narada-owned preference with three explicit states plus null (unrated). Existing `TrackLike` data from Spotify and Last.fm seeds the initial preferences.

**What this unlocks**:
- Narada-owned preference state independent of any service
- Negative signal ("dislike") for filtering tracks out of workflows — "never show me this"
- Distinction between "like" (good) and "love" (always welcome) for finer-grained curation
- Foundation for preference-aware workflow nodes (v0.7.3) and Spotify playlist mapping (v0.7.2)

**Persona**: Weekly Curator — "show me loved tracks unplayed 6mo" becomes possible. Dislike filtering means workflow playlists never include tracks you've explicitly rejected.

**Key Design Decisions**:
- **TrackLike continues to exist** — it tracks per-service sync state (Spotify liked, Last.fm loved). `TrackPreference` is the unified Narada-owned canonical opinion. They coexist.
- **Source priority for conflict resolution**: manual > playlist_mapping > spotify_import/lastfm_import > migration. Manual preference always wins.
- **Migration strategy**: Spotify `is_liked=True` → `preference=like`. Last.fm `is_liked=True` → `preference=love` (Last.fm "love" is the highest explicit signal). When both exist, the higher preference wins.
- **Import side-effects**: After `ImportSpotifyLikesUseCase` writes a `TrackLike`, it also upserts `TrackPreference(state=like)` unless a higher preference already exists.

### v0.7.1: Tagging System
**Goal**: Add freeform tagging with optional namespaces to tracks, enabling user-defined categorization orthogonal to preferences.

**Context**: Preferences answer "how much do I like this?" Tags answer "what IS this?" — mood, energy, context, occasion, user-defined genre. The namespace convention (`mood:chill`, `energy:high`) provides optional structure while allowing uncategorized tags (`banger`, `learn on guitar`). Tags are Narada-owned, manually applied in this version, or bulk-imported from Spotify playlists in v0.7.2.

**What this unlocks**:
- Freeform categorization: `banger`, `mood:chill`, `energy:high`, `context:workout`, `vibe:late-night`
- Foundation for tag-based workflow filtering and sorting (v0.7.3)
- Playlist-to-tag mapping in v0.7.2

**Persona**: Weekly Curator — tags enable "all my chill tracks with 5+ plays" workflows. This is the categorization layer the user's existing Spotify playlists approximate but can't query across.

**Key Design Decisions**:
- **Normalization**: Tags stored lowercase, trimmed, consecutive whitespace collapsed. Max 64 chars. Allowed characters: `[a-z0-9 _-:]` after normalization.
- **Namespace convention**: `namespace:value` format, split on first colon only (`mood:chill:vibes` → namespace `mood`, value `chill:vibes`). Tags without a colon have no namespace (null).
- **No predefined taxonomy**: Namespaces are freeform. UI suggests common ones (`mood:`, `energy:`, `context:`, `genre:`) via autocomplete but does not enforce them.
- **Batch operations**: Must support bulk tag/untag at 15k scale — "tag all tracks in this playlist as mood:chill" as a single operation.

### v0.7.2: Spotify Playlist Mapping (Preference & Tag Import)
**Goal**: Designate specific Spotify playlists as preference or tag sources, so Narada reads them during sync to populate preferences and tags automatically.

**Context**: The Weekly Curator already has Spotify playlists that function as preference/tag sources — a "Loved" playlist, a "Disliked" playlist, playlists named after moods or vibes. Instead of manually tagging 15k tracks one by one, the user designates these playlists and Narada ingests the signals during sync. This is one-way read: Narada reads the playlist membership, not bidirectional.

**What this unlocks**:
- Bulk preference import from existing Spotify playlists ("Loved" playlist → all tracks get `preference=love`)
- Bulk tag import from playlists ("Workout Bangers" playlist → all tracks get `context:workout`)
- Incremental sync: re-reading the playlist picks up additions/removals
- Leverages years of existing curation investment without manual re-entry

**Persona**: Weekly Curator — leverages existing curation investment. Multiple Spotify playlists (due to API/playlist size limitations) map to single preference states or tags.

**Key Design Decisions**:
- **One-way read**: Narada reads the Spotify playlist but does not modify it. Setting `preference=love` in Narada does NOT add the track to the "Loved" Spotify playlist.
- **Conflict resolution for preferences**: When a track appears in both a "loved" and "disliked" mapped playlist, the most recently synced mapping wins. The UI warns about contradictions. Manual preference always overrides any import.
- **Removal tracking**: If a track is removed from a mapped playlist, the preference/tag that mapping set is cleared (tracked via `source` field). Manually-set preferences/tags are never auto-cleared.
- **Multiple mappings per playlist**: A single playlist can map to both a preference AND a tag (e.g., "Workout Loved" → `preference=love` AND `tag: context:workout`).

### v0.7.3: Workflow Integration & Quick Filters
**Goal**: Expose preferences and tags as workflow nodes (source, filter, sort) and add quick ad-hoc filtering in the web UI that can be saved as workflows.

**Context**: The data model exists (v0.7.0–v0.7.1), the data is populated (migration + v0.7.2 import), and now the user can DO things with it. New workflow nodes enable "loved tracks unplayed 6mo" and "all mood:chill tracks sorted by play count." Quick filters provide instant ad-hoc exploration without building a full workflow — with a "Save as Workflow" bridge for patterns you want to keep.

**What this unlocks**:
- "Show me loved tracks unplayed 6 months" as a workflow
- "Exclude all disliked tracks" as a standard filter in any workflow
- "All mood:chill tracks" quick filter in the Library page
- "Save as Workflow" from quick filter results for recurring curation
- Updated workflow templates demonstrating preference and tag nodes

**Persona**: Weekly Curator — this is the payoff. Every existing workflow that starts with `source.liked_tracks` can now be refined with preference levels and tags.

**Key Design Decisions**:
- **Parameterized workflows deferred**: Workflows with user-provided runtime parameters ("how many days back?") are a natural fit but add significant complexity. Deferred to v0.9.x (alongside LLM workflows, which also benefit from parameterization). v0.7.3 provides fixed-config nodes.
- **Quick filters are NOT workflows**: They are direct API queries (`GET /api/v1/tracks?preference=love&tag=mood:chill`). "Save as Workflow" serializes the filter state into a workflow definition — a one-way conversion. The saved workflow is independent.
- **Enricher nodes required**: Filter/sort nodes for preferences and tags need data on the TrackList metadata. `enricher.preferences` and `enricher.tags` follow the existing enricher pattern (like `enricher.play_history`).

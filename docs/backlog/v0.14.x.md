## User Workflow Context

Years of imports have accumulated mapping debt the user can now see and fix: tracks without mappings to a connector, stale connector metadata, and — since v0.10.2 made mapping history append-only — a change history worth displaying. This cycle ships the data-quality surface (manual mapping control, unmapped/stale detection, shared-table concurrency) and the Rekordbox connector: locally-owned, purchased music unified with streaming history, carrying the audio-quality metadata (BPM, key, codec, lossless) no streaming API provides. Pre-gate positioning: this is the "trust your library" hardening that should exist before other people are invited in (v1.0.0).

Persona view: the **Weekly Curator** fixes mappings and trusts automated results; the **Tinkerer** gets Rekordbox — data sovereignty in its purest form — and workflow filters like "lossless only, BPM 120–130"; the **Casual** listener benefits indirectly (fewer wrong-song moments).

Cross-cutting contract (stated once — per-epic Specs reference it): every new use case this cycle must be classified in the chat-tool parity registry (`src/application/tools/registry.py`) — CI (`tests/unit/application/tools/test_registry_parity.py`) fails on any unclassified use case. Reads get a `ToolSpec` or an explicit classification; mutations additionally get two-phase confirmation wiring like the existing `*_write` dispatchers.

### v0.14.0: Data Quality
**Goal**: Build data quality tools for the existing library. With two connectors already producing mapping scenarios (Spotify + Last.fm), the DQ tooling is useful immediately — and becomes the foundation that new connectors benefit from.

**Persona**: Weekly Curator — bad data means bad playlists. Fixing mappings, finding unmapped tracks, and detecting stale metadata are core to trusting automated results.

#### Data Quality Epics

- [ ] **Unmapped Tracks Query**
    - Effort: S
    - What: `GetUnmappedTracksUseCase` — find tracks without mappings to specific connector
    - Why: Identify gaps like "tracks liked on Spotify but not mapped to Last.fm"
    - Dependencies: None
    - Status: Not Started
    - Notes:
        - Input: connector_name (optional), limit, offset
        - Output: Paginated list of tracks without mappings

- [ ] **Stale Tracks Detection**
    - Effort: S
    - What: `GetStaleTracksUseCase` — identify tracks with outdated connector metadata
    - Why: Automated data quality monitoring
    - Dependencies: None
    - Status: Not Started
    - Notes:
        - Configurable staleness threshold (e.g., 30 days)
        - Could trigger refresh workflows

- [ ] **Optimistic Concurrency on Shared Tables**
    - Effort: M
    - What: Add `version` columns and `WHERE version = :expected` guards to shared (non-user-scoped) tables that multiple users write to concurrently
    - Why: Shared tables (`connector_tracks`, `connector_playlists`, `track_metrics`) are the global metadata cache — two users importing the same Spotify track simultaneously could silently overwrite each other's connector data. User-scoped tables are protected by isolation; shared tables are not.
    - Dependencies: v0.6.2 Optimistic Concurrency on Tracks (establishes the pattern)
    - Status: Not Started
    - Notes:
        - Tables: `connector_tracks`, `connector_playlists`, `track_metrics`, `playlist_mappings`
        - Pattern established in v0.6.0: `Track.version: int = 0` on domain entity, `DBTrack.version` column, mapper reads actual version from DB
        - Extend to `ConnectorTrack`, `ConnectorPlaylist`, `TrackMetric` domain entities
        - Repository `upsert()` for shared tables: `SET version = version + 1 ... WHERE version = :expected`, raise `ConcurrentModificationError` on conflict
        - Retry strategy: on conflict, re-read from DB and retry (shared metadata is append/update, not user-critical — retry is safe)
        - Consider: `SELECT ... FOR UPDATE SKIP LOCKED` for high-contention connector track upserts as an alternative to optimistic retry

- [ ] **Manual Track Mapping & Data Quality Management**
    - Effort: L
    - What: User control over track mapping — manual overrides, duplicate merging, confidence-based review
    - Why: Music services disagree on identity; users need authority over their library
    - Dependencies: None
    - Status: Not Started
    - Notes:
        - API: `PATCH /tracks/{id}/mappings/{id}`, `DELETE /tracks/{id}/mappings/{id}`, `GET /connectors/{connector}/search`
        - Frontend: Manual mapping correction UI on Track Detail, "Unmapped" filter in Library, dashboard DQ alerts
        - Interactive step-through for bulk operations

### v0.14.1: Rekordbox Connector
**Goal**: Import locally-owned music from Rekordbox and enrich tracks with audio quality metadata unavailable from streaming APIs.

**Context**: Rekordbox is a local music library, not a cloud API. It represents tracks the user has *purchased or owns* — a different class of data from streaming services. The connector reads Rekordbox's local database (SQLite) or XML export, matches tracks to existing canonical entries, and enriches them with metadata that streaming APIs don't provide: BPM, musical key, codec, bitrate, lossless status. This is data sovereignty in its purest form — your purchased music collection unified with your streaming history.

**Key differences from streaming connectors**: No HTTP client, no OAuth, no rate limiting. Instead: local file parsing (like Spotify GDPR import). Identity resolution via ISRC (Rekordbox stores ISRCs), artist+title fallback, or MusicBrainz ID. The connector is read-only — mixd doesn't write back to Rekordbox.

**Precedent**: `spotify/personal_data.py` + `spotify/play_importer.py` already parse local files (Spotify GDPR export). Rekordbox follows the same pattern: parse local source → resolve to canonical tracks → enrich with source-specific metadata.

#### Rekordbox Connector Epics

- [ ] **Rekordbox Library Parser**
    - Effort: M
    - What: Parse Rekordbox's SQLite database (`master.db` in Rekordbox 6+) or XML export (`rekordbox.xml`) into typed domain objects
    - Why: Foundation for all Rekordbox operations — must handle both storage formats
    - Dependencies: None
    - Status: Not Started
    - Notes:
        - Rekordbox 6+ uses SQLite (`master.db`) — preferred path, richer data
        - Rekordbox 5 and user exports use XML — fallback for compatibility
        - Pydantic models at the boundary (like all other connectors)
        - Extract per-track: title, artist, album, BPM, key, codec, bitrate, duration, file path, file size, ISRC (if tagged), rating, color, date added, play count
        - File path detection: user provides Rekordbox library path or mixd auto-detects common locations

- [ ] **Rekordbox Track Resolution & Matching**
    - Effort: M
    - What: `RekordboxInwardResolver` — match Rekordbox tracks to existing canonical tracks
    - Why: Connect owned/purchased tracks to streaming library for unified view
    - Dependencies: Rekordbox Library Parser
    - Status: Not Started
    - Notes:
        - ISRC-first (many purchased tracks have ISRCs in file tags)
        - Artist+title fallback via existing `BaseMatchingProvider` pattern
        - MusicBrainz ID if embedded in file tags
        - Duration cross-check to reduce false positives (local files have exact durations)
        - Reuse: `_shared/inward_track_resolver.py`, `_shared/matching_provider.py`, `_shared/isrc.py`

- [ ] **Audio Quality Enrichment**
    - Effort: S
    - What: New enricher node (`enricher.audio_quality`) that attaches Rekordbox-sourced metadata to tracks
    - Why: BPM, key, codec, lossless status are unavailable from streaming APIs — only local files have this data
    - Dependencies: Rekordbox Track Resolution & Matching
    - Status: Not Started
    - Notes:
        - Metrics added to track metadata: `bpm`, `musical_key`, `codec`, `bitrate`, `is_lossless`, `file_size_bytes`
        - Enables workflow filters: "lossless only", "BPM 120–130", "key of Cm"
        - Enables workflow sorters: sort by BPM, sort by bitrate
        - `is_lossless` derived from codec (FLAC, ALAC, WAV, AIFF → true; MP3, AAC, OGG → false)

- [ ] **Rekordbox Connector Facade & Registration**
    - Effort: S
    - What: `RekordboxConnector` — read-only connector that plugs into the connector ecosystem
    - Why: Lets existing use cases and workflows access Rekordbox data transparently
    - Dependencies: Audio Quality Enrichment
    - Status: Not Started
    - Notes:
        - Read-only: implements `TrackMetadataConnector` but NOT `PlaylistConnector` or `LikedTrackConnector`
        - No `aclose()` needed (no HTTP client to clean up)
        - Config: `rekordbox_library_path` in settings

- [ ] **Rekordbox Connector Tests**
    - Effort: M
    - What: Unit tests for parser (both SQLite and XML), integration tests for resolution
    - Why: File parsing is fragile — Rekordbox schema changes between versions
    - Dependencies: All above epics
    - Status: Not Started
    - Notes:
        - Test fixtures: sample SQLite DB and XML export with known tracks
        - Test ISRC resolution, artist+title fallback, duration cross-check
        - Test codec → lossless derivation

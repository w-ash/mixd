# v0.6.x: Multi-User Data Isolation

For strategic overview, see the [planning overview](README.md).

**Supersedes**: The former v1.0.x "User Authentication System" epic (deleted). Neon Auth already handles authentication (JWT validation, session management, email allowlist). This milestone is purely about **data isolation** and **per-user service connections** — the missing piece that prevents sharing the deployed instance.

**Multi-Layer Isolation Strategy**: Application-level filtering (WHERE clauses in repositories) is the primary isolation mechanism. PostgreSQL Row-Level Security (RLS) is the defense-in-depth safety net — even if a repo method misses a WHERE clause, the database itself won't return other users' rows. The implementation is phased: v0.6.0 creates RLS policies (inert), v0.6.2 sets `app.user_id` session variable per-transaction via `SET LOCAL` + contextvar (RLS active but table owner bypasses), v0.6.4 applies `FORCE ROW LEVEL SECURITY` (RLS enforced on table owner, after isolation tests pass). Uses portable PostgreSQL features (`current_setting()`, `SET LOCAL`) — no extensions required.

### v0.6.0: Schema + User Identity Foundation
**Goal**: Add `user_id` columns to all user-owned tables and establish the `CurrentUser` dependency for extracting identity from Neon Auth JWTs.

**Context**: Currently all 18 database tables are single-user. `BusinessLimits.DEFAULT_USER_ID = "default"` is hardcoded throughout. The Neon Auth middleware already validates JWTs and attaches `scope["auth_user"]` with claims including `sub` (user ID) and `email`, but no route handler reads these claims. This phase adds the schema foundation and identity extraction without changing application behavior.

**What this unlocks**:
- Data written from this point forward is tagged with the real user ID
- Foundation for repository filtering (v0.6.2) and per-user OAuth (v0.6.3)
- Existing `'default'` data can be claimed by a real user via backfill (v0.6.4)

**Persona**: Weekly Curator — invisible change (behavior unchanged), but data is now tagged correctly for future multi-user enforcement.

**Key Design Decisions**:
- **No local `users` table** — Neon Auth manages user identities externally. We only need the JWT `sub` claim as a string foreign key. No user CRUD, no password management, no session table.
- **`user_id` is a string** — matches Neon Auth JWT `sub` claim format (UUID string). Not an integer FK to a local users table.
- **Auth disabled (local dev) = `DEFAULT_USER_ID`** — when `NEON_AUTH_URL` is empty, `get_current_user_id()` returns `"default"`. Local dev operates as single-user seamlessly.

**Data Ownership Model**:

| Category | Tables | user_id column? |
|----------|--------|-----------------|
| **Shared** (canonical metadata cache) | `connector_tracks`, `track_metrics`, `connector_playlists` | No |
| **User-scoped** (owned data) | `tracks`, `track_mappings`, `match_reviews`, `track_likes`, `track_plays`, `playlists`, `workflows`, `oauth_tokens`, `user_settings`, `connector_plays` | Yes (added) |
| **Transitively scoped** (FK to user-scoped parent) | `playlist_tracks`, `playlist_mappings`, `workflow_runs`, `workflow_run_nodes`, `workflow_versions` | No (scoped via FK) |
| **Already has user_id** | `sync_checkpoints` | Exists |

**Context**: Neon Auth handles sign-up and sign-in, but the app had no way to view your profile, log out, or self-service delete your account. ### v0.6.2: Repository + Use Case Scoping
**Goal**: Thread `user_id` through the full request lifecycle — route handlers extract it, use cases pass it to repos, repos filter queries by it. After this phase, users see only their own data.

**Context**: The schema has `user_id` columns (v0.6.0) but nothing enforces isolation yet. This phase adds `user_id` parameters to repository protocols, implements filtering in all repo queries, threads `user_id` through use case Commands, and injects it from route handlers. This is the largest phase — it touches all layers of the architecture — but each individual change is mechanical.

**What this unlocks**:
- User A cannot see User B's playlists, likes, plays, or workflows
- Dashboard stats are per-user
- Workflow execution operates on the current user's data only
- Sharing the deployed instance with friends becomes safe

**Persona**: All personas — data isolation is the prerequisite for any multi-user scenario. The Weekly Curator's data is protected when friends sign up.

**Key Design Decisions**:
- **Reads filter by user_id, writes carry it on the entity** — read methods (`list_playlists`, `get_all_liked_tracks`) take an explicit `user_id` parameter. Write methods (`save_playlist`) get user_id from the entity's field (set by the use case).
- **ID-based lookups verify ownership** — `get_playlist_by_id(id, user_id)` returns 404 if the playlist belongs to a different user. Prevents IDOR via guessed IDs.
- **Template workflows use NULL user_id** — seeded templates (`is_template=True`) have `user_id=None` and are visible to all users. `list_workflows(user_id)` returns `WHERE user_id = ? OR user_id IS NULL`.
- **CLI always uses DEFAULT_USER_ID** — the CLI is the instance owner's tool. No JWT context available.
- **list_tracks liked_track_ids** — the `TrackRepository.list_tracks()` method joins to `track_likes` for the `liked_track_ids` set. This subquery must be scoped to the current user.
- **Defense-in-depth with RLS** — Application-level WHERE clauses are the primary isolation mechanism. PostgreSQL RLS policies (created in v0.6.0) act as a safety net via `SET LOCAL app.user_id` at transaction start. If a repo query forgets the filter, RLS catches it. Standard practice in multi-tenant PostgreSQL applications.

### v0.6.3: Per-User OAuth
**Goal**: Each user connects their own Spotify and Last.fm accounts. OAuth tokens are stored per-user, and connector operations use the current user's credentials.

**Context**: Currently `oauth_tokens` has `UNIQUE(service)` — one Spotify token, one Last.fm token, per database. A second user literally cannot authenticate with Spotify because the INSERT would conflict. This phase makes token storage user-scoped and threads user context through the connector auth flows.

**What this unlocks**:
- Each user connects their own Spotify/Last.fm accounts
- Each user sees their own connection status on the Integrations page
- Import operations use the current user's credentials
- Friends can sync their own liked tracks and play history

**Persona**: All personas — connecting your own service accounts is the minimum for a usable shared instance.

**Key Design Decisions**:
- **User ID in CSRF state** — the OAuth callback needs to know which user initiated the flow. Encode user_id in the CSRF state token (the `state` parameter), or recover it from the session cookie present on the redirect.
- **Session cookie preferred for callback** — Neon Auth's session cookie is present on the callback redirect. The middleware can extract user identity from it. This avoids encoding user_id in the state parameter (which could be tampered with).
- **Token storage uses standalone sessions** — `DatabaseTokenStorage` creates its own short-lived sessions (not UoW) because token refresh happens inside httpx auth flows with no UoW context. This pattern stays the same, just with an added `user_id` parameter.

### v0.6.4: Testing + Data Purge
**Goal**: Comprehensive multi-user isolation tests, test fixture updates, FORCE RLS defense-in-depth, and clean-slate purge of legacy `user_id='default'` data.

**Context**: After v0.6.0–v0.6.3, the application is multi-user capable. This phase proves correctness through isolation testing and completes the RLS enforcement. Early dev mode — clean breaks over backward compatibility, so legacy data is purged rather than migrated.

**What this unlocks**:
- Confidence that data isolation actually works (20 isolation tests across 10 repository areas)
- FORCE RLS on all 11 user-scoped tables — defense-in-depth even for the table owner role
- Clean test factories with explicit `user_id` support for future multi-user test scenarios
- Clean database — no orphaned `user_id='default'` rows

**Persona**: Weekly Curator — starts fresh with a real Neon Auth identity. No legacy data migration needed.

**Key Design Decisions**:
- **Two-user isolation tests** — insert as user_a, query as user_b, verify empty. Covers tracks, likes, plays, playlists, workflows, checkpoints, stats, and OAuth tokens.
- **Factory defaults match entity defaults** (`"default"`) — zero cascading test changes. Isolation tests use explicit user_ids. Two-tier fixture design: existing `db_session` for SQL correctness, future `rls_session` for true RLS testing.
- **FORCE RLS on all 11 tables** — backlog originally said 8, but migration 007 applied RLS to 11 tables. FORCE applied to all for consistency.
- **Clean-slate purge** over data claim — Alembic data migration deletes `user_id='default'` rows. No claim-data command needed.
- **Bug found and fixed** — `TrackRepository.save_track()` wasn't including `user_id` in its INSERT values dict; all tracks were saved with `user_id='default'` regardless of the entity's `user_id` field.

**Findings for future work**:
- Template workflows use `user_id='default'` (NOT NULL), not shared via NULL. The `list_workflows` query's `OR user_id IS NULL` clause is dead code. Making templates truly shared requires nullable `user_id` on workflows.
- Resolver `_save_with_connector_mappings` has the same user_id propagation gap (creates Track entities with default `user_id` instead of the passed parameter).

### v0.6.8: Resilient Imports & Real-Time Progress
**Goal**: Fix a critical data loss bug where long-running imports lose all progress when the Fly.io machine auto-stops, and improve progress reporting so users get immediate visual feedback during imports.

**Context**: First real user triggered a Spotify likes import (~1200+ liked tracks). The backend successfully fetched all tracks from the Spotify API (batches of 50, offset 0 → 1150 over ~3 minutes), but the Fly.io machine auto-stopped mid-import (`"API server shutting down — cancelling background tasks"`). **Zero tracks were persisted to the database.** Neon confirmed: 0 tracks, 0 likes, 0 plays — only the 2 OAuth tokens survived.

Root cause: all import use cases run as a single PostgreSQL transaction that only commits at the very end (`uow.commit()` on the last line of the use case). When the machine dies (Fly auto-stop, user navigates away dropping the SSE connection, network hiccup), the UoW `__aexit__` sees a `CancelledError` and rolls back the entire transaction — every batch fetched is lost.

Secondary issue: the UI showed "Connecting..." for the entire 3+ minute import instead of showing per-batch progress, giving no indication that work was happening.

**What this unlocks**:
- Imports survive machine restarts — data persists incrementally, at most one batch lost on crash
- Re-running an interrupted import picks up where it left off (checkpoint + duplicate detection)
- Users see real-time batch-level progress: "Fetched 500 of ~1200 tracks (10 batches)"

**Persona**: The Casual Enthusiast — importing a large Spotify library should feel safe and progressive, not a gamble that silently loses everything on any interruption.

**Key Design Decisions**:
- **`commit_batch()` not `flush_and_commit()`** — renamed per reviewer feedback. `flush_and_commit` conflates SQLAlchemy's `flush()` (sends SQL, keeps transaction open) with `commit()` (ends transaction, durable). The method calls `session.commit()` — a real PostgreSQL COMMIT — and intentionally does NOT set `_committed = True` so the UoW `__aexit__` auto-commit still fires on clean exit (a benign no-op).
- **Not on the domain Protocol** — `commit_batch()` is an infrastructure-level method on `DatabaseUnitOfWork` only, not on `UnitOfWorkProtocol`. The domain Protocol should only express `commit()` and `rollback()`. Use cases access it through the concrete type or a narrower application-layer protocol.
- **Rollback contract is voided** — after `commit_batch()`, rollback cannot undo committed batches. Callers MUST NOT rely on rollback for already-committed data. All writes must be idempotent (upserts via `ON CONFLICT`).
- **Checkpoint per batch** — checkpoint advances at the same granularity as commits, not every 10 batches. If the machine dies after batch 7, the checkpoint reflects batch 7 and re-run resumes from there.
- **`expire_on_commit=False` dependency** — the session config has `expire_on_commit=False` (confirmed in `db_connection.py`), which prevents `DetachedInstanceError` on ORM objects held across commit boundaries. `commit_batch()` docstring must note this dependency.

### v0.6.9: SSE Auth & Connection Resilience
**Goal**: Fix a production bug where all SSE progress streams (likes import, history import, workflow execution) fail silently because the SSE client doesn't send the Neon Auth Bearer token, and add connection resilience so failures are surfaced instead of hanging the UI.

**Context**: Spotify likes import hangs on "Connecting..." in production. Root cause: `connectToSSE()` in `web/src/api/sse-client.ts` used raw `fetch()` without an `Authorization` header. In production, `NeonAuthMiddleware` intercepts all `/api/` endpoints and returns 401 for unauthenticated requests. The SSE endpoint got rejected, but the frontend hooks had no mechanism to detect the failure — `useOperationProgress` ignored the `error` state from `useSSEConnection` and never transitioned out of the "pending" state.

This affected ALL SSE-based features (likes import, history import, likes export, workflow execution, playlist sync) — any operation that uses the real-time progress stream.

**What this unlocks**:
- SSE progress streams work in production (were completely broken since Neon Auth was added in v0.6.0)
- Connection failures show "Connection failed" instead of hanging forever
- Premature stream closure shows "Connection lost" instead of hanging
- 30-second connection timeout prevents indefinite hangs on unreachable servers

**Persona**: The Casual Enthusiast — clicking "Import" should show progress or a clear error, never an infinite spinner.

**Key Design Decisions**:
- **Shared `getAuthToken()` helper** — extracted from duplicated auth blocks in `customFetch` and `connectToSSE` into `auth.ts`. Single callsite for token retrieval prevents future divergence.
- **Connection timeout, not stream timeout** — the 30s timeout covers only the initial HTTP response (server reachable?), not the event stream. Imports can run for minutes; the stream lifetime is controlled by the caller's `AbortController`.
- **`failIfActive()` guard** — both `onStreamEnd` and `useEffect(error)` use a shared updater that only transitions to "failed" if the operation is still `pending` or `running`. This prevents clobbering a terminal state (completed/failed/cancelled) that arrived via a race.
- **No `AbortSignal.any()`** — the timeout and caller abort route through a single `AbortController` instead. `AbortSignal.any()` has cross-realm issues in jsdom test environments.

### v0.6.10: Neon Platform Integration & Multi-User Hardening
**Goal**: Adopt Neon's 2026 CI/CD features (branching, schema diff), add Neon Auth webhooks, harden for multi-user traffic, and modernize the frontend toolchain.

**Context**: Mixd is no longer a single-user app. Several assumptions from the early codebase — scale-to-zero deployment, unlocked JWKS cache, unprotected child tables, `@/` import alias — needed updating. Neon's early-2026 features (branch-per-PR, schema diff action, auth webhooks) enabled tighter CI integration. The frontend was also overdue for TypeScript 6.0 and Node.js subpath imports.

**What this unlocks**:
- CI tests run against real Neon branches (not ephemeral containers) — validates migrations against production schema shape
- Schema changes auto-post as PR comments for reviewer visibility
- Neon Auth webhook endpoint ready for signup validation and user lifecycle events
- JWKS cache handles concurrent users without thundering-herd fetches
- `track_metrics` and `playlist_mappings` have direct RLS policies (defense-in-depth)
- Frontend uses standards-based `#/` imports (Node.js subpath imports, no bundler magic)
- TypeScript 6.0 with ES2025 target

**Persona**: The Tinkerer — CI improvements and infrastructure hardening serve the developer experience. The Casual Enthusiast — always-on machine means signup never times out.

**Key Design Decisions**:
- **One Neon branch, two purposes** — CI branch serves both test execution AND schema diffing, avoiding the cost of creating two branches per PR.
- **EdDSA, not HMAC** — Neon Auth webhooks use the same Ed25519/JWKS infrastructure as JWT validation, so the webhook endpoint reuses `get_jwk_set()` from `auth_gate.py`.
- **Double-check lock on JWKS** — fast path reads without lock; slow path uses `asyncio.Lock` with inner re-check. Matches existing patterns (`SpotifyTokenManager._refresh_lock`, `SSEProgressRegistry._lock`).
- **Selective child table RLS** — only `track_metrics` and `playlist_mappings` get `user_id` + RLS (queried independently in repos). The remaining 6 tables are always accessed through user-scoped parent FKs and don't need direct policies.
- **`#/` over `@/`** — Node.js subpath imports (`"imports": {"#/*": "./src/*"}`) are a standard, unlike `@/` which requires bundler-specific config. TypeScript 6.0 resolves `#/` natively.
- **No upper version bounds** — dependency ceilings (e.g., `<3`) removed from pyproject.toml. They caused more harm (blocked security patches) than good (breaking changes are caught by CI).

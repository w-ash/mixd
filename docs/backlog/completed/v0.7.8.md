### v0.7.8: Mobile Responsiveness

**Goal**: Mixd works as a first-class app from a phone ŌĆö every page reads well,
every primary action is reachable, and nothing requires horizontal scrolling on
iPhone-class widths. The responsive pass is *measured*: every primary page gains a
Playwright visual-regression baseline at iPhone 15 Pro and 1280px desktop, and any
unintended layout drift fails CI. The one explicit exception is the workflow graph
editor: on a screen too small to operate React Flow comfortably, the editor refuses
gracefully with a "open this on a larger screen" placeholder that points the user
to runs/history instead.

**Context**: The current web UI was built desktop-first (v0.3.x onward) and has
never had a deliberate responsive pass. Most pages technically render on mobile,
but several break in real use ŌĆö data tables overflow horizontally, filter rows
push content off-screen, dialogs spill below the keyboard, and the workflow editor
is unusable. The Casual Enthusiast persona is mobile-first by definition (phone is
their primary device); the Curator persona reaches for their phone after a sync to
confirm "did it work?" without booting a laptop. v0.7.8 closes that gap before
v0.8.x scheduling, which leans heavily on the "wake up, glance at phone, confirm
overnight runs succeeded" loop. Couplefins (the sibling project) has working
`BottomSheet`, `BottomNav`, `useIsMobile`, and `useDialogSync` primitives; the
implementer ports those rather than reinventing them. v0.7.8 also lifts mixd's
testing posture by adding pixel-perfect visual regression baselines to Playwright
ŌĆö a 2026-standard guard against silent layout drift.

**What this unlocks**:
- Phone-first review of imports, syncs, runs, library browsing, tag management, and settings
- Casual Enthusiast persona becomes daily-active without ever opening a laptop
- Foundation for the v0.8.x scheduling experience (post-run notifications + glanceable status are inherently mobile use cases)
- Portable design tokens ŌĆö the responsive patterns we settle on here become the baseline for every page added in v0.8+
- Visual regression baselines that protect every future change (a typo in a Tailwind class can no longer silently shift the dashboard)
- Explicit, defensible exemption for the workflow editor ŌĆö sets the precedent that "graph-canvas UX is desktop-only" rather than chasing degraded mobile graph interactions

**Persona**: Casual Enthusiast ŌĆö phone is the primary device. Weekly Curator
secondary ŌĆö uses phone for status checks, laptop for deep curation. Tinkerer
tertiary ŌĆö self-host UX should feel polished on any viewport.

**Manual test**:
1. Open mixd on an iPhone-class viewport (Safari, ~390px wide) ŌåÆ sign in ŌåÆ land on the dashboard with no horizontal scrolling, all primary nav reachable.
2. Tap into Library ŌåÆ filters apply via a mobile-appropriate surface (bottom sheet, not desktop sidebar) ŌåÆ track list scrolls cleanly with each row's primary action reachable by thumb.
3. Run a Spotify sync ŌåÆ progress events render legibly ŌåÆ completion toast renders above the bottom nav, deep-link to Operation Run Log works.
4. Open Tags ŌåÆ manage tags (rename, merge, delete) ŌåÆ confirm dialogs render as bottom sheets, not center modals.
5. Open Workflows ŌåÆ see the list and run history ŌåÆ tap into a workflow's runs ŌåÆ confirm the run-detail page is fully usable on mobile.
6. Tap "Edit workflow" ŌåÆ land on a placeholder screen that explains the editor is desktop-only and offers a "View runs" CTA back to the run history. **The graph canvas is never rendered on mobile.**
7. Rotate to landscape on a phone ŌĆö content reflows, no broken layout.
8. iPad portrait (~820px wide) ŌĆö confirm bottom-nav + sheet behaviour applies (mobile shell extends to `lg:`). iPad landscape (~1180px) ŌĆö confirm desktop shell.
9. iPhone-class viewport with the on-screen keyboard up (focus a search field) ŌĆö input remains visible, layout uses `svh`/`dvh` so the address bar's show/hide doesn't jank.
10. iPhone with notch (Safari) ŌĆö the bottom-nav sits *above* the home indicator, not under it (safe-area insets working).
11. `pnpm --prefix web test:e2e` ŌĆö visual regression suite passes; intentional design changes require an explicit baseline-update PR.

**Key Design Decisions**:
- **One breakpoint matters: `lg:` (1024px)** ŌĆö below = mobile shell; at-or-above = desktop shell. iPad portrait (Ōēż1023px) gets the mobile experience; iPad landscape gets desktop. This is *not* the couplefins convention (couplefins uses `sm`/`md`); we're choosing the simpler one-breakpoint model and accepting that iPad portrait users will see bottom-nav. Container queries (`@container`) handle within-component density at intermediate widths so iPad portrait doesn't feel "phone-scaled-up."
- **Mobile-first refactor, desktop unchanged** ŌĆö Tailwind breakpoint conventions flip from `sm:/md:/lg:` overrides on desktop styles to mobile baseline + `lg:` desktop overrides. Existing desktop pixel-perfect layouts are preserved; the change is additive for narrow viewports.
- **Workflow editor explicitly opts out** ŌĆö the React Flow canvas is unusable below ~1024px and degraded interactions (touch-friendly node dragging, mobile minimap) are not worth the implementation cost. Below the breakpoint, render a placeholder component with copy ("Workflow editing requires a larger screen") and a CTA to the run history page. List/browse/run flows for workflows remain fully mobile-supported.
- **Port from couplefins, don't invent** ŌĆö couplefins's `useIsMobile`, `useDialogSync`, `BottomSheet`, and `BottomNav` are the source of truth. The implementer audits couplefins first, ports patterns directly, and only deviates with a written reason. Couplefins paths: `couplefins/web/src/hooks/useIsMobile.ts`, `couplefins/web/src/hooks/useDialogSync.ts`, `couplefins/web/src/components/BottomSheet.tsx`, `couplefins/web/src/components/BottomNav.tsx`, `couplefins/web/src/layouts/AppLayout.tsx`. Note: couplefins uses a `sm` cutoff; mixd ports the *primitives* but applies them at the `lg:` cutoff.
- **Bottom sheet over center modal on mobile** ŌĆö confirm dialogs, filter panels, and detail drawers render as bottom-anchored sheets below `lg:`. Implemented via a new `ResponsiveDialog` primitive that switches between center `Dialog` (desktop) and bottom `Sheet` (mobile) with the same children contract. The `Sheet` primitive is ported from couplefins's `BottomSheet` (native `<dialog>` + `useDialogSync`); we are *not* adding the shadcn `Sheet` from scratch.
- **Touch targets Ōēź 44px (WCAG 2.5.5 / Apple HIG)** ŌĆö every interactive element on mobile meets the AAA minimum, exceeding the WCAG 2.5.8 AA legal floor of 24px. Existing `icon-xs` / `icon-sm` button variants get padded hit-targets on mobile via a `mobile-tap-area` utility rather than being visually resized.
- **Persistent visibility for the post-run toast** ŌĆö sonner toast container is repositioned: bottom-right on desktop, top-center on mobile (above bottom-nav). z-index audited so the deep-link action is always tappable.
- **Visual regression in CI via Playwright `toHaveScreenshot`** ŌĆö every primary page gets a baseline at iPhone 15 Pro (390├Ś844) and desktop (1280├Ś800), light and dark themes. Baselines are generated in CI's Docker image (Playwright's official image) to lock font rendering. Diff threshold: `maxDiffPixelRatio: 0.005` (0.5%). On regression, CI uploads the diff image as an artifact; intentional changes require an explicit `--update-snapshots` PR. No external SaaS ŌĆö everything stays in-tree under `web/e2e/__screenshots__/`.
- **Safe-area insets for iPhone notch / home indicator** ŌĆö viewport meta gets `viewport-fit=cover`. New Tailwind utilities `pb-safe`, `pt-safe`, `px-safe` map to `env(safe-area-inset-*)`. Bottom-nav and any sticky-bottom surface use `pb-safe`. Bottom sheets respect `safe-area-inset-bottom` for the close-handle gutter.
- **Dynamic viewport units (`svh`/`dvh`)** ŌĆö full-height surfaces (mobile shell, bottom sheets in fullscreen mode) use `100svh` (small viewport, fits-on-load) instead of `100vh`. Modals that should grow with the visible viewport use `100dvh`. iOS Safari address-bar show/hide no longer causes layout jank.
- **Container queries for component-scoped responsive** ŌĆö data tables on Library, Playlists, ImportHistory, and WorkflowRunDetail use `@container` (Tailwind v4 native) to collapse to card lists based on the *content area's* width, not the viewport. This makes iPad portrait look intentional rather than phone-scaled-up.
- **View Transitions API for route changes** ŌĆö React Router v7's `<Link viewTransition>` (and `<NavLink viewTransition>`) prop is applied on primary navigation surfaces (Library, Playlists, Workflows list rows, Sidebar, MobileBottomNav). Reduced-motion preferences are respected via a `prefers-reduced-motion` override on `::view-transition-*` pseudo-elements. Per-route names suffixed with stable IDs to avoid the "duplicate name crashes the transition" gotcha.
- **No new runtime dependencies** ŌĆö Tailwind v4, Radix, React Router v7, and the ported couplefins primitives cover everything. Visual regression uses Playwright's built-in `toHaveScreenshot`. If something doesn't fit, escalate before adding a dep.
- **Test viewport matrix** ŌĆö Playwright gains `iPhone 15 Pro` and `iPad Air landscape` projects alongside the existing desktop project. Vitest gets a `useIsMobile` mock helper and a `mockMatchMedia(width)` test utility so component-level responsive branches are verifiable without a browser.

- [x] **View Transitions API for Route Changes** *(2026-05-09)*

    **Story**: When the user taps from Library to a track detail
    page, the transition feels native ŌĆö no flash of empty layout,
    no jarring full-page swap. The same applies to sidebar/bottom-
    nav navigation. Reduced-motion users get instant transitions
    instead.

    **Decisions**:
    - React Router v7's `<Link viewTransition>` (and `<NavLink
      viewTransition>`) prop is applied on primary navigation surfaces:
      Library/Playlists/Workflows list-row links and the Sidebar +
      MobileBottomNav nav links. The prop wraps each navigation in
      `document.startViewTransition()`.
    - `prefers-reduced-motion: reduce` respected via a `theme.css`
      override on the `::view-transition-*` pseudo-elements ŌĆö those
      live on a separate tree under `:root` and aren't reached by the
      existing `*` blanket selector, so they need their own rule.
    - Browser support: Chrome/Edge/Opera + Safari 18+ (~75% global as
      of May 2026). Non-supporting browsers no-op gracefully ŌĆö falls
      back to instant navigation.
    - Per-route names suffixed with stable IDs (`track-${id}`,
      `playlist-${id}`) to avoid the "duplicate name crashes the
      transition" gotcha.

    **Spec**:
    - Modified: `web/src/components/layout/Sidebar.tsx` and
      `web/src/components/layout/MobileBottomNav.tsx` ŌĆö `<NavLink
      viewTransition>` on every nav link.
    - Modified: `web/src/pages/Library.tsx`, `Playlists.tsx`,
      `Workflows.tsx` ŌĆö list-row `<Link viewTransition>` on the
      title-link in both card and table renders.
    - Modified: `web/src/theme.css` ŌĆö add `@media
      (prefers-reduced-motion: reduce) { ::view-transition-group(*),
      ::view-transition-old(*), ::view-transition-new(*) { animation:
      none !important; } }`.
    - Modified: `web/src/pages/Library.tsx` and `Playlists.tsx`
      ŌĆö list rows tag themselves with stable
      `view-transition-name` per item.
    - Modified: `web/src/pages/TrackDetail.tsx` and
      `PlaylistDetail.tsx` ŌĆö receiving page tags the same name
      on the detail header.

    **Tests**:
    - (vitest) Reduced-motion media query disables the transition

- [x] **v0.7.8.16** (2026-05-10) ŌĆö structural JSONB encoder: register orjson at the psycopg driver layer (eliminates the v0.7.8.14 / .15 bug class)

    **What:** Registered orjson as the psycopg JSONB dumper via `psycopg.types.json.set_json_dumps(orjson.dumps)` at module import of `db_connection.py`. orjson natively encodes UUID, datetime, date, time, dataclasses, Enum, and numpy types ŌĆö a strict superset of any hand-rolled coverage we could maintain. psycopg 3.3 (ticket #569) accepts bytes directly from JSON dumpers, so `orjson.dumps` plugs in without a `.decode()` wrapper. Wishful column comments on `output_tracks` and `node_details` were rewritten to describe the layered safety. New integration test class `TestJsonbEncoderUuidDatetime` (5 cases: UUID, datetime, date+time, nested UUID, unsupported-type-raises) at `tests/integration/repositories/test_jsonb_roundtrip.py`. New integration test class `TestWorkflowRunJsonbWrites` (4 cases: realistic `build_playlist_changes` round-trip through `update_node_status`, raw UUID/datetime regression guard through `update_node_status`, realistic `serialize_output_tracks` round-trip through `update_run_status`, raw UUID regression guard through `update_run_status`) at `tests/integration/repositories/test_workflow_run_repository.py` ŌĆö fills the prior coverage gap where the workflow run repository had 421 lines of tests with zero JSONB-column assertions.

# v0.8.x: Scheduling & Advanced Workflow Features

For strategic overview, see the [planning overview](README.md).

The v0.8.x cycle modernizes the workflow engine (reliability hardening → Prefect removal) and then layers automation on top (workflow + sync scheduling), finishing with authoring polish (templates, import/export, sub-flows, playlist browse). Each sub-version below is an independently shippable, manually-testable slice. The cross-cutting substrate investigation that motivates v0.8.0–v0.8.2 is captured in the [workflow-execution-substrate decision memo](../architecture/workflow-execution-substrate.md).

---

## User Workflow Context

**What the Curator does today.** Every week, the Curator opens mixd, manually triggers their Last.fm and Spotify imports, waits for them to finish, then runs each smart-playlist workflow by hand and eyeballs the result. The ritual works — but it is entirely hands-on, and it depends on the user being at the keyboard at the right moment.

**Where it breaks down.**
- **No automation.** "Refresh my dinner-party playlist every Sunday morning with songs I loved this month" is a thing the user wants the *system* to do, not a chore they want to remember to do. Today there is no way to say it.
- **The run log isn't fully trustworthy.** A worker dying (OOM/SIGTERM) and a workflow's own logic failing both record as `failed`, so the user can't tell "mixd broke" from "my pipeline is wrong." A sweeper/worker race can overwrite a run's recorded outcome. A CPU-bound node can get a healthy run falsely reaped. Until the log is trustworthy, the user can't stop babysitting.
- **Misconfigured workflows fail silently.** A few validation gaps let a pipeline produce *wrong output without erroring* — the hardest failure to trace.
- **The engine is a poor scheduling substrate.** Prefect's embedded ephemeral server is a fragile base for the concurrent runs scheduling will drive (it's documented to hit stuck-states under concurrent flows).

**The cycle's job, in order.** (1) Make runs trustworthy and reject bad configs *(v0.8.0)*; (2) swap the engine to stdlib asyncio so scheduling sits on a clean substrate *(v0.8.1)*; (3) deliver scheduling CLI-first, then web, then background sync *(v0.8.2–v0.8.4)*; (4) close design-debt correctness + hardening gaps from the 2026-06 review *(v0.8.5–v0.8.6)*; (5) rearchitect the broken import/sync flow around one reconciliation primitive — backend + CLI then web UI *(v0.8.7–v0.8.8)*; (6) reduce authoring friction *(v0.8.9–v0.8.10)*; (7) enable manual playlist editing *(v0.8.11)*.

**Headline acceptance — the compound capability this cycle exists to deliver:**
> **Given** a Curator in their own timezone who has built a "Fresh Faves" workflow and connected Last.fm,
> **when** they schedule the Last.fm sync for "every night at 2am" and the workflow for "every Sunday at 6am,"
> **then** their data refreshes overnight and the playlist rebuilds Sunday morning — in *their* local time, surviving restarts and DST — and **if** an overnight run fails, they see it the next time they open mixd rather than discovering a stale playlist days later.

**Personas across the cycle.** Curator — primary (automates the weekly ritual). Tinkerer — a simpler engine to self-host and reason about; templates and sub-flows for complex pipelines. Casual — playlists stay fresh without effort (v0.8.4) and next-run visibility builds confidence (v0.8.3).

### v0.8.0: Run Reliability & Validation Hardening
**Goal**: Make workflow run history trustworthy and reject misconfigured workflows before they run.

**Context**: Auditing the workflow engine (against Prefect's own source) surfaced latent run-state bugs and a class of silent-wrong-result validation gaps. These live in `workflow_runs.py` / the sweeper / `validation.py` / the executor's cleanup path — and ship independently of the engine swap, delivering user-visible trust first and de-risking the swap (v0.8.1) and scheduling (v0.8.2+) that follow.

**What this unlocks**: A run marked `failed` means "the workflow logic broke"; `crashed` means "the worker died" — distinct triage. No more last-writer-wins corruption of a run's recorded outcome, no false-positive reaps of healthy runs, no leaked HTTP pools when a deploy interrupts cleanup, and misconfigured pipelines fail loudly at validation instead of silently producing wrong output.

**Persona**: Curator (trusts the run log enough to stop babysitting), Tinkerer (validation catches taxonomy mistakes early).

- [x] **Run-state write hardening: terminal-write guard + distinct `crashed` status** *(2026-05-30)*

    **Story**: The Curator needs the run log to mean what it says. When they glance at a run marked `failed`, that must reliably mean "my workflow logic broke" — something *they* can fix — not "the worker was killed mid-run," which is an operational event. Today those collapse together, and worse, two writers can race to record a single run's outcome, leaving a run with a self-contradictory `duration_ms`/`error_message`. Without this, every "failed" badge invites a babysitting session to figure out what actually happened.

    **Decisions**:
    - **First-writer-wins on terminal state.** The sweeper (`workflow_run_sweeper.py:104`) and the run's own `except`/complete path (`workflow_runs.py:367,416`) can both write a terminal status to the same row (last-writer-wins, divergent fields). Guard `update_run_status` (+ repo writer) with `WHERE status NOT IN (terminal_set)` so the first terminal write wins — mirrors Prefect's `HandleFlowTerminalStateTransitions` (`core_policy.py:1589-1607`). Implement as the established conditional-UPDATE / `rowcount == 0` idiom (`track/core.py:327-340`, `playlist/links.py:192`); the v0.8.2 scheduler claim reuses the same shape.
    - **`crashed` is a distinct status, not a flavor of `failed`.** `failed` = workflow logic raised; `crashed` = worker died (OOM/SIGTERM). This is a triage-relevant distinction the sweeper currently flattens (Prefect models it as `StateType.CRASHED`).
    - Use name-overlays on existing statuses for any *future* sub-states (retrying/cached) rather than minting more top-level statuses (Prefect `states.py:883`).

    **Spec**:
    - Add `RUN_STATUS_CRASHED = "crashed"` to `src/config/constants.py` (today only `pending/running/completed/failed/cancelled` exist — `crashed` is new); include it in the terminal set and treat it as fail-class in run-level rollups (Prefect `states.py:686-687`).
    - `update_run_status` + the repository writer add `WHERE status NOT IN (<terminal_set>)`.
    - Sweeper writes `RUN_STATUS_CRASHED` (not `failed`); reclassify the SIGTERM/server-reload path (`workflow_runs.py:393`, currently writes `failed`) to `crashed`.
    - Surface cold-start latency as a first-class field: `started_at − created_at` (Prefect's `estimated_start_time_delta`, `objects.py:660`).

    **Tests**:
    - (integration) Terminal-write race: two concurrent sessions — sweeper marks `crashed` while the completion path marks `completed` on the same run; assert first-writer-wins, the loser is a silent no-op, fields stay self-consistent. (Build a two-session helper on the existing testcontainers harness; none exists yet.)
    - (unit) `crashed` is counted as fail-class in rollups; `failed` and `crashed` are distinguishable.
    - (unit) SIGTERM/reload path records `crashed`, not `failed`.

    **Implementation note**: The terminal-write guard branches on whether the
    *target* status is terminal — terminal writes use `WHERE status NOT IN
    (terminal set)` and treat `rowcount == 0` as a silent no-op (first-writer
    wins); the `pending → running` write keeps the strict `NotFoundError`. The
    two-session race is exercised sequentially (a single guarded UPDATE is
    atomic in Postgres). `crashed` flows to the frontend via the shared
    `RunStatus` (distinct amber badge). _(The optional cold-start-latency field
    was dropped as unused — `created_at`/`started_at` already persist if it's
    ever wanted.)_

    Effort: S | Dependencies: None | Status: Completed (2026-05-30)

- [x] **Crash-detection hardening: heartbeat on an OS thread** *(2026-05-30)*

    **Story**: A long, CPU-bound enrichment node shouldn't get a *healthy* run killed out from under the Curator. Today the heartbeat ticker shares the asyncio loop, so a node that blocks the loop stops the heartbeat — and the sweeper reaps a run that was actually fine. The user sees a spurious `crashed` and re-runs work that didn't need re-running.

    **Decisions**:
    - **Emit heartbeats from an OS daemon thread**, not an asyncio task — mixd's own sweeper docstring names "event loop blocked → heartbeat ticker can't fire" as a failure mode, and Prefect emits from a daemon thread for exactly this reason (`flow_engine.py:303-306`).
    - **Derive the stale threshold from the heartbeat interval** (a multiple), not a hardcoded 60s (mirrors Foreman, `foreman.py:88-94`).
    - **Stop heartbeating once terminal** (`flow_engine.py:330-336`), and **batch-cap each sweep** (Prefect caps at 400, `late_runs.py:68`).

    **Spec**:
    - Move heartbeat emission to a daemon thread; thread stops when the run reaches a terminal state.
    - Stale threshold = `heartbeat_interval × multiple` in the sweeper.
    - Sweeper processes at most N rows per cycle.

    **Tests**:
    - (unit) A blocked event loop still heartbeats (thread keeps ticking); the run is not reaped.
    - (unit) Heartbeat stops after terminal transition.
    - (unit) Sweep batch is capped.

    **Implementation note**: The story's premise — "a CPU-bound node blocks the
    loop, so the async ticker can't fire" — does **not** hold for this codebase:
    the only genuinely CPU-bound nodes (transforms, combiners) are already
    offloaded via `await asyncio.to_thread(...)` (`node_factories.py:136,201`),
    and enricher/source/destination work is I/O that awaits. The event loop is
    never starved by node execution, so the OS-thread design was unnecessary
    complexity (it also forced a second, synchronous DB write path). The ticker
    therefore stays a simple `asyncio` task (`_heartbeat_loop`), cancelled in the
    run's `finally`; its docstring records the to_thread dependency it relies on.
    The valuable, premise-independent hardening shipped: the stale threshold is
    `HEARTBEAT_INTERVAL_SECONDS x HEARTBEAT_STALE_MULTIPLE` (3x floor per
    Temporal's ratio; 12 preserves the historical 60s), the ticker interval
    reads the same shared constant, and `list_stalled_runs` is `limit`-capped at
    `SWEEP_MAX_BATCH` (400). _(Title kept as planned; the OS-thread approach was
    dropped during implementation — see above.)_

    Effort: S | Dependencies: None | Status: Completed (2026-05-30)

- [x] **Close pre-execution validation gaps** *(2026-05-30)*

    **Story**: When the Tinkerer wires up a workflow wrong, they want to be told *at validation time* — not to get a quietly-wrong playlist they then have to reverse-engineer. mixd's declarative validator already exceeds Prefect's (which does no static DAG validation), but three gaps produce wrong output *without erroring*, which are the hardest failures to trace.

    **Decisions**:
    - Reject the three silent-wrong-result cases the engine audit found; add each to both the blocking `validate_workflow_def` and the editor's `validate_workflow_def_detailed`.
    - These are *correctness* checks, not style — each one today yields wrong output that looks plausible.

    **Spec** (new `_validate_*` helpers in `src/application/workflows/validation.py`, called from both validators at `:126-166` and `:297-355`):
    - **`primary_input`:** reject when `config["primary_input"]` is set but not present in `upstream` (today silently falls back to `upstream[0]`, `prefect.py:285-289`).
    - **`result_key`:** reject a `result_key` that **collides with any task id**, and reject duplicate `result_key`s across tasks. (Precise mechanism: `task_results[result_key] = result` is stored as an *alias* alongside `task_results[task_id]` at `prefect.py:404-409`; if task B's `result_key` equals task A's id, A's stored result is silently overwritten by B's alias — not a "duplicate result_key" per se. The check guards against both collisions.)
    - **Non-source node with no upstream:** reject a non-source node placed at level 0 (today passes validation, then fails/empties at runtime).

    **Tests**:
    - (unit) `primary_input` set-but-not-in-upstream is rejected by both validators.
    - (unit) `result_key` that shadows a task id is rejected; duplicate `result_key`s are rejected.
    - (unit) Non-source node at level 0 is rejected.
    - (unit) Valid workflows still pass (no false positives).

    **Implementation note**: Three message-returning helpers
    (`_check_primary_input`, `_check_source_placement`, `_find_result_key_problems`)
    are shared by both `validate_workflow_def` (raises) and
    `validate_workflow_def_detailed` (structured per-field entries) — one
    detection path, two surfaces. Source placement is detected via the node
    registry `metadata["category"]`. A `result_key` equal to the task's *own*
    id is allowed (harmless self-overwrite).

    Effort: S | Dependencies: None | Status: Completed (2026-05-30)

- [x] **Shield the connector-cleanup `finally` against SIGTERM** *(2026-05-30)*

    **Story**: When a deploy or autoscale event sends SIGTERM mid-run, the Curator shouldn't pay for it with leaked HTTP connection pools that degrade the next process. This is a *present-tense* bug in today's executor, independent of the v0.8.1 swap — it's pulled into v0.8.0 with the other reliability fixes so the leak doesn't ride through a release.

    **Decisions**:
    - **Wrap the cleanup await in `asyncio.shield()`.** A `CancelledError` landing on `await workflow_context.connectors.aclose()` (`prefect.py:471`) during SIGTERM currently aborts the close and leaks httpx pools. Prefect shields its finalizers (`flow_engine.py:1491-1497`).
    - Note for v0.8.1: this lives in the executor cleanup path, so when `prefect.py` becomes `executor.py` the shield carries over unchanged — but it ships *now*, not with the swap.

    **Spec**:
    - `await asyncio.shield(workflow_context.connectors.aclose())` in the cleanup `finally` at `prefect.py:471`.

    **Tests**:
    - (unit) A `CancelledError` raised during shutdown still completes `aclose()` (connectors closed exactly once, no leak).

    **Implementation note**: At `prefect.py`'s cleanup `finally`, the idempotent
    `aclose()` is wrapped as `cleanup = asyncio.ensure_future(...)` + `await
    asyncio.shield(cleanup)` — **not** a bare `await asyncio.shield(coro())`. The
    explicit task + held reference avoids the documented `shield` weak-reference
    gotcha (the loop keeps only a weak ref to a bare shielded coroutine, so it
    can be GC'd mid-close). The regression test would time out if the shield were
    removed (cancellation mid-close leaves the connectors open). Carries into
    `executor.py` in v0.8.1 — **carry this exact form, not the one-line version**.

    Effort: XS | Dependencies: None | Status: Completed (2026-05-30)

### v0.8.1: Workflow Engine Swap (Prefect to stdlib asyncio)
**Goal**: Replace Prefect's `@flow`/`@task` with a stdlib-`asyncio` executor — lighter runtime, no embedded server, a clean substrate for scheduling.

**Context**: v0.7.8.x burned five revisions fighting Prefect 3's import weight + ephemeral SQLite-backed server. The prework investigation (below) confirmed mixd uses none of Prefect's distinctive value and decided to remove it before scheduling layers concurrent runs on top. Manual test: workflows run identically, `prefect` is not imported at app boot, cold start is faster.

**What this unlocks**: No new end-user feature, but a real user-visible outcome the Tinkerer cares about — **faster cold start, a self-host with no embedded server process to babysit, and a simpler engine to reason about** — plus the enabler the rest of the cycle needs: it drops the ephemeral server (~80–120 MB at run time), the 222-module import, and two env-var workarounds, so scheduling's concurrent runs sit on stdlib asyncio rather than a fragile embedded server (Prefect's embedded SQLite server is documented to hit stuck-states under concurrent flows). _(Per the backlog "every slice ships user value" rule: this slice is justified by the Tinkerer-visible runtime improvement and the concurrency forward-risk it removes, with a regression-test manual scenario rather than a new UI surface.)_

**Persona**: Tinkerer (a simpler, server-free engine to self-host and reason about).

- [x] **Level executor on `asyncio.TaskGroup` with explicit failure semantics** *(2026-05-30)*

    **Story**: The house convention mandates `TaskGroup` over `asyncio.gather`, but because `_run_node_lifecycle` never raises (it returns tuples), a mechanical swap would change *nothing* about failure handling — and quietly pick a sibling-cancellation behavior by accident. This story makes that behavior a *conscious* choice so a fatal node does the right thing to its level-siblings.

    **Decisions**:
    - **Deliberate semantics, not a mechanical swap.** With the tuple-return contract a `gather`→`TaskGroup` swap delivers zero sibling cancellation. Choose explicitly: **(a)** keep tuple-return — siblings run to completion, matching today and Prefect's own choice (`futures.py:557-576`); or **(b)** let fatal (non-enricher) nodes raise inside the group and handle the `ExceptionGroup` with `except*`. **Default: (a)** preserve current behavior unless fast-fail sibling cancellation is explicitly wanted. **This decision must be closed before the story is implemented** (its test can't be written until it is).
    - Distinct from the scheduler's isolation model (v0.8.2): there, each dispatch is wrapped *inside its own coroutine's* try/except so one schedule's failure never reaches the group — opposite intent, same primitive.
    - **Keep `CancelledError` out of the degrade/failure path.** Verify (don't assume) that `except (TimeoutError, Exception)` (`prefect.py:313`) can't catch cancellation — in Python 3.8+ `CancelledError` is a `BaseException`, so `except Exception` already won't catch it; confirm this holds and migrate to PEP 758 no-parens `except` syntax per `python-conventions.md` (already adopted in the use-case cancellation path — continue it, don't re-invent). Ensure `WorkflowCancelledError` can't enter `_is_failure_recoverable` (Prefect's `Exception` vs `BaseException` fault line, `task_engine.py:1499-1513`).
    - **The `crashed`-vs-`failed` status mapping is one layer up — leave it alone.** The executor only raises `WorkflowCancelledError` / propagates `CancelledError`; the `CancelledError → crashed` mapping lives in `use_cases/workflow_runs.py:383` (the run's `except` path), not the executor. The swap can't lose the v0.8.0 crashed distinction, and the existing `test_workflow_runs` CancelledError→crashed test stays the regression net. Don't migrate status-setting into `executor.py`.

### v0.8.2: Workflow Scheduling - Engine & CLI
**Goal**: Schedule a workflow (or background sync) to run automatically, drivable end-to-end from the CLI.

**Context**: The scheduling engine — DB-stored schedules + an in-process polling loop in the FastAPI lifespan — plus CLI commands as the first user surface (CLI-before-Web, mixd's established pattern). The web UI follows in v0.8.3. Depends on the engine swap (v0.8.1) so scheduling's concurrent runs sit on the stdlib executor.

**What this unlocks**: `mixd workflow schedule <id> --weekly sunday --at 06:30 --tz America/Los_Angeles` and `mixd sync schedule lastfm:plays --daily --at 02:00` — schedules persist across restarts, fire automatically, and respect the user's local time across DST.

**Persona**: Curator/Tinkerer (terminal-first power use); the foundation every other scheduling slice builds on.

**Key Tech Choices**:
- **Substrate = hand-roll (resolved).** In-process async scheduler — database-stored schedules + an async poll loop in the FastAPI lifespan — not Prefect server / APScheduler / PgQueuer. Rationale: single-VM single-instance today (so PgQueuer's multi-instance `SKIP LOCKED` is latent), APScheduler 4 is alpha and 3.x duplicates our schedule store, and mixd already runs `run_sweeper_loop()` as a precedent. The hardening (forward-from-now next-run, cadence correction, stuck-start reaper, optimistic lock) is mandatory. Full comparison: [decision memo](../architecture/workflow-execution-substrate.md) (Decision B).
- **Cadence = daily or weekly only** *(simplified 2026-06-01)*. No freeform cron, no sub-daily intervals. A schedule sets `hour`/`minute` (and `day_of_week` for weekly) at minute granularity; `schedule_type` ("daily"/"weekly") is **derived** from whether `day_of_week` is set. `croniter` is kept **only as an internal next-occurrence engine** (`compute_next_run` builds `"{minute} {hour} * * {dow}"`) for DST-correct math — the user never sees cron.
- **Single `schedules` table** with a `target_type` discriminator (`"workflow"` | `"sync"`) — one schema/migration/repository/CRUD surface for both kinds. **No RLS** (mirrors `workflow_runs`): per-user isolation is enforced by repo `WHERE user_id` filtering; the scheduler's cross-tenant poll reads all users' due rows. *(Resolved after 2026 web research rejected the policy-sentinel hatch; dedicated-`BYPASSRLS`-role is the documented upgrade path for multi-instance.)*
- **Per-schedule IANA timezone** — compute `next_run_at` in that zone, persist UTC. The polling query stays UTC-simple; `zoneinfo` + croniter own DST.
- **Pre-computed `next_run_at`** — indexed `(status, next_run_at)` makes polling a simple `WHERE status='enabled' AND next_run_at <= NOW()` index scan.
- **Run linkage** — scheduled runs carry `triggered_by_schedule_id` (FK→schedules, `ON DELETE SET NULL`) on both `workflow_runs` and `operation_runs`, so a schedule can show its run history.
- **60-second poll interval** — balances responsiveness with Neon keep-alive on the 1 GB / shared-cpu-2x Fly.io VM.

- [x] **Scheduler Service (Polling Loop)** *(2026-06-07)*

    **Story**: This is the engine that makes "every Sunday at 6am" actually happen — the background loop that wakes up, finds what's due, runs it on the same code path as a manual trigger, and does so resiliently: one schedule's failure can't take down the others, a dead dispatch can't wedge a schedule forever, and a token expiry surfaces as a failure the Curator will see rather than silent staleness.

    **Decisions**:
    - `src/application/services/scheduler.py` — an **application-layer background service** (peer of `run_sweeper_loop()`), not an infrastructure bridge.
    - **Per-schedule failure isolation.** Use `async with asyncio.TaskGroup()` but wrap each dispatch **inside its own coroutine's** try/except so a failure never propagates to the group's `ExceptionGroup` (a TaskGroup is a cancel-scope; an unhandled child exception would cancel sibling dispatches — [Python docs](https://docs.python.org/3/library/asyncio-task.html)). This is the inverse of v0.8.1's level-executor intent.
    - **Cadence correction:** `asyncio.sleep(max(0, poll_interval − cycle_duration))` to prevent drift toward ever-longer real intervals (Prefect docket `_perpetual.py:130`).
    - **Concurrency gate:** `asyncio.Semaphore(settings.scheduler.max_concurrent_scheduled_runs)`.
    - **Missed-run policy:** on startup, schedules with `next_run_at` in the past advance to the next future fire (not once per missed interval). `catchup=False` (default) skips all missed fires; `catchup=True` runs one catchup then advances.
    - **Per-user token isolation:** dispatch binds `user_id` from the schedule row to the use case (assertion / typed user-scoped command) — a scheduled job for user A can never run with user B's OAuth tokens.
    - **Stuck-start reaper — reuse the sweeper's mechanism.** A schedule whose `started_at` marker exceeds `stuck_start_timeout_seconds` without completing is `mark_schedule_failed` + advanced (Prefect's `MarkLateRuns`, `late_runs.py:69-85`). This is the same operation as `sweep_stalled_runs`/`list_stalled_runs` (`workflow_run_sweeper.py`) on a different table — follow that pattern, including the v0.8.0 *threshold-as-multiple-of-interval* + *batch-cap* hardening, rather than writing a parallel reaper. _(A generic two-table staleness-reaper is available but over-engineered for two callers — skip it.)_
    - **`consecutive_failures` resets to 0 on success** (in `mark_schedule_completed`) and increments on failure — without the reset, the v0.8.3 dashboard banner would latch on forever after a single failure.

    **Spec**:
    - Main loop: `find_due_schedules(now)` → dispatch (under semaphore + per-task try/except) → cadence-corrected sleep.
    - Workflow dispatch: check `is_workflow_running()` (in-process guard) then `RunWorkflowUseCase` → `ExecuteWorkflowRunUseCase` (same path as `POST /workflows/{id}/run`). `WorkflowAlreadyRunningError` → advance to next fire, no retry storm, **no `consecutive_failures` increment**.
    - Sync dispatch: map `sync_target` → import use case (`lastfm:plays`→`ImportTracksUseCase`, `spotify:likes`→`ImportSpotifyLikesUseCase`, `lastfm:likes`→`ExportLastFmLikesUseCase`).
    - Optimistic `mark_schedule_started` prevents double-execution across poll cycles.

- [x] **Schedule Management API** *(2026-06-07)*

    **Story**: The web UI and any external client need HTTP access to schedule CRUD — and that access must be airtight per-user, since one user must never see or mutate another's schedules.

    **Decisions**:
    - **PUT for idempotent create-or-replace** — a cron expression has no meaningful partial update.
    - **User scoping everywhere.** All endpoints pass `user_id` from the JWT to use cases; `GET /api/v1/schedules` is user-scoped (NOT the cross-user `find_due_schedules`). Sync mutations filter by `(user_id, target_id)`.
    - **Validation enforced in the use case, not just here.** The path-param regex is defense-in-depth; the real guarantee is `SyncTarget` enum + cron + tz + config validation at the use-case layer (so the CLI can't bypass it). `croniter.is_valid(strict=True)` at the boundary returns 422 early for UX.

    **Spec**:
    - `PUT/GET/DELETE /api/v1/workflows/{workflow_id}/schedule`; `PUT/GET/DELETE /api/v1/sync/schedules/{target_id}`; `GET /api/v1/schedules` (all of caller's schedules).
    - `target_id` path param constrained to `[a-z0-9_:.-]+` (defense-in-depth; enum is the real check). `config` serialized-size ≤4 KB.
    - Thin handlers: parse → Command → `execute_use_case(lambda uow: ..., user_id=user_id)` → Response.

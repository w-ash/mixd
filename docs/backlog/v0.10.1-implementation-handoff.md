## Gotchas discovered en route (do not re-derive)

- **`import_source` MUST be `"spotify_api"`** тАФ the `ChannelSpec ("spotify","spotify_api")` already exists in `src/domain/matching/play_projection.py` (priority 1, `time_semantics="start"`, `tolerance_override=180s` parked until first-poll calibration). `channel_for` raises on unknown pairs. Do not mint a different source string.
- **Checkpoint keys on the mixd `user_id`** тАФ the Last.fm importer keys its checkpoint by Last.fm *username*; that is an explicit quirk NOT to copy.
- **A xact-scoped advisory lock cannot guard the poll** тАФ the importer commits per batch, releasing `pg_try_advisory_xact_lock` at the first internal commit. Single-flight is the checkpoint claim column (user-approved deviation from the backlog's advisory-lock wording; update the backlog Decision text at completion).
- **Scheduled syncs are currently mis-attributed** тАФ `scheduler._dispatch_sync` omits `initiated_by`, so they land as `"manual"`. The runner extraction fixes this (pass `initiated_by="schedule"`).
- **`scheduled_sync:*` rows are hidden from the run-log default view** (`_IMPORT_LIKE_TYPES` in `routes/operation_runs.py`) тАФ intentional for poll runs (dozens/day); they show under type=all.
- **The `Schedule` entity is deliberately daily/weekly-only** тАФ the 30-min heartbeat needs the new `interval_minutes` cadence arm (user-approved).
- **`SYNC_DISPATCH` is a flat `dict[str, Callable[[str], Awaitable[object]]]`** тАФ no decision-function concept exists; the descriptor rework introduces it.
- **Resolver reuse is mechanical** тАФ `_extract_spotify_id_from_connector_play` (spotify/play_resolver.py) reads `service_metadata["track_uri"]`; the duration skip-filter fires only when `ms_played is not None`, so API rows (always NULL) pass through. No new resolver (hard constraint).
- **`launch_background` is interface-layer** тАФ the MCP/chat demand hook lives in application-layer `registry.py`, so demand fire-and-forget needs an application-local spawner (create_task + strong-ref set + done-callback), not an interface import.
- **No `users` table exists** (`user_settings` is preference KV) тАФ the activity signal gets its own `user_activity` table.
- **Migration head is `042_play_sources`** тАФ next is `043`; run `uv run alembic heads` first.

## The full approved plan

### Context

The GDPR export delivers depth but arrives weeks late (user's export stops 2025-02-28); Last.fm is live but identity-weak. v0.10.1 adds Spotify's recently-played API as a fresh, strong-identity observation channel (`spotify_api`) тАФ play counts land within the hour, and the v0.10.0 convergence projection guarantees a later GDPR upload deepens those same plays instead of duplicating them. Missed polls are permanent data loss (~50-play API window, PDR-001), so coverage polling + demand-pulled freshness are both required.

**Verified groundwork (v0.10.0 already shipped it):** `ChannelSpec ("spotify","spotify_api")` exists (see Gotchas). `("spotify","plays")` is declared in `CHECKPOINT_COMBINATIONS`; `sync_checkpoints` already has `cursor`/`last_timestamp`/`remote_total`.

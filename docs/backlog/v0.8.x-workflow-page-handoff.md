## Why this work exists (the problem)

The workflow **detail** page (`web/src/pages/WorkflowDetail.tsx`) has two distinct
problems, plus a forward-looking requirement the user asked for explicitly.

### Problem 1 — redundancy with run history

The page currently renders, top to bottom: a loose `PipelineStrip`, then
`LastRunCard`, then an inline `RunHistoryTable`, then `WorkflowScheduleCard`.

`LastRunCard` (`web/src/components/shared/LastRunCard.tsx`) is a **strict subset**
of the first row of the run-history table — status badge, track count, completion
time all appear twice. Its *only* unique contribution is the version-mismatch
warning (last run's `definition_version` < the workflow's current
`definition_version`). The duplication is pure noise; the run history is the
canonical log.

### Problem 2 — an active run is not legible

When a workflow is running, the user can only *infer* it: a spinner on the Run
button + the `PipelineStrip` animating. There is **no explicit "running now"
state**. Worse:

- **Navigating away and back, or reloading mid-run, loses the live view entirely.**
  The execution context only tracks runs *started in the current tab/session*; on
  remount it reports `isExecuting: false` and an empty node map, so the page looks
  idle even though the backend run is still in flight.

The user's words: *"If you navigate away from a running workflow, and navigate back
to it, you should see the current state of the run. If we need webhooks or
something to do this, we should."* (You do **not** need webhooks — see "What
already exists".)

### Forward-looking requirement — a shared active-run source

The user also wants, **eventually**, a sidebar indicator that "a run is happening,"
and was explicit that it should **hook into the same system** as the detail-page
reconnection — i.e. there should be one source of truth for "what is running,"
not two. Build the detail-page reconnection on top of an app-global active-runs
source now, so the future sidebar is a drop-in (the sidebar itself is **out of
scope** for this work — just don't paint yourself into a corner that prevents it).

## Decisions already locked with the user (constraints — do not re-litigate)

1. **Full reconnection on navigate-back/reload.** Not a degraded "it's running, go
   look elsewhere" placeholder — the page should show the *actual current state*
   of the in-flight run. (Live-streaming fidelity after reconnect is yours to
   scope — see the SSE note below — but the state shown must be real and current.)
2. **Build the app-global active-runs source now** (the shared "what's running"
   query/state), so the future sidebar reuses it. One source of truth.
3. **No new server push infrastructure** (no webhooks/websockets). The existing
   DB snapshot + SSE-resume primitives are sufficient — confirmed below.
4. **Remove the `LastRunCard`/run-history redundancy.** How you reshape the page is
   open; the redundancy must go.

Everything else — component decomposition, naming, whether the live region is a
"hero" or something else, how state is detected — is your call.

## What already exists (your toolbox — reuse before building)

### Backend: the reconnection primitives are already there

- **Snapshot endpoint** — `GET /api/v1/operations/{operation_id}/snapshot`
  (`src/interface/api/routes/operations.py:127`). Returns the **full per-node state
  rebuilt from the DB** (`workflow_run_nodes`): each node's status, execution
  order, durations, in/out track counts, error. This is what lets a reconnecting
  client show correct step N/M + progress **without** having been connected live.
- **SSE stream with resume** — `GET /api/v1/operations/{operation_id}/progress`
  (`operations.py:54`). Honors the `Last-Event-ID` header to resume from where a
  dropped connection left off. **Caveat (read it):** the event queue is an
  **in-memory, per-instance** `asyncio.Queue` — live SSE only exists on the API
  instance that started the run. The DB snapshot works from *any* instance, and
  there's already a poll-fallback when SSE stalls. So: snapshot = always correct;
  live SSE = best-effort, same-instance. Don't design as if cross-instance live
  streaming exists.
- **Precedent for rebuilding node state from a snapshot already exists** in
  `web/src/pages/WorkflowRunDetail.tsx:381-395` (builds a `nodeStatuses` Map from
  persisted node records). Study it — your reconnection likely does the same thing.
- **There is NO DB-backed "list active runs" endpoint yet.** `GET /api/v1/operations`
  (`operations.py:119`) lists only *in-memory, this-instance* SSE queues — not a
  cross-instance, DB-truth answer. **You will need to add** a small
  `GET` endpoint that returns the user's runs with `status='running'` (use case +
  `WorkflowRunRepository` query + a `status` index — mirror the existing run
  routes/use-cases). This is the backend half of the app-global active-runs source.
  After adding it, run `pnpm --prefix web sync-api` to regenerate the Orval hook.

## The gaps to close (goals, not a design)

1. **Kill the redundancy.** `LastRunCard`'s data is duplicated by the run-history
   table; its only unique value (version-mismatch warning) must survive somewhere.
   Reshape the top of the detail page so "current state" and "the run log" are
   distinct, non-overlapping concerns.
2. **Make an active run unmistakable**, including on reload/navigate-back: detect a
   live run from **server truth** (`latestRun.status === 'running'`), hydrate its
   current state from the **snapshot** endpoint, and resume SSE (`Last-Event-ID`)
   for live updates where available. The page must never look idle while the
   backend run is live.
3. **Introduce an app-global active-runs source** (the new endpoint + an app-level
   query/state) that the detail page reads, and that a future sidebar indicator can
   subscribe to unchanged. Invalidate it on run start/complete (the execution
   context already invalidates the per-workflow run queries on completion — extend
   that).
4. **Fold in the small DRY cleanups** noted above (`formatNextRun` → `format.ts`;
   `SchedulePicker` raw input → `ui/input`).

*Non-binding background:* an earlier planning pass sketched a single state-aware
"status hero" component that swaps between an ACTIVE branch (promoted
`PipelineStrip` + liveness + "view live run") and an IDLE branch (compact last-run
summary + next-run line), with the run-history table as the canonical ledger below
it. That's **one** shape that satisfies the goals — treat it as a reference point,
not a spec. If your research surfaces something cleaner/more DRY (e.g. a different
state model, colocating with the run-detail page's snapshot logic, a shared
`useActiveRun(workflowId)` hook), prefer it.

## Open questions for you to resolve (via research + the codebase)

- **How live must "reconnected" be?** Snapshot-on-mount gives correct *current*
  state cheaply. True live re-streaming after reload means teaching
  `WorkflowExecutionContext` to *adopt* an in-progress `operation_id` on mount and
  reopen SSE. Decide how far to take it — the constraint is "shows the real current
  state," and live updates after reconnect are a quality bar you can scope. (Note
  the per-instance SSE caveat: in a multi-instance deploy, reconnect may land on an
  instance without the live queue → fall back to the snapshot/poll path, which
  already exists.)
- **Where should active-run state live** so the detail page and a future sidebar
  share it without the detail page's in-session execution context becoming a
  bottleneck? (A separate app-global active-runs context/query alongside the
  existing execution context is one option; research the cleanest Tanstack-native
  pattern — e.g. a single polling query + `select`, or query + SSE invalidation.)
- **Reconnection trigger**: `latestRun.status==='running'` from the runs query is
  the obvious signal; confirm there's no cleaner source and how it interacts with
  the in-session `isExecuting` flag (don't double-drive the same run).

## Verification / acceptance

Functional (manual, via `pnpm dev` and the `run`/`verify` skills):
- Start a run → the page shows an unmistakable "running" state with live progress.
- Navigate away and back **while it's running** → the page shows the **current**
  state (correct step N/M, status), not an idle page.
- **Reload** mid-run → same: current state, not idle.
- Run completes → the page collapses back to an idle/last-run state automatically
  (the execution context already invalidates the run queries on completion).
- The version-mismatch warning still appears when the last run's definition is stale.
- No place on the page shows the same run's status/duration/tracks twice with no
  added purpose (the `LastRunCard` redundancy is gone).
- A future sidebar could read your active-runs source with no rework (sanity-check
  the shape; don't build the sidebar).

Gates (must pass before shipping):
- `pnpm --prefix web test` (your new component tests for the active/idle/empty/error
  states + reconnect-on-mount + the active-runs source)
- `pnpm --prefix web check && pnpm --prefix web build`
- If you added the backend endpoint: `uv run pytest` for the new use-case/route
  tests, `uv run basedpyright src/`, `uv run ruff check .`, then
  `pnpm --prefix web sync-api` to regenerate the client.

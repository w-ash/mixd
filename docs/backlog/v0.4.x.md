### v0.4.0: Workflow Persistence & Visualization (Vertical Slice 5a)
**Goal**: Persist workflow definitions to the database and display them as interactive DAGs in the web UI. Foundation for execution and editing in subsequent milestones.

**Context**: Workflows are Narada's core value proposition — declarative pipelines composing user-defined criteria. Currently stored as JSON files and run via CLI only. This milestone adds database persistence, a CRUD API, a template system for built-in workflows, and React Flow DAG visualization. No execution from the web yet — users can see and manage workflows visually, but run them via CLI until v0.4.1.

**What this unlocks**: Web-based workflow management, visual pipeline understanding, template-based onboarding for new users.

**Key tech choices**:
- **React Flow (xyflow) v12+** with **ELKjs** for layered auto-layout (superior to Dagre for complex DAGs)
- **Zustand** for React Flow canvas state (nodes, edges, viewport) — React Flow's officially recommended state management pattern. All server state remains in Tanstack Query.
- **7 custom node components** color-coded by category: Source (blue), Enricher (purple), Filter (orange), Sorter (gold), Selector (teal), Combiner (pink), Destination (green)

### v0.4.1: Workflow Execution & Run History (Vertical Slice 5b)
**Goal**: Execute workflows from the web with live per-node status on the DAG and persist complete run history.

**Context**: v0.4.0 delivered workflow persistence and read-only visualization. This milestone adds one-click execution with SSE progress streaming, per-node status visualization on the DAG (the "live pipeline" experience), and run history with per-node inspection. Leverages SSE infrastructure from v0.3.1 and Prefect execution engine from v0.2.7.

**What this unlocks**: Full web-based workflow lifecycle — create, visualize, execute, review history. Users see their pipeline executing in real time, node by node.

**Innovative patterns** (borrowed from n8n, Temporal, Prefect):
- **Per-node live status on DAG**: Pending -> Running (animated pulse) -> Completed (green + track count) -> Failed (red + error)
- **Definition snapshots**: Each run stores the exact `WorkflowDef` JSON used, so editing a workflow never breaks historical run views
- **Pre-flight connector validation**: Checks required connectors before execution starts, not after the first node fails
- **Execution timeline**: Horizontal bar chart of per-node durations showing where time was spent

### v0.4.2: Run-First Workflow UX (Vertical Slice 5c)
**Goal**: Restructure workflow pages around the primary user intent — running workflows and checking results — rather than visualizing definitions.

**Context**: v0.4.0–v0.4.1 built workflow persistence, DAG visualization, live execution, and run history. But the current UX treats the full React Flow DAG as the hero element on every page. In practice, users come to workflows primarily to **run them** ("refresh Hidden Gems"), not to inspect pipeline topology. The full DAG is an authoring/debugging tool, not a running tool. This milestone reorients the workflow pages around execution, adds a lightweight version counter, and reserves the full DAG for the editor (v0.4.3) and run inspection.

**What this unlocks**: A workflow "control panel" experience where running and checking results is the default. Per-track reasoning gives users full auditability over why their playlists contain what they contain. Prepares the page structure for the visual editor (v0.4.3) and future scheduling (v0.5.x+).

**Key design decisions** (informed by research across n8n, Windmill, Prefect, Dagster):
- **Run-focused detail page**: compact pipeline strip replaces full DAG as the default view
- **Full DAG lives in the editor only** (and on `WorkflowRunDetail` for historical inspection)
- **Inline execution progress**: pipeline strip animates node-by-node, no page navigation on Run
- **List page as control panel**: Run button per row with inline status updates
- **Lightweight version counter**: `definition_version` auto-incremented on edit, enables "changed since last run" indicator
- **Per-track run output**: every run persists its output tracklist with metrics + per-node filtering/sorting decisions. Users can see exactly why each track was included or removed, both at completion and historically.
- **Playlist change tracking**: when a run updates an existing playlist, the diff (tracks added/removed with reasons) is captured in the run record
- **DAG component preservation**: All v0.4.0–v0.4.1 React Flow infrastructure (7 custom node components, `BaseWorkflowNode`, ELKjs layout, `ExecutionOverlay`, `WorkflowCanvas`, Zustand store) is retained. `WorkflowRunDetail` continues to render the full DAG with execution overlay. The v0.4.3 editor builds directly on these components — upgrading `WorkflowCanvas` from read-only to interactive, not replacing it.
- **Forward-looking for scheduling**: component boundaries designed so daily/weekly scheduling UI slots in without restructuring

### v0.4.3: Visual Workflow Editor & Preview (Vertical Slice 5d)
**Goal**: Create and edit workflows visually with a drag-and-drop graph editor, real-time validation, and dry-run preview.

**Context**: v0.4.2 restructured the detail page around execution and moved the full DAG off the default view. This milestone brings the DAG back as an interactive editor on its own route. The editor is the only place the full React Flow graph appears for the current definition — users navigate here when they want to understand, create, or modify a workflow.

**What this unlocks**: Full self-service workflow lifecycle from the web. Users no longer need CLI access or filesystem editing to create/modify workflows.

**Key UX patterns** (borrowed from n8n, Windmill, React Flow Pro):
- **Editor is a separate route**: `/workflows/:id/edit` and `/workflows/new` — distinct pages with three-panel layout
- **Three-panel layout**: Node Palette (left) | React Flow Canvas (center) | Config Panel (right)
- **Drag-and-drop from palette**: DnD context + `screenToFlowPosition` for precise node placement
- **Undo/redo history stack**: Zustand state snapshots for every editor action
- **Edge validation**: Prevents self-loops, duplicate edges, and invalid node connections
- **Auto-layout on demand**: ELKjs re-arranges nodes after manual edits

### v0.4.6: Track Provenance & Merge (Vertical Slice 7a)
**Goal**: Surface mapping provenance on track detail pages and let users merge duplicate tracks from the web UI.

**Context**: Narada's automated track resolution creates canonical tracks and maps them to external service tracks. Sometimes the system creates duplicates (same song imported from different services). Users can't see HOW mappings were established, and have no way to fix duplicates. The `MergeTracksUseCase` exists but has no web API. This milestone adds **mapping transparency** and the highest-value identity correction tool: merge.

**What this unlocks**: Users can see why their tracks are linked (provenance), and fix the most common identity problem (duplicates). The `origin` column foundation enables v0.4.7 relink/unlink.

**Key design decisions** (informed by MDM golden record patterns, MusicBrainz canonical data model, Reltio pinned values, and Senzing entity resolution):
- **Manual overrides are sticky** (Reltio "pinned values" pattern) — New `origin` column on `track_mappings` (`"automatic"` vs `"manual_override"`) ensures re-matching never undoes user corrections. Two guard sites: `ingest_external_tracks_bulk()` and `map_track_to_connector()`.
- **Leverage existing provenance fields** — `match_method`, `confidence`, and `confidence_evidence` already exist on `TrackMapping` but aren't surfaced in the API/UI. Only `origin` is truly new. Enhanced TrackDetail shows all four provenance dimensions.
- **Build provenance from existing data** — The earliest `match_method="direct"` mapping on a track records which connector track originally created it. `ingest_external_tracks_bulk()` already sets `match_method="direct"` only during initial track creation.
- **Enhanced TrackDetail page, not new pages** — All identity management UI lives on the existing TrackDetail page in an expanded "Connectors" section. Correction happens in the context of the data being viewed.
- **Frontend DRY** — Merge dialog follows the exact mutation+toast+invalidation pattern established in PlaylistDetail. Only truly new components: track search combobox (for merge target) and side-by-side comparison grid.

**Research references**:
- [MusicBrainz Canonical Metadata](https://blog.metabrainz.org/2023/06/12/new-dataset-musicbrainz-canonical-metadata/) — redirect tables for provenance tracking
- [Golden Record in MDM](https://www.informatica.com/blogs/golden-record.html) — survivorship rules, data stewardship, manual override patterns
- [Reltio Pinned Values](https://docs.reltio.com/) — sticky manual corrections that override all survivorship rules
- [Senzing Entity Resolution](https://senzing.com/) — split/merge with audit trail, negative constraints
- [Entity Resolution reference architecture](https://arxiv.org/html/2503.08087v3) — merge, split, rearrange with manual override persistence
- [Record Linkage](https://en.wikipedia.org/wiki/Record_linkage) — fundamental entity resolution patterns
- [W3C VoID](https://www.w3.org/TR/void/) — link provenance metadata (method, confidence, timestamp, creator)

### v0.4.7: Track Relink & Unlink (Vertical Slice 7b)
**Goal**: Let users correct mis-linked connector tracks — change which canonical track a connector maps to, or dissociate it entirely.

**Context**: v0.4.6 delivered mapping provenance and duplicate merge. This milestone completes the identity management story with relink (move a connector mapping to a different canonical track) and unlink (break a mapping, auto-creating an orphan canonical track). These are lower-frequency than merges but critical for data correctness when automated matching gets it wrong.

**What this unlocks**: Full data stewardship — users can correct any automated matching decision. Combined with v0.4.6 merge, all three identity operations are available: merge duplicates, relink mis-matches, unlink incorrect associations.

**Key design decisions**:
- **Negative constraints via orphan mappings** — When unlinking, the orphan canonical track's mapping inherits `origin="manual_override"`. Since `ingest_external_tracks_bulk()` finds connector tracks by `(connector_name, external_id)` and the connector track remains mapped to the orphan, re-ingestion cannot re-create the broken link. No separate rejection table needed.
- **Orphaned connector tracks auto-create** — When unlinking a connector track that has no other canonical track, a new canonical track is automatically created from the connector data. No review queue complexity. (Follows Informatica's "linear unmerge" pattern.)
- **Frontend DRY** — Relink dialog reuses the `TrackSearchCombobox` from v0.4.6 merge. Unlink confirmation copies the `DeletePlaylistDialog` pattern. Mapping row actions reuse the hover-reveal button pattern from LinkedServicesSection.

### v0.4.9: CLI Unification & Polish
**Goal**: Unify CLI and web data paths, add onboarding templates, and polish CLI UX. Clean state before v0.5.0 PostgreSQL migration.

**Context**: CLI workflow commands still use file-based `workflow_loader` while the web uses database-backed use cases — two sources of truth. This milestone unifies them and adds polish: onboarding templates, shell completion, and matcher feedback.

**What this unlocks**: Single source of truth for workflows (CLI and web read/write the same database), inspiring templates for new users, and a polished CLI experience heading into v0.5.0.

#### CLI Unification Epic

- [ ] **CLI Workflow Commands → Database-Backed Use Cases**
    - Effort: M

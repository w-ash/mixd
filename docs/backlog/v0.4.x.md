### v0.4.0: Workflow Persistence & Visualization (Vertical Slice 5a)
**Goal**: Persist workflow definitions to the database and display them as interactive DAGs in the web UI. Foundation for execution and editing in subsequent milestones.

**Context**: Workflows are Mixd's core value proposition ŌĆö declarative pipelines composing user-defined criteria. Currently stored as JSON files and run via CLI only. This milestone adds database persistence, a CRUD API, a template system for built-in workflows, and React Flow DAG visualization. No execution from the web yet ŌĆö users can see and manage workflows visually, but run them via CLI until v0.4.1.

**What this unlocks**: Web-based workflow management, visual pipeline understanding, template-based onboarding for new users.

**Key tech choices**:
- **React Flow (xyflow) v12+** with **ELKjs** for layered auto-layout (superior to Dagre for complex DAGs)
- **Zustand** for React Flow canvas state (nodes, edges, viewport) ŌĆö React Flow's officially recommended state management pattern. All server state remains in Tanstack Query.
- **7 custom node components** color-coded by category: Source (blue), Enricher (purple), Filter (orange), Sorter (gold), Selector (teal), Combiner (pink), Destination (green)

### v0.4.1: Workflow Execution & Run History (Vertical Slice 5b)
**Goal**: Execute workflows from the web with live per-node status on the DAG and persist complete run history.

**Context**: v0.4.0 delivered workflow persistence and read-only visualization. This milestone adds one-click execution with SSE progress streaming, per-node status visualization on the DAG (the "live pipeline" experience), and run history with per-node inspection. Leverages SSE infrastructure from v0.3.1 and Prefect execution engine from v0.2.7.

**What this unlocks**: Full web-based workflow lifecycle ŌĆö create, visualize, execute, review history. Users see their pipeline executing in real time, node by node.

**Innovative patterns** (borrowed from n8n, Temporal, Prefect):
- **Per-node live status on DAG**: Pending -> Running (animated pulse) -> Completed (green + track count) -> Failed (red + error)
- **Definition snapshots**: Each run stores the exact `WorkflowDef` JSON used, so editing a workflow never breaks historical run views
- **Pre-flight connector validation**: Checks required connectors before execution starts, not after the first node fails
- **Execution timeline**: Horizontal bar chart of per-node durations showing where time was spent

### v0.4.2: Run-First Workflow UX (Vertical Slice 5c)
**Goal**: Restructure workflow pages around the primary user intent ŌĆö running workflows and checking results ŌĆö rather than visualizing definitions.

**Context**: v0.4.0ŌĆōv0.4.1 built workflow persistence, DAG visualization, live execution, and run history. But the current UX treats the full React Flow DAG as the hero element on every page. In practice, users come to workflows primarily to **run them** ("refresh Hidden Gems"), not to inspect pipeline topology. The full DAG is an authoring/debugging tool, not a running tool. This milestone reorients the workflow pages around execution, adds a lightweight version counter, and reserves the full DAG for the editor (v0.4.3) and run inspection.

**What this unlocks**: A workflow "control panel" experience where running and checking results is the default. Per-track reasoning gives users full auditability over why their playlists contain what they contain. Prepares the page structure for the visual editor (v0.4.3) and future scheduling (v0.5.x+).

**Key design decisions** (informed by research across n8n, Windmill, Prefect, Dagster):
- **Run-focused detail page**: compact pipeline strip replaces full DAG as the default view
- **Full DAG lives in the editor only** (and on `WorkflowRunDetail` for historical inspection)
- **Inline execution progress**: pipeline strip animates node-by-node, no page navigation on Run
- **List page as control panel**: Run button per row with inline status updates
- **Lightweight version counter**: `definition_version` auto-incremented on edit, enables "changed since last run" indicator
- **Per-track run output**: every run persists its output tracklist with metrics + per-node filtering/sorting decisions. Users can see exactly why each track was included or removed, both at completion and historically.
- **Playlist change tracking**: when a run updates an existing playlist, the diff (tracks added/removed with reasons) is captured in the run record
- **DAG component preservation**: All v0.4.0ŌĆōv0.4.1 React Flow infrastructure (7 custom node components, `BaseWorkflowNode`, ELKjs layout, `ExecutionOverlay`, `WorkflowCanvas`, Zustand store) is retained. `WorkflowRunDetail` continues to render the full DAG with execution overlay. The v0.4.3 editor builds directly on these components ŌĆö upgrading `WorkflowCanvas` from read-only to interactive, not replacing it.
- **Forward-looking for scheduling**: component boundaries designed so daily/weekly scheduling UI slots in without restructuring

### v0.4.3: Visual Workflow Editor & Preview (Vertical Slice 5d)
**Goal**: Create and edit workflows visually with a drag-and-drop graph editor, real-time validation, and dry-run preview.

**Context**: v0.4.2 restructured the detail page around execution and moved the full DAG off the default view. This milestone brings the DAG back as an interactive editor on its own route. The editor is the only place the full React Flow graph appears for the current definition ŌĆö users navigate here when they want to understand, create, or modify a workflow.

**What this unlocks**: Full self-service workflow lifecycle from the web. Users no longer need CLI access or filesystem editing to create/modify workflows.

**Key UX patterns** (borrowed from n8n, Windmill, React Flow Pro):
- **Editor is a separate route**: `/workflows/:id/edit` and `/workflows/new` ŌĆö distinct pages with three-panel layout
- **Three-panel layout**: Node Palette (left) | React Flow Canvas (center) | Config Panel (right)
- **Drag-and-drop from palette**: DnD context + `screenToFlowPosition` for precise node placement
- **Undo/redo history stack**: Zustand state snapshots for every editor action
- **Edge validation**: Prevents self-loops, duplicate edges, and invalid node connections
- **Auto-layout on demand**: ELKjs re-arranges nodes after manual edits

### v0.4.6: Track Provenance & Merge (Vertical Slice 7a)
**Goal**: Surface mapping provenance on track detail pages and let users merge duplicate tracks from the web UI.

**Context**: Mixd's automated track resolution creates canonical tracks and maps them to external service tracks. Sometimes the system creates duplicates (same song imported from different services). Users can't see HOW mappings were established, and have no way to fix duplicates. The `MergeTracksUseCase` exists but has no web API. This milestone adds **mapping transparency** and the highest-value identity correction tool: merge.

**What this unlocks**: Users can see why their tracks are linked (provenance), and fix the most common identity problem (duplicates). The `origin` column foundation enables v0.4.7 relink/unlink.

**Key design decisions** (informed by MDM golden record patterns, MusicBrainz canonical data model, Reltio pinned values, and Senzing entity resolution):
- **Manual overrides are sticky** (Reltio "pinned values" pattern) ŌĆö New `origin` column on `track_mappings` (`"automatic"` vs `"manual_override"`) ensures re-matching never undoes user corrections. Two guard sites: `ingest_external_tracks_bulk()` and `map_track_to_connector()`.
- **Leverage existing provenance fields** ŌĆö `match_method`, `confidence`, and `confidence_evidence` already exist on `TrackMapping` but aren't surfaced in the API/UI. Only `origin` is truly new. Enhanced TrackDetail shows all four provenance dimensions.
- **Build provenance from existing data** ŌĆö The earliest `match_method="direct"` mapping on a track records which connector track originally created it. `ingest_external_tracks_bulk()` already sets `match_method="direct"` only during initial track creation.
- **Enhanced TrackDetail page, not new pages** ŌĆö All identity management UI lives on the existing TrackDetail page in an expanded "Connectors" section. Correction happens in the context of the data being viewed.
- **Frontend DRY** ŌĆö Merge dialog follows the exact mutation+toast+invalidation pattern established in PlaylistDetail. Only truly new components: track search combobox (for merge target) and side-by-side comparison grid.

**Research references**:
- [MusicBrainz Canonical Metadata](https://blog.metabrainz.org/2023/06/12/new-dataset-musicbrainz-canonical-metadata/) ŌĆö redirect tables for provenance tracking
- [Golden Record in MDM](https://www.informatica.com/blogs/golden-record.html) ŌĆö survivorship rules, data stewardship, manual override patterns
- [Reltio Pinned Values](https://docs.reltio.com/) ŌĆö sticky manual corrections that override all survivorship rules
- [Senzing Entity Resolution](https://senzing.com/) ŌĆö split/merge with audit trail, negative constraints

### v0.4.7: Track Relink & Unlink (Vertical Slice 7b)
**Goal**: Let users correct mis-linked connector tracks ŌĆö change which canonical track a connector maps to, or dissociate it entirely.

**Context**: v0.4.6 delivered mapping provenance and duplicate merge. This milestone completes the identity management story with relink (move a connector mapping to a different canonical track) and unlink (break a mapping, auto-creating an orphan canonical track). **What this unlocks**: Full data stewardship ŌĆö users can correct any automated matching decision. Combined with v0.4.6 merge, all three identity operations are available: merge duplicates, relink mis-matches, unlink incorrect associations.

**Key design decisions**:

- **Frontend DRY** ŌĆö Relink dialog reuses the `TrackSearchCombobox` from v0.4.6 merge. Unlink confirmation copies the `DeletePlaylistDialog` pattern. Mapping row actions reuse the hover-reveal button pattern from LinkedServicesSection.

### v0.4.8: Usability & Self-Explanatory Interface Pass
**Goal**: Make every interface explain itself ŌĆö clear status communication, consequence previews before destructive actions, plain-language labels, and consistent interaction patterns across all pages so the user never changes mental models.

**Context**: After shipping v0.4.4 (playlist linking), v0.4.6 (track provenance), and v0.4.7 (relink/unlink), the app has powerful features behind interfaces that don't communicate what they do. A full UX audit revealed six cross-cutting problems: (1) hover-hidden actions everywhere, (2) technical jargon without definitions, (3) status indicators that are just colored dots, (4) no consequence preview before destructive actions, (5) stats without context, and (6) form options without explanations. These problems appear on every page, not just playlist sync.

**What this unlocks**: An app where every button, status, and label answers "what does this do?" and "what will happen if I click it?" without requiring prior knowledge. Consistent patterns across tracks, playlists, and settings mean learning one page teaches you the rest.

**Design principles** (sourced from NNG, Smashing Magazine, Carbon Design System):
- **Status = icon + color + text** ŌĆö never color alone. At minimum, pair a colored indicator with a human-readable label.
- **Confirmation dialogs restate consequences** ŌĆö "Sync 12 tracks to Spotify" not "Confirm". Action-specific button labels, not Yes/No.
- **Visual consequence preview** ŌĆö show what changes before committing. Affected entity counts, direction, service name.
- **Microcopy: titles standalone, explain the "why"** ŌĆö titles must make sense without reading body text. Explain reasoning before requiring action.
- **Progressive disclosure** ŌĆö basics visible at rest, details expandable on demand. Don't hide, don't overwhelm.
- **Discoverable actions** ŌĆö never hide primary actions behind hover. Every action persistently visible.
- **Consistent patterns** ŌĆö same type of action should look and behave the same way regardless of which page you're on.

**Key tech choices**:
- **Backend preview endpoint** reusing existing `calculate_playlist_diff()` from `domain/playlist/diff_engine.py` ŌĆö pure domain logic, no new external API calls
- **Direction change** via new `PATCH /playlists/{id}/links/{link_id}` endpoint (already documented but not implemented)
- **Shared connector UI components** extracted from TrackDetail and PlaylistDetail into `components/shared/`

### v0.4.10: Cross-Service Play History Deduplication
**Goal**: Support simultaneous Spotify export + Last.fm API play history imports without double-counting listening events that appear in both sources.

**Context**: Spotify extended streaming history exports are rich (ms_played, platform, shuffle, skip) but arrive infrequently via a legally-mandated manual request process. Last.fm scrobble data is good for incremental updates via API, but most scrobbles originate from Spotify ŌĆö they're the same listening events. Previously the system used Spotify up to a cutoff date, then Last.fm after. This milestone enables both sources simultaneously with robust cross-source deduplication.

**Dev mode**: All existing play data can be blown away and re-imported. No backward compatibility or migration backfill needed.

**Key design decisions** (confirmed via Phase 0 research):
- **Data model**: Enrich existing `track_plays` record ŌĆö one row per canonical listening event. Spotify wins as primary when both sources match (richer behavioral data). Last.fm metadata (MBIDs, loved status) merged into context. New `source_services` column tracks provenance.
- **Timing**: Inline dedup at import time ŌĆö after Phase 2 resolution but before `bulk_insert_plays()`. No post-hoc reconciliation needed since we start fresh.
- **Timestamp normalization**: Spotify `ts` = END time, Last.fm `date.uts` = START time. Must convert Spotify to start time via `played_at - ms_played` before comparison.
- **Tolerance window**: 30 seconds (after timestamp normalization). Fallback: 180 seconds when `ms_played` unavailable.
- **Source priority**: `spotify > lastfm` (configurable for future connectors).

### v0.5.0: CI/CD & Environment Hardening
**Goal**: Automated safety net for testing, linting, and type checking â€” plus env config audit to prepare for containerization.

**Context**: Every subsequent milestone in v0.5.x involves irreversible infrastructure changes (database migration, deployment, OAuth). CI/CD must land first so regressions are caught before merge. Environment hardening has no dependencies and naturally pairs here.

**What this unlocks**:
- **Regression protection**: Automated test + quality gates before the PostgreSQL migration
- **Contributor confidence**: PR checks catch issues before review
- **Container-ready config**: All settings driven by env vars with fail-fast validation

#### CI/CD Pipeline Epic

- [ ] **Unified CI Workflow**
    - Effort: S
    - What: Single `.github/workflows/ci.yml` with two parallel jobs (python, web)
    - Why: Catch regressions before merge â€” critical before the irreversible PostgreSQL migration and production deployment
    - Dependencies: None
    - Status: Not Started
    - Notes:
        - **Single workflow, two parallel jobs** â€” right-sized for a hobby project; separate workflows add checkout+install overhead that exceeds the parallelism gain for a ~32s test suite
        - **`python` job** (timeout 10min):
            - `astral-sh/setup-uv@v7` (caching ON by default, handles Python install from `.python-version` â€” no `actions/setup-python` needed)
            - `uv sync --frozen` (fails if lockfile stale)
            - `uv run ruff check .` + `uv run ruff format --check .` (lint before test â€” fast checks fail first)
            - `uv run basedpyright src/`
            - `uv run pytest -m "" --cov --cov-report=term-missing --cov-fail-under=80`
        - **`web` job** (timeout 10min):
            - `pnpm/action-setup@v4` + `actions/setup-node@v4` (cache: pnpm)
            - `pnpm install --frozen-lockfile`
            - `pnpm check` (Biome lint + tsc) + `pnpm test` (Vitest) + `pnpm build`
        - Trigger: push to main, PR to main
        - Concurrency: cancel-in-progress on PRs
        - **No Codecov** â€” `--cov-fail-under=80` is sufficient for hobby project
        - **No bandit** â€” ruff's `S` rules already cover flake8-bandit equivalents
        - **No Playwright E2E** â€” defer to v0.5.2 (needs API server + Vite â€” complex without Docker)
        - **Coverage at 80%** â€” no baseline exists yet; start here, tighten after measuring

- [ ] **Automated Changelog Generation**
    - Effort: XS
    - What: git-cliff config + release workflow for automated GitHub Release notes on tag push
    - Why: Clear release notes, semantic versioning
    - Dependencies: None
    - Status: Not Started
    - Notes:
        - **git-cliff v2.12** â€” single Rust binary, no Node.js toolchain; lighter than semantic-release (npm publishing), release-please (self-approve PRs), or standard-version (deprecated)
        - `cliff.toml` config: group by type (featâ†’Features, fixâ†’Bug Fixes, etc.), skip submodule noise
        - `.github/workflows/release.yml`: trigger on `v*` tags, `orhun/git-cliff-action@v4` â†’ `softprops/action-gh-release@v2`

### v0.5.1: PostgreSQL Migration
**Goal**: Swap SQLite for PostgreSQL â€” driver, schema, and query compatibility.

**Context**: The Repository + UoW pattern already fully abstracts database access â€” only connection config, driver, and a handful of SQLite-specific SQL constructs change. This was deferred from v0.3.0 because the web UI works identically on SQLite for local development.

**What this unlocks**:
- **Remote hosting**: PostgreSQL connection pool handles concurrent web requests
- **MVCC concurrency**: Foundation for Prefect parallel execution (v0.5.4)
- **JSONB columns**: `@>` containment queries and GIN indexes for workflow definitions

#### Database Migration Epic

- [ ] **Evaluate and Select Database**
    - Effort: S
    - What: Document ADR (Architecture Decision Record) for PostgreSQL selection
    - Why: Commit to one technology before migration
    - Dependencies: None
    - Status: Not Started
    - Notes:
        - **Leading candidate â€” PostgreSQL**: `asyncpg` driver, MVCC concurrency, JSONB columns, managed hosting (Neon/Supabase dev, Fly.io prod)
        - **Alternative â€” Turso (LibSQL)**: near-zero SQL migration cost, but smaller ecosystem
        - **Expected outcome**: PostgreSQL via `asyncpg`

- [ ] **Driver and Connection Config**
    - Effort: S
    - What: Swap `aiosqlite` -> `asyncpg`, update engine/session config, remove SQLite PRAGMAs
    - Why: Core connectivity change
    - Dependencies: Database Selection
    - Status: Not Started
    - Notes:
        - `db_connection.py`: URL -> `postgresql+asyncpg://`, switch `NullPool` -> `AsyncAdaptedQueuePool` (pool_size=5, max_overflow=10)
        - Remove `@event.listens_for` PRAGMA hook; remove SQLite connect args
        - `alembic.ini` + `alembic/env.py`: remove `render_as_batch=True`
        - Add `DATABASE_URL` env var

- [ ] **Schema and Query Compatibility**
    - Effort: M
    - What: Audit and migrate SQLite-specific SQL constructs
    - Why: Bulk upsert dialect, JSON -> JSONB, DateTime -> TIMESTAMPTZ
    - Dependencies: Driver and Connection Config
    - Status: Not Started
    - Notes:
        - `sqlite_insert()` -> `postgresql_insert()` in `base_repo.py`
        - JSON -> JSONB column types in `db_models.py`; unlocks `@>` containment queries and GIN indexes
        - Verify no tz-naive inserts leak through
        - Generate fresh `initial_schema` migration for PostgreSQL; archive SQLite chain
        - Test fixtures: replace `sqlite+aiosqlite:///:memory:` with `pytest-postgresql`

**Context**: With PostgreSQL in place (v0.5.1) and CI/CD catching regressions (v0.5.0), the app is ready to leave the developer's machine. Docker provides reproducible builds; Fly.io provides managed hosting at hobby tier.

**What this unlocks**:

### v0.5.3: OAuth & Credentials
**Goal**: Persist OAuth tokens in the database and implement Spotify's web OAuth flow so the deployed app can authenticate without a local browser.

**Context**: The CLI currently uses `webbrowser.open()` for Spotify OAuth and stores tokens in a `.spotify_cache` file. Neither works in a headless container. This milestone adds database-backed token storage and proper web OAuth callback routes.

**What this unlocks**:
- **Headless auth**: Spotify OAuth works without a local browser
- **Token durability**: Tokens survive container restarts
- **Graceful degradation**: Unauthenticated endpoints return `503` with a link to connect

#### OAuth Epic

- [ ] **Spotify Token Persistence**
    - Effort: S
    - What: Move tokens from `.spotify_cache` file to `oauth_tokens` database table
    - Why: Containers have no persistent filesystem; tokens must survive restarts
    - Dependencies: v0.5.1 (PostgreSQL migration)
    - Status: Not Started
    - Notes:
        - New `oauth_tokens` table: `service`, `access_token`, `refresh_token`, `expires_at`, `scope`, `updated_at`
        - `TokenStorage` protocol: `FileTokenStorage` (CLI) and `DatabaseTokenStorage` (hosted)
        - Local CLI unchanged: `FileTokenStorage` + localhost server

- [ ] **Spotify OAuth Web Flow**
    - Effort: M
    - What: Replace browser-on-localhost OAuth with hosted callback
    - Why: `SpotifyTokenManager._run_browser_auth()` uses `webbrowser.open()` â€” non-functional headless
    - Dependencies: Spotify Token Persistence, v0.5.2 (deployed callback URL)
    - Status: Not Started
    - Notes:
        - Routes: `GET /auth/spotify` (redirect), `GET /auth/spotify/callback` (exchange code, store via `DatabaseTokenStorage`)
        - Config: `SPOTIFY_REDIRECT_URI = https://{FLY_APP_HOSTNAME}/auth/spotify/callback`
        - Graceful degradation: Spotify-dependent endpoints return `503` with link to `/auth/spotify` when no valid token
        - User flow: [01-user-flows.md](../web-ui/01-user-flows.md#11-connect-spotify)

### v0.5.4: Parallel Execution & Performance
**Goal**: Unlock Prefect parallel task execution (now that PostgreSQL provides MVCC) and add simple caching for the web UI.

**Context**: `SharedSessionProvider` was a SQLite workaround â€” all Prefect tasks shared one session because SQLite can't handle concurrent writers. With PostgreSQL's connection pool, each task gets its own session and `.submit()` parallelism becomes safe. Caching is lightweight optimization work that polishes the deployed experience.

**What this unlocks**:
- **Faster workflows**: Source and enricher nodes run concurrently where the DAG allows
- **Simpler internals**: Remove `SharedSessionProvider` and manual topological sort
- **Snappier UI**: Reduced database queries via caching, measurable performance targets

#### Parallel Execution Epic

- [ ] **Prefect Parallel Execution**
    - Effort: M
    - What: Remove `SharedSessionProvider`, enable per-task sessions, `.submit()` parallelism
    - Why: `SharedSessionProvider` was a SQLite workaround; PostgreSQL handles concurrent sessions
    - Dependencies: v0.5.1 (PostgreSQL MVCC)
    - Status: Not Started
    - Notes:
        - Each Prefect task creates its own session from the pool
        - Replace manual topological sort with `.submit()` + Prefect-native future resolution
        - Source and enricher nodes become concurrently executable where DAG allows

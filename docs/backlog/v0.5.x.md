### v0.5.0: CI/CD & Environment Hardening
**Goal**: Automated safety net for testing, linting, and type checking — plus env config audit to prepare for containerization.

**Context**: Every subsequent milestone in v0.5.x involves irreversible infrastructure changes (database migration, deployment, OAuth). CI/CD must land first so regressions are caught before merge. Environment hardening has no dependencies and naturally pairs here.

**What this unlocks**:
- **Regression protection**: Automated test + quality gates before the PostgreSQL migration
- **Contributor confidence**: PR checks catch issues before review
- **Container-ready config**: All settings driven by env vars with fail-fast validation

#### CI/CD Pipeline Epic

- [x] **Unified CI Workflow**
    - Effort: S

    - Why: Catch regressions before merge — critical before the irreversible PostgreSQL migration and production deployment
    - Dependencies: None
    - Status: Completed
    - Notes:
        - **Single workflow, two parallel jobs** — right-sized for a hobby project; separate workflows add checkout+install overhead that exceeds the parallelism gain for a ~32s test suite
        - **`python` job** (timeout 10min):
            - `astral-sh/setup-uv@v7` (caching ON by default, handles Python install from `.python-version` — no `actions/setup-python` needed)
            - `uv sync --frozen` (fails if lockfile stale)
            - `uv run ruff check .` + `uv run ruff format --check .` (lint before test — fast checks fail first)
            - `uv run basedpyright src/`
            - `uv run pytest -m "" --cov --cov-report=term-missing --cov-fail-under=80`
        - **`web` job** (timeout 10min):
            - `pnpm/action-setup@v4` + `actions/setup-node@v4` (cache: pnpm)
            - `pnpm install --frozen-lockfile`
            - `pnpm check` (Biome lint + tsc) + `pnpm test` (Vitest) + `pnpm build`
        - Trigger: push to main, PR to main
        - Concurrency: cancel-in-progress on PRs
        - **No Codecov** — `--cov-fail-under=80` is sufficient for hobby project
        - **No bandit** — ruff's `S` rules already cover flake8-bandit equivalents
        - **No Playwright E2E** — defer to v0.5.2 (needs API server + Vite — complex without Docker)
        - **Coverage at 80%** — no baseline exists yet; start here, tighten after measuring

### v0.5.1: PostgreSQL Migration
**Goal**: Swap SQLite for PostgreSQL — driver, schema, and query compatibility.

**Context**: The Repository + UoW pattern already fully abstracts database access — only connection config, driver, and a handful of SQLite-specific SQL constructs change. This was deferred from v0.3.0 because the web UI works identically on SQLite for local development.

**What this unlocks**:
- **Remote hosting**: PostgreSQL connection pool handles concurrent web requests
- **MVCC concurrency**: Foundation for Prefect parallel execution (v0.5.4)
- **JSONB columns**: `@>` containment queries and GIN indexes for workflow definitions

#### PostgreSQL Optimization Epic (post-migration cleanup)

- [x] **PostgreSQL Optimization Audit**
    - Effort: M
    - What: Replace remaining SQLite-tuned patterns with PostgreSQL-native equivalents
    - Why: The initial migration swapped the driver and fixed SQL dialect issues, but left SQLite-era query patterns: in-memory aggregations, expression tree batching, generic JSON, ILIKE without index support, N+1 query loops
    - Dependencies: Schema and Query Compatibility
    - Status: Completed
    - Notes:
        - **Phase 1 — Query fixes** (`plays.py`):
            - `get_play_aggregations()`: DB-side `GROUP BY` + `COUNT/MIN/MAX` replaces loading all plays into Python defaultdict (50-100x faster, 99% less memory)
            - `total_plays_desc` sort: CTE + window function replaces N+1 loop (51 queries → 1)
            - `_find_existing_plays()`: `tuple_.in_()` replaces batched OR chains with SQLite expression tree limit comments (chunks of 200 → single query)
            - `find_plays_in_time_range()`: Removed chunked batching
            - Same patterns applied to `play/connector.py`
        - **Phase 2 — JSON → JSONB** (`db_models.py`, migration):
            - All 15 JSON columns → JSONB (25% smaller, GIN-indexable)
            - `source_services` JSON → `ARRAY(String())` for native array semantics
            - GIN index on `tracks.artists` with `jsonb_path_ops`
        - **Phase 3 — Search** (`core.py`, `db_models.py`, migration):
            - `pg_trgm` extension + GIN trigram indexes on `title`, `album`, `artists_text`
            - New `artists_text` denormalized column replaces `cast(artists, String).ilike()` — populated in `save_track()` and via migration
        - **Phase 4 — Indexing** (migration):
            - BRIN index on `track_plays.played_at` (99% smaller than B-tree for time-series)
            - Status indexes on `workflow_runs.status`, `playlist_mappings.sync_status`
            - Removed unnecessary `VARCHAR(N)` constraints on high-traffic columns
        - **Phase 5 — Docs**: `database.md` rewritten — removed all SQLite sections (WAL, NullPool, pragmas), added PostgreSQL patterns (JSONB, pg_trgm, BRIN, connection pooling)
        - Migration: `002_pg_opt` with full downgrade support

- [x] **Keyset Pagination**
    - Effort: M
    - What: Replace offset-based pagination with cursor-based keyset pagination on `list_tracks`
    - Why: Offset pagination degrades at scale (`OFFSET 10000` scans and discards 10k rows). Keyset pagination uses a `WHERE (sort_col, id) > (:value, :last_id)` pattern that's O(1) regardless of page depth — critical for the 15k+ track library
    - Dependencies: PostgreSQL Optimization Audit
    - Status: Completed
    - Notes:
        - **Hybrid cursor+offset**: API accepts both `cursor` and `offset` params — cursor takes precedence when present, offset fallback for page jumps and deep links
        - **Scoped to `list_tracks`** only — playlists, workflows, match reviews use in-memory or low-volume pagination where offset is fine
        - **Page-based UI preserved**: kept `TablePagination` page numbers (better for Weekly Curator browsing). Frontend caches cursors in `useRef<Map>` for sequential next-page navigation, falls back to offset for random page jumps
        - **Cursor encoding**: opaque base64 JSON of `(sort_column, sort_value, id)` — column name embedded to reject stale cursors on sort change
        - **Architecture**: cursor encode/decode in `src/application/pagination.py` (application layer); repository accepts raw `after_value`/`after_id` primitives (respects dependency rule — infrastructure doesn't import application)
        - **`PaginatedResponse`**: added `next_cursor: str | None = None` (backward compatible default)
        - **Row-value comparison**: `tuple_(col, DBTrack.id) > tuple_(literal(value), literal(id))` — PostgreSQL natively supports `(a, b) > (x, y)`
        - **Secondary ID tiebreaker**: ORDER BY now includes `DBTrack.id` as second sort key for stable ordering across pages
        - **Composite indexes**: migration `003_keyset_idx` adds `(title, id)`, `(created_at, id)`, `(artists_text, id)`
        - **Orval types regenerated**: `cursor` query param and `next_cursor` response field auto-generated

- [x] **Track Merge CTE Refactor**
    - Effort: S
    - What: Consolidate the multi-statement track merge logic (`move_references_to_track`, `merge_mappings_to_track`, `merge_metrics_to_track`) into PostgreSQL CTEs
    - Why: Current approach used multiple sequential raw SQL statements with Python loops for conflict resolution. CTE chains provide single atomic operations with clearer declarative logic
    - Dependencies: PostgreSQL Optimization Audit
    - Status: Completed
    - Notes:
        - **`move_references_to_track`**: 3 sequential statements + `_merge_track_likes` (6 potential SQL ops in Python loop) → 1 CTE chain with 6 operations. `_merge_track_likes` private method deleted.
        - **`merge_mappings_to_track`**: raw SQL + Python loop with if/else branching → 1 CTE with 2 SQL-level branches (`WHERE loser_ct_id = winner_ct_id` vs `!=`). Same-ext-ID: keep higher confidence, delete loser. Different-ext-ID: keep both, winner=primary, loser=secondary.
        - **`merge_metrics_to_track`**: raw SQL + Python loop + `isinstance()` defensive type checks → 1 CTE. Eliminates string-to-datetime coercion hacks (PostgreSQL handles timestamps natively).
        - **CTE snapshot semantics**: explicit `AND id NOT IN (SELECT loser_id FROM [read-only conflict CTE])` prevents double-operations from PostgreSQL's same-snapshot data-modifying CTE behavior
        - **Net result**: ~170 lines of Python loop logic → ~100 lines of SQL CTEs. Method signatures unchanged. Service orchestrator unchanged. All existing integration tests pass without modification.

- [x] **SQLAlchemy & PostgreSQL Best Practices Audit (March 2026)**
    - Effort: S
    - What: Research current SQLAlchemy 2.x and PostgreSQL best practices (as of March 2026) and apply any relevant patterns to the narada persistence layer
    - Why: The codebase was built over many months — newer SQLAlchemy features, PostgreSQL patterns, or psycopg3 capabilities may simplify code, improve performance, or increase robustness. Final sweep before calling v0.5.1 complete.
    - Status: Completed
    - Dependencies: Keyset Pagination, Track Merge CTE Refactor
    - Status: Not Started
    - Notes:
        - Web research: SQLAlchemy 2.1 changelog, psycopg3 best practices, PostgreSQL 17 features
        - Areas to audit: connection pool tuning, session lifecycle patterns, bulk operation APIs, type annotation improvements, async best practices
        - DRY focus: identify any remaining boilerplate in repository layer that newer APIs could eliminate
        - Performance: review `EXPLAIN ANALYZE` on critical queries, check for missing index opportunities
        - Robustness: connection retry, health checks, graceful degradation patterns
        - Goal is to close gaps, not introduce new features — keep scope tight
    - **Known issues from keyset pagination + CTE work** (investigate during audit):
        - **Duplicate sort-column mapping**: `_SORT_MAP` in `track/core.py` and `_SORT_BY_TO_COLUMN` in `list_tracks.py` duplicate the same sort-key→column knowledge. Consolidate into a single source of truth. Related: `sort_by` flows as a raw `str` through four layers (API route → use case → interface → repository). A `Literal` union (like `PlaySortBy` already does for plays) would catch invalid sort keys at type-check time instead of silently falling through to the default.
        - **`list_tracks` parameter sprawl**: Signature grew to 8 parameters + 4-element return tuple `(list[Track], int, set[int], tuple[Any, int] | None)` from keyset pagination additions (`after_value`, `after_id`). The opaque return tuple requires `_, _, _, _` destructuring at every call site. Consider a typed `@define` pagination params object and a `ListTracksResult` at the repository boundary (one already exists in the use case layer).
        - **Count query on keyset pages**: `list_tracks` always runs `SELECT count(*)` before the data query. For cursor-paginated requests (page 2+), the frontend already has the total from page 1 — the count query is redundant. With a 15k+ track library and trigram ILIKE filters, the count can be the most expensive part of the request. Consider an `include_total` parameter the use case sets to `False` when a cursor is present.
        - **`text()` result typing**: CTE `fetchone()` results require `# type: ignore[index]` (6 occurrences in `track/core.py`); `fetchall()` with `row[0]` indexing appears 10+ times across `connector.py`, `plays.py`, `likes.py`. Check if SQLAlchemy 2.1 `Row` typing or named tuples eliminate these suppressions.
        - **ORDER BY missing ID tiebreaker**: `list_tracks` was fixed, but `match_review.py:74-82` (confidence/created_at sort) and `workflow/runs.py:161` (created_at sort) still lack a secondary ID sort key — unstable ordering for rows with equal values.
        - **EXPLAIN ANALYZE on keyset queries**: Verify the new composite indexes `(title, id)`, `(created_at, id)`, `(artists_text, id)` from migration `003_keyset_idx` are actually used by the keyset WHERE clause query plans.
        - **Document CTE snapshot pattern**: Add the `AND id NOT IN (SELECT ... FROM read_only_cte)` pattern for data-modifying CTEs to `database.md` — prevents the same-snapshot double-operation pitfall in future CTEs.
    - **DRY issues from code review** (investigate during audit):
        - **Duplicate batch-tuple-IN pattern**: `_find_existing_plays()` in `track/plays.py` and `_find_existing_connector_plays()` in `play/connector.py` are structurally identical — extract keys as tuples, batch by `TUPLE_IN_BATCH_SIZE`, `tuple_(...).in_(batch)`, collect into set with `ensure_utc` normalization. Only the model class and column names differ. Extract a generic helper to `_shared/` parameterized by model, columns, and key extractor.
        - **`artists_display` property on Track**: The pattern `", ".join(a.name for a in track.artists)` appears inline in 7+ locations across domain, application, infrastructure, and interface layers (`diff_engine.py`, `track_commands.py`, `workflow_runs.py`, `playlist_results.py`, `track/core.py`). Add a `@property` on the `Track` entity to provide a single source of truth.

### v0.5.4: Parallel Execution & Performance
**Goal**: Unlock Prefect parallel task execution (now that PostgreSQL provides MVCC) and add simple caching for the web UI.

**Context**: `SharedSessionProvider` was a SQLite workaround — all Prefect tasks shared one session because SQLite can't handle concurrent writers. With PostgreSQL's connection pool, each task gets its own session and `.submit()` parallelism becomes safe. Caching is lightweight optimization work that polishes the deployed experience.

**What this unlocks**:
- **Faster workflows**: Source and enricher nodes run concurrently where the DAG allows
- **Simpler internals**: Remove `SharedSessionProvider` and manual topological sort
- **Snappier UI**: Reduced database queries via caching, measurable performance targets

#### Parallel Execution Epic

- [ ] **Prefect Parallel Execution**
    - Effort: M
    - What: Remove `SharedSessionProvider`, enable per-task sessions, `.submit()` parallelism
    - Why: `SharedSessionProvider` was a SQLite workaround; PostgreSQL handles concurrent sessions
    - Dependencies: v0.5.1 (PostgreSQL MVCC)
    - Status: Not Started
    - Notes:
        - Each Prefect task creates its own session from the pool
        - Replace manual topological sort with `.submit()` + Prefect-native future resolution
        - Source and enricher nodes become concurrently executable where DAG allows

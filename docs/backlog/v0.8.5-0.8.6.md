### v0.8.5: Operation & Surface Reliability
**Goal**: Make the things mixd shows the user *true* — a failed import/sync/export/workflow looks failed, a disconnected connector fails cleanly, and the editor's controls do what they show.

**Context**: The 2026-06 design-debt review found that this cycle's own headline acceptance — *"if an overnight run fails, they see it the next time they open mixd rather than discovering a stale playlist days later"* — is **broken on the web path**: a single dropped-result seam turns five different half-failures into reported successes, and several flows that the spec marks "shipped" lie about their state. These are correctness bugs, not polish; they directly undermine the trust the v0.8.0 run-log hardening set out to build. The CLI is correct throughout (it renders the same `OperationResult` the web seam discards), so this is web-surface correctness. Full evidence in [design-debt-findings.md](design-debt-findings.md) §3, §5, §6, §10.

**What this unlocks**: The Curator can trust the web UI's success/failure signal on every long-running operation — the precondition for actually stepping away from the keyboard that the whole scheduling cycle assumes.

**Persona**: Curator (trusts the overnight ritual), Tinkerer (clean connector errors instead of hangs), Casual (next-run + failure confidence).

> **Why this ships first** (pulled forward 2026-06-15): these are correctness fixes that close this cycle's own headline-acceptance gap ("if a run fails, they see it"), so they are scheduled ahead of the templates/editor authoring work (now v0.8.7–v0.8.8).

- [x] **Propagate operation results across the SSE seam** (the root fix)

    **Story**: The user wants a failed or partial import/sync/export to *look* failed in the web UI, so they never trust corrupted data. Today `run_sse_operation` does `await coro` and **discards the returned `OperationResult`** (`sse_operations.py:123`) — nothing maps `summary_metrics` into the terminal SSE event or the `OperationRun` audit counts. So use cases defensively convert exceptions to returned results (which the seam also can't read), and the web reports success while the CLI is correct.

    **Decisions**:
    - **Fix once, fix all**: this single change makes six surfaces report the real result (the four findings below + Result Summaries + the dead frontend count machinery). It is the dependency for the per-flow fixes.
    - Capture `result = await coro`; map `OperationResult.summary_metrics` → both the terminal SSE event and `finalize_run`'s counts/issues; stop use cases swallowing exceptions into results.
    - Delete ~50 lines of dead frontend count-selection machinery (`toasts.ts:107`, keys with zero backend producers).

    **Tests**: net `tests/unit/interface/api/services/test_progress.py`, `tests/integration/api/test_imports.py`; build the SSE-seam characterization (result→terminal-event mapping) first.

    **Shipped — 1a, the durable half (2026-06-15)**: `OperationResult.is_failure` + `to_counts()` (`domain/entities/operations.py`) are now the single failure definition; `sync_targets.sync_result_failed` reads through `is_failure` (scheduler + web + CLI agree). `run_sse_operation` captures `result = await coro` and finalizes the `OperationRun` audit row with the run's real status + counts via the new `_audit_outcome(result)` helper. **This delivers the headline acceptance** ("see it next time they open mixd") because the audit row is the durable, re-attachable surface. New characterization: `tests/unit/interface/api/services/test_sse_operations.py` (the seam was previously untested).

    **Remaining — 1b, the live terminal event (bundled with Story 2)**: the *live* SSE toast still reflects the use-case-emitted status, because the terminal `complete`/`error` event fires from **inside the coro** via `tracked_operation` (`domain/entities/progress.py:327`) → `complete_operation` → `SSEProgressSubscriber.on_operation_completed`, *before* `run_sse_operation` ever sees the result. Putting status + counts in the live event requires the parent/child lifecycle inversion that **is** Story 2 — so 1b and Story 2 are one unit of work. Frontend half also remaining: `useWorkflowSSE.ts` must read `counts` off the terminal event; `toasts.ts primaryCount()` keys must be reconciled to the real `summary_metrics` names the seam will emit (not blind-deleted — the backend will now produce counts). See Handoff Notes.

#### Story 2/1b Implementation Log (2026-06-16)

**The spec's root-cause diagnosis was incomplete — re-verified against code.** The web import doesn't fail at the phase-2 *"already being tracked"* step the spec describes; it fails *earlier*. Under any real coordinator-backed emitter (the web `OperationBoundEmitter`), the base importer emits a 0–100 pipeline meter (`base_play_importer.py:174` `current=20/100`) while Last.fm's `_fetch_data` emits a per-day meter (`current=1/total_days`) on the **same operation_id** → the coordinator's monotonicity check raises *"Progress went backwards: 1 < 20"*, which `AsyncProgressManager.emit_progress` **re-raised** straight into the importer pipeline, aborting ingestion (0 connector_plays, 0 track_plays). The rebinding made it worse (collapsed every op onto one id) but was not the sole cause. CLI/scheduler stay immune (NullProgressEmitter → no coordinator).

**What shipped (all unit-verified; integration pending Docker):**
- **1.1 — `OperationBoundEmitter` injects `parent_operation_id` instead of rebinding** (`interface/api/services/progress.py`). Each use-case op keeps its own id and routes as a sub-op of the request. Guard: an already-parented op forwards untouched (no child-of-child).
- **1.2 — `ProgressCoordinator.complete_operation` now evicts** (`del self._operations[id]`) so ids are reusable and the map can't grow unbounded.
- **Progress is now non-fatal telemetry** — `AsyncProgressManager.emit_progress` logs+drops an invalid event instead of re-raising (mirrors the existing subscriber error-isolation). *This is the actual fix for the monotonicity-abort* and hardens every importer against progress quirks. **No caller depended on the raise** (verified).
- **1.3 — `SSEProgressSubscriber.on_operation_completed` no longer emits the registered-op terminal+sentinel.** The seam owns it now.
- **1.4 — `run_sse_operation` owns the request-op lifecycle**: starts the parent op (top-level `started` event), and on completion pushes the terminal event via `build_terminal_event(..., counts=...)` using `_audit_outcome` (same status+counts as 1a's audit row), then `_safe_complete_parent` drives eviction; `finalize_sse_operation` pushes the single sentinel. - [ ] **Persistence-layer eager-load hardening: global `lazy="raise_on_sql"` + eager-coverage audit**

    **Story**: The user wants every playlist and synced library mixd shows them to be *complete and correct* — never a playlist that's silently missing tracks, or a track missing its connector IDs, because a query somewhere forgot to eager-load a relationship its mapper reads. The cycle's persistence refactor made the read primitives `loaded_list`/`loaded_one` (`db_models.py`) deliberately *no-I/O*: a forgotten `selectinload` degrades to `[]`/`None` rather than crashing — graceful, but **silent**, so a wrong playlist surfaces days later instead of at the failing query. That refactor scoped `lazy="raise_on_sql"` to the 7 mapper-traversed relationships only; this epic extends fail-loud coverage to the **rest** of the relationship graph, finishing the architecture's own stated invariant ("selectinload() for ALL relationships", `.claude/rules/infrastructure-patterns.md`). It is a developer-trust story in the same spirit as v0.8.0's run-log hardening: catch the mistake in dev/test, never ship silent-wrong data.

    **Decisions**:
    - **Audit before flip — the audit *is* the work.** `raise_on_sql` only fails loud; it's safe only where every reader already eager-loads. The load-bearing deliverable is an eager-coverage audit (every `relationship()` × every reader call site across mappers / repos / services / use cases), classifying each site into one of three buckets — (a) already eager-loaded, (b) deliberately tolerates-absence via a `loaded_*` primitive, (c) genuine lazy-reliant read that must be migrated. The flip is one line per relationship; the audit is the engineering. The v0.8.x scoped change earned its 7 via exactly this method (call-graph trace + full-suite proof) — generalize it.
    - **`raise_on_sql`, not `raise`.** Permit zero-IO identity-map hits (common for many-to-one), forbid only SQL-emitting loads — the async-correct guard (a bare `raise` over-triggers on safe identity-map reads). Matches the scoped v0.8.x choice; confirmed against current SQLAlchemy docs.
    - **Reciprocals decided case-by-case.** The scoped change guarded only the 7 traversed *directions*, not their `back_populates` reciprocals (`DBPlaylistTrack.playlist`, `DBTrackMapping.track`, …). The audit dispositions each reciprocal: one never read stays lazy; one that is read must be eager-or-guarded. No blanket "both sides."
    - **Primitives remain the graceful path, by design.** `loaded_*` read `loaded_value` and are *immune* to `raise_on_sql`, so deliberate tolerates-absence reads keep working (e.g. `find_tracks_by_title_artist` in `track/core.py` maps tracks without their connector mappings on purpose). The guard covers *direct instrumented* access; the primitives cover *deliberate-absence* reads. The audit's job is to prove every site is one or the other — never an accidental lazy load.
    - **Cascade safety is a precondition, not an afterthought.** A `cascade="all, delete-orphan"` relationship without `passive_deletes=True` can make the UoW lazy-load the collection during ORM `session.delete(parent)` — which `raise_on_sql` would then block. Before guarding each such relationship, confirm it's deleted via Core bulk `delete()` + DB `ON DELETE CASCADE` (as `delete_playlist`/track-delete already are) **or** add `passive_deletes=True`. The two delete-orphan relationships guarded in v0.8.x (`DBPlaylist.tracks`, `DBTrack.likes`) cleared this bar; the rest must be re-checked.

    **Spec**:
    - Produce the audit artifact (a table/doc): every `relationship()` in `db_models.py` × reader call sites × current load strategy × disposition (eager / primitive / migrate). This is the reviewable output even before any flip.
    - Apply `lazy="raise_on_sql"` to every relationship whose readers are confirmed eager (or migrated). Record any deliberately-excluded relationship with its reason — no silent gaps.
    - Migrate each lazy-reliant reader the audit finds to either an eager `selectinload` in the repo's loader (prefer routing through the mapper's `get_default_relationships`, the established single-source-of-truth) or a `loaded_*` primitive.
    - Grep gate: once complete, assert no `relationship(` lacks an explicit `lazy=` except an allowlisted, justified set.

    **Tests**:
    - (integration) Full suite green with the global guard on — the existing cross-repo CRUD/cascade/delete suite is the primary net (the scoped v0.8.x flip was proved this way: 3214 passed, incl. slow).
    - (integration) Per-repo completeness regression: fetch via each production loader and assert the mapped entity has no silently-empty collection.
    - (integration) A representative "forgot the eager-load" query raises `sqlalchemy.exc.InvalidRequestError` — proves the guard actually fires, not just that nothing broke.
    - (unit/integration) Cascade delete still succeeds for every delete-orphan relationship under the guard.

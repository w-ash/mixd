# v0.13.x: Completing the Collection

For strategic overview, see the [planning overview](README.md).

## User Workflow Context

The foundations (v0.11.x) and the entities (v0.12.x) converge: Apple Music graduates from bones to a full peer connector, the physical collection catalogued on Discogs finally lands in the library as ownership records on real albums, and manual listens close the last hole in the canonical history — the vinyl play that no service observes. After this cycle, the collection is *complete*: every service the user touches, every format they own, every listen they log.

Persona view (from the [alignment table](README.md#persona-alignment)): the **Weekly Curator** gets the full reclamation story — streaming, physical, and manual history unified; the **Tinkerer** gets the re-resolution machinery and collection import to script; the **Casual** listener gets "connect Apple Music" at parity with Spotify.

Cross-cutting contract (stated once — per-epic Specs reference it): every new use case this cycle must be classified in the chat-tool parity registry (`src/application/tools/registry.py`) — CI (`tests/unit/application/tools/test_registry_parity.py`) fails on any unclassified use case. Reads get a `ToolSpec` or an explicit classification; mutations additionally get two-phase confirmation wiring like the existing `*_write` dispatchers.

### v0.13.0: Apple Music Integration

**Goal**: Apple Music reaches parity as a library connector — likes, playlists, catalog search — and the plays v0.11.0 deliberately deferred converge into the canonical history.

**Context**: The v0.11.0 bones (auth, client, ISRC-only resolution, play channel) have been accruing observations; the v0.12.x entity work shipped the alias-aware comparator and album context that full resolution needs. This milestone finishes the connector: the library surface (Apple has no "love" API — Add to Library is the like analogue) and the designed exit for every deferred observation.

**What this unlocks**: "like on Spotify, appears in Apple Music library" cross-service sync; Apple playlists in workflows; the deferred Apple plays finally resolved and projected — the payoff of ingest-early/resolve-conservatively.

**Persona**: Curator-primary (full Apple parity); Casual secondary (Apple Music users become first-class citizens).

**Key Design Decisions**:
- **No `LoveTrackConnector`** — Apple has no love API; `LikedTrackConnector` maps to library additions, and pretending otherwise would fabricate a signal the service doesn't have.
- **Re-resolution is the ledger's designed exit, not a repair** — v0.11.0 recorded fallback candidates unresolved on purpose; resolving them now with the full matcher and replaying the projection (`mixd plays rebuild`) is the plan completing, with every decision logged in the v0.10.2 event log with matcher version.

### v0.13.1: Physical Media & Discogs

**Goal**: Track which albums the user owns and on which media. Discogs as the primary source for physical release metadata — pressings, formats, labels, catalog numbers.

**Context**: Your music identity isn't just streaming — it's the vinyl on your shelf, the CDs in your car, the albums you've purchased digitally. Discogs has 15M+ releases with rich physical media metadata (format, label, catalog number, country, pressing notes). Importing a user's Discogs collection instantly populates "what I own and in what format."

**What this unlocks**: the physical half of the library — ownership records that v0.13.2's manual listens and future collection features build on.

**Persona**: Curator-primary (the record collector's data reclamation); Tinkerer secondary (a new connector to explore). Casual: not served — physical collections are a Curator concern.

**Key Design Decisions**:
- **Release ≠ Album**: An album (release group) can have many releases (pressings). "OK Computer" is one album; the 1997 UK vinyl, 2008 Japanese CD, and 2017 OKNOTOK remaster are three releases.
- **Discogs as primary physical source**: MusicBrainz has releases too, but Discogs has stronger collector community data (condition, notes, marketplace). Start with Discogs; MusicBrainz release data can supplement later.
- **Read-only initially**: Import from Discogs, don't write back. User manages their Discogs collection on Discogs.

#### Discogs Matching Epics


- [ ] **Discogs Release Matching**

    **Story**: The user's Discogs collection is only useful once each release connects to the canonical album it's a pressing of.

    **Decisions**:
    - Discogs "master release" ≈ MusicBrainz "release group" ≈ mixd "album" — that's the mapping spine.
    - Prefer the MusicBrainz↔Discogs cross-reference (MusicBrainz stores Discogs URLs as external links); fall back to artist+title fuzzy matching with duration cross-check.

    **Spec**:
    - Match Discogs releases to canonical albums, and Discogs master releases to MusicBrainz release groups
    - Match by: artist+title normalization, MB↔Discogs cross-reference, duration cross-check fallback

    **Tests**:
    - (unit) match ladder: cross-reference hit, normalized-title hit, fuzzy fallback

    Effort: M | Dependencies: Discogs API Client & Auth (v0.11.1), v0.12.2 (first-class albums) | Status: Not Started

#### Physical Media Epics

- [ ] **Release Version Model**

    **Story**: "Which pressing do I own?" needs a release entity — format, label, catalog number, country are the physical identity of the object on the shelf.

    **Decisions**:
    - **Releases deliberately skip the three-layer mapping pattern** — `discogs_release_id`/`musicbrainz_release_id` live inline on `album_releases`, because a pressing is single-source catalog data with no cross-service identity to resolve; canonical/mapping/connector here would be ceremony without a matching problem. (Documented so a future refactor doesn't "fix" it.)

    **Spec**:
    - Domain: `AlbumRelease(id, album_id, format, label, catalog_number, country, year, discogs_release_id, musicbrainz_release_id, notes)`
    - `format`: "Vinyl", "CD", "Cassette", "Digital", "Vinyl (12\")", "Vinyl (7\")", "Box Set", etc. (Discogs format taxonomy)
    - DB: `album_releases` table with `user_id` + FK to `albums` (per-user chain)
    - Multiple releases per album is the norm — a well-known album might have 50+ pressings on Discogs

    **Tests**:
    - (integration) many releases per album round-trip

    Effort: M | Dependencies: v0.12.2 (first-class albums) | Status: Not Started

- [ ] **Ownership Tracking**

    **Story**: "What do I own, and on what format?" is the core question for physical media tracking — sourced from Discogs import or entered by hand.

    **Spec**:
    - Domain: `OwnedRelease(id, album_release_id, source, acquired_date, notes, discogs_instance_id: str | None, is_active: bool)` — `source` is "discogs_import" or "manual"
    - DB: `owned_releases` table with `user_id`, FK to `album_releases`, `discogs_instance_id` (nullable — the collection API's stable per-item id; unique per user where present), `is_active`

    - Manual entry: web UI form on album detail page — pick a known release or add custom
    - Future: Discogs condition grades, purchase price (available in collection API)

    **Tests**:

    Effort: M | Dependencies: Release Version Model | Status: Not Started

- [ ] **Collection Import from Discogs**

    **Story**: One import populates the collector's entire physical collection — the "instant gratification" moment of the milestone.

    **Decisions**:
    - **Incremental by `instance_id` set-diff, not timestamp** — Discogs collection items carry a stable `instance_id`; diffing against stored ids makes re-import idempotent and makes **removals** visible. Removals mark `owned_releases` inactive rather than delete — the user may have sold the record, but the listen history it explains stays.

    **Spec**:
    - Paginated API: `/users/{username}/collection/folders/0/releases` returns all releases
    - Per release: match to canonical album → create `AlbumRelease` if new → create `OwnedRelease`
    - Progress tracking via existing SSE infrastructure
    - API surface: `GET /api/v1/collection` (paginated, format filter); `POST /api/v1/connectors/discogs/collection/import` (long-op, SSE)
    - Parity (file-header contract): collection reads get `ToolSpec`s; the import follows the existing long-op classification pattern

    **Tests**:
    - (integration, mocked transport) full + incremental import; unmatched releases surface, not vanish
    - (integration) re-import with one removal + one addition converges — removed item inactive, not deleted

    Effort: M | Dependencies: Discogs Release Matching, Ownership Tracking | Status: Not Started

- [ ] **Collection Web UI**

### v0.13.2: Manual Listens

**Goal**: Record physical album listens — "I put this record on at 3pm" — scrobble to Last.fm and store as canonical plays in mixd.

**Context**: When you listen to a vinyl record, that listening event is invisible to your digital history. Mixd should let you log it: pick the album, enter when you started, and the system figures out the per-track timestamps from track order and durations. The plays become part of your canonical history — indistinguishable from streaming plays in workflows and metrics.

**What this unlocks**: complete listening history — physical plays join the canonical record, so "loved but forgotten" workflows and play metrics stop under-counting vinyl listeners.

**Persona**: Curator-primary (the vinyl listener whose history has holes); Tinkerer secondary (CLI + scripting surface). Casual: not served by this milestone.

**Key Design Decisions**:
- **Album-level input, track-level output**: User says "I listened to this album starting at 3pm." System calculates per-track timestamps and creates individual `TrackPlay` records. Each track gets its own play entry because that's what Last.fm expects and what mixd's play history model requires.
- **Optional Last.fm scrobble**: User chooses whether to also scrobble to Last.fm or keep it mixd-only. Some people want to keep physical and digital scrobbles separate.
- **Retroactive and real-time**: Support both "I listened yesterday at 3pm" and "I'm listening right now."

#### Listen Infrastructure Epics

- [ ] **Last.fm Scrobble API**

    **Story**: Sending play events to Last.fm is the foundation for physical listens reaching the user's scrobble history.

    **Decisions**:
    - **Scrobble failure never blocks mixd persistence** — Last.fm is maintenance-mode and observably flaky *(2026-07 check: API operational, `track.scrobble` batch-50 current, no deprecations — but integrations report intermittent auth errors)*; the canonical play lands regardless, the scrobble retries via `_shared/retry_policies.py` and degrades to a per-track warning, because the user's own history must not depend on someone else's uptime.

    **Spec**:
    - Implement `track.scrobble` on `LastFMAPIClient` — authenticated POST, batch up to 50 tracks per request
    - Last.fm `track.scrobble` API: `method=track.scrobble`, params: `artist`, `track`, `timestamp` (Unix epoch), optional `album`, `duration`
    - Batch: up to 50 scrobbles per request via indexed params (`artist[0]`, `track[0]`, `timestamp[0]`, etc.)
    - Requires same session key auth as `track.love` (already implemented)
    - Scrobble rules: track must be >30s duration, listen must be >50% of duration or >4min

    **Tests**:
    - (unit) batch param encoding; (integration, mocked transport) partial-failure reporting

    Effort: S | Dependencies: None (builds on existing Last.fm auth — `track.love` write already works) | Status: Not Started

- [ ] **Album Listen Calculator**

    **Story**: Converting "album listen at 3pm" into individual track plays with correct timestamps is the core logic of manual listens — and it must be right, because it writes permanent history.

    **Decisions**:
    - **Missing durations: use any connector-known duration; otherwise skip that track's scrobble but keep the mixd play** (timestamp = previous track's end) — fabricating durations would corrupt permanent history, and dropping the play entirely would under-count, the exact defect this milestone fixes.
    - **"Listening right now" means "just finished", not "starting now"** — the real-time input is listen *end* = now, timestamps computed backward; computing forward from a now-start would write future-dated plays that Last.fm rejects and mixd can't honestly claim happened.

    **Spec**:
    - Pure domain function: `calculate_listen_timestamps(tracks: list[Track], start_time: datetime) -> list[tuple[Track, datetime]]`
    - Track N+1 starts at: start_time + sum(duration of tracks 1..N)
    - Handle multi-disc albums: disc order → track order (already modeled in `album_tracks` via disc_number + track_number)
    - Pure function, no side effects — testable in isolation

    **Tests**:
    - (unit) timestamp arithmetic incl. multi-disc ordering and missing-duration handling

    Effort: S | Dependencies: v0.12.2 (first-class albums with ordered track listings) | Status: Not Started

- [ ] **Record Album Listen Use Case**

    **Story**: The user logs one album listen; the system resolves tracks, calculates timestamps, persists canonical plays, and (optionally) scrobbles — one command, full flow.

    **Decisions**:
    - Plays land in the same table as streaming plays — canonical history is unified.
    - **Manual plays are `service='mixd'` + `import_source='manual'`, physical detail in `context`** — `service` records the observing system and mixd is the observer; inventing a 'physical' service value would leak provenance into the service axis and break existing `service` queries (the dedup constraint keys on it).
    - **Scrobble targets sit behind a `ScrobbleSink` seam** — Last.fm is the first sink; a ListenBrainz sink (token auth, batch-1000 submits — filed in [unscheduled](unscheduled.md#enrichment-sources)) slots in later without reopening this use case.

    **Spec**:
    - `RecordAlbumListenCommand(album_id, started_at, scrobble_to_lastfm: bool = True)`
    - Appends `connector_plays` observations per track (see **Manual Plays Join the Ledger** below — no direct canonical writes); the v0.10.0 projection creates the `track_plays` rows: `service='mixd'`, `import_source='manual'`, `source_services=['mixd']`, `context` JSONB `{"listen_type": "physical", "album_release_id": ...}` carried onto the projected row (winner-flat context rule; `mixd` outranks `lastfm`), so the album page's "Recent listens" query reads canonical history unchanged
    - Parity (file-header contract): mutation → `ToolSpec` + two-phase confirm
    - If `scrobble_to_lastfm`: batch-scrobble to Last.fm via `track.scrobble`
    - Result includes: tracks played, timestamps, Last.fm scrobble success/failure per track

    **Tests**:
    - (unit + mock UoW) plays persisted with calculated timestamps; scrobble skipped when toggled off; per-track scrobble failure reported, not swallowed
    - (unit) `test_registry_parity.py` stays green

    Effort: M | Dependencies: Album Listen Calculator, Last.fm Scrobble API | Status: Not Started

- [ ] **Manual Plays Join the Ledger**

    **Story**: The user logs a vinyl listen and scrobbles it to Last.fm; the next Last.fm import returns that same scrobble — one event, seen twice. The projection must know they're the same play without a special-case merge path, or every manual listen eventually double-counts, silently inflating the history this milestone exists to make honest.

    **Decisions**:
    - **Manual plays are ledger observations, not direct canonical writes** — `RecordAlbumListenUseCase` appends to `connector_plays` and lets the v0.10.0 projection create the canonical rows, because the round-trip merge then falls out of channel-keyed grouping (`mixd` and `lastfm` are just two observers of one event) instead of needing bespoke dedup code.
    - **The manual listen's timestamp and context survive the round-trip** — `mixd` outranks `lastfm` in channel priority, so the returning scrobble enriches (`source_services` gains `'lastfm'`) rather than displaces.

    **Spec**:
    - `RecordAlbumListenUseCase` writes `connector_plays` observations: `connector_name='mixd'`, `import_source='manual'`, identifier = `artist::title`, `resolved_track_id` set immediately, physical detail in `raw_metadata`
    - Migration backfills ledger observations for any pre-existing `service='mixd'` plays

    **Tests**:
    - (integration) log listen → scrobble → simulated Last.fm import of the same scrobble → play count unchanged, `source_services == ['mixd', 'lastfm']`, `played_at` stays the mixd timestamp

    Effort: M | Dependencies: Record Album Listen Use Case, Projection Pipeline in the Import Path (v0.10.0) | Status: Not Started

# v0.12.x: First-Class Artists & Albums

For strategic overview, see the [planning overview](README.md).

## User Workflow Context

The user's music identity is bigger than their track list, but mixd models only tracks today. Artists are `{"names": ["..."]}` JSON strings — "Radiohead" on Spotify and "Radiohead" on Last.fm are unrelated text, so there is no artist page, no "all tracks by X", no favorites. Albums are a bare `VARCHAR(255)` on tracks. This cycle promotes both to first-class entities on the same three-layer identity pattern as tracks — and it starts with a spike, because with the v0.11.x connector foundations in place, entity design can be measured against **five services' real catalog data** (Spotify, Last.fm, Apple Music, Discogs, MusicBrainz) instead of assumed from two. The play-convergence work proved the method: investigate first, cite findings in every design decision.

Persona view (from the [alignment table](README.md#persona-alignment)): the **Weekly Curator** gets artist favorites and album browsing built on identity that holds up across services; the **Tinkerer** gets a rich relational model designed in the open (findings doc); the **Casual** listener gets the browse-by-artist/album navigation every music app is expected to have.

Cross-cutting contract (stated once — per-epic Specs reference it): every new use case this cycle must be classified in the chat-tool parity registry (`src/application/tools/registry.py`) — CI (`tests/unit/application/tools/test_registry_parity.py`) fails on any unclassified use case. Reads get a `ToolSpec` or an explicit classification; mutations additionally get two-phase confirmation wiring like the existing `*_write` dispatchers.

### v0.12.0: Entity Representation Spike

**Goal**: Before artists and albums get schemas, measure how every connected service actually represents them — with real catalog data, not assumptions.

**Context**: The play-convergence findings ([play-import-convergence-findings.md](play-import-convergence-findings.md)) turned guesses about timestamps and identity into measured facts that reshaped the design; this spike does the same for entity modeling. The v0.11.x clients make it possible: sample entities from the user's *existing* library and look the same artists/albums up in each service's **catalog** APIs — no personal Apple history or Discogs collection required. Docs-only milestone (precedent: v0.8.12).

**What this unlocks**: v0.12.1 and v0.12.2 Key Design Decisions that cite measurements instead of assumptions — alias handling sized against Discogs ANVs and MusicBrainz aliases, album modeling tested against real release-group/master divergence, multi-artist credit handling grounded in observed join-phrase chaos.

**Persona**: indirectly all three — this is the milestone that keeps the entity model from being redesigned after the third connector.

**Key Design Decisions**:
- **Catalog lookups, not personal history** — the sample comes from the user's existing `tracks` table; each service is queried for the *same* entities, so representations are directly comparable. Personal Apple/Discogs data is welcome when it exists but not required.
- **Findings doc is the deliverable and the gate** — v0.12.1 does not start until every census question has a recorded answer with a citing table (the play-findings pattern).

### v0.12.1: First-Class Artists

**Goal**: Promote artists from embedded JSON strings to first-class entities with identity resolution, favorites, web UI, and workflow integration.

**Context**: Artists are currently stored as `{"names": ["..."]}` JSON on Track and ConnectorTrack records — "Radiohead" on Spotify and "Radiohead" on Last.fm are unrelated text. This milestone gives artists their own identity, cross-service mapping, favorites, and dedicated pages, following the same three-layer pattern established for tracks.

**What this unlocks**: artist browsing and favorites in the web UI, "tracks by my favorite artists" workflows, and the artist-identity substrate that v0.12.2 albums and the v1.1.x social layer build on.

**Persona**: all three — the Curator favorites artists and builds artist-driven playlists; the Tinkerer explores the mapping model; the Casual listener finally gets browse-by-artist.

**Key Design Decisions**:
- **Favorites are Mixd-only** — users favorite artists in the UI, because the favorite is *their* curation signal, not a mirror of platform state. Service sync (e.g., Spotify followed artists) deferred to a future milestone.
- **`track_artists` join table** replaces JSON artist embedding on tracks — enables efficient "all tracks by artist X" queries and proper relational integrity.
- **Canonical artists are per-user, not global** — `tracks` is per-user today and cross-user identity is deliberately deferred to v1.1.0; a global artists table would smuggle in cross-user identity semantics (merge rights, shared edits) this cycle hasn't earned. `connector_artists` stays global like `connector_tracks` — service records are facts about the service, not the user.
- **The JSON→`track_artists` move is expand/contract, not a cutover** — the matching layer reads `artist_normalized`/`artists_text` precomputed from the JSONB today; a one-shot swap would put match confidence at risk mid-cycle. Expand (dual-write) → repoint readers → contract (drop the JSONB) as a separate final step, only after matching characterization tests pass against the new source.

- [ ] **Artist Database Schema**

    **Story**: The user's "all tracks by artist X" and favorites queries need a relational model — the JSON embedding can't answer them without full scans.

    **Decisions**:
    - Backfill follows the file-level expand/contract decision — the JSONB `artists` column stays authoritative until the matching layer's precomputed reads repoint; the contract (drop) migration ships separately, later.
    - `artist_favorites` has no `service` column — Mixd-only by design.

    **Spec**:
    - Replicates three-layer pattern from `db_models.py`: `DBArtist` ↔ `DBArtistMapping` ↔ `DBConnectorArtist`
    - `artists`: `id`, `user_id`, `name`, `mbid` (nullable), `created_at`, `updated_at`. `name` deliberately **non-unique** (same-name artists are real — see the Last.fm capped-confidence decision below); uniqueness on `(user_id, mbid)` where MBID present, mirroring `uq_tracks_user_mbid`
    - `connector_artists`: `id`, `connector_name`, `connector_artist_identifier`, `name`, `raw_metadata` (JSONB + GIN). Global — no `user_id`, like `connector_tracks`
    - `artist_mappings`: `user_id`, `artist_id` → `artists`, `connector_artist_id` → `connector_artists`, `match_method`, `confidence`, `is_primary` — mirrors `track_mappings` incl. tenancy
    - `artist_favorites`: `user_id`, `artist_id` → `artists`, `is_favorited`, `favorited_at`; unique `(user_id, artist_id)`
    - `track_artists`: `user_id`, `track_id` → `tracks`, `artist_id` → `artists`, `position` (ordering)
    - Migration (expand phase): create tables; **batched backfill outside the migration transaction** (10–50k rows per commit); FKs added `NOT VALID` then `VALIDATE CONSTRAINT`; indexes via `CREATE INDEX CONCURRENTLY`. Contract (drop tracks JSONB `artists`) is a separate later migration, after matching reads repoint

    **Tests**:
    - (integration) migration backfill: JSON artists → `track_artists` rows with positions preserved
    - (integration) backfill idempotent/resumable: re-running converges, no duplicate rows
    - (integration) matching characterization: `artist_normalized`/`artists_text` inputs identical before/after readers repoint
    - (integration) cascade behavior on artist/track delete

    Effort: M | Dependencies: Artist Entity & Domain Model | Status: Not Started

### v0.12.2: First-Class Albums

**Goal**: Promote `album` from a string field on Track to a proper entity with identity resolution, cross-service mapping, and dedicated web UI.

**Context**: Albums are currently a bare `VARCHAR(255)` on the tracks table — "OK Computer" on Spotify and "OK Computer" on Last.fm are unrelated strings. This milestone gives albums their own identity, following the same three-layer pattern (canonical → mapping → connector) established for tracks and extended for artists in v0.12.1.

**What this unlocks**: album browsing with ordered track listings, and the album entity that physical media (v0.13.1) and manual listens (v0.13.2) require.

**Persona**: all three — album browsing is core navigation for everyone; the Curator additionally needs albums as the unit of physical ownership.

**Key Design Decisions**:
- **Album ≠ Release**: An "album" is the abstract work (MusicBrainz calls this a "release group"). A "release" is a specific pressing/edition (UK vinyl 1997, Japanese CD with bonus tracks). This milestone covers albums; v0.13.1 adds the release/pressing layer.
- **MusicBrainz release groups as canonical** — but release-group data is **not** in today's responses: `get_recording_by_isrc` requests `inc=artist-credits+releases` and `MusicBrainzRelease` models only `id`+`title`. This milestone extends the lookup to `inc=release-groups` and widens the model — same call, richer `inc`, 1 req/s budget unchanged.
- **Album artists vs track artists**: An album has its own artist(s) (e.g., "Various Artists" for compilations). `album_artists` join table parallels `track_artists` from v0.12.1.

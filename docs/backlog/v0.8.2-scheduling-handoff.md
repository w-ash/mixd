### Notable deviations from the plan text (call the user's attention)
- **Safe failure message**: the scheduler records `type(exc).__name__` into
  `schedules.last_error` rather than routing through a connector `ErrorClassifier`.
  Reason: the classifier lives in `infrastructure/connectors/` and the scheduler is
  application-layer (can't import it without breaking layer rules); the class name
  is guaranteed leak-free and enough to triage (full detail is in the linked run).
- **`mark_schedule_skipped`** added (not in the original repo list) so
  already-running / missed-window skips advance `next_run_at` **without** bumping
  `run_count` or `consecutive_failures` — keeps the failure banner honest.
- **catchup honored**: a missed window (> 2× poll interval late) is skipped+advanced
  by default (`catchup=False`), or fired once when `catchup=True`.
- **Workflow run-status → schedule failure**: a scheduled workflow that *runs* but
  ends `failed`/`crashed` increments the schedule's `consecutive_failures` (so the
  banner reflects a chronically-broken pipeline), while a `cancelled` drain does not.

## The model — read this first (two deliberate simplifications vs. the old backlog text)

1. **Cadence = daily or weekly only.** No freeform cron, no sub-daily intervals.
   A schedule has `hour` (0–23), `minute` (0–59), `day_of_week` (`None` ⇒ daily;
   `0`=Sunday…`6`=Saturday ⇒ weekly). `schedule_type` ("daily"/"weekly") is a
   **derived** `@property`, never stored — like `target_type`. Minute granularity.
   `croniter` is kept **only as an internal engine**: `compute_next_run` builds
   `"{minute} {hour} * * {dow}"` for DST-correct next-occurrence math. Users never
   see cron. (Dropped: `cron_expression`, `interval_seconds`, `anchor_at`,
   `min_interval_seconds`, min-gap simulation, and the `cron-parser`/`cronstrue`
   frontend deps.)
2. **No RLS on the `schedules` table.** It mirrors `workflow_runs` (which also has
   no RLS and is swept cross-tenant). Per-user isolation = explicit `WHERE user_id`
   in every CRUD repo method; the scheduler's cross-tenant poll reads all rows.
   This was the user's call after 2026 web research rejected the policy-sentinel
   hatch as a security smell. (Dedicated-`BYPASSRLS`-role is the documented
   upgrade path if mixd ever goes multi-instance.)

**Exclusive arc:** exactly one of `workflow_id` / `sync_target` is set.
**Run linkage:** `triggered_by_schedule_id` (FK→schedules, `ON DELETE SET NULL`)
exists on `workflow_runs` and `operation_runs` (columns + migration done; the
*threading* — setting it on run creation — is deferred to the scheduler task).

## The DRY spine (the user's prime directive — do not violate)

Every surface (CLI now, API next, **future MCP**) is a thin adapter over **one**
codepath. Never put validation/dispatch logic in an edge:

```
parse args/JSON  →  shared validators (raise ValueError)  →  frozen Command
  →  await execute_use_case(lambda uow: SomeUseCase().execute(cmd, uow), user_id=…)
  →  render
```

`execute_use_case` (`src/application/runner.py`) is the single funnel; it sets the
RLS `app.user_id` ContextVar when `user_id` is passed. `user_id` source per surface:
CLI = `get_cli_user_id()`; API = `Depends(get_current_user_id)` (JWT); MCP = request
context. Template to copy: `normalize_tag` (`domain/entities/tag.py:32`) →
`validate_tag` (`cli_helpers.py`) → app-layer use.

- **CRUD** (all `WHERE user_id`): `create`, `update`, `delete`, `get_by_id`,
  `get_for_target`, `list_for_user`. Map partial-unique `IntegrityError` → new
  `ScheduleAlreadyExistsError` (mirror `runs.py:69-72`).
- **System hot-path** (cross-tenant, no user filter): `find_due_schedules(now, *, limit)`
  (`WHERE status='enabled' AND next_run_at<=now`, cap `DUE_BATCH_MAX`),
  `mark_schedule_started`, `mark_schedule_completed`, `mark_schedule_failed`,
  `list_stuck_started(timeout, *, limit)`.
- **LOAD-BEARING — the optimistic claim guard:**
  `mark_schedule_started` = `UPDATE … WHERE id=:id AND status='enabled' AND
  next_run_at=:expected AND started_at IS NULL` → `cast('int', result.rowcount) > 0`
  (reuse the v0.8.0 idiom at `workflow/runs.py:183-194`). **`started_at IS NULL` is
  not optional** — without it a dispatch longer than the 60s poll gets re-claimed on
  the next tick (next_run_at only advances on completion). `mark_schedule_completed`
  (sets `started_at=NULL`, `next_run_at=<advanced>`, `last_run_id`, resets
  `consecutive_failures=0`, `run_count+1`) and `mark_schedule_failed` (increments
  `consecutive_failures`, sets `last_error` via `truncate_error_message`, advances)
  should themselves guard on `started_at` so a reaped-then-late-completing run can't
  resurrect state.
- **Race fixture** (does not exist yet): a `two_db_sessions` testcontainers fixture
  (two independent committed `AsyncSession`s on separate connections, `TEST_%`-namespaced
  + `finally` cleanup) for the optimistic-claim test (exactly one of two concurrent
  claimers wins). Integration tests: claim race, disabled-in-window guard,
  `find_due_schedules` cross-user, partial-unique one-per-target, workflow-delete
  CASCADE, schedule-delete SET-NULL preserves run history, cross-user CRUD isolation.

# v0.6.x: Multi-User Data Isolation

For strategic overview, see the [planning overview](README.md).

**Supersedes**: The former v1.0.x "User Authentication System" epic (deleted). Neon Auth already handles authentication (JWT validation, session management, email allowlist). This milestone is purely about **data isolation** and **per-user service connections** â€” the missing piece that prevents sharing the deployed instance.

**Multi-Layer Isolation Strategy**: Application-level filtering (WHERE clauses in repositories) is the primary isolation mechanism. PostgreSQL Row-Level Security (RLS) is the defense-in-depth safety net â€” even if a repo method misses a WHERE clause, the database itself won't return other users' rows. The implementation is phased: v0.6.0 creates RLS policies (inert), v0.6.2 sets `app.user_id` session variable per-transaction via `SET LOCAL` + contextvar (RLS active but table owner bypasses), v0.6.4 applies `FORCE ROW LEVEL SECURITY` (RLS enforced on table owner, after isolation tests pass). Uses portable PostgreSQL features (`current_setting()`, `SET LOCAL`) â€” no extensions required.

### v0.6.0: Schema + User Identity Foundation
**Goal**: Add `user_id` columns to all user-owned tables and establish the `CurrentUser` dependency for extracting identity from Neon Auth JWTs.

**Context**: Currently all 18 database tables are single-user. `BusinessLimits.DEFAULT_USER_ID = "default"` is hardcoded throughout. The Neon Auth middleware already validates JWTs and attaches `scope["auth_user"]` with claims including `sub` (user ID) and `email`, but no route handler reads these claims. This phase adds the schema foundation and identity extraction without changing application behavior.

**What this unlocks**:
- Data written from this point forward is tagged with the real user ID
- Foundation for repository filtering (v0.6.2) and per-user OAuth (v0.6.3)
- Existing `'default'` data can be claimed by a real user via backfill (v0.6.4)

**Persona**: Weekly Curator â€” invisible change (behavior unchanged), but data is now tagged correctly for future multi-user enforcement.

**Key Design Decisions**:
- **No local `users` table** â€” Neon Auth manages user identities externally. We only need the JWT `sub` claim as a string foreign key. No user CRUD, no password management, no session table.
- **`user_id` is a string** â€” matches Neon Auth JWT `sub` claim format (UUID string). Not an integer FK to a local users table.
- **Auth disabled (local dev) = `DEFAULT_USER_ID`** â€” when `NEON_AUTH_URL` is empty, `get_current_user_id()` returns `"default"`. Local dev operates as single-user seamlessly.

**Data Ownership Model**:

| Category | Tables | user_id column? |
|----------|--------|-----------------|
| **Shared** (canonical metadata cache) | `connector_tracks`, `track_metrics`, `connector_playlists` | No |
| **User-scoped** (owned data) | `tracks`, `track_mappings`, `match_reviews`, `track_likes`, `track_plays`, `playlists`, `workflows`, `oauth_tokens`, `user_settings`, `connector_plays` | Yes (added) |
| **Transitively scoped** (FK to user-scoped parent) | `playlist_tracks`, `playlist_mappings`, `workflow_runs`, `workflow_run_nodes`, `workflow_versions` | No (scoped via FK) |
| **Already has user_id** | `sync_checkpoints` | Exists |

### v0.6.2: Repository + Use Case Scoping
**Goal**: Thread `user_id` through the full request lifecycle â€” route handlers extract it, use cases pass it to repos, repos filter queries by it. After this phase, users see only their own data.

**Context**: The schema has `user_id` columns (v0.6.0) but nothing enforces isolation yet. This phase adds `user_id` parameters to repository protocols, implements filtering in all repo queries, threads `user_id` through use case Commands, and injects it from route handlers. This is the largest phase â€” it touches all layers of the architecture â€” but each individual change is mechanical.

**What this unlocks**:
- User A cannot see User B's playlists, likes, plays, or workflows
- Dashboard stats are per-user
- Workflow execution operates on the current user's data only
- Sharing the deployed instance with friends becomes safe

**Persona**: All personas â€” data isolation is the prerequisite for any multi-user scenario. The Weekly Curator's data is protected when friends sign up.

**Key Design Decisions**:
- **Reads filter by user_id, writes carry it on the entity** â€” read methods (`list_playlists`, `get_all_liked_tracks`) take an explicit `user_id` parameter. Write methods (`save_playlist`) get user_id from the entity's field (set by the use case).
- **ID-based lookups verify ownership** â€” `get_playlist_by_id(id, user_id)` returns 404 if the playlist belongs to a different user. Prevents IDOR via guessed IDs.
- **Template workflows use NULL user_id** â€” seeded templates (`is_template=True`) have `user_id=None` and are visible to all users. `list_workflows(user_id)` returns `WHERE user_id = ? OR user_id IS NULL`.
- **CLI always uses DEFAULT_USER_ID** â€” the CLI is the instance owner's tool. No JWT context available.
- **list_tracks liked_track_ids** â€” the `TrackRepository.list_tracks()` method joins to `track_likes` for the `liked_track_ids` set. This subquery must be scoped to the current user.
- **Defense-in-depth with RLS** â€” Application-level WHERE clauses are the primary isolation mechanism. PostgreSQL RLS policies (created in v0.6.0) act as a safety net via `SET LOCAL app.user_id` at transaction start. If a repo query forgets the filter, RLS catches it. Standard practice in multi-tenant PostgreSQL applications.

### v0.6.3: Per-User OAuth
**Goal**: Each user connects their own Spotify and Last.fm accounts. OAuth tokens are stored per-user, and connector operations use the current user's credentials.

**Context**: Currently `oauth_tokens` has `UNIQUE(service)` â€” one Spotify token, one Last.fm token, per database. A second user literally cannot authenticate with Spotify because the INSERT would conflict. This phase makes token storage user-scoped and threads user context through the connector auth flows.

**What this unlocks**:
- Each user connects their own Spotify/Last.fm accounts
- Each user sees their own connection status on the Integrations page
- Import operations use the current user's credentials
- Friends can sync their own liked tracks and play history

**Persona**: All personas â€” connecting your own service accounts is the minimum for a usable shared instance.

**Key Design Decisions**:
- **User ID in CSRF state** â€” the OAuth callback needs to know which user initiated the flow. Encode user_id in the CSRF state token (the `state` parameter), or recover it from the session cookie present on the redirect.
- **Session cookie preferred for callback** â€” Neon Auth's session cookie is present on the callback redirect. The middleware can extract user identity from it. This avoids encoding user_id in the state parameter (which could be tampered with).
- **Token storage uses standalone sessions** â€” `DatabaseTokenStorage` creates its own short-lived sessions (not UoW) because token refresh happens inside httpx auth flows with no UoW context. This pattern stays the same, just with an added `user_id` parameter.

### v0.6.4: Testing + Data Purge
**Goal**: Comprehensive multi-user isolation tests, test fixture updates, FORCE RLS defense-in-depth, and clean-slate purge of legacy `user_id='default'` data.

**Context**: After v0.6.0â€“v0.6.3, the application is multi-user capable. This phase proves correctness through isolation testing and completes the RLS enforcement. Early dev mode â€” clean breaks over backward compatibility, so legacy data is purged rather than migrated.

**What this unlocks**:
- Confidence that data isolation actually works (20 isolation tests across 10 repository areas)
- FORCE RLS on all 11 user-scoped tables â€” defense-in-depth even for the table owner role
- Clean test factories with explicit `user_id` support for future multi-user test scenarios
- Clean database â€” no orphaned `user_id='default'` rows

**Persona**: Weekly Curator â€” starts fresh with a real Neon Auth identity. No legacy data migration needed.

**Key Design Decisions**:
- **Two-user isolation tests** â€” insert as user_a, query as user_b, verify empty. Covers tracks, likes, plays, playlists, workflows, checkpoints, stats, and OAuth tokens.
- **Factory defaults match entity defaults** (`"default"`) â€” zero cascading test changes. Isolation tests use explicit user_ids. Two-tier fixture design: existing `db_session` for SQL correctness, future `rls_session` for true RLS testing.
- **FORCE RLS on all 11 tables** â€” backlog originally said 8, but migration 007 applied RLS to 11 tables. FORCE applied to all for consistency.
- **Clean-slate purge** over data claim â€” Alembic data migration deletes `user_id='default'` rows. No claim-data command needed.
- **Bug found and fixed** â€” `TrackRepository.save_track()` wasn't including `user_id` in its INSERT values dict; all tracks were saved with `user_id='default'` regardless of the entity's `user_id` field.

**Findings for future work**:
- Template workflows use `user_id='default'` (NOT NULL), not shared via NULL. The `list_workflows` query's `OR user_id IS NULL` clause is dead code. Making templates truly shared requires nullable `user_id` on workflows.
- Resolver `_save_with_connector_mappings` has the same user_id propagation gap (creates Track entities with default `user_id` instead of the passed parameter).

### v0.6.5: First-Class CLI
**Goal**: Make the CLI a first-class interface â€” fully functional in both local (Docker PostgreSQL) and remote (Neon) modes, with near-complete feature parity with the web UI (~95%).

**Context**: After v0.6.4, multi-user data isolation is complete: RLS policies enforced via `FORCE ROW LEVEL SECURITY`, per-user OAuth tokens encrypted at rest. But the CLI is broken against Neon: it hardcodes `user_id="default"` in 24 call sites across 8 command files, never passes `user_id` to `execute_use_case()` for RLS, and has no connector auth commands. Beyond the identity fix, the CLI is a strict subset of web functionality (~70%) â€” missing workflow run history, versioning, match reviews, track mapping mutation, and playlist link management. All these use cases already exist; the CLI just lacks commands to invoke them.

**Known bug**: `track_commands.py` â€” `ListTracksCommand(query=query, ...)` is missing required `user_id` field (first positional attr, no default). Runtime TypeError on every `mixd tracks list` call.

**What this unlocks**:
- CLI works against remote Neon database (same data visible in web UI)
- CLI-only users can set up connectors without needing the web server
- Workflow run inspection and version management from terminal
- Match review resolution (interactive mode) from terminal
- Track mapping mutation (relink, unlink, set-primary) from terminal
- Playlist link management and sync from terminal
- Clear documentation for both local and remote operation modes

**Persona**: The Tinkerer â€” prefers CLI workflows, wants to self-host OR connect to the hosted instance from their terminal. A first-class CLI means choosing the terminal doesn't mean losing functionality.

**Key Design Decisions**:
- **Direct DB for both modes** â€” The CLI connects directly to PostgreSQL in both local and remote modes. DB credentials are the trust boundary. No API-based remote mode (would require running a web server, defeating CLI independence). Direct access enables power-user operations the REST API can't provide.
- **Trusted user_id via config** â€” `MIXD_USER_ID` env var. No JWT ceremony needed. When unset, falls back to `"default"` (local single-user, unchanged from today).
- **Existing browser OAuth for connectors** â€” `SpotifyTokenManager._run_browser_auth()` already implements local HTTP server on :8888. After v0.6.3, it just needs the user_id passed through for token storage.
- **No profiles** â€” `.env.local` with `DATABASE_URL` and `MIXD_USER_ID` is sufficient. Profiles can be layered on later if needed.
- **Centralized user resolution** â€” A single `get_cli_user_id()` function replaces all 24 hardcoded `DEFAULT_USER_ID` references.
- **No SSE/streaming in CLI** â€” Long-running operations run synchronously with Rich progress bars (existing pattern). No need to mirror the web's SSE infrastructure.
- **Feature parity via existing use cases** â€” Every new CLI command wraps an existing use case. Zero new business logic required.

# v0.6.x: Multi-User Data Isolation

For strategic overview, see the [planning overview](README.md).

**Supersedes**: The former v1.0.x "User Authentication System" epic (deleted). Neon Auth already handles authentication (JWT validation, session management, email allowlist). This milestone is purely about **data isolation** and **per-user service connections** — the missing piece that prevents sharing the deployed instance.

---

### v0.6.0: Schema + User Identity Foundation
**Goal**: Add `user_id` columns to all user-owned tables and establish the `CurrentUser` dependency for extracting identity from Neon Auth JWTs.

**Context**: Currently all 18 database tables are single-user. `BusinessLimits.DEFAULT_USER_ID = "default"` is hardcoded throughout. The Neon Auth middleware already validates JWTs and attaches `scope["auth_user"]` with claims including `sub` (user ID) and `email`, but no route handler reads these claims. This phase adds the schema foundation and identity extraction without changing application behavior.

**What this unlocks**:
- Data written from this point forward is tagged with the real user ID
- Foundation for repository filtering (v0.6.1) and per-user OAuth (v0.6.2)
- Existing `'default'` data can be claimed by a real user via backfill (v0.6.3)

**Persona**: Weekly Curator — invisible change (behavior unchanged), but data is now tagged correctly for future multi-user enforcement.

**Key Design Decisions**:
- **No local `users` table** — Neon Auth manages user identities externally. We only need the JWT `sub` claim as a string foreign key. No user CRUD, no password management, no session table.
- **`user_id` is a string** — matches Neon Auth JWT `sub` claim format (UUID string). Not an integer FK to a local users table.
- **Auth disabled (local dev) = `DEFAULT_USER_ID`** — when `NEON_AUTH_URL` is empty, `get_current_user_id()` returns `"default"`. Local dev operates as single-user seamlessly.

**Data Ownership Model**:

| Category | Tables | user_id column? |
|----------|--------|-----------------|
| **Shared** (canonical metadata cache) | `tracks`, `connector_tracks`, `track_mappings`, `match_reviews`, `track_metrics`, `connector_playlists` | No |
| **User-scoped** (owned data) | `track_likes`, `track_plays`, `playlists`, `workflows`, `oauth_tokens`, `user_settings`, `connector_plays` | Yes (added) |
| **Transitively scoped** (FK to user-scoped parent) | `playlist_tracks`, `playlist_mappings`, `workflow_runs`, `workflow_run_nodes`, `workflow_versions` | No (scoped via FK) |
| **Already has user_id** | `sync_checkpoints` | Exists |

### v0.6.1: Repository + Use Case Scoping
**Goal**: Thread `user_id` through the full request lifecycle — route handlers extract it, use cases pass it to repos, repos filter queries by it. After this phase, users see only their own data.

**Context**: The schema has `user_id` columns (v0.6.0) but nothing enforces isolation yet. This phase adds `user_id` parameters to repository protocols, implements filtering in all repo queries, threads `user_id` through use case Commands, and injects it from route handlers. This is the largest phase — it touches all layers of the architecture — but each individual change is mechanical.

**What this unlocks**:
- User A cannot see User B's playlists, likes, plays, or workflows
- Dashboard stats are per-user
- Workflow execution operates on the current user's data only
- Sharing the deployed instance with friends becomes safe

**Persona**: All personas — data isolation is the prerequisite for any multi-user scenario. The Weekly Curator's data is protected when friends sign up.

**Key Design Decisions**:
- **Reads filter by user_id, writes carry it on the entity** — read methods (`list_playlists`, `get_all_liked_tracks`) take an explicit `user_id` parameter. Write methods (`save_playlist`) get user_id from the entity's field (set by the use case).
- **ID-based lookups verify ownership** — `get_playlist_by_id(id, user_id)` returns 404 if the playlist belongs to a different user. Prevents IDOR via guessed IDs.
- **Template workflows use NULL user_id** — seeded templates (`is_template=True`) have `user_id=None` and are visible to all users. `list_workflows(user_id)` returns `WHERE user_id = ? OR user_id IS NULL`.
- **CLI always uses DEFAULT_USER_ID** — the CLI is the instance owner's tool. No JWT context available.
- **list_tracks liked_track_ids** — the `TrackRepository.list_tracks()` method joins to `track_likes` for the `liked_track_ids` set. This subquery must be scoped to the current user.

### v0.6.2: Per-User OAuth
**Goal**: Each user connects their own Spotify and Last.fm accounts. OAuth tokens are stored per-user, and connector operations use the current user's credentials.

**Context**: Currently `oauth_tokens` has `UNIQUE(service)` — one Spotify token, one Last.fm token, per database. A second user literally cannot authenticate with Spotify because the INSERT would conflict. This phase makes token storage user-scoped and threads user context through the connector auth flows.

**What this unlocks**:
- Each user connects their own Spotify/Last.fm accounts
- Each user sees their own connection status on the Integrations page
- Import operations use the current user's credentials
- Friends can sync their own liked tracks and play history

**Persona**: All personas — connecting your own service accounts is the minimum for a usable shared instance.

**Key Design Decisions**:
- **User ID in CSRF state** — the OAuth callback needs to know which user initiated the flow. Encode user_id in the CSRF state token (the `state` parameter), or recover it from the session cookie present on the redirect.
- **Session cookie preferred for callback** — Neon Auth's session cookie is present on the callback redirect. The middleware can extract user identity from it. This avoids encoding user_id in the state parameter (which could be tampered with).
- **Token storage uses standalone sessions** — `DatabaseTokenStorage` creates its own short-lived sessions (not UoW) because token refresh happens inside httpx auth flows with no UoW context. This pattern stays the same, just with an added `user_id` parameter.

    - Effort: S

    - Why: These fields are stored as plaintext. A database compromise (leaked backup, SQL injection, Neon breach) gives direct access to users' Spotify and Last.fm accounts. The tokens are long-lived (Spotify refresh tokens never expire; Last.fm session keys are permanent).

        - `DatabaseTokenStorage.load_token()` and `save_token()` are the only read/write paths

- [ ] **OAuth State Durability**
    - Effort: S

    - Why: Currently an in-memory Python dict with 5-minute TTL. If the server restarts mid-OAuth flow, the callback fails with "invalid state." Multi-worker uvicorn (if ever needed) would route the callback to a different worker that doesn't have the state.
    - Dependencies: OAuth Callback User Association
    - Status: Not Started
    - Notes:
        - Current implementation: `_csrf_states: dict[str, float] = {}` — lost on restart, per-worker only
        - The state only needs to live ~5 minutes (OAuth round-trip time)
        - Single-worker Fly.io deployment means cross-worker isn't urgent, but restart durability is (Fly.io restarts VMs for scaling/maintenance)
        - Natural to address alongside callback rework since both touch the same code

### v0.6.3: Testing + Data Claim
**Goal**: Comprehensive multi-user isolation tests, update all existing test fixtures, and provide a mechanism for the instance owner to claim existing `'default'` data.

**Context**: After v0.6.0–v0.6.2, the application is multi-user capable. This phase ensures correctness through testing and handles the migration of existing data from `user_id='default'` to the real user.

**What this unlocks**:
- Confidence that data isolation actually works (not just "user_id is in the WHERE clause")
- Existing 15k tracks of imported data claimed by the instance owner's real Neon Auth user ID
- Clean test fixtures that work with multi-user semantics

**Persona**: Weekly Curator — their existing data seamlessly transfers to their real account on first login.

**Key Design Decisions**:
- **Two-user isolation tests** — the gold standard for data isolation testing. Insert as user_a, query as user_b, verify empty. For every scoped repository method.
- **Data claim on first login** — a one-time operation that updates all `user_id='default'` rows to the first real user's ID. Could be an admin API endpoint, a CLI command, or automatic on first authenticated request.
- **Template workflows stay NULL** — seeded templates are not claimed. Their `user_id` remains NULL (or a system constant).

    - Notes:
        - Pattern: create data as `user_a`, query as `user_b`, assert empty/404
        - Test every scoped repo: likes, plays, playlists, workflows, settings, stats

### v0.8.0: Workflow & Sync Scheduling
**Goal**: Automate the Weekly Curator's ritual — playlists and source data stay fresh without manual triggers.

**Context**: Without scheduling, every workflow run and data import requires a manual click. The Curator's Sunday ritual works, but it shouldn't require babysitting. Scheduling is the single highest-QoL feature for power users — it turns mixd from a tool you operate into a tool that works for you. Separated from editor polish (v0.8.1) because scheduling alone is a huge win and shouldn't be blocked behind editor features.

**Key Tech Choices**:
- **In-process async scheduler, not Prefect server** — Prefect 3.6 runs in ephemeral mode (no server). Native scheduling (`CronSchedule`, `IntervalSchedule`) requires a Prefect server or deployments, doubling infrastructure for a single-VM app. Instead: database-stored schedules + async polling loop in the FastAPI lifespan. Prefect stays for workflow execution (`@flow`/`@task`).
- **`croniter` for cron parsing** — lightweight (~400KB, no transitive deps), provides `croniter.get_next(datetime)` and `croniter.is_valid()`. No APScheduler/Celery needed.
- **Single `schedules` table** — unified table with `target_type` discriminator (`"workflow"` | `"sync"`) avoids duplicating schema, migration, repository, and CRUD use cases.
- **Pre-computed `next_run_at`** — indexed column makes the polling query a simple `WHERE status = 'enabled' AND next_run_at <= NOW()` index scan, not O(all-schedules) croniter evaluation.
- **60-second poll interval** — balances responsiveness with Neon keep-alive (prevents cold starts) and resource efficiency on the 512MB Fly.io VM.

- [ ] **Schedule CRUD Use Cases**
    - Effort: M
    - What: Create, update, delete, get, list, toggle use cases following Command/Result pattern, plus `compute_next_run()` helper
    - Why: Application-layer business rules for schedule management — validation, target existence checks, next_run_at computation
    - Dependencies: Schedule Repository, Add `croniter` Dependency

    - Notes:
        - `compute_next_run(schedule_type, cron_expression, interval_seconds, after)` lives here in the application layer (not on the domain entity) because it imports `croniter` — domain purity requires zero third-party deps
        - `CreateScheduleUseCase`: validates cron expression (`croniter.is_valid()`), validates target exists (workflow via `workflow_repo`, sync via `SyncTarget` enum), computes initial `next_run_at`, enforces one-schedule-per-target

        - `UpdateScheduleUseCase`: update cron/interval, **must recompute `next_run_at`** from the new expression (not carry over the old value)
        - `ToggleScheduleUseCase`: recomputes `next_run_at` on enable (from now, not from last run)
        - Known sync targets validated via `SyncTarget` domain literal: `"lastfm:plays"`, `"spotify:likes"`, `"lastfm:likes"`

- [ ] **Scheduler Service (Polling Loop)**
    - Effort: M
    - What: Async background task that polls `schedules` table for due entries and triggers existing use cases
    - Why: The engine that makes scheduling work — bridges schedule config to actual execution
    - Dependencies: Schedule CRUD Use Cases, Scheduler Config
    - Status: Not Started
    - Notes:
        - `src/application/services/scheduler.py` — approved infrastructure bridge (like `runner.py`, `prefect.py`)
        - Main loop: `asyncio.sleep(poll_interval)` → `find_due_schedules(now)` → dispatch due schedules
        - **Structured concurrency**: use `async with asyncio.TaskGroup() as tg:` with `try/except` per task inside the group body — isolates failures per schedule while maintaining structured cancellation on shutdown (per `python-conventions.md`)
        - **Concurrency gate**: `asyncio.Semaphore(settings.scheduler.max_concurrent_scheduled_runs)` limits concurrent scheduled executions — excess schedules wait for a slot rather than firing unbounded
        - **Missed-run policy**: on startup, schedules with `next_run_at` in the past are advanced to the next future fire time and executed once (not once per missed interval). Configurable via `catchup: bool = False` in SchedulerConfig — when False (default), skip all missed fires and compute next future time; when True, run one catchup execution then advance.
        - Workflow dispatch: check `is_workflow_running()` first (in-process guard), then `RunWorkflowUseCase` → `ExecuteWorkflowRunUseCase` (same path as `POST /workflows/{id}/run`)
        - Sync dispatch: map `target_id` → import use case (`ImportTracksUseCase` for `"lastfm:plays"`, `ImportSpotifyLikesUseCase` for `"spotify:likes"`, `ExportLastFmLikesUseCase` for `"lastfm:likes"`)
        - Optimistic lock via `mark_schedule_started` prevents double-execution across poll cycles — `WHERE status = 'enabled' AND next_run_at = :expected` ensures a disabled schedule can't be started in the race window
        - `WorkflowAlreadyRunningError` caught gracefully — schedule advances to next fire time, no retry storm
        - OAuth tokens: import use cases handle refresh internally; full expiry recorded as failure via `mark_schedule_failed`, schedule stays enabled but `consecutive_failures` increments

#### Workflow Scheduling Interface Epic

- [ ] **Schedule Management API**
    - Effort: M
    - What: REST endpoints for managing workflow and sync schedules
    - Why: Web UI and external clients need HTTP access to schedule CRUD
    - Dependencies: Schedule CRUD Use Cases
    - Status: Not Started
    - Notes:
        - `PUT /api/v1/workflows/{workflow_id}/schedule` — create-or-replace (PUT for idempotent upsert semantics, returns 201 on create / 200 on update)
        - `GET /api/v1/workflows/{workflow_id}/schedule` — get current schedule
        - `DELETE /api/v1/workflows/{workflow_id}/schedule` — remove schedule
        - `PUT /api/v1/sync/schedules/{target_id}` — create/update sync schedule
        - `GET /api/v1/sync/schedules` — list sync schedules
        - `DELETE /api/v1/sync/schedules/{target_id}` — remove sync schedule
        - `GET /api/v1/schedules` — list all schedules for the current user (both types)
        - Thin handlers: parse request → build Command → `execute_use_case(lambda uow: ..., user_id=user_id)` → serialize Response
        - **User scoping**: all endpoints pass `user_id` from JWT to use cases. `GET /api/v1/schedules` is user-scoped (NOT the cross-user `find_due_schedules` used by the internal scheduler). Sync schedule mutations filter by `(user_id, target_id)` composite — user A cannot modify user B's sync schedule.
        - **Input validation**: `target_id` path param constrained to `[a-z0-9_:.-]+` regex (prevents path traversal). `cron_expression` validated by `croniter.is_valid()` at API boundary — reject with 422 before reaching the database. `config` JSONB validated for max serialized size (4KB).
        - Uses PUT (not PATCH) because schedules are atomic create-or-replace — there's no meaningful partial update of a cron expression

### v0.8.1: Editor Polish, Templates & Playlist Browse
**Goal**: Power-user editor features and curated starting points that make workflow creation faster and more discoverable.

**Context**: v0.4.3 delivered the core visual drag-and-drop editor. This milestone adds polish: sub-flows for complex pipelines, import/export for sharing, curated templates for common patterns, and a browse UI for linking playlists. Templates also inform LLM prompt engineering in v0.9.0. Depends on v0.8.0 scheduling infrastructure being in place.

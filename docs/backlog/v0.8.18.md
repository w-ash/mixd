### v0.8.18: Identity Integrity

**Goal**: Stop the identity-resolution layer from silently corrupting its own confidence data, close the ISRC merge holes that overwrite user metadata, and give the operator a way to notice matching degradation — the "harden in place" direction (D1) from the 2026-07 identity research.

**Prod baseline (2026-07-03, `scripts/sql/identity-quantification.sql`)**: all 13,874 mappings sit at exactly confidence 100 (12,869 constant-assigned `direct`; 1,005 Last.fm rows self-scored to 100 — the FM1c signature); 363 tracks (~2.9%) have a stale/unbacked `spotify_id` column (FM4d confirmed live); 844 multi-identifier ISRCs of which only 7 cross the 10s suspect threshold; all 583 Last.fm connector rows are URL-keyed (the play-import paths have never run against prod); 450 duplicate-canonical groups (901 tracks). Baseline also surfaced two findings beyond the memo: the Neon owner role has BYPASSRLS so RLS is inert in prod (PDR-002 evidence row added), and the Last.fm enrichment re-save inserts an orphaned duplicate canonical on every successful getInfo enrichment (pinned in the characterization net; fixed by the epic-3 reorder).

**Post-deploy verification (2026-07-09, same pack re-run read-only against prod at migration 035)**: migration 035 folded all 583 Last.fm rows to the composite scheme (Q8: 100% `url` → 100% `composite`) with zero rows lost (identical track/mapping/connector counts) and zero merge conflicts (as Q8b's zero case variants predicted); Q6 stays at zero divergent rows — no new confidence corruption since the fix; Q7's stale-`spotify_id` stock is unchanged at 318+45, as expected — the flow is fixed and the stock drains via read-path healing, watched by `stats --health` check #7; the review queue is empty pending the first re-import that touches one of the 7 suspect ISRCs.

**Context**: The identity research pass ([design-space memo](identity-resolution-design-space.md)) found the matching *engine* sound but the machinery around it eroding what it computes: every playlist re-import silently promotes mappings to confidence 100 (`_bump_confidence_if_needed`, live corruption today), the re-resolution fast path discards stored provenance for a synthetic 90, Last.fm's unreliable MBIDs earn ISRC-grade evidence, `save_track` overwrites existing track metadata on ISRC collision, Last.fm identity fragments across three identifier schemes, and a stale-sync bug leaves dead Spotify IDs in the fast-path column after every redirect. None of this is visible to anyone — there are no resolution-quality metrics.

**Why now (sequencing)**: the corruption is active — it compounds with every import. Downstream, v0.10.0's Artist Identity Resolution mirrors the track-matching patterns (building it on broken confidence semantics clones the defects — memo §7), v1.0.1 Apple Music adds a third connector to the same substrate, and v1.1.x/v1.2.x public sharing resolves from this data (prerequisite named in both specs). This is also classic hardening work — a natural continuation of the v0.8.x series.

**What this unlocks**:
- Confidence scores that still mean "how likely is this the right track" months after import
- Safe remaster/ISRC-reuse handling (review instead of silent metadata clobber)
- One Last.fm identity per track (plays and loves stop fragmenting)
- `mixd stats` visibility into resolution health — the "we'd notice if matching degraded" capability
- A characterization net that lets all of the above land without silent behavior change

**Persona**: Weekly Curator — "the foundation has to be trustworthy or nothing built on it is" (data quality is a named core behavior). Every fix here protects the library the curator's workflows depend on.

**Key Design Decisions**:
- **Characterization tests first, deliberately pinning bugs**: each fix commit flips an assertion instead of changing silence. The 9 tests are enumerated in the [design-space memo §4](identity-resolution-design-space.md).
- **Re-encounter is a freshness signal, not evidence** — record it; never overwrite confidence with it.
- **Missing metadata is neutral, not a mismatch** — MISSING comparison levels for title/artist mirror the existing duration handling.
- **Reuse the engine's own suspect check at every ISRC merge site** — no new heuristics.
- **Counters, not dashboards** for drift metrics; baseline-then-alert, no magic thresholds.

#### Identity Integrity Epics

- [x] **Characterization test net for identity flows**

    **Story**: The user wants to harden identity resolution without silently changing what their library does today — several flows this version touches (confidence bumping, fast-path synthesis, redirect handling, Last.fm identifier creation) have *no* test witnessing their current behavior, so any fix could regress matching invisibly. Pin the behavior first; then each fix commit flips an assertion instead of changing silence.

    **Decisions**:
    - **Pin bugs deliberately** (stale denorm sync, dangling cross-discovery canonical) — a failing-later assertion is the cheapest proof a fix landed.
    - Tests live at the level of their subject (integration for repo/import flows, unit for scoring) so they survive refactors of the layer above.

    **Spec**: the 9 characterization tests enumerated in the memo §4 (bump-to-100, fast-path synthesis, redirect denorm ordering, healing mask, Last.fm identifier trifecta, save_track ISRC clobber, cross-discovery dangling canonical, MB empty-metadata auto-accept, promotion-policy divergence). Use `tests.fixtures` factories; testcontainers for the integration set.

    **Tests**: this story *is* tests; done = all 9 green against current behavior.

    **Implementation notes (2026-07-03)**: net landed as `tests/integration/characterization/` (one file per flipping epic) + colocated unit pins in `tests/unit/.../track/test_mapper.py` (FM4b/FM4c) and `tests/unit/.../musicbrainz/test_matching_provider.py` (FM1g, pinned at confidence == 98 exactly). Grew to 10 pins: baseline work surfaced an unlisted defect — the Last.fm enrichment re-save (`Track(id=…)` without `version`) INSERTS an orphaned duplicate canonical — pinned in `test_isrc_merge_sites.py::TestEnrichmentResaveDuplicatesCanonical`, flipped by the epic-3 reorder. Co-flip casualties recorded in docstrings: `test_mapping_origin.py::test_automatic_confidence_is_updated`, `test_lastfm_connector_id.py::TestInwardResolverUsesUrlFormat`.

- [x] **Confidence integrity repair (memo D1: FM1a/1b/1d/1g)**

    **Story**: The user wants a mapping's confidence score to still mean "how likely this is the right track" months after import — today re-imports silently promote 70-confidence guesses to a permanent 100 (FM1a), the re-resolution fast path replaces every stored score with a synthetic 90 (FM1b), and Last.fm's stale MBIDs earn near-certain scores (FM1d) — so the review queue and any future "verified" badge are built on corrupted numbers.

    **Decisions**:
    - **Re-encounter is a freshness signal, not evidence of correctness** — record `last seen` instead of overwriting confidence; the connector track existing again proves nothing about the canonical match.
    - **Provenance over synthesis** — the fast path loads the real mapping row (confidence/method are already in `track_mappings`); no invented constant can stand in for it.
    - **Last.fm MBIDs are hints, not identity** (MetaBrainz's own guidance) — score Last.fm matches as `artist_title` unless the MBID verifies against MusicBrainz (Track-vs-Recording + 301-merge aware).
    - **Missing metadata is neutral, not a mismatch** — add MISSING comparison levels for title/artist; require minimal recording metadata before a MusicBrainz ISRC match can auto-accept (today it auto-accepts at ~98 with empty everything and the suspect check unreachable).

    **Spec**: `repositories/track/connector.py` (`_bump_confidence_if_needed`), `track_identity_service_impl.py` (`get_existing_identity_mappings`), `lastfm/matching_provider.py` (method labeling), `domain/matching/probabilistic.py` + `algorithms.py` (MISSING levels), `musicbrainz/matching_provider.py` (metadata fetch before ISRC accept).

    **Tests**: characterization flips for FM1a/1b/1g + new unit coverage for MISSING levels and Last.fm method demotion; existing `tests/unit/domain/matching/*` extended.

    **Implementation notes (2026-07-03)**: Landed as C1–C5 (commits `882b83af`→`e663d1d2`): MISSING title/artist comparison levels + ISRC-grade evaluation gate; MB `/isrc/{isrc}` returns the full recording; Last.fm MBIDs demoted to `artist_title` (kept in `service_data`); re-encounter records `last_seen_at` (migration 034) instead of bumping confidence; the re-resolution fast path returns stored provenance (`PrimaryMappingDetail`), not a synthetic 90.

    Effort: M | Dependencies: Characterization test net | Status: Completed (2026-07-03)

- [x] **ISRC guard at merge sites (memo D1: FM2a/2c)**

    **Story**: The user imports a 2024 remaster whose label reused the original ISRC and their original track's title, album, and duration get silently overwritten (`save_track` upserts by ISRC and replaces metadata) — or a Last.fm skeletal track and a Spotify match of the same recording end up as two half-mapped canonicals. ISRC reuse across versions is a documented industry reality (IFPI rules are violated at scale — ~295k multi-recording ISRCs in MusicBrainz's own report); the engine has a suspect check for exactly this, but none of the merge sites use it.

    **Decisions**:
    - **Reuse the existing domain check** (`assess_isrc_match_reliability`) at every ISRC-keyed merge/reuse site — no new heuristic; the engine already knows how.
    - **On suspect: don't merge silently** — route to the review queue (or create a distinct canonical where the schema decision in memo D2.5 allows), because a wrong merge destroys user metadata and a queued item doesn't.
    - Fix the cross-discovery dangling-canonical path (FM3b) in the same change — the reuse paths must merge-or-map, never both-and-neither.

    **Spec**: `repositories/track/core.py:save_track` (upsert path), `spotify/inward_resolver.py` (ISRC dedup), `cross_discovery.py` (both reuse paths).

    **Tests**: characterization flips for FM2a/FM3b; new integration case: same-ISRC different-duration import → review, not clobber.

    **Implementation notes (2026-07-03)**: Landed as 3a–3e (commits `5e36ff3b`→`3631abc0`): `MatchMethod.ISRC_SUSPECT` + `queue_isrc_collision_review` (any-status dedupe); `save_track`, Spotify inward dedup, and cross-discovery all reuse `assess_isrc_match_reliability` to route suspect collisions to review + a distinct ISRC-stripped canonical; ingest pre-collects owners and accept-merges the deferred canonical; cross-discovery rewritten to return a `DiscoveryOutcome` and the Last.fm resolver reuses-before-create — one canonical holds both connectors' mappings, killing the FM3b dangling/orphan canonical.

    Effort: M | Dependencies: Characterization test net | Status: Completed (2026-07-03)

- [x] **Last.fm identifier unification (memo D1: FM3a/3c)**

    **Story**: The user's Last.fm plays, loves, and matches for one track can fragment across up to four `connector_tracks` rows because three code paths mint identifiers differently (MBID-or-URL, URL + lowercased `artist::title`, case-preserved `artist::title`) — so play counts split and mappings disagree about which row is "the" track.

    **Decisions**:
    - **One scheme everywhere**: normalized `artist::title` via `make_lastfm_identifier`, applied at all three creation sites — it is the only identifier Last.fm itself keeps stable (URLs encode names and break on rename; its MBIDs are type-confused and stale).
    - **Normalize through `track.getCorrection` before keying** — Last.fm's own canonicalization is the closest thing it has to an identity function, and it is currently unused.
    - One-shot data migration folds existing variant rows (case variants, URL-vs-composite duplicates) with mapping/play reattachment.

    **Spec**: `lastfm/conversions.py:144`, `lastfm/inward_resolver.py`, `cross_discovery.py:102`; Alembic data migration for the fold.

    **Tests**: characterization flip for FM3a (three paths → one row); migration test: seeded variants fold without losing plays.

    **Implementation notes (2026-07-03)**: Landed as 4a–4b (commits `f78e9f1d`, `b51d56e2`): all four mint sites (inward resolver, conversions, matching provider, cross-discovery) now key on `make_lastfm_identifier` of Last.fm-corrected names — new `get_track_correction` fallback when `getInfo` fails; a `LASTFM_RAW_ALIAS` secondary mapping caches the raw spelling when autocorrect diverges. Migration 035 folds existing URL/mbid/`lastfm:`/case variants (RLS-bracketed; mappings, reviews, and playlist pointers reattached; `folded_from` provenance). Trifecta 4→1.

    Effort: M | Dependencies: Characterization test net | Status: Completed (2026-07-03)

- [x] **Healing correctness: denormalized-ID sync + one promotion policy (memo D1: FM4b/4c/4d)**

    **Story**: The user resolves an old Spotify export and every redirected track ends up with its *dead* ID in the fast-path `spotify_id` column (the secondary stale mapping is synced last), which then masks the read-path healing that would have fixed it — and depending on which code path notices a missing primary, a different mapping gets promoted.

    **Decisions**:
    - **Sync the denormalized column only for primary mappings** — it exists to accelerate the fast path, and the fast path must see the live ID.
    - **One promotion policy: highest confidence** — the mapper's first-in-iteration-order promotion contradicts `ensure_primary_for_connector`; two policies for one repair is worse than either.

    **Spec**: `repositories/track/connector.py` (`map_track_to_connector` sync gating), `repositories/track/mapper.py` (promotion selection).

    **Tests**: characterization flips for FM4c/4d; regression: redirect resolution leaves `spotify_id` = current ID.

    **Implementation notes (2026-07-03)**: Landed as C6 (commit `20613555`): the denormalized `spotify_id` column syncs only for primary mappings; the mapper walks mappings first (columns are post-walk fallbacks); promotion delegates to a single highest-confidence policy in `ensure_primary_for_connector` — the mapper's iteration-order promotion is gone.

    Effort: S | Dependencies: Characterization test net | Status: Completed (2026-07-03)

- [x] **Matching drift metrics (memo D1: FM5a)**

    **Story**: The user wants to notice when matching silently degrades (a provider changes an API shape, a normalization regresses) *before* months of plays attach to wrong tracks — today failure surfaces only in debug logs and an unvisited review queue. Production ER systems converge on a small run-over-run metric set; mixd has none of it.

    **Decisions**:
    - **Counters, not dashboards**: fallback-resolution rate, read-path promotion count, dead-ID hits on touch, review inflow, confidence-band distribution per run — surfaced via the existing `stats --health` / integrity-check scaffold (v0.4.9), because that's where an operator already looks.
    - Baseline-then-alert (record normal ranges empirically) rather than fixed thresholds — the published practice; no defensible magic numbers exist.

    **Spec**: counters emitted from the four identity-write entry points; `get_match_method_stats` (`connector.py:1340`) extended; CLI/API surface via existing stats endpoints.

    **Tests**: (unit) counters increment per outcome class; (integration) stats query returns per-method band distribution.

    **Implementation notes (2026-07-03)**: Landed as 6a–6b (commits `ea770544`, `ca4fb32f`): `get_match_method_stats` gains confidence-band counts; new stale-denorm (Q7) + evidence-divergence (Q6) + review-inflow queries feed a "Drift Signals" panel in `stats --matching` and `drift` on `GET /stats/matching`; data-integrity check #7 watches stale denorm IDs. Run-attached counters (redirects, fallbacks, dead-IDs, suspect-deferrals) flow into `operation_runs.counts` via `OperationResult.to_counts()`; the mapper emits a grep-able `read_path_promotion` structured event.

    Effort: M | Dependencies: None (parallel to repairs) | Status: Completed (2026-07-03)

#### Post-Deploy Revisions

- [x] **v0.8.18.1** (2026-07-04) — identity-review hardening: migration 035 collision-safe rename, Last.fm MBID quarantine, per-user review routing, unified promotion tie-break

    **Why:** A post-ship identity/matching review of v0.8.18 found four latent defects behind the shipped behavior: the matching layer's distrust of Last.fm MBIDs was undone in persistence; a survivor-rename cycle could abort migration 035 with an `IntegrityError`; `create_review` dropped non-`default` tenants' reviews onto the RLS server-default; and display vs. promotion could still disagree on equal-confidence ties.
    **What:**
    - **Migration 035** — two-phase park-then-assign rename, so a survivor whose current identifier equals another group's target can't trip the `connector_tracks` unique constraint mid-fold; the fold key is recomputed in Python (`_norm` = `strip().lower()`) to match the runtime mint, since SQL `btrim` leaves the Unicode whitespace the mint strips.
    - **Last.fm MBID quarantine** — `getInfo`'s untrusted *track* MBID no longer writes `connector_track_identifiers['musicbrainz']` (it fed `save_track`'s `uq_tracks_user_mbid` merge key and could collapse distinct recordings); the raw-spelling alias mapping mints `auto_set_primary=False` so it never demotes the corrected import primary.
    - **Per-user review routing** — `create_review`'s upsert keys on `(user_id, track_id, connector_name, connector_track_id)`; non-`default` tenants' reviews were silently taking the server-default `default` (invisible to the owner, rejected under prod RLS).
    - **One promotion policy** — display (`TrackMapper`) and promotion (`ensure_primary_for_connector`) share a single `(confidence desc, id asc)` total order; a new convergence test pins that both pick the same row on a tie.
    - **DRY** — new `compute_duration_diff_ms` domain helper folds the ISRC duration-diff guard from 5 call sites; the drift panel reuses `list_pending_reviews`'s total instead of a second identical `count_pending()` query.

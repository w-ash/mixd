### v0.7.8: Mobile Responsiveness

**Goal**: Mixd works as a first-class app from a phone — every page reads well,
every primary action is reachable, and nothing requires horizontal scrolling on
iPhone-class widths. The responsive pass is *measured*: every primary page gains a
Playwright visual-regression baseline at iPhone 15 Pro and 1280px desktop, and any
unintended layout drift fails CI. The one explicit exception is the workflow graph
editor: on a screen too small to operate React Flow comfortably, the editor refuses
gracefully with a "open this on a larger screen" placeholder that points the user
to runs/history instead.

**Context**: The current web UI was built desktop-first (v0.3.x onward) and has
never had a deliberate responsive pass. Most pages technically render on mobile,
but several break in real use — data tables overflow horizontally, filter rows
push content off-screen, dialogs spill below the keyboard, and the workflow editor
is unusable. The Casual Enthusiast persona is mobile-first by definition (phone is
their primary device); the Curator persona reaches for their phone after a sync to
confirm "did it work?" without booting a laptop. v0.7.8 closes that gap before
v0.8.x scheduling, which leans heavily on the "wake up, glance at phone, confirm
overnight runs succeeded" loop. Couplefins (the sibling project) has working
`BottomSheet`, `BottomNav`, `useIsMobile`, and `useDialogSync` primitives; the
implementer ports those rather than reinventing them. v0.7.8 also lifts mixd's
testing posture by adding pixel-perfect visual regression baselines to Playwright
— a 2026-standard guard against silent layout drift.

**What this unlocks**:
- Phone-first review of imports, syncs, runs, library browsing, tag management, and settings
- Casual Enthusiast persona becomes daily-active without ever opening a laptop
- Foundation for the v0.8.x scheduling experience (post-run notifications + glanceable status are inherently mobile use cases)
- Portable design tokens — the responsive patterns we settle on here become the baseline for every page added in v0.8+
- Visual regression baselines that protect every future change (a typo in a Tailwind class can no longer silently shift the dashboard)
- Explicit, defensible exemption for the workflow editor — sets the precedent that "graph-canvas UX is desktop-only" rather than chasing degraded mobile graph interactions

**Persona**: Casual Enthusiast — phone is the primary device. Weekly Curator
secondary — uses phone for status checks, laptop for deep curation. Tinkerer
tertiary — self-host UX should feel polished on any viewport.

**Manual test**:
1. Open mixd on an iPhone-class viewport (Safari, ~390px wide) → sign in → land on the dashboard with no horizontal scrolling, all primary nav reachable.
2. Tap into Library → filters apply via a mobile-appropriate surface (bottom sheet, not desktop sidebar) → track list scrolls cleanly with each row's primary action reachable by thumb.
3. Run a Spotify sync → progress events render legibly → completion toast renders above the bottom nav, deep-link to Operation Run Log works.
4. Open Tags → manage tags (rename, merge, delete) → confirm dialogs render as bottom sheets, not center modals.
5. Open Workflows → see the list and run history → tap into a workflow's runs → confirm the run-detail page is fully usable on mobile.
6. Tap "Edit workflow" → land on a placeholder screen that explains the editor is desktop-only and offers a "View runs" CTA back to the run history. **The graph canvas is never rendered on mobile.**
7. Rotate to landscape on a phone — content reflows, no broken layout.
8. iPad portrait (~820px wide) — confirm bottom-nav + sheet behaviour applies (mobile shell extends to `lg:`). iPad landscape (~1180px) — confirm desktop shell.
9. iPhone-class viewport with the on-screen keyboard up (focus a search field) — input remains visible, layout uses `svh`/`dvh` so the address bar's show/hide doesn't jank.
10. iPhone with notch (Safari) — the bottom-nav sits *above* the home indicator, not under it (safe-area insets working).
11. `pnpm --prefix web test:e2e` — visual regression suite passes; intentional design changes require an explicit baseline-update PR.

**Key Design Decisions**:
- **One breakpoint matters: `lg:` (1024px)** — below = mobile shell; at-or-above = desktop shell. iPad portrait (≤1023px) gets the mobile experience; iPad landscape gets desktop. This is *not* the couplefins convention (couplefins uses `sm`/`md`); we're choosing the simpler one-breakpoint model and accepting that iPad portrait users will see bottom-nav. Container queries (`@container`) handle within-component density at intermediate widths so iPad portrait doesn't feel "phone-scaled-up."
- **Mobile-first refactor, desktop unchanged** — Tailwind breakpoint conventions flip from `sm:/md:/lg:` overrides on desktop styles to mobile baseline + `lg:` desktop overrides. Existing desktop pixel-perfect layouts are preserved; the change is additive for narrow viewports.
- **Workflow editor explicitly opts out** — the React Flow canvas is unusable below ~1024px and degraded interactions (touch-friendly node dragging, mobile minimap) are not worth the implementation cost. Below the breakpoint, render a placeholder component with copy ("Workflow editing requires a larger screen") and a CTA to the run history page. List/browse/run flows for workflows remain fully mobile-supported.
- **Port from couplefins, don't invent** — couplefins's `useIsMobile` (renamed `useBreakpoint` in mixd), `useDialogSync`, `BottomSheet`, and `BottomNav` are the source of truth. The implementer audits couplefins first, ports patterns directly, and only deviates with a written reason. Couplefins paths: `couplefins/web/src/hooks/useIsMobile.ts`, `couplefins/web/src/hooks/useDialogSync.ts`, `couplefins/web/src/components/BottomSheet.tsx`, `couplefins/web/src/components/BottomNav.tsx`, `couplefins/web/src/layouts/AppLayout.tsx`. Note: couplefins uses a `sm` cutoff; mixd ports the *primitives* but applies them at the `lg:` cutoff.
- **Bottom sheet over center modal on mobile** — confirm dialogs, filter panels, and detail drawers render as bottom-anchored sheets below `lg:`. Implemented via a new `ResponsiveDialog` primitive that switches between center `Dialog` (desktop) and bottom `Sheet` (mobile) with the same children contract. The `Sheet` primitive is ported from couplefins's `BottomSheet` (native `<dialog>` + `useDialogSync`); we are *not* adding the shadcn `Sheet` from scratch.
- **Touch targets ≥ 44px (WCAG 2.5.5 / Apple HIG)** — every interactive element on mobile meets the AAA minimum, exceeding the WCAG 2.5.8 AA legal floor of 24px. Existing `icon-xs` / `icon-sm` button variants get padded hit-targets on mobile via a `mobile-tap-area` utility rather than being visually resized.
- **Persistent visibility for the post-run toast** — sonner toast container is repositioned: bottom-right on desktop, top-center on mobile (above bottom-nav). z-index audited so the deep-link action is always tappable.
- **Visual regression in CI via Playwright `toHaveScreenshot`** — every primary page gets a baseline at iPhone 15 Pro (390×844) and desktop (1280×800), light and dark themes. Baselines are generated in CI's Docker image (Playwright's official image) to lock font rendering. Diff threshold: `maxDiffPixelRatio: 0.005` (0.5%). On regression, CI uploads the diff image as an artifact; intentional changes require an explicit `--update-snapshots` PR. No external SaaS — everything stays in-tree under `web/e2e/__screenshots__/`.
- **Safe-area insets for iPhone notch / home indicator** — viewport meta gets `viewport-fit=cover`. New Tailwind utilities `pb-safe`, `pt-safe`, `px-safe` map to `env(safe-area-inset-*)`. Bottom-nav and any sticky-bottom surface use `pb-safe`. Bottom sheets respect `safe-area-inset-bottom` for the close-handle gutter.
- **Dynamic viewport units (`svh`/`dvh`)** — full-height surfaces (mobile shell, bottom sheets in fullscreen mode) use `100svh` (small viewport, fits-on-load) instead of `100vh`. Modals that should grow with the visible viewport use `100dvh`. iOS Safari address-bar show/hide no longer causes layout jank.
- **Container queries for component-scoped responsive** — data tables on Library, Playlists, ImportHistory, and WorkflowRunDetail use `@container` (Tailwind v4 native) to collapse to card lists based on the *content area's* width, not the viewport. This makes iPad portrait look intentional rather than phone-scaled-up.
- **View Transitions API for route changes** — Tanstack Router's `viewTransition: true` is enabled per-route on the primary navigation surfaces (Library, Playlists, Workflows, Dashboard). Reduced-motion preferences are respected. Per-route names suffixed with stable IDs to avoid the "duplicate name crashes the transition" gotcha.
- **No new runtime dependencies** — Tailwind v4, Radix, Tanstack Router, and the ported couplefins primitives cover everything. Visual regression uses Playwright's built-in `toHaveScreenshot`. If something doesn't fit, escalate before adding a dep.
- **Test viewport matrix** — Playwright gains `iPhone 15 Pro` and `iPad Air landscape` projects alongside the existing desktop project. Vitest gets a `useBreakpoint` mock helper and a `mockMatchMedia(width)` test utility so component-level responsive branches are verifiable without a browser.

- [ ] **Visual Regression Baseline (Playwright)**

    **Story**: A Tailwind-class typo three weeks from now no longer
    silently shifts the dashboard 4px to the right and ships unnoticed.
    Every primary page has a pixel-baseline at the two reference
    viewports, and unintended drift fails CI with a side-by-side diff
    artifact attached to the PR.

    **Decisions**:
    - Baselines stored in-tree at `web/e2e/__screenshots__/` so diffs
      are reviewable in PR file-changes.
    - Baselines generated in CI's Docker image (Playwright's official
      image) — local generation is *forbidden* to prevent
      OS-font-render drift. Local devs see "baseline missing" and
      open a PR titled `chore: visual baseline update` to regenerate
      via CI.
    - Diff config: `maxDiffPixelRatio: 0.005`,
      `animations: 'disabled'`, `caret: 'hide'`,
      `await document.fonts.ready` before capture.
    - Light + dark theme captured for every screen.
    - Two viewports: iPhone 15 Pro (390×844) and Desktop (1280×800).
    - Test data is deterministic — a Playwright fixture seeds a
      fixed user account with stable timestamps so screenshots
      reproduce across runs.
    - Skeleton borrowed from `couplefins/e2e/screenshots.spec.ts`,
      upgraded from snapshot capture to `toHaveScreenshot` assertions.

    **Spec**:
    - New: `web/e2e/visual.spec.ts` — iterates the page list and
      asserts `toHaveScreenshot` per route × viewport × theme.
    - Modified: `web/playwright.config.ts` — adds `iPhone 15 Pro`
      and `iPad Air landscape` projects, screenshot defaults
      (`maxDiffPixelRatio`, `animations`, `caret`).
    - New: `web/e2e/fixtures/visual-fixtures.ts` — seeds
      deterministic test data (frozen-time clock,
      fixed user, fixed sample tracks/playlists/tags).
    - New: `.github/workflows/visual-regression.yml` (or extend the
      existing web CI) — runs E2E in Playwright Docker image,
      uploads diff artifacts on failure.
    - New: `web/e2e/README.md` — one-page contributor guide:
      "how to update a baseline" (push branch, run `--update-snapshots`
      in CI, commit the regenerated PNGs).

    **Tests**:
    - (playwright visual) Each page in the in-scope list has a
      passing baseline at both viewports × both themes (initial
      seed PR generates ~52 PNGs: 13 pages × 2 viewports × 2 themes).
    - (playwright) Deterministic fixture verified — running the
      visual suite three times in a row produces zero diffs.

- [ ] **Page-by-Page Responsive Pass (with Container Queries + Dynamic Viewport)**

    **Story**: Every primary page reads cleanly on a phone — no
    horizontal scroll, no overlapping controls, no critical action
    pushed off-screen. The pass is page-by-page rather than a global
    "responsive utility CSS" sweep, because each page has its own
    information density and content shape. iPad portrait looks
    intentional, not phone-scaled-up, because tables and cards use
    container queries to densify based on their own width.

    **Decisions**:
    - Data tables collapse to card lists on mobile via container
      queries (`@container`), not media queries. The card shows
      the same primary fields as the table row plus the row's
      primary action; secondary fields collapse into an expand
      control or move to the detail page.
    - Filter rows that take a horizontal toolbar on desktop become
      a bottom-sheet "Filters" panel on mobile, opened by a single
      filter button.
    - Multi-column forms stack to single-column on mobile.
    - Content max-width is removed on mobile (full-bleed); preserved
      at `lg:` to match the desktop reading experience.
    - Full-height surfaces use `100svh` (not `100vh`) to avoid iOS
      Safari address-bar jank.
    - Search/filter inputs that may scroll the page when the keyboard
      opens use `100dvh` for their containing scrollable region so
      content remains reachable.

    **Spec**:
    - Pages in scope (route + page file):
      - `/` → `web/src/pages/Dashboard.tsx`
      - `/library` → `web/src/pages/Library.tsx` (+ table components in `web/src/components/library/`)
      - `/library/:id` → `web/src/pages/TrackDetail.tsx`
      - `/playlists` → `web/src/pages/Playlists.tsx`
      - `/playlists/:id` → `web/src/pages/PlaylistDetail.tsx`
      - `/settings/integrations` → `web/src/pages/settings/Integrations.tsx`
      - `/settings/sync` → `web/src/pages/settings/Sync.tsx`
      - `/settings/tags` → `web/src/pages/settings/Tags.tsx`
      - `/settings/imports` → `web/src/pages/settings/ImportHistoryPage.tsx`
      - `/settings/account` → `web/src/pages/settings/Account.tsx`
      - `/workflows` → `web/src/pages/Workflows.tsx`
      - `/workflows/:id` → `web/src/pages/WorkflowDetail.tsx`
      - `/workflows/:id/runs/:runId` → `web/src/pages/WorkflowRunDetail.tsx`
    - Pages explicitly out of scope: `WorkflowEditor` (covered by
      its own epic).
    - New shared component: `web/src/components/shared/ResponsiveTable.tsx`
      — wraps a tabular dataset in `@container` and renders rows as
      cards under a configurable container-width threshold.
    - Modified: `web/src/theme.css` — add `view-transition-name`
      utility set for use in this epic and the View Transitions epic.

    **Tests**:
    - (vitest) `ResponsiveTable` switches between table and card
      layouts based on a mocked `ResizeObserver` width.
    - (vitest) Each page imports without crashing at 390px and
      1280px (responsive smoke).
    - (playwright/iPhone 15 Pro) smoke run hits every page in scope
      and asserts `document.body.scrollWidth === window.innerWidth`
      (no horizontal scroll).
    - (playwright/iPhone 15 Pro) Library mobile flow: filter chips →
      bottom-sheet filters → apply → result count updates.
    - (playwright visual) baselines updated for every page touched.

- [ ] **Workflow Editor Mobile Gate**

    **Story**: When the user opens a workflow editor URL on a
    too-small screen, they don't get a broken React Flow canvas —
    they get an explicit "this needs a larger screen" placeholder
    with a one-tap path to the workflow's run history (the next
    most useful thing they could be doing).

    **Decisions**:
    - The gate is viewport-based, not feature-flag based. Below
      `lg:`, the editor route renders the placeholder; at-or-above,
      the editor renders normally. Resizing the window past the
      breakpoint hot-swaps to the editor without a route change.
    - Placeholder copy is concrete and actionable: "Workflow editing
      needs a larger screen. View this workflow's runs instead →"
      with a button to `/workflows/{id}/runs`.
    - The mobile gate also applies to the workflow *creation* flow —
      "+ New workflow" on mobile hits the same placeholder. List/run
      remain reachable.

    **Spec**:
    - Modified: `web/src/pages/WorkflowEditor.tsx` (95 LOC) early-returns
      a new `WorkflowEditorMobilePlaceholder` component below `lg:`.
    - New: `web/src/pages/WorkflowEditorMobilePlaceholder.tsx`.
    - Mobile bottom-nav's "Workflows" entry deep-links to the list
      view, not the editor — already the case, just verified.

    **Tests**:
    - (vitest) `WorkflowEditor` renders placeholder below `lg:`.
    - (vitest) Placeholder's CTA links to `/workflows/{id}/runs`.
    - (playwright/iPhone 15 Pro) navigating to `/workflows/{id}/edit`
      shows placeholder, never mounts React Flow (assert via
      `expect(page.locator('.react-flow')).not.toBeVisible()`).
    - (playwright visual) baseline screenshot of the placeholder.

- [ ] **View Transitions API for Route Changes**

    **Story**: When the user taps from Library to a track detail
    page, the transition feels native — no flash of empty layout,
    no jarring full-page swap. The same applies to sidebar/bottom-
    nav navigation. Reduced-motion users get instant transitions
    instead.

    **Decisions**:
    - Tanstack Router's `viewTransition: true` enabled per-route on
      primary navigation surfaces (`/`, `/library`, `/library/:id`,
      `/playlists`, `/playlists/:id`, `/workflows`, `/workflows/:id`).
    - `prefers-reduced-motion: reduce` respected — transitions
      become instant, not slowed-down.
    - `view-transition-name` utility added to `theme.css` for the
      common shared elements (page header, primary list item).
    - Per-route names suffixed with stable IDs (`track-${id}`,
      `playlist-${id}`) to avoid the "duplicate name crashes the
      transition" gotcha.

    **Spec**:
    - Modified: `web/src/router.tsx` — enable `viewTransition: true`
      on the routes listed above.
    - Modified: `web/src/theme.css` — add `view-transition-name-*`
      utilities + a global `@media (prefers-reduced-motion: reduce)`
      override that sets `view-transition-name: none`.
    - Modified: `web/src/pages/Library.tsx` and `Playlists.tsx`
      — list rows tag themselves with stable
      `view-transition-name` per item.
    - Modified: `web/src/pages/TrackDetail.tsx` and
      `PlaylistDetail.tsx` — receiving page tags the same name
      on the detail header.

    **Tests**:
    - (vitest) Reduced-motion media query disables the transition
      utility (DOM has `view-transition-name: none`).
    - (playwright/iPhone 15 Pro) clicking a Library row navigates
      without console errors; `document.startViewTransition` is
      called once per navigation.
    - (manual) Visual review of the transition on iPhone — judge
      whether it feels native; baseline an "after-transition"
      screenshot to catch regressions.

    Effort: S | Dependencies: Responsive Layout Shell, Page-by-Page Responsive Pass | Status: Not Started

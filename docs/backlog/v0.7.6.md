### v0.7.6: Spotify Flow Polish
**Goal**: Smooth the rough edges left in the v0.7.2 tag UX and v0.7.3 Spotify playlist flow so the end-to-end bootstrap (browse → import → map → tag) feels finished, not "ships but has visible gaps."

**Context**: v0.7.2 shipped tagging and v0.7.3 shipped the Spotify Playlist Browser, but both landed with explicitly-deferred polish — cold-start tag suggestions, a tag management page, a pre-import track-resolution preview, per-playlist sync-direction overrides, and proper route-level integration tests. The Weekly Curator who just finished the v0.7.4 bulk bootstrap is the one who notices these gaps first: they want to rename `mood:chill` → `mood:ambient` across 400 tracks, they want to see how many tracks will resolve *before* importing a playlist, and first-time users staring at an empty tag input with no guidance drop off. This sub-version ships the delta that makes the v0.7.x flow feel intentional instead of MVP.

**What this unlocks**:
- First-time taggers see namespace guidance (`mood:`, `energy:`, `context:`, `genre:`) instead of an empty input
- A dedicated Tag Management page for rename / merge / delete at scale
- Pre-import preview: "25 of 30 tracks resolvable" before the user confirms
- Mixed-direction batch imports: Pull some playlists, Push others, in one operation
- Reliable route-level integration tests for the Spotify playlist routes (no more "unit-only" gaps)
- Infrastructure cleanup that unblocks future bulk flows

**Persona**: Weekly Curator — "I've imported everything and I want to maintain it. Renaming a tag across my library should be one click, not a workaround."

**Manual test**:
1. Open Tag Management (new page under Library) → see all tags with track counts → rename `mood:chill` → `mood:ambient` → confirm → open a track previously tagged `mood:chill` → now shows `mood:ambient`.
2. Open Spotify playlist picker → select 3 playlists → Import → see per-playlist track-resolution counts in the confirm dialog ("Chill Vibes: 247 of 250 resolvable"). Toggle one row from Pull to Push → confirm → observe the mixed import runs correctly.
3. First-time user with no tags: open Track Detail → tag input shows the Suggested Tags panel with `mood:`/`energy:`/`context:`/`genre:` namespaces and one-click examples.

**Key Design Decisions**:
- **Polish, not new primitives** — every epic here extends something that already shipped. No new domain concepts, no new connectors. If an epic grows into "we need a new domain model," it moves to a later sub-version.
- **The pre-import preview respects metadata-only browsing** — the preview column in the confirm dialog fetches full tracks **only for playlists with a known snapshot match** (i.e., cached items from a prior import). Unchanged playlists are a free read; unknown playlists say "Resolvable on import" and fall back to post-import counts. Keeps the picker's fast-browse contract intact.
- **Tag rename is an atomic bulk update, not a two-phase add-then-remove** — `UPDATE track_tags SET tag = :new WHERE tag = :old` plus event rows. Two phases would double-emit events; atomicity matters because the tag history is the audit trail.
- **Route integration tests get a reliable connector seam** — the current tests were dropped in v0.7.3 because real OAuth tokens in the test env leak to live Spotify. This epic establishes a pytest fixture that overrides `ServiceConnectorProvider` in the FastAPI app's DI container, usable by every future connector-adjacent integration test.

#### Polish Epics

- [ ] **Cold-Start Suggested Tags Panel**

    **Story**: A new user opens Track Detail for the first time and
    sees an empty tag input with no guidance. They don't know the
    namespace convention (`mood:chill`, `energy:high`), so they
    either type freeform tags with inconsistent formatting or drop
    off. The suggested-tags panel teaches the system on first use:
    show common namespaces with one-click examples, fade away once
    the user has ≥5 tags of their own (the system has learned).

    **Decisions**:
    - Panel visible when `ListTagsResult` is empty — deterministic
      trigger, no time-based heuristics.
    - Suggestions are static, not ML — `mood:`, `energy:`, `context:`,
      `genre:` are the namespaces the Weekly Curator persona uses.
      Machine-suggested tags can come later if a user actually asks.
    - One-click fills the input with `mood:` (cursor after the colon),
      not `mood:chill` — the user picks the value; the system suggests
      the structure.

    **Spec**:
    - `SuggestedTagsPanel` component at `web/src/components/shared/`.
    - Mount inside `TagEditor` on Track Detail when
      `useListTagsApiV1TagsGet` returns `data.length === 0`.
    - Four namespace chips + brief caption "Tags can be namespaced
      like `mood:chill`. Try one of these starters."

    **Tests**:
    - (vitest) Panel renders when tag list is empty.
    - (vitest) Panel hidden once any tag exists.
    - (vitest) Clicking a namespace chip prefills the input with
      `namespace:` and focuses after the colon.

    Effort: S | Dependencies: v0.7.2 (Tag UI) | Status: Not Started

- [ ] **Tag Management Page**

    **Story**: The curator ends up with 200+ tags after bootstrap and
    wants to maintain them — rename `mood:chill` to `mood:ambient`
    after deciding that's the more accurate label, merge
    `context:workout` and `context:gym` (one was a typo), delete
    `TODO:check` once they've processed the queue. All of this needs
    to be one atomic operation across potentially thousands of tagged
    tracks, not a script.

    **Decisions**:
    - Rename is `UPDATE track_tags SET tag = :new WHERE user_id = :u
      AND tag = :old`, plus one event row per affected track with
      `action="rename"`. Single statement, atomic.
    - Merge = rename (target) + delete source, wrapped in one UoW.
    - Delete cascades to the event log — tag removal events are not
      retained when the tag itself is deleted (the audit trail's
      subject no longer exists). This is a product call, not a
      technical necessity; revisit if compliance says otherwise.
    - Bulk operations protected by a confirmation dialog showing
      affected track count before committing.

    **Spec**:
    - `GET /api/v1/tags` already returns tag + count; extend to
      return `{tag, namespace, value, track_count, last_used_at}`.
    - `PATCH /api/v1/tags/{tag}` body `{new_tag: str}` — rename.
    - `DELETE /api/v1/tags/{tag}` — bulk delete across all tracks.
    - `POST /api/v1/tags/merge` body `{source: str, target: str}`.
    - New page at `/library/tags` — sortable table, rename/merge/delete
      actions, confirmation dialogs with affected-track count.
    - Extends `TagRepositoryProtocol` with `rename_tag`, `delete_tag`,
      `merge_tags` (all accept `user_id` kwarg).

    **Tests**:
    - (integration) Rename: 50 tracks tagged `mood:chill` → after
      rename, 50 tracks tagged `mood:ambient`, 0 still on old tag.
    - (integration) Merge: all tracks previously on `source` now on
      `target`; source tag removed.
    - (integration) Delete: cascade removes tag rows but leaves
      tracks intact.
    - (vitest) Confirmation dialog shows affected-track count.

    Effort: M | Dependencies: v0.7.2 (Tag API + UI) | Status: Not Started

- [ ] **Pre-Import Track-Resolution Preview**

    **Story**: The curator selects 5 playlists to import and wants
    to know **before** confirming: "how many of these tracks does
    Mixd already have? How many will be unresolved?" The current
    import dialog shows counts only after the operation runs.
    This epic surfaces the counts up-front where possible without
    breaking the v0.7.3 metadata-only browse contract.

    **Decisions**:
    - Preview only runs for playlists whose `snapshot_id` matches
      a cached `DBConnectorPlaylist` with `len(items) > 0` — i.e.,
      playlists we've previously imported with full tracks. For
      those, the resolution is a cache-only read via
      `connector_repo.find_tracks_by_connectors` on the cached
      items. No Spotify API call.
    - Playlists without a cached full-items snapshot show
      "Resolvable on import" as the preview — honest about the
      unknown rather than making a speculative number up.
    - The preview column replaces the current "247 tracks" count
      on the confirm dialog when a preview is available.

    **Spec**:
    - New `PreviewPlaylistImportsUseCase` — read-only, returns
      `{connector_playlist_id: {total, resolvable, unresolvable} | "unknown"}`.
    - `POST /api/v1/connectors/spotify/playlists/import/preview`
      body `{connector_playlist_ids: [...]}`.
    - `ImportPlaylistsConfirmDialog` fires the preview on mount
      and renders the per-row counts.

    **Tests**:
    - (unit) Preview for uncached playlist → `"unknown"`, no connector call.
    - (unit) Preview for cached playlist with items → exact counts,
      no connector call (cache-only).
    - (integration) `POST /preview` returns correct shape.
    - (vitest) Confirm dialog renders preview counts per row.

    Effort: M | Dependencies: v0.7.3 (Import use case + confirm dialog) | Status: Not Started

- [ ] **Per-Playlist Sync-Direction Override in Batch Import**

    **Story**: The curator's selected batch contains 3 playlists
    they want Mixd to manage (push their edits to Spotify) and 2
    they want Spotify to own (pull any external changes). Today
    this requires two separate import runs because the confirm
    dialog applies one direction to the whole batch. Per-row
    override eliminates the round-trip.

    **Decisions**:
    - Default stays batch-wide (most imports are same-direction) —
      per-row override is a progressive disclosure. Radio at the
      top sets the default; each row has a small toggle to diverge.
    - Backend already accepts one direction per request; this epic
      widens the request schema to `list[{id, sync_direction}]`
      **in addition to** the existing `{ids, sync_direction}` shape.
      Old callers (CLI, tests) keep working.

    **Spec**:
    - `ImportSpotifyPlaylistsRequest` gets an optional
      `overrides: list[{connector_playlist_id: str, sync_direction: Literal["pull","push"]}] | None`
      field. Falls back to the batch-level `sync_direction` when
      no override for a given ID.
    - Use case: applies the override per-playlist in Phase 6 link
      construction.
    - Confirm dialog: radio for default + per-row toggle.

    **Tests**:
    - (unit, use case) Override present → that playlist's link gets
      the overridden direction; others get the default.
    - (unit, use case) No overrides → identical to pre-epic behavior.
    - (integration) `POST /import` accepts overrides and applies them.
    - (vitest) Per-row toggle changes the override for that row only.

    Effort: S | Dependencies: v0.7.3 (Import use case + confirm dialog) | Status: Not Started

      v0.7.3 (cache-hit path + error-status path) and extend with

    **Tests**:

    - Loading strings: "Tagging N tracks…" / "Importing N playlists…".

    **Tests**:
    - (vitest) Loading toast shown while mutation pending, success

#### Infrastructure Polish (ships alongside the UX epics)

- [ ] **Extract `BaseRepository.bulk_insert_returning_inserted`**

    **Story**: Two repositories now hand-roll the same
    `pg_insert(...).on_conflict_do_nothing(...).returning(id)` +
    filter-by-inserted-id pattern: `track/tags.py:add_tags` and
    `playlist/links.py:create_links_batch`. Rule of 3 isn't met,
    but the pattern is stable and proven — extract now so the third
    caller (a future bulk-upsert for preferences or plays) doesn't
    add a fourth divergent copy.

    **Spec**:
    - `BaseRepository.bulk_insert_returning_inserted(entities, conflict_keys, returning_col) -> list[UUID]`
      in `base_repo.py`.
    - Refactor `add_tags` and `create_links_batch` to use it;
      behavior unchanged.

    **Tests**:
    - (integration) Existing tests for both call sites continue to
      pass without modification.

    Effort: XS | Dependencies: — | Status: Not Started

- [ ] **Generic `_model_to_values(db_model)` via SQLAlchemy `inspect()`**

    **Story**: `ConnectorPlaylistMapper.to_values_dict` is hand-written
    — every mapper that adopts bulk upsert will write its own. When a
    third mapper needs this (likely the preferences or plays mappers
    once they get bulk variants), replace all three with a runtime-
    inspected base helper.

    **Spec**:
    - `BaseModelMapper.default_values_dict(db_model) -> dict` using
      `inspect(type(db_model)).mapper.column_attrs` to iterate
      columns.
    - Mappers with custom serialization (like `ConnectorPlaylist`'s
      `items` dict-list) override it; others get it for free.
    - Refactor `ConnectorPlaylistMapper.to_values_dict` to use the
      base helper for all non-`items` columns.

    **Tests**:
    - (unit) Generic helper produces correct dict for a simple
      mapper (no JSON columns, no derived fields).
    - (integration) Bulk upsert on a refactored mapper still round-
      trips correctly.

    Effort: S | Dependencies: A third mapper needing bulk upsert | Status: Not Started

- [ ] **CTE-Based `create_links_batch` (single round-trip)**

    **Story**: `PlaylistLinkRepository.create_links_batch` currently
    does two round-trips: one SELECT to resolve
    `connector_playlist_id` from `(connector_name,
    connector_playlist_identifier)`, then one INSERT. A CTE folds
    these into a single statement. Worth doing only if profiling
    shows the extra RTT matters at realistic batch sizes (current
    max is ~50 links per import, ~10ms extra).

    **Decisions**:
    - Don't do this eagerly — the current two-query version has an
      explicit pre-insert `missing` `ValueError` that a CTE would
      lose. The clarity is worth the RTT until measurement shows
      otherwise.
    - Trigger to prioritize: a user-facing import of 200+ playlists
      (the `--all` flag from the browse/import CLI) consistently
      sees >500ms latency attributable to the two-query pattern.

    **Spec**:
    - `WITH resolved AS (SELECT id FROM connector_playlists WHERE (connector_name, connector_playlist_identifier) IN (...)) INSERT INTO playlist_mappings ... SELECT ... FROM resolved ...`
    - Preserve the `missing` error: post-hoc check on insert count
      vs input count, raise `ValueError` with the missing identifiers.

    **Tests**:
    - (integration) Existing `test_playlist_link_repository.py` tests
      continue to pass.
    - (integration) Performance test: 200 links inserted in <2
      round-trips (verify via query trace / timing).

    Effort: S | Dependencies: Measurement showing the RTT matters | Status: Not Started

#### v0.7.4 Deferred Polish

These epics complete the v0.7.4 mapping flow with UX polish that was
explicitly deferred from the engine, CLI, and Web UI epics. Each
extends already-shipped infrastructure — no new domain primitives.

- [ ] **SSE Progress for Metadata Import**

    **Story**: When the Weekly Curator hits "Import All" on 200
    metadata mappings, the engine takes ~10-20 seconds (HTTP-bounded,
    not DB-bounded) but the UI today shows a spinner and a final
    result. They want live per-mapping progress — "Applying mapping 47
    of 200 — Workout Bangers (47 tracks tagged)" — so they know the
    operation is alive and roughly when it'll finish. The CLI gets the
    same treatment via the existing `progress_coordination_context`
    Rich progress bar.

    **Decisions**:
    - **Reuse the existing `prepare_sse_operation` + `launch_background`
      pattern** from `routes/imports.py`. The engine emits one
      `progress` event per mapping processed (numerator/denominator),
      one `conflict` event per detected conflict, and a terminal
      `complete` / `error` event with the full
      `ImportPlaylistMetadataResult` payload.
    - **`ImportPlaylistMetadataUseCase` accepts an optional
      `ProgressEmitter`** in its constructor (matches the v0.7.0
      pattern from `import_play_history.py`). When None (the
      default), no events fire — keeps unit tests + the CLI's
      synchronous path unchanged.
    - **CLI integration**: `mixd playlist import-metadata` opens a
      `progress_coordination_context(show_live=True)` around the
      run, passing the resulting `RichProgressProvider` as the
      emitter. Same Rich progress bar the v0.7.0 history import uses.
    - **The Web UI subscribes via the existing `useSSEOperation` hook**
      — same hook the playlist import flow uses. Conflict events
      append to a running list rendered as a banner above the
      progress bar.

    **Spec**:
    - Modified: `src/application/use_cases/import_playlist_metadata.py`
      — accept `progress_emitter: ProgressEmitter | None = None` on
      the use case constructor; emit per-mapping `progress` + per-
      conflict `conflict` events.
    - Modified: `src/interface/api/routes/playlist_mappings.py`
      `POST /import` — wrap the use case call in `prepare_sse_operation`
      + `launch_background`; return `{operation_id}` immediately.
    - New: `src/interface/api/routes/imports.py` SSE endpoint already
      handles `playlist_metadata_import` operation type — no new
      streaming infrastructure.
    - Modified: `src/interface/cli/playlist_commands.py` `import-metadata`
      — wrap `run_import_playlist_metadata` in
      `progress_coordination_context`.
    - Modified: `web/src/components/shared/PlaylistMappingsList.tsx`
      "Import All" — invoke via `useSSEOperation`, render progress +
      conflict banner.

    **Tests**:
    - (unit) Use case emits one `progress` event per mapping when an
      emitter is passed; zero events when emitter is None.
    - (integration) `POST /api/v1/playlist-mappings/import` returns
      `{operation_id}`; subsequent SSE stream emits `progress` →
      `complete` events with the same `operation_id`.
    - (vitest) UI shows incrementing progress on each event; renders
      conflict banner when `conflict` events are received.

    Effort: M | Dependencies: v0.7.4 Epic 2 (engine), v0.7.4 Epic 3 (web UI) | Status: Not Started

- [ ] **UI-Surfaced Conflict Warnings**

    **Story**: The metadata-import engine detects when two mappings
    contradict each other (a track is in both a "Starred" and a "Nah"
    playlist) and logs a warning, but the user has no way to see this
    in the UI today. They want an inline banner BEFORE running the
    import — "3 tracks have conflicting preferences across your
    mappings: track-x is in Starred AND Nah" — so they can fix the
    contradiction (drop one mapping, fix the playlist) before
    committing to a stale-feeling result. They also want the resolved
    "winner" surfaced after the run so they can audit the engine's
    most-recently-imported tiebreak.

    **Decisions**:
    - **Pre-import preview reuses the engine's pass-1 aggregation**:
      a new dry-run mode on `ImportPlaylistMetadataUseCase`
      (`dry_run: bool = False`) builds the desired-state dicts and
      detects conflicts WITHOUT writing anything. The route exposes
      this as `POST /api/v1/playlist-mappings/import/preview` —
      mirrors the v0.7.6 *Pre-Import Track-Resolution Preview*
      pattern.
    - **Post-import detail comes from the SSE `conflict` events**
      streamed by *SSE Progress for Metadata Import*. The UI just
      collects and renders them.
    - **Conflict resolution stays log-only at the engine layer** —
      the engine's preference-order tiebreak logic is unchanged. The
      UI gives the user a chance to see + fix; the engine doesn't
      gain new resolution policies.

    **Spec**:
    - Modified: `ImportPlaylistMetadataUseCase` — add `dry_run` flag.
      When True, run pass-1 only; return `(conflicts, projected_counts)`
      without committing.
    - New: `POST /api/v1/playlist-mappings/import/preview` route.
    - New: `web/src/components/shared/ImportConflictsBanner.tsx` —
      collapsible list of conflicts with "fix in mappings list" links.
    - Modified: `MapPlaylistDialog` — when creating a preference
      mapping, run preview against the new mapping + existing
      mappings; warn inline if it would create a new conflict.

    **Tests**:
    - (unit) `dry_run=True` returns conflicts without calling
      `set_preferences` / `add_tags` / `replace_members`.
    - (vitest) Conflict banner renders when preview returns
      conflicts; hidden otherwise.
    - (vitest) Map dialog shows inline warning when adding a
      mapping would create a conflict.

    Effort: M | Dependencies: SSE Progress for Metadata Import | Status: Not Started

- [ ] **Force-Refresh Spotify Playlist Import**

    **Story**: After the v0.7.4 bootstrap, the user updates their
    "Chill Vibes" Spotify playlist (adds 5 tracks, removes 2) and
    wants Mixd to pick up the changes. Today, `mixd playlist
    import-spotify "Chill Vibes"` short-circuits because the cached
    snapshot exists — they have to either run `mixd playlist sync
    <link-id>` (different mental model — link-based) or manually NULL
    the cached snapshot. They want one command/button: "I know this
    Spotify playlist changed; re-fetch and re-apply mappings."

    **Decisions**:
    - **`--refresh` / `force=true` bypasses the snapshot short-circuit
      in both UC1 (`RefreshConnectorPlaylistsUseCase`) and UC2
      (`ImportConnectorPlaylistsAsCanonicalUseCase`).** Skip-check is
      gated on `not force`. Default stays opt-out (preserves API
      quota on routine re-runs).
    - **Force-refresh chains into metadata re-import**: the CLI's
      `import-metadata --refresh` first refreshes affected connector
      playlists, then runs the engine. UI parallel: a "Refresh and
      Re-import" button on the mapping list.
    - **Per-playlist scope, not global**: `--refresh <playlist-name-or-id>`
      refreshes a specific playlist; `--all-mappings --refresh`
      refreshes every playlist that has an active mapping.

    **Spec**:
    - Modified: `RefreshConnectorPlaylistsCommand` — add `force: bool = False`.
    - Modified: `ImportConnectorPlaylistsAsCanonicalCommand` — add `force: bool = False`.
    - Modified: `ImportPlaylistMetadataCommand` — add `refresh_first: bool = False`; when True, calls UC1 with `force=True` for the connector_playlist_ids of all active mappings before running pass 1.
    - Modified: CLI `import-spotify --refresh`, `import-metadata --refresh`.
    - Modified: API `POST /connectors/spotify/playlists/import` accepts `{force: bool}`; `POST /api/v1/playlist-mappings/import` accepts `{refresh_first: bool}`.
    - Modified: Web UI Spotify picker dialog gains "Force re-fetch" toggle; mapping list gains "Refresh & Re-import All".

    **Tests**:
    - (unit) `force=True` skips the `_already_linked` / `has_fresh_cache`
      check; always fetches.
    - (unit) `force=False` (default) preserves snapshot short-circuit.
    - (integration) `import-metadata --refresh` calls UC1 before the
      engine; cached snapshots get updated mid-run.

    Effort: S | Dependencies: v0.7.4 Epics 2+7 | Status: Not Started

- [ ] **Standalone Refresh-Connector-Playlists CLI**

    **Story**: Power users + scripts want to refresh the connector-
    playlist cache without forking into canonical. `RefreshConnector
    PlaylistsUseCase` (UC1, shipped v0.7.4 Epic 7) has no CLI surface
    today — only the heavier `import-spotify` (UC2) is exposed. Adding
    `mixd playlist refresh-spotify` makes the lightweight cache
    refresh scriptable for cron jobs and pre-mapping flows.

    **Decisions**:
    - **Same surface shape as `import-spotify`**: takes name/ID args
      OR `--all`, optional `--refresh` (which here means "force
      bypass even fresh cache"). Reuses
      `_resolve_spotify_playlist_refs` and
      `SpotifyPlaylistView.connector_playlist_db_id` — same plumbing
      the existing commands use.
    - **No new application code** — wraps `run_refresh_connector_playlists`
      that's already exported.

    **Spec**:
    - New CLI command in `src/interface/cli/playlist_commands.py`:
      `mixd playlist refresh-spotify <name-or-id>...` /
      `--all` / `--refresh`.
    - Output via `BatchOperationResult` + `render_batch_summary` —
      "Refreshed / Skipped (already fresh) / Failed" rows.

    **Tests**:
    - (unit) `refresh-spotify <name>` resolves via existing helper +
      calls `run_refresh_connector_playlists` with the right ids.
    - (unit) `refresh-spotify --all` includes every cached playlist.

- [ ] **Per-Mapping `last_applied_at` for Conflict Tiebreak**

    **Story**: When two mappings contradict each other (Star + Nah on
    the same track), the engine today resolves by preference order
    (star > yah > hmm > nah) — but if both are `yah` from different
    playlists, the tiebreak is ill-defined ("first encountered in
    iteration order"). Adding `last_applied_at: datetime` to
    `PlaylistMetadataMapping` lets the engine apply a stable
    "most-recently-imported wins" rule and surfaces it to the UI as
    "this preference came from your most recent import of <playlist
    name>".

    **Decisions**:
    - **`last_applied_at` updates atomically with `replace_members`**
      — same UoW transaction so the snapshot timestamp + member set
      always agree.
    - **Schema migration adds the column nullable** + backfills NULL
      to current `created_at` of the mapping row, then enforces NOT
      NULL.
    - **Conflict tiebreak**: for same-state contradictions, the
      mapping with the most recent `last_applied_at` wins. Stable
      across re-runs.

    **Spec**:
    - Migration: `017_playlist_metadata_mappings_last_applied_at.py`.
    - Modified: `PlaylistMetadataMapping` domain entity — add
      `last_applied_at: datetime | None`.
    - Modified: `ImportPlaylistMetadataUseCase` — pass-1 conflict
      tiebreak uses `last_applied_at`; replace_members_for_mappings
      atomically writes new `last_applied_at` per mapping.

    **Tests**:
    - (integration) Migration backfills existing rows.
    - (unit) Same-state conflict resolves to mapping with later
      `last_applied_at`.
    - (unit) Re-running the engine updates `last_applied_at` on
      processed mappings.

### v0.7.6: Tag Maintenance & Single-Playlist Polish
**Goal**: Smooth the rough edges in tag taxonomy maintenance and single-playlist Spotify import so the curator can confidently import a playlist, map it to tags or preferences, and maintain their tag library — without "ships but has visible gaps."

**Context**: v0.7.2 shipped tagging and v0.7.3 shipped the Spotify Playlist Browser, but both landed with explicitly-deferred polish — a dedicated tag management page and proper route-level integration tests. v0.7.4 shipped the playlist-mapping bootstrap but left the single-playlist refresh path awkward (forces users into the link-based `sync` mental model). The Weekly Curator notices these gaps first: they want to rename `mood:chill` → `mood:ambient` across 400 tracks, and they want a one-command "Spotify changed this playlist; pull it in fresh" flow. This sub-version ships the delta that makes single-playlist preference/tag work feel intentional instead of MVP. Bulk-playlist downloading affordances (multi-select batches, `--all` flows, cross-mapping coordination) are deliberately out of scope and live in `unscheduled.md` until bulk-flow demand becomes observable.

**What this unlocks**:
- A dedicated Tag Management page for rename / merge / delete at scale
- One-command "Spotify changed this playlist; pull it in fresh" flow (`refresh-spotify` + `import-spotify --refresh`)
- Reliable route-level integration tests for the Spotify playlist routes — and a reusable connector-test seam for every future connector
- A "something is happening" loading toast on bulk-tag operations

**Persona**: Weekly Curator — "I've imported everything and I want to maintain it. Renaming a tag across my library should be one click, not a workaround."

**Manual test**:
1. Open Tag Management (new page under Settings) → see all tags with track counts → rename `mood:chill` → `mood:ambient` → confirm → open a track previously tagged `mood:chill` → now shows `mood:ambient`. Merge `context:gym` → `context:workout` → tracks consolidate. Delete `TODO:check` → tag gone, tracks intact.
2. External change to a single Spotify playlist → `mixd playlist refresh-spotify "Chill Vibes"` updates the cache without forking → `mixd playlist import-spotify "Chill Vibes" --refresh` re-fetches and re-imports with the latest state, bypassing the snapshot short-circuit.
3. Bulk-tag UX: select 15 tracks → bulk-tag with `mood:chill` → loading toast appears immediately, swaps to success on completion.

**Key Design Decisions**:
- **Polish, not new primitives** — every epic here extends something that already shipped. No new domain concepts, no new connectors. If an epic grows into "we need a new domain model," it moves to a later sub-version.
- **Tag rename is an atomic bulk update, not a two-phase add-then-remove** — `UPDATE track_tags SET tag = :new WHERE tag = :old` plus event rows. Two phases would double-emit events; atomicity matters because the tag history is the audit trail.
- **Route integration tests get a reliable connector seam** — the Spotify playlist routes ship today with no integration coverage because real OAuth tokens in the test env leak to live Spotify. This epic establishes a pytest fixture that overrides `ServiceConnectorProvider` in the FastAPI app's DI container, usable by every future connector-adjacent integration test (Apple Music, Tidal, etc.).

#### Polish Epics

- [x] **Tag Management Page**

    **Story**: The curator ends up with 200+ tags after bootstrap and
    wants to maintain them — rename `mood:chill` to `mood:ambient`
    after deciding that's the more accurate label, merge
    `context:workout` and `context:gym` (one was a typo), delete
    `TODO:check` once they've processed the queue. All of this needs
    to be one atomic operation across potentially thousands of tagged
    tracks, not a script.

    **Decisions**:
    - Rename is `UPDATE track_tags SET tag = :new WHERE user_id = :u
      AND tag = :old`, plus one event row per affected track with
      `action="rename"`. Single statement, atomic.
    - Merge = rename (target) + delete source, wrapped in one UoW.
    - Delete cascades to the event log — tag removal events are not
      retained when the tag itself is deleted (the audit trail's
      subject no longer exists). This is a product call, not a
      technical necessity; revisit if compliance says otherwise.
    - Bulk operations protected by a confirmation dialog showing
      affected track count before committing.

    **Spec**:
    - `GET /api/v1/tags` already returns tag + count; extend to
      return `{tag, namespace, value, track_count, last_used_at}`.
    - `PATCH /api/v1/tags/{tag}` body `{new_tag: str}` — rename.
    - `DELETE /api/v1/tags/{tag}` — bulk delete across all tracks.
    - `POST /api/v1/tags/merge` body `{source: str, target: str}`.
    - New page at `/settings/tags` — sortable table, rename/merge/delete
      actions, confirmation dialogs with affected-track count.
    - Extends `TagRepositoryProtocol` with `rename_tag`, `delete_tag`,
      `merge_tags` (all accept `user_id` kwarg).

    **Tests**:
    - (integration) Rename: 50 tracks tagged `mood:chill` → after
      rename, 50 tracks tagged `mood:ambient`, 0 still on old tag.
    - (integration) Merge: all tracks previously on `source` now on
      `target`; source tag removed.
    - (integration) Delete: cascade removes tag rows but leaves
      tracks intact.
    - (vitest) Confirmation dialog shows affected-track count.

    Effort: M | Dependencies: v0.7.2 (Tag API + UI) | Status: Completed (2026-04-22)

- [x] **Route-Level Integration Tests for Spotify Playlist Routes**

    **Story**: The Spotify playlist routes —
    `GET /api/v1/connectors/spotify/playlists` and
    `POST /api/v1/connectors/spotify/playlists/import` — ship today
    with no route-level integration coverage because the
    integration-test environment carries real Spotify OAuth tokens
    that leak to live Spotify on cache-miss. The use cases are
    fully unit-tested, but the route handlers (response envelopes,
    auth dep injection, schema validation) aren't. Every future
    connector-adjacent integration test (Apple Music, Tidal,
    Deezer) will hit the same wall.

    **Decisions**:
    - Fix the seam once, use it everywhere — pytest fixture that
      overrides `ServiceConnectorProvider` in the FastAPI app's DI
      container with a test double. Applies to any route that
      resolves a connector.
    - Override at the **provider** level, not the token-storage
      level. Future connector tests in unscheduled
      (Apple Music in v1.0.1, Tidal/Deezer) inherit a working seam.
    - Tests assert the **route contract** (status code, response
      envelope, validation behavior); use cases stay covered by
      their existing unit tests.

    **Spec**:
    - New fixture `mock_connector_provider` in
      `tests/integration/api/conftest.py` that installs a test
      double on the FastAPI app's DI.
    - New `TestSpotifyPlaylistBrowse` class in
      `tests/integration/api/test_connectors.py`:
      cache-hit path (seeded `DBConnectorPlaylist` rows →
      `from_cache: true`, correct payload shape) + error-status
      path (stubbed connector raises → assert error envelope).
    - New `TestSpotifyPlaylistImport` class for the POST route —
      covers 202 `OperationStartedResponse` shape on a one-element
      `connector_playlist_ids` list (single-playlist case) + 422
      validation error on empty `connector_playlist_ids`.

    **Tests**:
    - (integration) `GET /connectors/spotify/playlists` cache-hit
      path → `from_cache: true`, correct payload shape.
    - (integration) `GET /connectors/spotify/playlists` with stubbed
      connector raising → returns the error envelope (no live API
      call leaks).
    - (integration) `POST /.../import` with single-element ids list
      → 202 + `OperationStartedResponse(operation_id)` shape.
    - (integration) `POST /.../import` with empty `connector_playlist_ids`
      → 422 before hitting the use case.

    Effort: S | Dependencies: — | Status: Completed (2026-04-22)

- [x] **`toast.promise()` Adoption for `BulkTagDialog`**

    **Story**: Bulk-tagging 15+ tracks takes >500ms but the
    `BulkTagDialog` shows a success toast only after the work
    completes — no "something is happening" signal in the interim.
    The toast-promise infrastructure already exists
    (`web/src/lib/toasts.ts:toasts.promise`) but has zero call
    sites. This epic adopts it at the bulk-tag dialog.

    **Decisions**:
    - **Scope: `BulkTagDialog` only** — multi-track-single-tag is a
      preference/tag flow (in scope). The parallel migration for
      `ImportPlaylistsConfirmDialog` is bulk-playlist UX and lives
      in unscheduled.

    **Spec**:
    - `BulkTagDialog` wraps its mutation handler in
      `toasts.promise(batchTag.mutateAsync(...), {
        loading: "Tagging N tracks…",
        success: ({tagged_count, tag}) => ...,
        error: formatApiError,
      })`.

    **Tests**:
    - (vitest) Loading toast shown while mutation pending, success
      toast on resolve, error toast on reject.

    Effort: XS | Dependencies: v0.7.2 (BulkTagDialog) | Status: Completed (2026-04-22)

#### Single-Playlist Refresh Polish

These epics complete the v0.7.4 mapping flow's single-playlist
refresh story so the curator can pick up external changes to one
Spotify playlist without dropping into the link-based `sync` mental
model. Bulk variants (`--all`, `--all-mappings --refresh`,
cross-mapping coordination) are out of scope and live in
`unscheduled.md`.

- [x] **Force-Refresh Single Spotify Playlist**

    **Story**: After the v0.7.4 bootstrap, the user updates one
    "Chill Vibes" Spotify playlist (adds 5 tracks, removes 2) and
    wants Mixd to pick up the changes. Today,
    `mixd playlist import-spotify "Chill Vibes"` short-circuits
    because the cached snapshot exists — they have to either run
    `mixd playlist sync <link-id>` (different mental model —
    link-based) or manually NULL the cached snapshot. They want
    one command: "I know this single playlist changed; re-fetch
    and re-import it fresh."

    **Decisions**:
    - **`--refresh` / `force=true` bypasses the snapshot short-circuit
      in both UC1 (`RefreshConnectorPlaylistsUseCase`) and UC2
      (`ImportConnectorPlaylistsAsCanonicalUseCase`).** Skip-check
      is gated on `not force`. Default stays opt-out (preserves
      API quota on routine re-runs).
    - **Single-playlist scope only** — no `--all-mappings --refresh`
      chain into the metadata engine, no
      `ImportPlaylistMetadataCommand.refresh_first` flag, no
      "Refresh & Re-import All" UI button. The bulk variants live
      in `unscheduled.md`. If a curator wants to refresh multiple
      playlists, they call the command in a loop or wait for the
      bulk epic.

    **Spec**:
    - Modified: `RefreshConnectorPlaylistsCommand` — add `force: bool = False`.
    - Modified: `ImportConnectorPlaylistsAsCanonicalCommand` — add `force: bool = False`.
    - Modified: CLI `import-spotify --refresh`.
    - Modified: API `POST /connectors/spotify/playlists/import` accepts `{force: bool}`.
    - Modified: Web UI Spotify picker dialog gains a per-row "Force re-fetch" toggle.

    **Tests**:
    - (unit) `force=True` skips the `_already_linked` /
      `has_fresh_cache` check; always fetches.
    - (unit) `force=False` (default) preserves snapshot short-circuit.
    - (integration) `POST /import` with `{force: true}` re-fetches
      a previously-imported playlist.

    Effort: S | Dependencies: v0.7.4 Epic 7 | Status: Completed (2026-04-22)

- [x] **Standalone `refresh-spotify` CLI (single-playlist)**

    **Story**: Power users + scripts want to refresh a single
    connector-playlist cache entry without forking into canonical.
    `RefreshConnectorPlaylistsUseCase` (UC1, shipped v0.7.4 Epic 7)
    has no CLI surface today — only the heavier `import-spotify`
    (UC2) is exposed. Adding `mixd playlist refresh-spotify <name>`
    makes the lightweight cache refresh scriptable for pre-mapping
    flows.

    **Decisions**:
    - **Same surface shape as single-playlist `import-spotify`**:
      takes one name/ID arg, optional `--refresh` (force bypass
      even fresh cache). Reuses `_resolve_spotify_playlist_refs`
      and `SpotifyPlaylistView.connector_playlist_db_id` — same
      plumbing the existing commands use.
    - **No `--all` flag** — bulk variant lives in `unscheduled.md`
      under "Bulk Playlist Operations." Single-playlist scope only
      for now.
    - **No new application code** — wraps the already-exported
      `run_refresh_connector_playlists`.

    **Spec**:
    - New CLI command in `src/interface/cli/playlist_commands.py`:
      `mixd playlist refresh-spotify <name-or-id>` / `--refresh`.
    - Output via `BatchOperationResult` + `render_batch_summary`
      for consistency with `import-spotify`, even though it's a
      one-row result.

    **Tests**:
    - (unit) `refresh-spotify <name>` resolves via existing helper
      + calls `run_refresh_connector_playlists` with the right id.
    - (unit) `refresh-spotify` with no args errors cleanly via
      `typer.BadParameter`.

    Effort: XS | Dependencies: v0.7.4 Epic 7 | Status: Completed (2026-04-22)

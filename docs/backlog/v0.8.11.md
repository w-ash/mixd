### v0.8.11: Manual Playlist Track Editing
**Goal**: Let the user edit a canonical playlist's contents directly — add, remove, and reorder tracks from its detail page.

**Context**: A playlist's detail page is read-only — manual curation (add / remove / reorder tracks) has no surface, so users are pushed to connector sync or workflows for something they should be able to do by hand. Web flows 3.4 (add) / 3.5 (reorder) / 3.6 (remove) were specced but never built.

*Already in place*: the DB (incident `e1045cab`: `DBPlaylistTrack.id` + `sort_key` for identity-preserving reorder) and the entry-identity threading the spec called for (v0.8.7 import/sync rearchitecture: `PlaylistEntry.id` `eq=False`, mapper population, entry-id-first matching in `playlist/core.py`). So this slice is purely the *editing surface* — use cases + endpoints + UI + CLI — on top of latent, already-wired identity. Design-space and the incident-hardened path it must respect: [design-debt-findings.md](design-debt-findings.md) §4 (F6). Break into sub-versions (add → remove → reorder) if it warrants.

**What this unlocks**: Hands-on curation from the playlist page itself — add a track you just discovered, drop one that doesn't fit, reorder for flow — without round-tripping through a connector or a workflow.

**Persona**: Curator (hands-on curation), Casual (basic playlist editing without learning workflows).

- [ ] **Playlist track add / remove / reorder via entry identity**

    **Story**: The user wants to add tracks to a playlist, remove individual or batch-selected tracks, and reorder by drag-and-drop or keyboard — directly from the detail page, not only via connector sync or workflow output.

    **Decisions**:
    - **Entry identity is already threaded (shipped v0.8.7).** `PlaylistEntry.id = field(factory=uuid7, eq=False)` exists, the mapper populates it from `DBPlaylistTrack.id`, and `playlist/core.py` matches by `entry.id` first (FIFO fallback for id-less new entries). **`eq=False` is load-bearing** — attrs equality stays value-based so the playlist diff/dedup engine doesn't distinguish otherwise-identical entries. This slice *consumes* that; it does not re-add it.
    - **Duplicate policy decided here**: honor the spec (manual add allows duplicates; the DB is designed for it). This must **not** disturb the workflow-append dedupe filter (`_append_entries`, `update_canonical_playlist.py`), which is justified for its only consumer (weekly append-mode re-runs). Manual add and workflow append are different paths with different correct behavior.
    - **Reorder is full-list semantics** (client sends complete ordered `entry_ids`; repo renumbers `sort_key`); single-track move is an optional later optimization.
    - Independent of the `UpdateCanonicalPlaylistUseCase` mode-split (tracked separately); keep decoupled.

    **Spec**:
    - Domain/repository/schema: `PlaylistEntry.id` + mapper population + entry-id-first matching + `PlaylistEntrySchema.id` are **already in place** (v0.8.7); verify, don't rebuild.
    - Three frozen-Command use cases: `AddPlaylistTracksUseCase` (`playlist_id`, `track_ids`, `position=None`), `RemovePlaylistEntriesUseCase` (`playlist_id`, `entry_ids`), `ReorderPlaylistEntriesUseCase` (`playlist_id`, ordered `entry_ids`).
    - Endpoints (corrected types): `POST /playlists/{id}/tracks`, `DELETE /playlists/{id}/tracks/{entry_id}` + batch `DELETE /playlists/{id}/tracks` (body `entry_ids: UUID[]`), `PATCH /playlists/{id}/tracks/reorder` (`entry_ids: UUID[]`). Handlers 5–10 lines.
    - Frontend: switch row keys from `entry.position` to `entry.id` (`PlaylistDetail.tsx` — latent reconciliation bug under optimistic reorder); Add-Tracks search modal, checkbox/X remove, drag + keyboard reorder with optimistic update + revert-on-failure.
    - CLI parity: `mixd playlist add-tracks` / `remove-tracks` / `reorder`.
    - Docs: flip flows 3.4/3.5/3.6 to Implemented; re-run `pnpm --prefix web sync-api`.

    **Tests**:
    - **(integration, write FIRST — characterization)** Pin current duplicate-FIFO consumption: save `[A,B,A]`, reorder to `[A,A,B]` via the existing replace path; lock record-id stability. Net: `tests/integration/test_playlist_update_preservation_bugs_v2.py` (record-id stability through reorder/duplicate/removal) — do not break it.
    - (integration) `PlaylistEntry.id` survives load→save→load; `eq=False` keeps two same-track entries value-equal.
    - (integration) add appends; batch-remove removes only the named entry_ids (removing one of two identical tracks leaves the other); full-list reorder renumbers `sort_key`, preserves `added_at`.
    - (integration) reorder/remove against a stale `entry_id` → 404, list refetches.
    - (component) optimistic reorder reverts on API failure; rows keyed by `entry.id`.

    Effort: L (reduced — entry-identity threading already shipped in v0.8.7) | Dependencies: v0.8.7 entry-identity threading (✅) | Status: Not Started

### v0.7.4: Tag & Preference Bootstrap
**Goal**: Let the user turn their hundreds of existing Spotify playlists into Mixd tags and preferences in bulk — so years of curation investment transfers into Mixd without re-doing it by hand, and Mixd can become the source of truth for categorization.

**Context**: The user has hundreds of themed Spotify playlists — "Chill Vibes", "Workout", "Late Night", plus "Nah" and "Starred" playlists that function as preference buckets. The playlist browser (v0.7.3) lets them see and import these playlists. Now they need to say "this playlist means `mood:chill`" or "this playlist means `star`" and have that applied across all its tracks. This is a **one-time migration tool, not a perpetual sync** — after the bootstrap, the user manages tags and preferences in Mixd directly. Re-import is available during the transition period (using `snapshot_id` to skip unchanged playlists), but the design optimizes for the initial bulk operation.

**What this unlocks**:
- Hundreds of themed playlists → hundreds of tags, in one bulk operation
- "Nah" playlist → bulk `nah`, "Starred" playlist → bulk `star`
- Tags carry the original Spotify `added_at` dates — "when did I add this to Chill Vibes?" is answerable
- Clean cutover: after bootstrap, Mixd owns the categorization

**Persona**: Weekly Curator — "I have hundreds of playlists organized by theme. I need all of that to become tags in Mixd without re-doing it by hand. And when I remove a track from a Spotify playlist during the transition, Mixd should notice."

**Manual test**: In playlist browser → select "Chill Vibes" playlist → "Map to tag" → type `mood:chill` (auto-suggested from playlist name) → "Import" → open Library → filter by `mood:chill` → see all tracks from that playlist with their Spotify `added_at` dates as `tagged_at`.

**Key Design Decisions**:
- **Thin layer on existing infrastructure** — `PlaylistMetadataMapping` links to `DBConnectorPlaylist` (our cache of external playlist state, always refreshed via Epic 7's `RefreshConnectorPlaylistsUseCase`). This decouples tag-mapping from canonical Playlist materialization: a user can tag-map "Chill Vibes" without forking it into a Mixd-owned editable Playlist.
- **One playlist can have multiple mappings** — "Workout Starred" maps to both `preference=star` AND `tag: context:workout`. A single playlist can carry both types of metadata.
- **Source timestamps from Spotify** — `tagged_at` and `preferred_at` come from the Spotify playlist's `added_at` for each track, not the import date. The user wants to know when they added a track to "Chill Vibes," not when they ran the Mixd import.
- **Removal tracking via membership snapshot** — `playlist_mapping_members` stores the track set from each import. On re-import, diff against snapshot to detect removals. Tracks removed from a Spotify playlist get their mapping-sourced metadata cleared — this is intentional signal (the user removed the track from that category). Manually-set preferences/tags (`source="manual"`) are never auto-cleared.
- **Conflict detection for preferences** — if a track appears in playlists mapped to contradictory preferences (e.g., both "Starred" and "Nah"), log a warning and use the most recently imported mapping. Manual preference always overrides.
- **Playlist name → tag suggestion** — auto-suggest a tag based on the playlist name (e.g., "Workout Bangers" → `context:workout`). Reduces friction during bulk setup.
- **`snapshot_id` for re-import efficiency** — skip playlists unchanged since last import. No need to re-fetch and diff the full track list.

#### Bootstrap Epics

- [x] **Connector Playlist Path Consolidation**

    **Story**: The user wants one code path for "know about an external
    playlist" and one code path for "fork it into Mixd" — not the
    pre-existing drift between `mixd playlist backup` (creates canonical
    `Playlist` with no `PlaylistLink`) and `mixd playlist import-spotify`
    (creates both, Spotify-only). The mental model the user wants:
    **`ConnectorPlaylist` is always created** — it's the cache of "what
    we know to be true about the external playlist (and when we last
    knew it)." **Canonical `Playlist` is optional** — the user can use
    the ConnectorPlaylist alone as a tag-source (v0.7.4 Epic 1+2 flow,
    mapping connector tracks → canonical tracks via the existing
    `ConnectorTrack` mapping) or fork it into a canonical Mixd Playlist
    (the original backup/import functionality). This split also sets up
    Apple Music / Tidal / Last.fm playlist imports later: Spotify is the
    first case of a generic connector-playlist capability.

    **Decisions**:
    - **Two use cases, one shared helper.** `RefreshConnectorPlaylistsUseCase`
      (UC1) refreshes the `DBConnectorPlaylist` cache only. `ImportConnector
      PlaylistAsCanonicalUseCase` (UC2) adds canonical `Playlist` +
      `PlaylistLink` on top. A shared helper `batch_refresh_connector_
      playlists()` in `services/connector_playlist_sync_service.py` holds
      the bounded-concurrent fetch + sequential cache upsert; both use
      cases compose it inside their own UoW.
    - **Naming rule: no unqualified "sync" or "import".** "Sync" collides
      with bidirectional `SyncPlaylistLinkUseCase`; unqualified "import"
      is ambiguous about end state. UC1 is `RefreshConnectorPlaylists`
      (refreshes the cache, specifically). UC2 is `ImportConnector
      PlaylistAsCanonical` (imports INTO Mixd AS a canonical Playlist,
      specifically). Both names are self-evident about side effects.
    - **Compose the existing primitives, don't reimplement.** UC2 calls
      the shared helper, then per-playlist `upsert_canonical_playlist()`
      (CREATE-or-UPDATE Playlist with diff + metrics + track resolution)
      — the same primitive `sync_playlist_link.py` uses for PULL.
      Per-playlist link creation runs if absent, after canonical upsert.
    - **Snapshot short-circuit at two layers.** UC2 first checks
      `(PlaylistLink exists AND cached snapshot non-NULL)` → full skip
      (no HTTP, no DB write). For ids past that check, the shared helper
      applies a cache-only skip: `cached snapshot non-NULL` → skip fetch,
      keep cache. Different skip criteria for different layers.
    - **Connector-side concurrency; DB side sequential.** Phase 1 (HTTP
      fetch) uses `TaskGroup` + `Semaphore` sized by
      `settings.api.spotify.concurrency`. Per-task `try/except` inside
      `_fetch_one` so one playlist's 404 doesn't cancel siblings.
      Phase 2 (DB writes) is sequential because SQLAlchemy async
      sessions aren't concurrency-safe.
    - **Same command creates OR updates.** Re-running `mixd playlist
      import-spotify "Chill Vibes"` after the Spotify playlist changed
      → `upsert_canonical_playlist` handles the diff → entries updated.
      Re-running on an unchanged playlist → short-circuits before fetch.
      No separate `backup` command needed.

    **Spec**:
    - New: `src/application/use_cases/refresh_connector_playlists.py`
      (UC1: `RefreshConnectorPlaylistsUseCase`, Command carries
      `connector_name`, `connector_playlist_ids`)
    - New: `src/application/use_cases/import_connector_playlist_as_canonical.py`
      (UC2: `ImportConnectorPlaylistAsCanonicalUseCase`, Command adds
      `sync_direction`; Result uses `CanonicalImportOutcome` with
      `resolved`/`unresolved` track counts)
    - Modified: `src/application/services/connector_playlist_sync_service.py`
      — added `batch_refresh_connector_playlists()` shared helper plus
      `RefreshedPlaylist` / `RefreshFailure` result types. Does NOT
      commit; caller owns the UoW.
    - Modified: `src/interface/api/routes/connectors.py` — delegates to
      UC2 with `connector_name=SPOTIFY_CONNECTOR`. URL unchanged:
      `POST /connectors/spotify/playlists/import`.
    - Modified: `src/interface/cli/playlist_commands.py` —
      `import-spotify` delegates to UC2; `backup` command removed.
    - Deleted: `src/application/use_cases/import_spotify_playlists.py`,
      `src/application/services/playlist_backup_service.py`, `POST
      /playlists/backup` route, `BackupPlaylistRequest` schema,
      `tests/integration/api/test_playlist_backup.py`, generated
      `backupPlaylistRequest.ts` (auto-removed via `pnpm --prefix web
      sync-api`).
    - Unchanged (composable primitives still consumed by the refactor):
      `sync_connector_playlist()` per-playlist, `upsert_canonical_playlist()`,
      `SyncPlaylistLinkUseCase`.

    **Tests**:
    - (unit) UC1 snapshot short-circuit: cached non-NULL snapshot → no
      fetch, `uow.commit` not called
    - (unit) UC1 null-snapshot triggers fetch + upsert + commit
    - (unit) UC1 per-playlist failure isolation: one 404 → others
      succeed (TaskGroup + per-task try/except)
    - (unit) UC1 connector_name threading → resolves to correct provider
    - (unit) UC2 skip-when-linked: PlaylistLink + snapshot → fully skip
      (no fetch, no canonical call, no link create)
    - (unit) UC2 create path: no link → refresh + canonical create +
      link create + commit
    - (unit) UC2 update path: existing canonical without link → canonical
      UPDATE + link create (fixes legacy backup-state playlists)
    - (unit) UC2 canonical-upsert failure captured as
      `CanonicalImportFailure`; no link created; no commit
    - Deleted: `tests/unit/application/use_cases/test_import_spotify_playlists.py`
      (replaced by UC1 + UC2 test files)

- [x] **Playlist Metadata Mapping Model**

    **Story**: The user wants to say "this Spotify playlist means
    `mood:chill`" and have that relationship stored so Mixd can apply
    the tag to all the playlist's tracks. They also want one playlist
    to carry multiple meanings — "Workout Starred" means both
    `preference=star` and `tag: context:workout`. The mapping model
    is the thin layer that connects an imported playlist (v0.7.3) to
    the metadata it represents.

    **Decisions**:
    - **FK target is `DBConnectorPlaylist`, not `PlaylistLink`.** (Revised
      after Epic 7.) Canonical `Playlist` + `PlaylistLink` are optional
      per the Epic 7 mental model — users can tag-map a Spotify playlist
      without forking it into a canonical Mixd Playlist. FK'ing to
      `PlaylistLink` would force canonical materialization; FK'ing to
      `DBConnectorPlaylist` lets the mapping drive tag application
      straight off the cached connector tracks via the existing
      `ConnectorTrack → Track` mapping.
    - `action_value` validated in the domain entity constructor and
      Pydantic API schema (422 before DB touch), NOT in the SQLAlchemy
      model — `normalize_tag()` and `PreferenceState` are domain
      concepts that can't be imported into infrastructure. This also
      ensures CLI and migration paths that bypass the API still validate.
    - `playlist_mapping_members` is a snapshot that is **replaced** on
      each import (DELETE + INSERT for that mapping_id), not accumulated.
      This keeps the diff correct — if membership rows accumulate,
      a track re-added after removal would produce duplicates and
      the removal diff would break.

    **Spec**:
    - `PlaylistMetadataMapping(user_id, connector_playlist_id, action_type, action_value, id)`
      — `connector_playlist_id` FKs to `DBConnectorPlaylist.id`
    - `action_type: Literal["set_preference", "add_tag"]`
    - `action_value`: `"hmm"|"nah"|"yah"|"star"` for preferences, normalized tag string for tags
    - DB: UNIQUE on `(connector_playlist_id, action_type, action_value)`
    - `playlist_mapping_members`: `mapping_id` (FK), `track_id` (FK→canonical `tracks.id`), `synced_at`, UNIQUE on `(mapping_id, track_id)`

    **Tests**:
    - (integration) UNIQUE constraint prevents duplicate mappings
    - (integration) Invalid `action_value` rejected at creation
    - (integration) FK links to existing `DBConnectorPlaylist` (not `PlaylistLink`)
    - (integration) Mapping can exist for a ConnectorPlaylist that has NO canonical `Playlist` / `PlaylistLink` (validates Epic 7 decoupling)
    - (integration) `action_value` validated against `PreferenceState` / `normalize_tag()`

- [x] **Metadata Import Use Case**

    **Story**: The user has mapped 50 Spotify playlists to tags and
    preferences and now wants to click "Import All" and have it just
    work — thousands of tracks getting tagged and rated based on their
    playlist membership, with the original Spotify dates preserved,
    conflicts warned about, and manual preferences left untouched.
    This is the engine that turns the playlist mappings into actual
    data across the user's entire library.

    **Decisions**:
    - Follows the same "sync from external source" pattern as
      `SyncPreferencesFromLikesUseCase` (v0.7.1): read source →
      check priority via `should_override()` → upsert with source
      timestamp → write event if changed → skip if idempotent.
    - Uses tracks already resolved during playlist import (v0.7.3) —
      no duplicate Spotify API calls.
    - Source timestamps: `preferred_at`/`tagged_at` = Spotify
      playlist's per-track `added_at` date (shared pattern #3).
    - Source priority via shared `should_override()` —
      `playlist_mapping` can't override `manual`.
    - Removal tracking: diff current playlist tracks against
      `playlist_mapping_members` snapshot. Tracks removed get their
      mapping-sourced metadata cleared. Manual metadata never cleared.
    - Conflict detection: contradictory playlists → log warning.
      "Most recently imported" determined by `synced_at` on members.
    - `snapshot_id` change detection: skip unchanged playlists.
    - Re-import warning in UI before running.
    - Progress reporting via existing SSE infrastructure.

    **Spec**:
    - `ImportPlaylistMetadataUseCase` — for each active mapping: read linked playlist tracks → apply action
    - Preference action: upsert `TrackPreference(state, source="playlist_mapping", preferred_at=added_at)`
    - Tag action: `add_tag(tag, source="playlist_mapping", tagged_at=added_at)`, idempotent
    - Removal: diff against `playlist_mapping_members`, clear mapping-sourced metadata for removed tracks
    - SSE progress events

    **Tests**:
    - (unit, mock uow) Applies tag to all resolved tracks with correct `tagged_at`
    - (unit, mock uow) Existing `source=manual` preference → not overwritten
    - (unit, mock uow) Unchanged `snapshot_id` → skip, no work done
    - (integration) Removal tracking: track removed from playlist → mapping-sourced preference cleared
    - (integration) Removal tracking: manually-set preference → not cleared
    - (integration) Conflict: track in contradictory playlists → warning logged AND resulting preference matches most recently imported mapping

    Effort: L | Dependencies: Playlist Metadata Mapping Model | Status: Completed (2026-04-16; SSE + UI-surfaced conflicts deferred to follow-up)

- [x] **Bounded Concurrency for Bulk Playlist Import**

    **Story**: The Weekly Curator has ~200 Spotify playlists and needs
    "Import all" to finish in a reasonable window. The v0.7.3 import
    use case fetches playlists sequentially — acceptable for the
    5-playlist picker flow, painful at 50–200. This epic makes the
    Spotify fetch phase bounded-concurrent so the metadata bootstrap
    from 200 playlists doesn't block the user for minutes.

    **Decisions**:
    - **Bounded, not unbounded** — Spotify enforces a rolling 30-second
      rate-limit per token (~20 req/sec target). Unbounded `gather` on
      200 playlists burns quota and risks 429s mid-batch. Use
      `asyncio.TaskGroup` + `asyncio.Semaphore(N)` with N tuned to the
      target throughput.
    - **DB phases stay bulk** — concurrency only applies to the
      `connector.get_playlist(id)` fetch loop in
      `ImportSpotifyPlaylistsUseCase`. Phases 2-6 already run as single
      bulk operations; no change needed there.
    - **Per-task exception isolation** — `TaskGroup` cancels siblings
      on any unhandled exception. Wrap each task body in try/except to
      preserve the existing "non-atomic fetch" semantic: one playlist's
      404 must not kill the batch.
    - **Concurrency limit is a constant, not a config knob** — pick a
      conservative default (4–8) based on Spotify's rate-limit guidance
      and measured import times. Config-knobs are YAGNI until a user
      demonstrably wants to tune.

    **Spec**:
    - Replace Phase 1 sequential for-loop in
      `src/application/use_cases/import_spotify_playlists.py` with
      `asyncio.TaskGroup` + `Semaphore` bounded concurrent fetch.
    - Preserve `_already_imported` short-circuit — skip checks run
      synchronously before spawning tasks.
    - Preserve `failed: list[ImportFailure]` capture on per-task error.

    **Tests**:
    - (unit, mock client) 20 playlists, semaphore(4) → max 4 concurrent
      `get_playlist` awaits observed via a counter on the mock.
    - (unit, mock client) One task raises → other 19 still succeed
      (no TaskGroup cancellation cascade).
    - (unit, mock client) Per-task order of results not guaranteed —
      assertions use sets or sort-by-id, not list equality.

    Effort: S | Dependencies: v0.7.3 (Import use case) | Status: Completed (2026-04-15)

### v0.7.5: Workflow Integration & Quick Filters
**Goal**: Let the user DO things with their preferences and tags — build preference-aware workflows that replace the Spotify-playlist-based ones, and browse tracks by mood and preference when deciding what to listen to right now.

**Context**: The data model exists (v0.7.0, v0.7.2), the data is populated (migration v0.7.1 + bootstrap v0.7.4), and now the user can act on it. This is where the Spotify-to-Mixd migration pays off: workflows that used to start with `source.liked_tracks` (a flat list) can now filter by preference level and tags. And when the user just wants to decide what to play right now — "show me my starred chill tracks" — quick filters provide instant exploration without building a workflow.

**What this unlocks**:
- "Starred tracks I haven't played in 6 months" — a workflow the user couldn't build before
- "Exclude all nah tracks" — a standard filter in any workflow, replacing the "Nah" playlist exclusion hack
- "Show me my chill tracks" — instant browsing when deciding what to listen to right now
- "Save as Workflow" — when a quick filter reveals a useful pattern, save it for recurring use
- Updated templates that demonstrate preference and tag nodes — onboarding for new workflows

**Persona**: Weekly Curator — "This is why I migrated everything. I want to build workflows using my actual opinions about tracks, not Spotify playlist membership. And when I just want to pick something to listen to, I want to browse by mood."

**Manual test**: Open Library → expand filter panel → select preference "star" + tag "mood:chill" → results update live to show only starred chill tracks → click "Save as Workflow" → workflow editor opens pre-populated → save and run → generates a playlist.

**Key Design Decisions**:
- **Quick filters are NOT workflows** — they are direct API queries (`GET /api/v1/tracks?preference=star&tag=mood:chill`). "Save as Workflow" serializes the filter state into a workflow definition — a one-way conversion. This keeps casual browsing fast (no workflow execution overhead) while providing a bridge to the pipeline system for patterns worth keeping.
- **Parameterized workflows deferred** — runtime parameters ("how many days back?") add significant complexity. Deferred to v0.9.x alongside LLM workflows. v0.7.5 provides fixed-config nodes.
- **Enricher nodes required** — filter/sort nodes need preference and tag data loaded onto the TrackList metadata first. `enricher.preferences` and `enricher.tags` follow the existing `enricher.play_history` pattern. Workflow validation enforces that the required enricher is upstream — same mechanism, no second enforcement system.

#### Workflow & Quick Filter Epics

- [ ] **Preference & Tag Enricher Nodes**

    **Story**: The user wants to build workflows that filter and sort
    by preference and tag — "starred tracks unplayed 6 months",
    "mood:chill tracks sorted by play count". But filter/sort nodes
    can only work with data that's been loaded onto the tracklist.
    Just like `enricher.play_history` loads play data before
    `filter.by_play_history` can use it, preference and tag enrichers
    need to batch-load their data onto the tracklist first.

    **Decisions**:
    - Follows existing enricher pattern from `enricher_nodes.py` —
      not a new mechanism. The pattern is proven and understood.
    - Enricher dependency enforced at workflow validation time — if
      a user adds `filter.by_preference` without an upstream
      `enricher.preferences`, the workflow is rejected with a clear
      error, not a cryptic runtime failure.
    - Batch-loads, not N+1 — at 15k tracks, per-track lookups would
      be catastrophic. Both enrichers use `get_preferences_batch` /
      `get_tags_batch`.

    **Spec**:
    - `enricher.preferences`: batch-loads preferences for all tracks, attaches to TrackList metadata
    - `enricher.tags`: batch-loads tags for all tracks, attaches to TrackList metadata
    - Dependency enforcement via existing `validation.py` mechanism

    **Tests**:
    - (unit) Empty preference table → tracks have null preference in metadata
    - (unit) Empty tags → tracks have empty tag list
    - (unit) Both enrichers call `get_preferences_batch`/`get_tags_batch` exactly once regardless of tracklist size (assert call count, not just result correctness)
    - (unit) Workflow validation rejects filter without upstream enricher

    Effort: S | Dependencies: v0.7.0 (Preferences), v0.7.2 (Tags) | Status: Not Started

- [ ] **Preference & Tag Workflow Nodes**

    **Story**: The user wants to build workflows like "starred tracks
    I haven't played in 6 months" and "all mood:chill tracks sorted
    by play count." These are the building blocks that make
    preference- and tag-aware playlists possible — replacing the
    Spotify-playlist-based workflow inputs the user relied on before.
    The user also wants to exclude nah'd tracks from any workflow
    with a single filter node, replacing the manual "Nah" playlist
    exclusion they do today.

    **Decisions**:
    - `source.preferred_tracks` as a new source node — the user wants
      to start a workflow from "all my starred tracks" without first
      loading all tracks and filtering. Direct DB query is faster
      than load-all → filter.
    - `filter.by_preference` supports both include and exclude modes —
      "only starred" and "everything except nah" are both common
      patterns. The user shouldn't have to negate their intent.
    - `filter.by_tag` supports `any` and `all` match modes —
      "tracks tagged mood:chill OR mood:melancholy" vs "tracks tagged
      BOTH mood:chill AND energy:low."
    - `sorter.by_preference` uses `PREFERENCE_ORDER` domain constant —
      single source of truth, same ordering everywhere.

    **Spec**:
    - **Source**: `source.preferred_tracks` — config `{state: "star", limit: 1000}`
    - **Filter**: `filter.by_preference` — config `{include: ["yah", "star"]}` or `{exclude: ["nah"]}`
    - **Filter**: `filter.by_tag` — config `{tags: ["mood:chill"], match_mode: "any"|"all"}`
    - **Filter**: `filter.by_tag_namespace` — config `{namespace: "mood", values: ["chill", "melancholy"]}`
    - **Sort**: `sorter.by_preference` — config `{order: ["star", "yah", "hmm", "nah"]}`
    - Register in `transform_definitions.py`, `node_catalog.py`, `node_config_fields.py`

    **Tests**:
    - (unit) `filter.by_preference(include=["star"])` keeps only starred
    - (unit) `filter.by_preference(exclude=["nah"])` removes nah'd, keeps unrated
    - (unit) `filter.by_tag(match_mode="all")` requires all tags present
    - (unit) `filter.by_tag(match_mode="any")` keeps tracks with either tag
    - (unit) `filter.by_preference(exclude=["nah"])` keeps tracks with NO preference (unrated) — exclude means "remove nah'd", not "require a preference"
    - (unit) `sorter.by_preference` orders star > yah > hmm > nah > unrated

    Effort: M | Dependencies: Preference & Tag Enricher Nodes | Status: Not Started

- [ ] **Updated Workflow Templates**

    **Story**: The user's first action after migrating is rebuilding
    their Spotify-playlist-based workflows as Mixd-native ones. They
    want templates that show what's now possible — "Starred &
    Forgotten" (loved tracks unplayed 6 months), "Triage Queue" (hmm
    tracks waiting for a listen), "Mood Playlist" (chill tracks
    shuffled). Templates are the bridge between "I have preferences
    and tags" and "I'm building playlists with them." Also serves the
    Tinkerer — well-lit paths for exploring the new node types.

    **Spec**:
    - New: "Triage Queue" — `source.preferred_tracks(state=hmm)` → sort by `preferred_at`
    - New: "Starred & Forgotten" — `source.preferred_tracks(state=star)` → `filter.by_play_history(max_days_back=180, max_plays=0)` → sort → limit 40
    - New: "No Nah Mix" — `source.liked_tracks` → `filter.by_preference(exclude=["nah"])` → existing sort/select
    - New: "Mood Playlist" — `source.preferred_tracks(state=yah)` → `filter.by_tag(tags=["mood:chill"])` → `sorter.weighted_shuffle`
    - Update: "Current Obsessions" to exclude nah'd tracks
    - Update: "Hidden Gems" to use `source.preferred_tracks(state=yah)` instead of `source.liked_tracks`

    Effort: S | Dependencies: Preference & Tag Workflow Nodes | Status: Not Started

- [ ] **Quick Filter UI**

    **Story**: When the user is deciding what to listen to right now —
    "show me my starred chill tracks" — they shouldn't have to build
    a workflow. Quick filters let them browse the Library by preference,
    tag, play count, and date range, with results updating live as
    filters change. When a filter combination turns out to be useful
    enough to reuse every week, "Save as Workflow" bridges the gap —
    serializing the current filters into a workflow definition that
    can be scheduled and run automatically.

    **Decisions**:
    - Quick filters are direct API queries, NOT workflow execution —
      no execution overhead for casual browsing. The user clicks a
      filter and sees results instantly.
    - Filters compose as AND — `preference=star AND tag=mood:chill`.
      OR composition adds UI complexity for minimal gain at this stage.
      The UI should state this: "Quick filters use AND. For OR queries
      (e.g., mood:chill OR mood:melancholy), use `filter.by_tag_namespace`
      in a workflow."
    - Query parameters validated before reaching repository —
      `preference` validated against `PreferenceState` literals,
      `tag` normalized via `normalize_tag()`. A client sending
      `tag=MOOD:CHILL` gets normalized results, not a silent miss.
    - "Save as Workflow" opens the workflow editor pre-populated with
      equivalent nodes. The saved workflow appears in the workflow list
      and is independent — editing the workflow doesn't change the
      quick filter, and vice versa.
    - Compound query performance verified at 15k scale — `preference`
      and `tag` query parameters use the indexed columns from v0.7.0
      and v0.7.2. Verify the query plan before shipping.

    **Spec**:
    - Library: "Filter" panel (expandable sidebar or drawer)
    - Controls: preference state (multi-select), tags (autocomplete multi-select), play count range, date range
    - Results update live as filters change
    - "Save as Workflow" button → opens editor pre-populated
    - API: extend `GET /api/v1/tracks` with `preference` and `tag` query parameters

    **Tests**:
    - (vitest) Filter panel renders and updates results on change
    - (vitest) "Save as Workflow" opens editor with pre-populated nodes
    - (integration) `GET /api/v1/tracks?preference=star&tag=mood:chill` returns only matching tracks
    - (integration) `GET /api/v1/tracks?preference=star` with no matches → empty list, not error
    - (integration) `GET /api/v1/tracks?tag=MOOD:CHILL` → normalized to `mood:chill`, returns correct results
    - (integration, performance) Compound `preference=star&tag=mood:chill` at 15k rows → sub-100ms (verify index usage)

    Effort: L | Dependencies: Preference & Tag Workflow Nodes, Tag Web UI (v0.7.2) | Status: Not Started

# v0.8.8 Import/Sync Web UI — UX Research

**Status**: Active — UX reference library; the anti-pattern catalogue and borrow-kit outlive v0.8.8.

> Companion to [v0.8.7–v0.8.8](v0.8.7-0.8.8.md). Feeds the rewritten v0.8.8 stories. **No code** — this is a reference library: a comparison matrix, distilled principles, a named anti-pattern catalogue, and a "borrow-kit" mapping concrete affordances (microcopy, layout, visual detail) to the Mixd components they'd land in.

## How to use this doc

Each v0.8.8 story cites entries here. When building a screen, find its archetype → read the principles it must satisfy and the anti-patterns it must avoid → pull the borrow-kit row for the exact component + microcopy to adapt. All quoted strings are verbatim from the cited products; adapt, don't paste — bend everything to Mixd's **Dark Editorial Music** aesthetic (`.claude/skills/frontend-design/SKILL.md`).

## The surface, de-costumed

v0.8.8 is four well-studied UX archetypes plus one cross-cutting concern. Every finding hangs off one.

| # | Archetype | v0.8.8 story | The user's question |
|---|---|---|---|
| A | **Preview-then-apply** | Import wizard · sync preview | "What exactly will this change?" |
| B | **Destructive gating** | Real 409 confirm round-trip | "Will I lose something — and is it stopping me?" |
| C | **Background job + re-attach** | Operations provider · sidebar · SSE | "Is it still running? Did it fail? Can I get back to it?" |
| D | **Conflict / unresolved / repair** | Unresolved-tracks list + repair | "Why didn't this match, and how do I fix it?" |
| ✕ | **Direction clarity** (cross-cut) | `SyncDirectionIndicator` / `DirectionChooser` | "Which way is data flowing, and what gets overwritten?" |

## Evaluation rubric (0–3 per dimension)

1. **Consequence clarity** — exact counts/names, not "items"
2. **Friction calibration** — gate proportional to blast radius
3. **Reversibility** — undo/recoverable vs. hard one-way
4. **Progress honesty** — real granular state vs. indeterminate spinner
5. **Discoverability & persistence** — findable from anywhere, survives reload
6. **Microcopy quality** — verb-labeled buttons, plain language
7. **State coverage** — empty / loading / partial / error / success / stale
8. **Visual scannability** — color semantics, counts, hierarchy, +/−/~ legibility
9. **Borrowability** — concrete layout/CSS/timing/copy worth lifting

## What separates good from bad

- **A — Preview:** The best (Terraform) lead with one scannable summary line, color/symbol-code the diff (`+/~/-`), and **collapse the unchanged behind a count** so the diff stays parseable. The worst dump every unchanged item (unparseable diff) or apply with no preview at all.
- **B — Gating:** The best (Vercel) lock the *verb* across title, button, and toast, interpolate the real resource name + counts, and fence irreversibility in a red band. The worst use a generic "Are you sure?" — zero decision-relevant information — which trains autopilot dismissal.
- **C — Re-attach:** The best (Chrome, Vercel, GitHub Actions) treat the stream as a *view over durable server state*: on reconnect they replay backlog then append live, and a global affordance is **silent at zero** but loud on failure. The worst lose progress on reload (orphaned job) or fail invisibly.
- **D — Repair:** The best (Roon, Picard) never drop an unmatched item — they give it a durable, visible status and re-match *in place*. The worst silently drop (iTunes class) or force a full re-import to fix one bad match.
- **✕ — Direction:** The best (GitHub Desktop, FreeFileSync) make the **verb the direction** ("Push to Spotify", per-row ←/→) and disclose the overwritten side *up front*. The worst hide direction behind a generic "Sync" until the irreversible commit (iTunes "Erase and Sync").

## UX principles (the rules every v0.8.8 screen must satisfy)

1. **Count the consequence, don't noun it.** "47 of 50 tracks", "12 to add · 3 to remove · 2 unresolved" — never "items"/"changes". *(Terraform, apt, macOS, Vercel)*
2. **The verb is the button.** Primary action = exact verb + target ("Remove 47 tracks", "Import to 'Roadtrip Mix'", "Push to Spotify"). Verb locked across title = button = success toast. Never "Confirm"/"OK"/"Sync". *(Vercel, GitHub Desktop)*
3. **Friction scales with blast radius.** One threshold, three behaviors: additive/safe → one click (or undo-toast); destructive → blocking modal with counts; near-total wipe → type-to-confirm. *(GitHub, Vercel, macOS)*
4. **The stream is a view, never the source of truth.** Server holds canonical operation state; SSE renders + resumes. On (re)connect, replay backlog *then* append live; add an explicit **"Reconnecting…"** micro-state (the one thing Vercel/GitHub lack). *(Vercel, GitHub Actions — and our `Last-Event-ID`/`evt_N` already supports replay)*
5. **Honest progress, never a lying spinner.** Real granular state ("312 of 1,204 tracks matched", "Importing playlist (step 2 of 4)"). Indeterminate spinners only for 2–10s waits. *(Chrome ring, LogRocket)*
6. **Absence = healthy.** A global indicator silent at zero, appearing only when something runs or fails. *(Linear, Figma)*
7. **Never silently drop; unresolved is first-class.** Unmatched items get a durable, visible status + a roll-up count. *(Roon "Unidentified" badge; the iTunes silent-exclusion anti-pattern)*
8. **Co-locate the remedy with the failure, repair in place.** "Retry failed only", "Fix Match" — never repair-by-reimport. *(GitHub "Re-run failed jobs", Chrome per-row Retry, Plex/Roon/Picard)*
9. **Direction leads with what gets overwritten.** Name the action, show ←/→, disclose the overwritten side up front — not at commit. *(FreeFileSync Mirror copy, GitHub ↑↓; vs the iTunes trap)*
10. **Collapse the unchanged behind a count.** Show adds/removes/unresolved; hide "kept" behind an expandable "45 unchanged tracks hidden". *(Terraform `# (N unchanged … hidden)`)*
11. **Reversibility where deferrable; gate where not.** Our overwrite is hard to undo → gate scaled by removal ratio, and *optionally snapshot prior state* so undo can become real rather than cosmetic. *(Gmail undo vs. confirm decision guidance)*

## Anti-pattern catalogue (named — what to design against)

| Name | Failure mode | Where v0.8.8 is at risk |
|---|---|---|
| **The silent overwrite** | Destructive sync applied with no real confirm | **Live today**: `PlaylistDetail.tsx:759` sends `confirmed: true` (an *ignored* field) — the shipped gate is bypassed |
| **The lying spinner** | Indeterminate spinner, no real state/ETA, for a 30s+ job | Import/sync progress UI |
| **The generic confirm** | "Are you sure?" with no counts/names → autopilot dismissal | The 409 confirm dialog |
| **The silent drop** | Unmatched items vanish — no count, no list, no badge | Unresolved-tracks surfacing (the cardinal sin v0.8.7 backend already fixed) |
| **The directionless sync** | A "Sync" button hides which side wins until the irreversible commit | Direction component |
| **The fire-and-forget** | No acknowledgment a job started → user re-triggers → duplicate work | Import/sync kickoff |
| **The orphaned job** | Progress bound to the tab; reload aborts/loses it | Re-attach provider |
| **The invisible failure** | Background job dies with no notification; past failure undiscoverable | Sidebar / failure surfacing |
| **The unparseable diff** | Every unchanged item dumped inline, burying real changes | Preview diff |
| **Confirm fatigue** | Over-gating routine (additive) actions → banner-blindness | Friction calibration |
| **Repair-by-reimport** | Fixing one bad match requires delete + re-add | Repair flow |

## Simplicity guardrails — what we deliberately did NOT borrow

Research surfaces *more* good ideas than a feature should adopt. The discipline that keeps v0.8.8 simple: **mirror the backend's binary signals; never add friction to a non-destructive action.** The exemplars below are excellent in their original context but would be over-engineering here — each is consciously deferred or dropped.

| Borrowed-from | Affordance | Why it's wrong *here* | What we do instead |
|---|---|---|---|
| GitHub repo delete | **Type-to-confirm** (type the playlist name) *(BK-8)* | Calibrated for *irreversible, identity-level* destruction. A playlist sync is recoverable — typing a name is a hoop, not a safeguard. | One clear confirm modal with the count + a verb button, only when `safety_flagged`. |
| Gmail / Linear | **Undo-toast + prior-state snapshot** *(BK-20)* | A *real* undo needs the engine to snapshot the pre-sync track set — a separate backend feature. A toast without it would lie. | Gate the destructive case up front; add genuine undo later only if we build snapshots. |
| Terraform / Lightroom | **Preview/diff before a first import** *(BK-1–BK-4)* | A first import is purely additive (CREATE path) — there's nothing to lose, so a confirm step is ceremony. The picker already shows counts. | Diff preview lives only on the destructive *sync* confirm, where it's earned. |
| MusicBrainz Picard | **4-color match-confidence ramp** *(BK-15)* | Picard models *fuzzy* match confidence; our entries are *binary* (resolved / unresolved). A ramp implies a spectrum that doesn't exist. | One clear "unresolved" treatment (icon + label) + a roll-up count. |
| Git / GitHub Desktop | **↑↓ ahead/behind counts** *(BK-19)* | A two-way-merge metaphor. Our sync is one-way *replace* (`direction_label` says "replaces X") — ahead/behind would mislead. | Lead with the `direction_label` and a plain `Nothing to sync` empty state. |
| Vercel / GitHub Actions | **"Reconnecting…" status chip** *(BK-10)* | Nice polish, but `useWorkflowSSE` already recovers via snapshot poll — the chip is cosmetic, not load-bearing. | Optional polish; not a required spec item. |
| Plex / Picard / Roon | **Per-track in-place "Fix match"** *(BK-17)* | Needs per-track repair endpoints that don't exist; `/repair` is bulk-only today. | Bulk `Repair unresolved (N)` now; per-track Fix Match → `unscheduled.md`. |

**The litmus test for any new affordance in this epic:** *does the backend expose a decision that needs this UI, or am I decorating a binary?* `safety_flagged` (gate or not), resolved/unresolved (binary), two `direction_label` strings (binary) — the UI should be exactly as simple as those signals.

## Key findings that change the plan

These came out of the codebase audit and **must** be reflected in the v0.8.8 stories:

1. **No import-preview endpoint exists.** There is *no* `POST .../import/preview`; the only pre-commit diff is the **sync** preview on the link side (`SyncPreviewResponse`). "Preview-first import" as originally written assumes a capability that doesn't exist. **Recommendation:** for *re-imports* (which route through `engine.apply(PULL)`), reuse the sync-preview machinery for a full diff; for *first-imports* (CREATE path, nothing to diff against), show a lightweight "what will be imported" summary from picker data (track counts, already-linked status) + an honest post-commit result stream. A full first-import diff requires a new backend endpoint — call it out as an explicit prerequisite if we want it.
2. **`confirmed: true` is an *ignored* field, not just a bypass.** `SyncLinkRequest`'s real field is `confirm_token: str | None`. The client sends `confirmed: true` (unrecognized → dropped), so the shipped staleness-token gate is **entirely unused client-side**. The fix isn't "stop hard-coding true" — it's "wire up the `confirm_token` round-trip that already exists server-side."
3. **`OperationsProvider` is absent.** No global operations context exists. Build it on the proven `WorkflowExecutionContext` split (domain context + SSE-liveness context). `useActiveRuns` is **workflow-scoped** (`/workflows/active-runs`) — a new hook over `GET /operation-runs?status=running` is needed.
4. **The `force` re-fetch toggle is still fully wired** (schema → route → use case → engine), surfaced as the "Force re-fetch" `Switch` in `ImportPlaylistsConfirmDialog.tsx`. Removing it is a real cross-layer task, not just hiding a toggle. (Distinct from the picker's header "Refresh" button, which is a legitimate browse-cache bypass to keep.)
5. **The Motion library is NOT a dependency** (only `tw-animate-css` + CSS), despite design docs referencing it. Keep motion specs CSS-first; don't spec orchestrated framer-motion sequences.
6. **`/repair` is bulk-only.** No request body, returns `{ repaired, still_unresolved }` (two ints, no lists). v0.8.8 scope = the bulk "Repair unresolved (N)" button + count toast. Per-track confidence colors / "Fix match" (Picard/Plex-grade) is a *future* enhancement — flag to `unscheduled.md`, don't over-promise.
7. **No distinct `cancelled` SSE event** — cancellation surfaces as `error`. The UI must distinguish *superseded/cancelled* from *failed* using `final_status`, not by waiting for a cancel event.

## Reusable inventory (what NOT to rebuild)

`OperationProgress` (+ `useOperationProgress`), `ConnectorPlaylistPickerDialog`, `SyncConfirmationDialog` (extendable — already shows +add/−remove/unchanged + `safety_flagged`), `ConfirmationDialog` (default-focus Cancel, destructive variant), `RunStatusBadge` (handles both vocabularies), `ImportPlaylistResultRow`, `Badge`, Sonner `toasts`, SSE transport (`sse-client.ts`, `Last-Event-ID` replay). Build new: `OperationsProvider` + sidebar badge, `SyncDirectionIndicator`/`DirectionChooser`, the client-side 409 `confirm_token` handling.

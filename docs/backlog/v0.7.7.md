### v0.7.7: Operation Run Log
**Goal**: Every long-running operation — playlist import, likes sync, history import, the v0.7.4 metadata engine — produces a persisted log row the user can audit after the fact. A post-run toast points the user to an Import History page where they can browse past runs and see per-item issues.

**Context**: Mixd runs a growing set of long-running operations via SSE (v0.6.8/9, v0.7.5.post2) — Spotify playlist import, likes sync, Last.fm history import, the v0.7.4 `apply-assignments` metadata engine. All of them stream rich progress events while running, but the stream evaporates when the page closes. Users who hit a silent issue — a track that couldn't be imported, a conflict the engine auto-resolved, a playlist that partially failed — have no way to find out after the fact. The Weekly Curator runs a bulk import before bed, closes the tab, and wakes up with no idea whether 200 of 200 succeeded or 197 of 200. This sub-version converts the ephemeral event stream into a persistent, queryable artifact.

**What this unlocks**:
- Persistent record of every import / sync / metadata engine run, scoped per user
- Post-run toast — "Imported 3 playlists · 2 items had issues · View log →" — that deep-links to the run
- `/settings/imports` page listing recent runs newest-first with expandable issue detail
- One choke point (the SSE operation wrappers) covers all current and future long-running operations — zero use-case changes
- Foundation for future observability work (retry-failed-items, notification badge, cross-run analytics) without committing to notification infra now

**Persona**: Weekly Curator — "I ran the bulk import last night. Did anything go wrong? Which tracks?"

**Manual test**:
1. Import 3 Spotify playlists (at least one with an unimportable or region-locked track) → run completes → toast appears reading "Imported 3 playlists · N items had issues · View log →".
2. Click the toast action → land on `/settings/imports` with the just-finished run expanded at the top → see per-item issues with reasons.
3. Navigate to `/settings/imports` directly later → see the last 10 runs sorted newest-first → expand any past run → see the same details.
4. Run a likes-sync → observe it also produces a run row with the same shape (proves the choke point captures more than just playlist imports).

**Key Design Decisions**:
- **Persist at the SSE-operation seam, not per use case** — `prepare_sse_operation` and `launch_background` in `routes/imports.py` are the single choke point every long-running operation already passes through. Writing one `OperationRun` row from there captures playlist imports, likes sync, history import, and metadata engine runs without touching any use case. Use cases stay pure — they keep emitting progress events; the recorder listens.
- **Counts and issues as JSONB, not normalized tables** — rule of 3 isn't met; every operation type's issue shape is different (failed track vs conflict vs rate-limit skip). Normalize when a third consumer needs to query across issue types.
- **Toast deep-links by run ID** — the toast fires on SSE `complete`/`error` and carries the run ID. Clicking opens `/settings/imports?run=<id>` which expands that run inline. No new route needed.
- **"Operation" domain, "Import History" user label** — domain vocab (`OperationRun`) matches existing SSE terminology; user-facing copy stays concrete ("Import History"). Translation happens at the UI seam only.
- **Tier 1 only** — scope is limited to persist-log + list-page + post-run toast. Persistent notification inbox, retry-failed-items, and cross-run analytics are explicitly out of scope; the goal is observability, not a full notification platform.

#### Epics

- [ ] **`OperationRun` Domain Entity + Repository**

    **Story**: Every long-running operation needs a persisted row
    capturing who ran it, what kind, when it started and ended, how
    it ended, and what issues surfaced. The row is written at
    operation start, updated on complete/error. This epic lands the
    domain entity, the table, and the repo — infrastructure only, no
    user-visible change yet.

    **Decisions**:
    - Single table; `counts` and `issues` as JSONB for flexibility
      across operation types.
    - `status` is a string enum in the domain: `running` / `complete`
      / `error` / `cancelled`.
    - `operation_type` is a string, not a DB-level enum — the registry
      of types lives in the application layer (SSE infrastructure)
      and can grow without migrations.
    - `user_id` scoping is mandatory; repo methods accept it as a
      kwarg like every other user-scoped repo.

    **Spec**:
    - New migration `018_operation_runs.py`:
        - `operation_runs`: `id` (UUIDv7 PK), `user_id` (FK→users,
          ON DELETE CASCADE), `operation_type` (str), `started_at`
          (timestamptz), `ended_at` (timestamptz, nullable),
          `status` (str), `counts` (JSONB, default `{}`),
          `issues` (JSONB, default `[]`), `created_at`.
        - Index on `(user_id, started_at DESC)` for the list query.
    - New domain entity `OperationRun` in
      `src/domain/entities/operation_run.py` (frozen, slots,
      attrs-defined).
    - New `OperationRunRepositoryProtocol` in
      `src/domain/repositories/interfaces.py` with methods:
      `create`, `update_status`, `append_issues`, `list_for_user`,
      `get_by_id`.
    - New SQLAlchemy impl under
      `src/infrastructure/persistence/repositories/operation_run/`.

    **Tests**:
    - (integration) Insert + read back preserves all fields
      including JSONB shapes.
    - (integration) `list_for_user` returns rows scoped to user,
      ordered by `started_at DESC`.
    - (integration) CASCADE on user delete removes rows.
    - (integration) `append_issues` is additive (doesn't clobber
      prior issues on repeated calls).

    Effort: S | Dependencies: — | Status: Not Started

- [ ] **SSE Operation Persistence Hook**

    **Story**: The existing `prepare_sse_operation` +
    `launch_background` pattern is the choke point for every
    long-running operation. Teaching that pair to write an
    `OperationRun` row captures observability for every existing
    and future SSE-backed use case in one place — no use case
    touched.

    **Decisions**:
    - `prepare_sse_operation` writes the row with `status=running`
      at kickoff and returns the run ID alongside the operation ID.
    - The background wrapper subscribes to the event stream; on
      terminal events (`complete` / `error` / `cancelled`) it updates
      the row with `ended_at`, final `status`, merged `counts`, and
      any accumulated `issues`. Non-terminal events with an `issue`
      or `error` payload append to `issues` incrementally so a
      disconnected client doesn't lose detail.
    - No changes to the shape of SSE events — the recorder reads
      what's already emitted. Use cases remain pure.

    **Spec**:
    - Modified: `src/interface/api/routes/imports.py` —
      `prepare_sse_operation` and `launch_background` accept a
      `UnitOfWork` for writing to `OperationRunRepository`, and
      return `(operation_id, run_id)`.
    - New module `src/application/services/operation_run_recorder.py`
      — thin service that subscribes to the event stream and
      updates the row through the repo.
    - Existing SSE event types augmented with optional
      `issue: dict | None` field (backwards-compatible addition).

    **Tests**:
    - (integration) Happy path: start → progress → complete → row
      has `status=complete`, final counts, zero issues.
    - (integration) Issues path: start → progress with issue →
      complete → row has `status=complete`, issues array populated.
    - (integration) Error path: start → error event → row has
      `status=error`, error captured in issues.
    - (integration) Orphan case: background task dies without a
      terminal event — row stays `status=running` (documented
      intentional behavior; sweep is future work).

    Effort: M | Dependencies: `OperationRun` Domain Entity + Repository | Status: Not Started

- [ ] **Import History API + Page**

    **Story**: The curator needs a place to browse past runs and
    drill into one to see issues. A list page at
    `/settings/imports` shows recent runs; expanding a row reveals
    the issue list inline.

    **Decisions**:
    - List is paginated (cursor-based, `limit=20` default) —
      matches the paginated-query pattern used elsewhere.
    - Detail is lazy: the list response returns summary
      (`id`, `operation_type`, `started_at`, `ended_at`, `status`,
      `counts`, `issue_count`) without the full issues payload;
      clicking to expand fires `GET /operations/{id}` for the
      full `issues` array.
    - Empty-state copy: "No imports yet. Run one from Settings →
      Connectors or the CLI."

    **Spec**:
    - `GET /api/v1/operations?limit=N&cursor=...` — paginated list,
      user-scoped via auth dep. Returns summary shape.
    - `GET /api/v1/operations/{id}` — full detail with issues. 404
      for other users' runs (not 403, to avoid leaking existence).
    - New page `web/src/pages/settings/ImportHistoryPage.tsx`
      mounted at `/settings/imports`.
    - Deep-link support: `?run=<id>` query param auto-expands that
      run on mount.
    - Navigation: new entry in the Settings nav labeled "Import
      History."

    **Tests**:
    - (integration) `GET /operations` paginates; returns only the
      requesting user's rows.
    - (integration) `GET /operations/{id}` — 200 for owner, 404 for
      non-owner (no existence leak).
    - (vitest) Page renders list; expanding a row fires the detail
      fetch and renders issues.
    - (vitest) Deep-link `?run=<id>` auto-expands matching run on
      load.
    - (vitest) Empty state renders when the list is empty.

    Effort: M | Dependencies: `OperationRun` Domain Entity + Repository | Status: Not Started

- [ ] **Post-Run Toast with Log Deep-Link**

    **Story**: When a long-running operation finishes, the user
    gets a toast that tells them whether anything went wrong and
    gives them a one-click path to the log. 0 issues → quiet
    success toast. ≥1 issues → warning toast with an action that
    deep-links to the run.

    **Decisions**:
    - Reuses the existing `useSSEOperation` hook; the terminal
      event carries both the final result and the run ID.
    - Toast variant driven by issue count: 0 → success (no
      action), 1+ → warning (action: "View log"), error →
      error (action: "View log").
    - Toast copy templates per operation type: "Imported N
      playlists", "Synced N likes", "Imported N scrobbles", "Applied
      N mappings" — keep them concrete instead of generic.

    **Spec**:
    - Modified: `useSSEOperation` returns `runId` and `issueCount`
      in the `onComplete` / `onError` callback payloads.
    - Modified call sites: `ImportPlaylistsConfirmDialog`, history
      import UI, likes sync UI, apply-assignments UI — each adopts
      a shared `formatRunCompletionToast` helper that reads the
      callback payload and emits the right toast variant.
    - New shared helper
      `web/src/lib/toasts.ts:toasts.runCompleted({ operationType, counts, issueCount, runId })`.

    **Tests**:
    - (vitest) Run with 0 issues → success toast, no action.
    - (vitest) Run with N issues → warning toast, action navigates
      to `/settings/imports?run=<id>`.
    - (vitest) Error run → error toast with action.
    - (vitest) Each of the four call sites (playlist import, likes
      sync, history import, apply-assignments) emits a correctly
      formatted toast.

    Effort: S | Dependencies: Import History API + Page | Status: Not Started

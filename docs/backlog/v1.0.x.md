### v1.0.0: Data Quality
**Goal**: Build data quality tools for the existing library. With two connectors already producing mapping scenarios (Spotify + Last.fm), the DQ tooling is useful immediately ŌĆö and becomes the foundation that new connectors benefit from.

**Persona**: Weekly Curator ŌĆö bad data means bad playlists. Fixing mappings, finding unmapped tracks, and detecting stale metadata are core to trusting automated results.

#### Data Quality Epics

- [ ] **Unmapped Tracks Query**
    - Effort: S
    - What: `GetUnmappedTracksUseCase` ŌĆö find tracks without mappings to specific connector
    - Why: Identify gaps like "tracks liked on Spotify but not mapped to Last.fm"
    - Dependencies: None
    - Status: Not Started
    - Notes:
        - Input: connector_name (optional), limit, offset
        - Output: Paginated list of tracks without mappings

- [ ] **Stale Tracks Detection**
    - Effort: S
    - What: `GetStaleTracksUseCase` ŌĆö identify tracks with outdated connector metadata
    - Why: Automated data quality monitoring
    - Dependencies: None
    - Status: Not Started
    - Notes:
        - Configurable staleness threshold (e.g., 30 days)
        - Could trigger refresh workflows

- [ ] **Optimistic Concurrency on Shared Tables**
    - Effort: M
    - What: Add `version` columns and `WHERE version = :expected` guards to shared (non-user-scoped) tables that multiple users write to concurrently
    - Why: Shared tables (`connector_tracks`, `connector_playlists`, `track_metrics`) are the global metadata cache ŌĆö two users importing the same Spotify track simultaneously could silently overwrite each other's connector data. User-scoped tables are protected by isolation; shared tables are not.
    - Dependencies: v0.6.2 Optimistic Concurrency on Tracks (establishes the pattern)
    - Status: Not Started
    - Notes:
        - Tables: `connector_tracks`, `connector_playlists`, `track_metrics`, `playlist_mappings`
        - Pattern established in v0.6.0: `Track.version: int = 0` on domain entity, `DBTrack.version` column, mapper reads actual version from DB
        - Extend to `ConnectorTrack`, `ConnectorPlaylist`, `TrackMetric` domain entities
        - Repository `upsert()` for shared tables: `SET version = version + 1 ... WHERE version = :expected`, raise `ConcurrentModificationError` on conflict
        - Retry strategy: on conflict, re-read from DB and retry (shared metadata is append/update, not user-critical ŌĆö retry is safe)
        - Consider: `SELECT ... FOR UPDATE SKIP LOCKED` for high-contention connector track upserts as an alternative to optimistic retry

- [ ] **Manual Track Mapping & Data Quality Management**
    - Effort: L
    - What: User control over track mapping ŌĆö manual overrides, duplicate merging, confidence-based review
    - Why: Music services disagree on identity; users need authority over their library
    - Dependencies: None
    - Status: Not Started
    - Notes:
        - API: `PATCH /tracks/{id}/mappings/{id}`, `DELETE /tracks/{id}/mappings/{id}`, `GET /connectors/{connector}/search`
        - Frontend: Manual mapping correction UI on Track Detail, "Unmapped" filter in Library, dashboard DQ alerts
        - Interactive step-through for bulk operations

- [ ] **Mapping Supersession + Resolution Event Log** *(scheduled 2026-07-02 from the identity research ŌĆö [memo D2](identity-resolution-design-space.md))*

    **Story**: When a mapping changes ŌĆö an ID dies, a re-match finds better, a human overrides ŌĆö the user wants "we used to believe X" preserved: today the upsert overwrites method/confidence/evidence in place, flip-flops are invisible, and below-threshold rejections leave no memory, so the same wrong candidate is re-fetched and re-rejected forever. This is also the substrate the Manual Mapping UI (above) displays as change history, the prerequisite for v1.0.3's observation ledger, and ŌĆö critically ŌĆö **v1.0.1 Apple Music needs it**: the supersession schema must represent platform-asserted succession (Apple equivalents, Spotify relinking, Tidal `replacement`) distinctly from inferred re-matching, or the second connector forces a migration (memo ┬¦5.3).

    **Decisions**:
    - **Supersede, never overwrite**: `superseded_by_id` / `superseded_at` / typed reason on `track_mappings` (the Wikidata "withdrawn vs deprecated" distinction: *ID died* vs *we were wrong* must stay distinguishable, or accuracy analytics are poisoned).
    - **Append-only resolution event log** with matcher version ŌĆö rejects become negative cache (don't re-propose superseded-as-mismatch candidates); ListenBrainz's production pattern (no-match rows re-checked on exponential backoff, 1dŌåÆ32d cap) is the model for retrying *absent* matches without hammering providers.
    - **Debounce death**: an external ID is suspect after one failure, dead only after N consecutive failures over ŌēźT days (InternetArchiveBot: 3 over 9) ŌĆö Spotify communities document spurious 404s on live IDs.
    - **Every reader becomes supersession-aware by default** (repository-level live-rows filter + partial index) ŌĆö half-aware tooling generating false alarms forever is Wikidata's documented cost of this pattern.

    **Spec**: 2 migrations (`track_mappings` columns; `resolution_events` table, JSONB payload, BRIN on occurred_at); all four identity-write sites route through one application-service seam (which also collapses the "four entry points" drift, memo FM6a). Constants (N=3, T=9d, backoff cap 32d) are starting points, revisit.

    **Tests**: (integration) re-match supersedes rather than overwrites, chain queryable; reject ŌåÆ negative-cache honored on next run; debounce: 2 failures ŌēĀ dead; (unit) supersession-aware default filters.

    Effort: L | Dependencies: v0.8.18 (Confidence integrity repair, ISRC guard) | Status: Not Started

### v1.0.2: Rekordbox Connector
**Goal**: Import locally-owned music from Rekordbox and enrich tracks with audio quality metadata unavailable from streaming APIs.

**Context**: Rekordbox is a local music library, not a cloud API. It represents tracks the user has *purchased or owns* ŌĆö a different class of data from streaming services. The connector reads Rekordbox's local database (SQLite) or XML export, matches tracks to existing canonical entries, and enriches them with metadata that streaming APIs don't provide: BPM, musical key, codec, bitrate, lossless status. This is data sovereignty in its purest form ŌĆö your purchased music collection unified with your streaming history.

**Key differences from streaming connectors**: No HTTP client, no OAuth, no rate limiting. Instead: local file parsing (like Spotify GDPR import). Identity resolution via ISRC (Rekordbox stores ISRCs), artist+title fallback, or MusicBrainz ID. The connector is read-only ŌĆö mixd doesn't write back to Rekordbox.

**Precedent**: `spotify/personal_data.py` + `spotify/play_importer.py` already parse local files (Spotify GDPR export). Rekordbox follows the same pattern: parse local source ŌåÆ resolve to canonical tracks ŌåÆ enrich with source-specific metadata.

### v1.0.3: Cross-User Track Identity

**Goal**: Build a global, append-only ledger of cross-user track-to-service mappings with confidence tiers so public surfaces (sharing, social, recommendations) can resolve tracks reliably without trusting any single user's local state ŌĆö and recover automatically when bad mappings land.

**Context**: Today's matching pipeline (Fellegi-Sunter scoring with ID > MBID > ISRC > Title+Artist hierarchy in `src/domain/matching/`) is sophisticated but per-user: each user's `track_mappings` are RLS-isolated to their own `user_id`. That works for personal libraries but breaks every cross-user surface mixd is heading toward ŌĆö public track sharing (v1.2.x), shareable playlist links (v1.1.3), social context on entity pages (v1.1.2), and any future recommendation system. Worse, it leaves the door open to two adversarial scenarios: (1) **gaming** ŌĆö a malicious sharer crafts a link that resolves to the wrong song on the viewer's preferred service, and (2) **drift** ŌĆö a streaming service rotates a track ID (takedown, re-upload, label change) and every user's stale mapping silently points at the wrong song. The fix is a layer above the existing matcher that treats each match decision as an *observation* rather than a fact, weights observations by contributor trust, and re-validates against external truth (MusicBrainz / ISRC) on a schedule. This is not a rewrite ŌĆö `connector_tracks` is already global, ISRCs are already extracted, MusicBrainz is already wired. v1.0.3 is the abstraction layer that makes the existing data shareable across users safely.

**What this unlocks**:
- Public share-link resolvers (v1.2.x) read canonical mappings without RLS bypass hacks
- v1.1.x social features can show cross-user track context safely
- "Verified" / "best available" UI hints in the v1.2.0 service picker
- Audit trail: every mapping change traceable to its source observation
- Self-healing: monthly re-validation flips bad mappings as new evidence arrives
- Foundation for any future mixd-to-MusicBrainz contribution (give back to the commons)

**Persona**: All three. Weekly Curator ŌĆö reliable mappings for shared playlists. Tinkerer ŌĆö full data sovereignty + audit trail of every mapping decision. Casual Enthusiast ŌĆö share-link resolutions stay correct over time without anyone noticing the machinery.

> **Research revision (2026-07-02)** ŌĆö the identity-resolution and governance research passes ([design-space memo](identity-resolution-design-space.md), [governance memo](identity-governance-design-space.md)) validated this feature's core principle (external APIs as arbiter, users as observers; append-only supersession) and contradicted parts of its machinery at mixd's scale. These revisions govern implementation; where a Key Design Decision below conflicts, this block wins:
> - **Shared tiers collapse to `verified` / `unverified`.** Only externally-verified identities (MusicBrainz-confirmed recording MBID with 301-merge handling, platform-confirmed ISRC, platform-asserted successors) are readable by public/cross-user surfaces. The `strong` tier (Ōēź3 trust-weighted observations) is removed: under open registration it is a three-sock-account quorum (governance memo ┬¦2-S3); MusicBrainz's own quorum rule inverts into an attack accelerator at small N.
> - **TrustScore is parked** (epic below marked Deferred). Activity-earned trust is farmable (Wikipedia extended-confirmed gaming record). Revisit trigger: a feature genuinely requires sharing *unanchored* identities AND invite-tree identity cost (v1.1.3 invite provenance) exists to gate it.
> - **ObservationLedger narrows** to provenance + a candidate queue for the verification worker ŌĆö not a voting substrate. This also removes the queue-rot risk: the queue's consumer is a worker, not a human.
> - **Re-validation mechanics**: MusicBrainz explicitly discourages API polling; trickle at 1 req/s works to ~100k mappings, CC0 dumps/canonical data beyond. "Batch ISRC lookups" here means sequential-under-rate-limit ŌĆö no batch endpoint exists.
> - **Backfill prerequisite**: the D1 confidence repairs ([v0.8.18 Identity Integrity](v0.8.18.md)) and the supersession substrate (v1.0.0's Mapping Supersession epic) must land first ŌĆö stored confidences currently include bump-to-100 inflation that a backfill would launder into tiers.
> - **Last.fm-sourced MBIDs never produce `verified`** without a MusicBrainz recording-level cross-check (they are frequently Track MBIDs, and merge-stale).
> - **Sequencing note**: v1.2.0 no longer hard-depends on this feature ŌĆö public share links resolve via the share-time fan-out cache (governance memo ┬¦2-S1); this feature upgrades that cache's inputs when it ships.

**Key Design Decisions**:
- **External APIs are arbiter, users are observers**: MusicBrainz / ISRC lookups produce `verified` tier evidence that outweighs any crowd-sourced count of user observations. Crowd consensus is the fallback when external truth is unavailable, not the primary signal. This is the gameability defense ŌĆö to corrupt a `verified` identity, an attacker would need to corrupt MusicBrainz itself.
- **Append-only observation ledger, never destructive deletes**: Retracted observations are flagged (`retracted_at`), not removed. Bad mappings are *superseded* (linked to their replacement via `superseded_by_id`), not deleted. This is the self-healing defense ŌĆö every change is reversible, every state is reachable from the audit trail.
- **Confidence tiers, not a single boolean**: `verified` (ISRC/MBID confirmed by MusicBrainz), `strong` (Ōēź3 distinct trust-weighted observations agree), `weak` (1ŌĆō2 observations, no external verification), `disputed` (contradicting observations not yet resolved). The picker UI surfaces the tier subtly so casual viewers can make their own trust decision when verification is incomplete.
- **TrustScore is computed, not user-controlled**: Account age + successful-observation history + disputed-observation penalty, computed by a background worker. Range `[0.0, 10.0]`. New users get `1.0`; can't bias the system on day one. This dilutes burner-account vote-stuffing.
- **No write API for canonical mappings**: Users contribute *observations*; they never directly write `ConnectorIdentity` rows. The aggregator job (or the same-transaction observation handler for new evidence) recomputes tier and winner. Even admin tools work by emitting high-trust observations, not by mutating identities.
- **Privacy by aggregation**: Public resolvers return aggregate counts only, never contributor identity. The `evidence_payload` JSONB is queryable internally for moderation but not exposed publicly.
- **GDPR-aware**: When a user deletes their account, their observations are retracted (not deleted) and the affected identities recompute. The canonical state stays correct because other users' observations remain.

#### Cross-User Identity Epics

- [ ] **ConnectorIdentity Entity & Repository**

    **Story**: The system wants to store a single, globally-readable assertion that "mixd `track_id={uuid}` Ōåö Spotify `track_id=abc123`" with a confidence tier so any non-user-scoped surface (public share links, social previews, recommendation prep) can resolve a track to its external service URLs without violating row-level security or trusting any single user.

    **Decisions**:
    - One row per `(track_id, connector_name, connector_track_id)` triple ŌĆö the identity is the relationship, not a property of one side.
    - **Globally readable** (no RLS) so public surfaces can resolve without bypass hacks.
    - `superseded_by_id` self-FK for soft-replacement when a `disputed` identity is resolved ŌĆö keeps history intact.
    - `confidence_tier` is a derived field, recomputed when observations change. Persisting it (vs computing on read) is the right trade because reads dominate writes.

    **Spec**:
    - New domain entity at `src/domain/entities/connector_identity.py` (frozen attrs, slots)
    - Fields: `id: UUID`, `track_id: UUID` (FKŌåÆtracks), `connector_name: ConnectorName`, `connector_track_id: str` (FKŌåÆconnector_tracks), `confidence_tier: Literal["verified", "strong", "weak", "disputed"]`, `total_trust_weight: Decimal`, `external_verification_at: datetime | None`, `superseded_by_id: UUID | None`, `created_at`, `updated_at`
    - DB table `connector_identities` ŌĆö **no `user_id` column**, no RLS policy
    - Unique on `(track_id, connector_name, connector_track_id)`
    - Index on `(track_id, confidence_tier)` for the public-resolver query
    - Repository protocol in `src/domain/repositories/`, SQLAlchemy impl in `src/infrastructure/persistence/repositories/`, wired into UoW

    **Tests**:
    - (unit) Tier ordering: `verified > strong > weak > disputed`
    - (integration) Two users observing the same `(track_id, connector_track_id)` produce one row, not two
    - (integration) Public resolver reads identity rows without any user session context (RLS bypass not required)

    Effort: M | Dependencies: None | Status: Not Started

- [ ] **ObservationLedger Entity & Repository**

    **Story**: When a user imports a track or a system process verifies a mapping, the system wants to record that decision as an immutable piece of evidence ŌĆö so we can recompute the canonical truth as new evidence arrives, audit any mapping change, and recover from bad data without losing history.

    **Decisions**:
    - Append-only ŌĆö `retracted_at` instead of DELETE ŌĆö because deletes destroy the audit trail and break post-hoc dispute resolution.
    - One observation per `(connector_identity_id, user_id, source_kind)` per user import. A user re-importing the same track doesn't multiply their voice.
    - System sources (`musicbrainz_lookup`, `spotify_api_isrc`) have `user_id = NULL` and `trust_weight_at_observation` set high enough to outweigh plausible user-vote totals.
    - `evidence_payload` is JSONB so we can carry source-specific context (the raw ISRC, the API response timestamp, the matcher's confidence score) without schema churn.

    **Spec**:
    - New domain entity at `src/domain/entities/observation.py`
    - Fields: `id: UUID`, `connector_identity_id: UUID` (FK), `source_kind: ObservationSource`, `user_id: UUID | None` (FKŌåÆusers, nullable for system sources), `trust_weight_at_observation: Decimal`, `evidence_payload: Mapping[str, JsonValue]`, `observed_at: datetime`, `retracted_at: datetime | None`, `retraction_reason: str | None`
    - `ObservationSource = Literal["user_import", "manual_override", "musicbrainz_lookup", "spotify_api_isrc", "merge_result", "admin_override"]`
    - DB table `observations` ŌĆö globally readable; index on `(connector_identity_id, retracted_at)`, BRIN index on `observed_at` for time-range queries
    - Unique on `(connector_identity_id, user_id, source_kind)` where `retracted_at IS NULL` ŌĆö partial unique index

    **Tests**:
    - (integration) Append-only enforcement: repository has no `delete` method
    - (integration) Re-importing a track creates one observation, not duplicates
    - (integration) System source observations can have null `user_id` and aren't constrained by the user-uniqueness rule when source_kind differs

    Effort: M | Dependencies: ConnectorIdentity Entity & Repository | Status: Not Started

- [ ] **TrustScore on Users**

    **Story**: A new burner account shouldn't have the same vote weight as a 2-year-old account with thousands of validated observations ŌĆö so the contribution of each observation to a `ConnectorIdentity`'s `total_trust_weight` is multiplied by the contributor's current `TrustScore`, computed on a schedule from validated history.

    **Decisions**:
    - Computed by background worker, not real-time. Real-time would create a write storm on every observation.
    - Range `[0.0, 10.0]`. New users start at `1.0`; can't manipulate from day one. Banned users at `0.0`. Aged contributors with strong history cap at `10.0`.
    - Stored on `users` table (scalar field), not in a separate `user_trust` table. Single column, no join overhead on observation writes.
    - Frozen at observation time (`trust_weight_at_observation` on the `observations` row). Recomputing all historical observations when a user's score changes would be quadratic; freezing is the right trade.

    **Spec**:
    - Migration: add `trust_score: Decimal NOT NULL DEFAULT 1.0` to `users`
    - New use case `RecomputeTrustScoresUseCase` in `src/application/use_cases/`
    - Inputs to scoring: account age (months capped at 24), count of validated observations (those that became part of `verified` or `strong` identities), count of retracted observations, count of observations on `disputed` identities not in the winning side, observation diversity (distinct days with activity)
    - Background worker schedule: weekly recomputation. Can be triggered ad-hoc by admin.
    - Helper at `src/domain/services/trust.py`: `compute_trust_score(metrics: TrustMetrics) -> Decimal` ŌĆö pure function, no I/O

    **Tests**:
    - (unit) `compute_trust_score` boundary cases: brand-new account ŌåÆ 1.0, old + zero observations ŌåÆ 1.0, old + many validated ŌåÆ high but capped at 10.0, account with many retractions ŌåÆ low
    - (integration) Recomputation is idempotent
    - (integration) Score changes don't retroactively alter past observation weights

    Effort: M | Dependencies: ObservationLedger Entity & Repository | Status: Deferred (2026-07-02 ŌĆö see Research revision block; revisit only with an unanchored-sharing need + invite-tree identity cost)

- [ ] **Backfill Existing Mappings into Observations**

    **Story**: Users have existing `track_mappings` rows from their imports ŌĆö those mappings represent real evidence we shouldn't throw away when v1.0.3 ships. The system wants to seed `ConnectorIdentity` and `ObservationLedger` from existing mapping rows so the new abstraction layer is populated on day one rather than empty.

    **Decisions**:
    - Single-shot migration script (not an ongoing process). After this runs, new observations come from imports going forward.
    - Initial tier from `match_method`: `isrc` or `mbid` ŌåÆ `verified` (after MusicBrainz cross-check), `direct` ŌåÆ `strong`, `artist_title` ŌåÆ `weak`. Tier may upgrade later as more users observe.
    - Each existing `track_mappings` row produces exactly one `user_import` observation with `trust_weight_at_observation = 1.0` (we don't have historical trust scores; default).
    - Existing `track_mappings` rows are NOT deleted ŌĆö they remain as the per-user "which connector identities have I matched" index. New writes during the v1.0.3 transition still write both the mapping and the observation; later versions can drop the dual-write.

    **Spec**:
    - Alembic data migration (separate from the schema migration) at `alembic/versions/0XX_backfill_observations.py`
    - Idempotent: re-running produces no duplicates (uses the partial unique index on `observations`)
    - Logs progress at INFO level ŌĆö backfill of millions of mappings should be observable
    - Runs in batches of 10,000 rows with explicit transaction commits to avoid lock pressure on large libraries

    **Tests**:
    - (integration) Backfilling N existing mappings produces N observations and ŌēżN identities (with deduplication where multiple users observed the same identity)
    - (integration) Re-running the backfill is a no-op
    - (integration) Tier assignment matches the source `match_method`

    Effort: M | Dependencies: ConnectorIdentity Entity & Repository, ObservationLedger Entity & Repository | Status: Not Started

- [ ] **Tier Re-computation & Conflict Resolution Worker**

    **Story**: When a new observation arrives, the system wants to immediately recompute the affected identity's tier (cheap aggregation). When a `disputed` identity exists, a periodic worker tries to resolve it via MusicBrainz lookup so the system self-heals without admin intervention.

    **Decisions**:
    - Tier re-computation is synchronous on observation write (cheap, single-row update). Conflict resolution is async (requires external API call).
    - Resolution outcomes: `disputed` may become `verified` (when MusicBrainz returns an authoritative answer) or stay `disputed` indefinitely (no external evidence available; falls back to the highest-trust-weighted observation).
    - Losing identities are not deleted ŌĆö `superseded_by_id` is set, preserving the audit trail. Future queries should follow the supersession chain.

    **Spec**:
    - Use case `RecomputeIdentityTierUseCase` ŌĆö given an identity, count distinct unretracted observations, sum trust weights, check for contradictions, set tier
    - Use case `ResolveDisputedIdentitiesUseCase` ŌĆö runs as background worker, picks up `disputed` identities, calls MusicBrainz, applies resolution
    - Worker registered alongside existing background tasks (likely the v0.8.0 scheduler if shipped, or a standalone async loop)
    - Rate-limited to MusicBrainz's 1 req/sec policy
    - Resolution algorithm follows the existing `merge_mappings_to_track` CTE pattern in `src/infrastructure/persistence/repositories/track/core.py`

    **Tests**:
    - (unit) Tier transitions: weak + 2 more observations ŌåÆ strong; strong + ISRC verification ŌåÆ verified; strong + contradicting evidence ŌåÆ disputed
    - (integration) Disputed identity resolved by MusicBrainz lookup transitions to verified; loser gets `superseded_by_id` set
    - (integration) Resolution is idempotent ŌĆö re-running on a resolved identity is a no-op

    Effort: M | Dependencies: TrustScore on Users | Status: Not Started

- [ ] **MusicBrainz Re-validation Worker**

    **Story**: Streaming services occasionally rotate track IDs (takedowns, re-uploads, label changes) which silently breaks our cached `verified` mappings ŌĆö the system wants to re-validate `verified` identities monthly so stale mappings get demoted before they cause user-visible failures.

    **Decisions**:
    - Monthly cadence ŌĆö Odesli's experience shows 24ŌĆō48h lag is normal; monthly is conservatively safe.
    - Re-validation only runs against `verified` tier (the only tier that needs upholding by MB). Other tiers are already revisited whenever new observations arrive.
    - On contradiction: demote `verified ŌåÆ disputed`, do NOT immediately pick a new winner. Let the conflict-resolution worker handle it on its next cycle. This keeps re-validation simple.
    - Spotify-side direct ISRC query (`/v1/tracks/{id}` returns ISRC) is cheaper than MusicBrainz round-trips when we're checking a Spotify-side identity. Use the cheapest available source per connector.

    **Spec**:
    - Use case `RevalidateVerifiedIdentitiesUseCase` running on a monthly cron via the v0.8.0 scheduler
    - Rate-limited and batched ŌĆö 100 identities per worker run, configurable
    - Reuses `src/infrastructure/connectors/musicbrainz/matching_provider.py` for batch ISRC lookups
    - Emits a `musicbrainz_lookup` observation on success (refreshes the verification timestamp), or demotes to `disputed` on contradiction

    **Tests**:
    - (integration) Verified identity with consistent MB result: `external_verification_at` updated, tier stays `verified`
    - (integration) Verified identity with contradictory MB result: tier becomes `disputed`, contradicting observation logged
    - (integration) MB lookup failure (network error, rate limit) doesn't change tier ŌĆö retried next cycle

    Effort: M | Dependencies: Tier Re-computation & Conflict Resolution Worker | Status: Not Started

- [ ] **Public Resolver Use Case**

    **Story**: A public, unauthenticated request for `/api/v1/public/tracks/{track_id}` wants the canonical list of service-resolved URLs for that track ŌĆö drawn from `ConnectorIdentity` (not any user's mappings), grouped by service, with the highest-tier identity per service winning.

    **Decisions**:
    - Read-only query ŌĆö never writes. Anti-abuse handled at the HTTP layer (v1.1.0's rate limits).

### v1.0.4: Data Sovereignty ŌĆö Archive & Exit Rights

**Goal**: The user's listening data exists at all times in a form they can hold ŌĆö scheduled full exports in documented open formats, to storage they control, with a verified round-trip import path.

**Context**: Deliberately sequenced as the last pre-social milestone: exit rights must be real *before* the v1.1.x era puts other people's data on hosted instances. This is the direction-neutral half of the open data-ownership decision ([PDR-001](../decisions/PDR-001-data-ownership-model.md)) ŌĆö every candidate answer, including a future local-first pivot, requires this machinery; nothing here forecloses that decision. It is also the concrete mechanism behind the "Users own their data" core principle: ownership through control-and-exit (the credibility model behind Obsidian's "your notes are just files"), delivered on a server-authoritative architecture.

**What this unlocks**:
- mixd (hosted or self-hosted) can disappear without taking years of play history, likes, and playlists
- The hosted-instance ownership story for non-technical users: account + automatic archive to their own storage + a documented way to walk away
- Groundwork for instance migration ("leave to a friend's server") and any future federation

**Persona**: All three ŌĆö the Tinkerer gets data sovereignty made tangible; the Casual Enthusiast gets ownership without self-hosting; the Weekly Curator gets disaster insurance for years of curation.

**Key Design Decisions**:
- **Scheduled, not on-demand only** ŌĆö ownership that requires remembering a button isn't ownership; a stale archive is a broken promise.
- **Usable without mixd** ŌĆö documented open formats (JSON/CSV per entity + one SQLite bundle); format docs ship with the feature.
- **Round-trip is the bar** ŌĆö an archive that can't import into a fresh instance is a backup, not an exit right. Full cross-instance migration (ID remapping) can follow later; single-user fidelity is required now.

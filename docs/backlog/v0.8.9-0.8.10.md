### v0.8.9: Workflow Templates & Import/Export
**Goal**: Give workflow creation curated starting points and portability.

**Context**: v0.4.3 (✅ Completed) delivered the visual editor. The template *infrastructure* — a file-backed template **gallery** (`GET /workflows/templates`, served from `definitions/*.json`), clone-on-use instantiation (`InstantiateWorkflowUseCase`), **Duplicate** on any workflow (`DuplicateWorkflowUseCase`, fetch+clone+save in one transaction), and the elimination of the read-only `is_template` kind so the `workflows` table holds only user-owned, editable workflows — was **built as a pre-cycle DRY consolidation** (migration `023`, code-complete 2026-05-30, pending ship). Cloning mints a fresh unique `definition.id` slug, so duplicates/template-instantiations never collide on the slug the CLI and personal seeder key on. What remains in this slice is the curated template *content* and the import/export path.

**What this unlocks**: Pick "Hidden Gems" and customize instead of building from scratch; export a workflow as JSON to back up or share; import one from a file or URL.

**Persona**: Tinkerer (templates as onboarding, sharing between instances), Curator (faster ritual setup).

- [x] **Workflow Templates (curate the gallery content)**

    **Story**: A new Tinkerer staring at a blank editor doesn't yet know the node catalog — they want to start from a working example that already expresses a good idea ("Hidden Gems"), see how it's built, and tweak it. The gallery + clone-on-use plumbing already exists (pre-cycle refactor); this story curates the *content* that fills it.

    **Decisions**:
    - Templates are **user-facing starting points** — curated empty-state inspiration and copy-and-modify seeds. They stand on their own user value. _(They are explicitly NOT the LLM's training data; v0.9.0 generates workflows via schema-constrained tool use, where the WorkflowDef JSON Schema is the prior. v0.8.9 and v0.9.0 are independent — see v0.9.x Key Design Decisions.)_
    - Templates are **file-backed `WorkflowDef` JSON** under `definitions/*.json`, served by the gallery and instantiated on use into a new user-owned, editable workflow — **not** seeded `workflows` rows, and **not** a read-only kind (that plumbing shipped in the pre-cycle refactor). Each carries a description explaining its curation logic. Some use scheduled refresh — pairs with v0.8.2+ scheduling.

    **Spec**:
    - 5–8 curated definitions: "Current Obsessions" (liked + 8+ plays last 30d → top 20), "Hidden Gems" (liked + 3+ plays, untouched 6mo), "Forgotten Favorites" (liked + 10+ plays all-time, 0 plays last year), "Recent Discovery Mix" (first played last 30d, 3+ plays), "Decade Deep Cuts" (liked, released 2010–2019, <5 plays), etc.
    - The "New Workflow" flow surfaces the gallery (✅ built) — this story populates it. Blank / from-template / from-import paths.

    **Implementation note (v0.8.9):** The pre-cycle refactor already left **13 production templates** + the parametrized `test_template_definitions.py` guard + the gallery UI. So this slice's actual work was the **two templates that weren't expressible** and the **two new filter nodes** they required:
    - `filter.by_release_year` (pure domain) — absolute year range, not the drifting age-in-days `filter.by_release_date`; powers "Decade Deep Cuts".
    - `filter.by_first_played_date` (application) — `filter_by_play_history` parameterized with `date_source="first_played"` (mirrors `sorter.by_first_played` / `sort_by_date`), no 130-line clone; powers "Recent Discovery Mix".
    - Tightened the guard to call the real `validate_workflow_def` per template (caught + fixed a pre-existing list-valued `tags` in `mood_playlist.json` that the runtime accepted but the validator rejected).

    **Tests**:
    - (unit) Every bundled template passes `validate_workflow_def`.
    - (component) Selecting a gallery template instantiates and opens an editable workflow.

    Effort: M (absorbs the two new filter nodes + their config/validation wiring) | Dependencies: v0.4.3 (✅ Visual Editor) | Status: Completed (2026-06-28)

- [x] **Workflow Import/Export**

    **Story**: The Tinkerer wants to back up a workflow they're proud of, move it to another mixd instance, or share it — and to import someone else's safely.

    **Decisions**:
    - Export downloads the `WorkflowDef` as `.json`. Import validates via the existing `validate_workflow_def` before creating — **no new endpoint**: the import flow seeds the editor store as an unsaved draft and Save reuses `CreateWorkflowUseCase`, which already validates before persisting.
    - Import **projects to clean `{name, description, version, tasks}`**, dropping `id` and any server-minted fields, so a file can never smuggle a foreign slug or stale identity into a freshly-created workflow — the server is the sole authority on identity.
    - **URL sharing dropped (considered, deferred).** A downloaded `.json` already satisfies all three real jobs — back up, move to another instance, share — because the file *is* the portable artifact. A query-param-encoded definition is a strictly worse container (URL length caps), and a *good* share-link is a different, larger feature (server-stored snapshots with short revocable tokens) that deserves its own story. _(If a server-side fetch is ever added, it must use an allowlist and block private CIDRs — SSRF guard.)_

    **Spec**:
    - Export: download `WorkflowDef` as `.json` (editor toolbar; pure client-side serialization of `toWorkflowDef()`).
    - Import: file picker on the editor toolbar **and** the list-page New-Workflow actions → parse + shape-guard → load as a draft → Save validates + creates.
    - ~~Share via URL~~ — deferred; see Decisions.

    **Tests**:
    - (unit) `parseWorkflowFile` round-trips, projects away `id`/server fields, and rejects garbage / non-workflow JSON with clear errors; `downloadWorkflowDef` names via `toWorkflowId` and revokes the blob URL.
    - (component) Editor + list-page import of a valid file seeds an unsaved draft; an invalid file surfaces an error toast and creates nothing.

    **Review hardening (v0.8.9)**: the shipped import path also got a deeper `parseWorkflowFile` task-shape guard (rejects tasks missing a string `id`/`type` before they reach the canvas), a `.catch` on the editor store's async layout, and a shared `useWorkflowImport` hook used by both entry points. Both entry points now navigate to `/workflows/new` with an `{ imported: true }` marker so importing while editing a saved workflow can't be clobbered by a background refetch. Follow-on validation/clarity items deferred to [v0.8.12](v0.8.12.md); editor-store lifecycle to [v0.8.10](#v0810-editor-polish---sub-flows--playlist-browse).

    Effort: S | Dependencies: v0.4.3 (✅ Visual Editor) | Status: Completed (2026-06-28)

### v0.8.10: Editor Polish - Sub-Flows & Playlist Browse
**Goal**: Reduce friction in building large workflows and in linking playlists.

**Context**: The two remaining editor-friction reducers. Sub-flows tame 10+ node workflows; browse-to-link removes the paste-an-ID power-user pattern. Depends on the v0.4.3 editor (✅) and v0.4.4 connector linking (✅).

**What this unlocks**: Collapse related nodes into named, reusable groups; pick a Spotify playlist to link from a searchable list instead of pasting its ID.

**Persona**: Tinkerer (sub-flows for complex pipelines), Casual (browse-to-link removes the only ID-knowledge requirement).

- [ ] **Predictable editor canvas on every entry point**

    **Story**: However the user opens the editor — blank "New Workflow", an existing workflow, an import, a template, or (this version) a sub-flow — they want the canvas to show exactly what they expect, never a stale draft left over from the last thing they edited.

    **Decisions**:
    - **Make canvas-entry intent explicit** — because the singleton editor store survives navigation and `/workflows/new` doesn't reset it. v0.8.9 import made store-survival load-bearing and it works only via an implicit chokepoint (`WorkflowEditor` resets on mount unless the navigation carried `{ imported: true }`); adding sub-flow/snippet entry points to that implicit rule is how it breaks.
    - **A small documented state machine** (blank → reset, existing → load, import/template → seed) — so each entry point *declares* its intent rather than relying on mount order. This is the prerequisite for the Sub-Flows story's new entry points, which is why it ships first.

    **Spec**:
    - Replace the implicit reset chokepoint in `WorkflowEditor.tsx` with an explicit entry-intent contract; `editor-store.ts` already exposes `resetWorkflow` (✅ v0.8.9) plus the seed path.
    - **Notes**: while in this SSE/editor code, opportunistically sweep up the latent web-suite test error tracked in [unscheduled.md](unscheduled.md) → Testing & CI (suspected in `useWorkflowSSE` / `useWorkflowPreview`).

    **Tests**:
    - (component) each entry mode lands the documented state; opening blank after editing an existing workflow shows an empty canvas, not the prior draft.

    Effort: S | Dependencies: v0.4.3 (✅ Visual Editor), v0.8.9 (✅ import + `resetWorkflow`) | Status: Not Started

- [ ] **Sub-Flows & Node Grouping**

    **Story**: As a Tinkerer's workflow grows past ~10 nodes it becomes hard to navigate; they want to collapse related nodes into a named group they can fold away — and reuse as a snippet.

    **Decisions**:
    - Use React Flow's `parentId` + `extent: 'parent'`; groups are collapsible, nameable, reusable as "snippets".

    **Spec**:
    - Labeled group nodes as collapsible containers; expand/collapse toggle.

    **Tests**:
    - (component) Grouping nodes, collapsing/expanding, and naming a group all work; a named group can be reused.

    Effort: M | Dependencies: v0.4.3 (✅ Visual Editor) | Status: Not Started

- [ ] **Browse/Search Playlists from Connector**

    **Story**: Most users don't know their playlist IDs — paste-ID-to-link is a power-user pattern that creates friction for everyone else. They want to browse and search their *own* playlists and click to link. This is the most broadly user-facing friction fix in the cycle: it touches everyone who links a playlist.

    **Decisions**:
    - New paginated `list_user_playlists()` connector method (name + ID + track count); browse UI replaces/supplements paste-ID.

    **Spec**:
    - Spotify: `GET /me/playlists` (OAuth scope already held). Apple Music: `GET /me/library/playlists` (when that connector ships in v1.0.1).
    - Browse UI: searchable/filterable list, click to select.

    **Tests**:
    - (integration) `list_user_playlists()` paginates correctly.
    - (component) Browse → search → select links the chosen playlist.

    Effort: M | Dependencies: v0.4.4 (✅ Connector Playlist Linking) | Status: Not Started

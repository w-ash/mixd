### v0.8.9: Workflow Templates & Import/Export
**Goal**: Give workflow creation curated starting points and portability.

**Context**: v0.4.3 (✅ Completed) delivered the visual editor. The template *infrastructure* — a file-backed template **gallery** (`GET /workflows/templates`, served from `definitions/*.json`), clone-on-use instantiation (`InstantiateWorkflowUseCase`), **Duplicate** on any workflow (`DuplicateWorkflowUseCase`, fetch+clone+save in one transaction), and the elimination of the read-only `is_template` kind so the `workflows` table holds only user-owned, editable workflows — was **built as a pre-cycle DRY consolidation** (migration `023`, code-complete 2026-05-30, pending ship). Cloning mints a fresh unique `definition.id` slug, so duplicates/template-instantiations never collide on the slug the CLI and personal seeder key on. What remains in this slice is the curated template *content* and the import/export path.

**What this unlocks**: Pick "Hidden Gems" and customize instead of building from scratch; export a workflow as JSON to back up or share; import one from a file or URL.

**Persona**: Tinkerer (templates as onboarding, sharing between instances), Curator (faster ritual setup).

- [x] **Workflow Templates (curate the gallery content)**

    **Story**: A new Tinkerer staring at a blank editor doesn't yet know the node catalog — they want to start from a working example that already expresses a good idea ("Hidden Gems"), see how it's built, and tweak it. The gallery + clone-on-use plumbing already exists (pre-cycle refactor); this story curates the *content* that fills it.

    **Decisions**:
    - Templates are **user-facing starting points** — curated empty-state inspiration and copy-and-modify seeds. They stand on their own user value. _(They are explicitly NOT the LLM's training data; v0.9.0 generates workflows via schema-constrained tool use, where the WorkflowDef JSON Schema is the prior. v0.8.9 and v0.9.0 are independent — see v0.9.x Key Design Decisions.)_
    - Templates are **file-backed `WorkflowDef` JSON** under `definitions/*.json`, served by the gallery and instantiated on use into a new user-owned, editable workflow — **not** seeded `workflows` rows, and **not** a read-only kind (that plumbing shipped in the pre-cycle refactor). Each carries a description explaining its curation logic. Some use scheduled refresh — pairs with v0.8.2+ scheduling.

    **Spec**:
    - 5–8 curated definitions: "Current Obsessions" (liked + 8+ plays last 30d → top 20), "Hidden Gems" (liked + 3+ plays, untouched 6mo), "Forgotten Favorites" (liked + 10+ plays all-time, 0 plays last year), "Recent Discovery Mix" (first played last 30d, 3+ plays), "Decade Deep Cuts" (liked, released 2010–2019, <5 plays), etc.
    - The "New Workflow" flow surfaces the gallery (✅ built) — this story populates it. Blank / from-template / from-import paths.

    **Implementation note (v0.8.9):** The pre-cycle refactor already left **13 production templates** + the parametrized `test_template_definitions.py` guard + the gallery UI. So this slice's actual work was the **two templates that weren't expressible** and the **two new filter nodes** they required:
    - `filter.by_release_year` (pure domain) — absolute year range, not the drifting age-in-days `filter.by_release_date`; powers "Decade Deep Cuts".
    - `filter.by_first_played_date` (application) — `filter_by_play_history` parameterized with `date_source="first_played"` (mirrors `sorter.by_first_played` / `sort_by_date`), no 130-line clone; powers "Recent Discovery Mix".
    - Tightened the guard to call the real `validate_workflow_def` per template (caught + fixed a pre-existing list-valued `tags` in `mood_playlist.json` that the runtime accepted but the validator rejected).

    **Tests**:
    - (unit) Every bundled template passes `validate_workflow_def`.
    - (component) Selecting a gallery template instantiates and opens an editable workflow.

    Effort: M (absorbs the two new filter nodes + their config/validation wiring) | Dependencies: v0.4.3 (✅ Visual Editor) | Status: Completed (2026-06-28)

- [x] **Workflow Import/Export**

    **Story**: The Tinkerer wants to back up a workflow they're proud of, move it to another mixd instance, or share it — and to import someone else's safely.

    **Decisions**:
    - Export downloads the `WorkflowDef` as `.json`. Import validates via the existing `validate_workflow_def` before creating — **no new endpoint**: the import flow seeds the editor store as an unsaved draft and Save reuses `CreateWorkflowUseCase`, which already validates before persisting.
    - Import **projects to clean `{name, description, version, tasks}`**, dropping `id` and any server-minted fields, so a file can never smuggle a foreign slug or stale identity into a freshly-created workflow — the server is the sole authority on identity.
    - **URL sharing dropped (considered, deferred).** A downloaded `.json` already satisfies all three real jobs — back up, move to another instance, share — because the file *is* the portable artifact. A query-param-encoded definition is a strictly worse container (URL length caps), and a *good* share-link is a different, larger feature (server-stored snapshots with short revocable tokens) that deserves its own story. _(If a server-side fetch is ever added, it must use an allowlist and block private CIDRs — SSRF guard.)_

    **Spec**:
    - Export: download `WorkflowDef` as `.json` (editor toolbar; pure client-side serialization of `toWorkflowDef()`).
    - Import: file picker on the editor toolbar **and** the list-page New-Workflow actions → parse + shape-guard → load as a draft → Save validates + creates.
    - ~~Share via URL~~ — deferred; see Decisions.

    **Tests**:
    - (unit) `parseWorkflowFile` round-trips, projects away `id`/server fields, and rejects garbage / non-workflow JSON with clear errors; `downloadWorkflowDef` names via `toWorkflowId` and revokes the blob URL.
    - (component) Editor + list-page import of a valid file seeds an unsaved draft; an invalid file surfaces an error toast and creates nothing.

    **Review hardening (v0.8.9)**: the shipped import path also got a deeper `parseWorkflowFile` task-shape guard (rejects tasks missing a string `id`/`type` before they reach the canvas), a `.catch` on the editor store's async layout, and a shared `useWorkflowImport` hook used by both entry points. Both entry points now navigate to `/workflows/new` with an `{ imported: true }` marker so importing while editing a saved workflow can't be clobbered by a background refetch. Follow-on validation/clarity items + editor-store lifecycle both landed in [v0.8.10](#v0810-editor-polish--trustworthy-play-history-config) below.

    Effort: S | Dependencies: v0.4.3 (✅ Visual Editor) | Status: Completed (2026-06-28)

### v0.8.10: Editor Polish + Trustworthy Play-History Config
**Goal**: Make the editor canvas predictable on every entry point, remove the paste-an-ID pattern from playlist linking, and make play-history workflow validation + field naming trustworthy.

**Context**: This ship **absorbed the play-history-config work originally scoped as a separate feature** (the v0.8.9-review follow-ons): config-aware validation + the breaking day-window key rename now land here rather than as their own version. Four stories: two editor-friction reducers (the entry-intent state machine makes the navigation-surviving editor store predictable; browse-to-link removes the paste-an-ID power-user pattern) and two play-history-config trust fixes (validation that reflects what a workflow actually produces; date-window field names that read the way they behave). Depends on the v0.4.3 editor (✅), v0.4.4 connector linking (✅), and v0.8.9 (✅ `_enricher_emits`, `DEFAULT_PLAY_HISTORY_METRICS`).

**Sub-flows / node grouping was CUT** (2026-06-29) — *wrong solution, not absent problem*. The navigation problem is real (the user has 40+ node workflows), but collapsible named groups + a persistence model + snippet reuse is too much overhead and would have added the cycle's only `WorkflowDef` schema change. The underlying need — *navigating large workflows* — is recorded in [unscheduled.md](unscheduled.md) for a lighter approach (minimap / node-search-jump / outline panel). Snippet reuse already overlaps v0.8.9 templates + import/export.

**What this unlocks**: A fresh "New Workflow" never shows a stale draft; pick a Spotify playlist to link from a searchable list instead of pasting its ID; the editor's green check stops certifying empty-result play-history workflows; a date-window field label matches the filter you get.

**Persona**: Tinkerer (predictable canvas across import/template/blank entry points; trusts the editor's validation feedback), Casual (browse-to-link removes the only ID-knowledge requirement).

- [x] **Predictable editor canvas on every entry point**

    **Story**: However the user opens the editor — blank "New Workflow", an existing workflow, an import, a template, or (this version) a sub-flow — they want the canvas to show exactly what they expect, never a stale draft left over from the last thing they edited.

    **Decisions**:
    - **Make canvas-entry intent explicit** — because the singleton editor store survives navigation and `/workflows/new` doesn't reset it. v0.8.9 import made store-survival load-bearing and it works only via an implicit chokepoint (`WorkflowEditor` resets on mount unless the navigation carried `{ imported: true }`); adding sub-flow/snippet entry points to that implicit rule is how it breaks.
    - **A small documented state machine** (blank → reset, existing → load, import/template → seed) — so each entry point *declares* its intent rather than relying on mount order. This is the prerequisite for the Sub-Flows story's new entry points, which is why it ships first.

    **Spec**:
    - Replace the implicit reset chokepoint in `WorkflowEditor.tsx` with an explicit entry-intent contract; `editor-store.ts` already exposes `resetWorkflow` (✅ v0.8.9) plus the seed path.
    - **Notes**: while in this SSE/editor code, opportunistically sweep up the latent web-suite test error tracked in [unscheduled.md](unscheduled.md) → Testing & CI (suspected in `useWorkflowSSE` / `useWorkflowPreview`).

    **Tests**:
    - (component) each entry mode lands the documented state; opening blank after editing an existing workflow shows an empty canvas, not the prior draft.

    **Implementation note (v0.8.10):** Typed entry-intent contract in `web/src/lib/editor-entry.ts` (`load`/`seed`/`blank`, resolved from `:id` param + a typed `EDITOR_SEED_STATE` location-state) replaces the implicit `{ imported: true }` reset chokepoint; the dispatch lives in `useEditorEntry`. Rode along: the latent web-suite "1 error" was traced to `useWorkflowSSE`'s snapshot-recovery effect crashing on a partial recovery snapshot (`snapshot.nodes` undefined when a mocked SSE stream's `streamEnded` fired a real, late-resolving snapshot query) — hardened with `snapshot.nodes ?? []` plus abort-signal guards in `useSSEConnection`'s consumer loop. Full `pnpm --prefix web test` now exits 0.

    Effort: S | Dependencies: v0.4.3 (✅ Visual Editor), v0.8.9 (✅ import + `resetWorkflow`) | Status: Completed (2026-06-29)

- [x] **~~Sub-Flows & Node Grouping~~ — CUT (wrong solution)**

- [x] **Browse/Search Playlists from Connector**

    **Story**: Most users don't know their playlist IDs — paste-ID-to-link is a power-user pattern that creates friction for everyone else. They want to browse and search their *own* playlists and click to link. This is the most broadly user-facing friction fix in the cycle: it touches everyone who links a playlist.

    **Decisions**:
    - New paginated `list_user_playlists()` connector method (name + ID + track count); browse UI replaces/supplements paste-ID.

    **Spec**:
    - Spotify: `GET /me/playlists` (OAuth scope already held). Apple Music: `GET /me/library/playlists` (when that connector ships in v1.0.1).
    - Browse UI: searchable/filterable list, click to select.

    **Tests**:
    - (component) Browse → search → select links the chosen playlist.

    **Implementation note (v0.8.10):** Reduced from M → S — the browse infra already shipped for the *import* flow (v0.8.7–v0.8.8): `ConnectorPlaylistPickerDialog`, the `fetch_user_playlists()` connector method, and `GET /connectors/{service}/playlists` (name + ID + track count + import status). No new connector method or endpoint. This slice added a single-select `mode` to the picker (`"import"` default keeps the import path byte-identical) and wired a "Browse" path into `LinkPlaylistDialog` (`PlaylistDetail.tsx`), retiring paste-an-ID as the primary surface (kept as a fallback). Gated on the connector's `playlist_import` capability.

    Effort: S (was M — browse infra already shipped in v0.8.7–v0.8.8) | Dependencies: v0.4.4 (✅ Connector Playlist Linking), v0.8.8 (✅ browse picker + endpoint) | Status: Completed (2026-06-29)

- [x] **Validation that reflects what a workflow will actually produce** _(pulled in from the v0.8.9 review follow-ons)_

    **Story**: A Tinkerer building a `filter.by_metric` over a play-history enricher wants the editor to warn them up front when the enricher isn't configured to emit that metric — not save clean and then return an empty playlist at run time.

    **Decisions**:
    - **Check the enricher's actual config, not its capability** — because `enricher.play_history` only emits the metrics named in its `metrics` config, so "this enricher *can* emit X" is not "this enricher *will*". A capability-based check certifies a workflow that produces nothing.
    - **Warn, don't block** — consistent with the existing enrichment-dependency warnings; the run still proceeds, but the editor flags the empty-result risk before the user is surprised by it.

    **Already in place (v0.8.9)**: the fixed-requirement path (`filter.by_first_played_date`) already inspects the upstream config via `_enricher_emits`; `DEFAULT_PLAY_HISTORY_METRICS` is the single source for the default metric set.

    **Tests**:
    - (unit) `filter.by_metric(metric_name="period_plays")` over a default enricher → validation warns; with `period_plays` requested → no warning.
    - (unit) `period_days` set without `period_plays` → warns.

    **Implementation note (v0.8.10):** Factored a shared `_enricher_emitted_metrics(task)` in `validation.py` (config-aware: play_history → its configured `metrics` or `DEFAULT_PLAY_HISTORY_METRICS`; other enrichers → capability set); the `filter.by_metric`/`sorter.by_metric` branch now unions emitted metrics per upstream task instead of capability sets, and `_enricher_emits` routes through it. Chose **warn** for `period_days` (a separate pass over enricher.play_history tasks) — runtime semantics unchanged. The warning surfaced that *all 9* play-history seed templates carried an inert `period_days`; stripped it (no-op — the runtime never reads it without `period_plays`). **Noted follow-on (→ unscheduled):** `enricher.play_history` has an empty editor config-field tuple, so `metrics`/`period_days` aren't settable in the editor — the warning is actionable for template/JSON authors but an editor user can't yet add the missing metric.

    Effort: S–M | Dependencies: v0.8.9 (✅ `_enricher_emits`, `DEFAULT_PLAY_HISTORY_METRICS`) | Status: Completed (2026-06-29)

- [x] **Day-window config keys that read the way they behave** _(pulled in from the v0.8.9 review follow-ons)_

    **Story**: A contributor (or a user reading the editor) wants the play-history date-window field name to match its behavior the first time — without re-learning that "min" means "older" and "max" means "within".

    **Decisions**:
    - **Rename to self-documenting keys** (`played_within_days` / `not_played_in_days`) — because `min_days_back` / `max_days_back` encode age backwards and already shipped inverted editor labels once (v0.8.9, fixed at the `_date_range_fields` source). A name that states the behavior stops the labels drifting again.

    **Tests**:
    - (unit) renamed keys flow through `calculate_time_window` / the play-history transforms unchanged (existing `test_play_history.py`, renamed in place).
    - (unit) migration 033's pure `_rename_definition_keys` renames day-window keys across all task configs, round-trips upgrade↔downgrade, and skips malformed rows (`tests/unit/migrations/`). Grep gate asserts the legacy keys are absent from `src/`.

    **Implementation note (v0.8.10):** Renamed `min_days_back → not_played_in_days`, `max_days_back → played_within_days` across `config_fields.py` (`_date_range_fields`), `_helpers.py` (`calculate_time_window`), `play_history.py`, `transform_definitions.py`, the 9 seed JSONs, and `docs/guides/workflows.md`. **Web needed no code change** — the editor renders fields dynamically from the backend `ConfigFieldSchema`, so the rename propagates via `sync-api`. Migration **033** (new territory — no prior JSONB *key*-rewrite) rewrites the three columns that hold a `WorkflowDef` (`workflows.definition`, `workflow_versions.definition`, `workflow_runs.definition_snapshot`) via a unit-tested pure helper + a row loop, with a symmetric downgrade. The integration DB builds via `create_all` (not migrations), so the live round-trip is the manual `alembic upgrade head` smoke step.

    Effort: M (breaking) | Dependencies: none | Status: Completed (2026-06-29)

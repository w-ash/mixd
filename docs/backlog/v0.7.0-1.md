## User Workflow Context

The Weekly Curator's current triage flow lives entirely in Spotify playlists as a workaround:

1. **Discovery** — A workflow aggregates new tracks from many sources (Pitchfork, Stereogum, Release Radar, Beatport, friends' and artists' playlists) into a **"Later"** playlist.
2. **Quick triage** — Listen to each "Later" track for a few seconds. Promising tracks move to an **"Interesting"** playlist; the rest go to a **"Nah"** playlist. Each track is removed from "Later" once triaged.
3. **Full listen** — Listen to everything on "Interesting" all the way through. Tracks that hold up get liked in Spotify and tagged by adding them to one or two themed playlist folders (e.g., "Chill Vibes", "Workout"). Tracks that don't hold up move to "Nah."
4. **Promotion** — At any point, tracks that stand out get added to a **"Starred"** playlist — the most curated tier.

These Spotify playlists are workarounds for missing Mixd concepts:

| Spotify workaround | Mixd replacement | Meaning |
|---|---|---|
| "Later" playlist | `hmm` preference | "Haven't decided yet, need to listen" |
| "Nah" playlist | `nah` preference | "Not for me, don't surface again" |
| Spotify liked / "Interesting" | `yah` preference | "Good, keep in rotation" |
| "Starred" playlist | `star` preference | "Always welcome, highly curated" |
| Themed playlist folders (hundreds) | Tags (`mood:chill`, `energy:high`, ...) | "What IS this track?" |

v0.7.x replaces the playlist workarounds with first-class Mixd concepts. Once migrated, the user manages preferences and tags in Mixd directly and stops using Spotify playlists as organizational infrastructure.

### Design Decision: Temporal History is Part of the Data

Music taste is a timeline, not a snapshot. "What was I into last October?" and "show me tracks I starred this year" are natural questions — they're how people think about their relationship with music. The user has years of curation history embedded in Spotify: when they liked a track, when they added it to a playlist, when they promoted it to their starred list. That timeline is part of their listening identity.

If Mixd stamps everything with today's date during migration, that history is destroyed. A track liked in 2019 would look like it was liked in 2026. The user's curation journey — what they were discovering when, how their taste evolved — would be flattened into a single import event. That contradicts Mixd's core promise of data sovereignty: you can't truly own your data if the history is erased during the transfer.

So preferences and tags both carry **source timestamps** (`preferred_at`, `tagged_at`) that preserve the original date from the source service. When migrating a Spotify like, `preferred_at` = Spotify's `added_at`. When importing tags from a playlist, `tagged_at` = the date the track was added to that playlist on Spotify. When the user acts manually in Mixd, the timestamp is now.

Both also have **append-only event logs** (`track_preference_events`, `track_tag_events`) that capture every change non-destructively. When a track goes from yah (October) to star (November), both dates are preserved. "When did I first like this?" and "when did I star it?" are both answerable — the promotion doesn't erase the original opinion.

### Design Decision: "Hmm" as a Preference State

The user's "Later" playlist represents "discovered but not yet evaluated" — a triage inbox. Rather than a separate entity or tag convention, `hmm` is a first-class preference state. It's the entry point of the triage flow: discovery workflows flag tracks as `hmm`, the user listens and upgrades to `nah` / `yah` / `star`.

This is the simplest model — one entity, one table, one set of filters. The trade-off is that `hmm` isn't really an "opinion" like the others, it's a triage state. But in practice the user treats it the same way: it's a label they put on a track that changes how it shows up in workflows and filters. "Show me my hmm tracks" is a valid query.

### Shared Patterns (established in v0.7.0, reused by v0.7.2 Tags and v0.7.4 Playlist Bootstrap)

The preference system establishes three patterns that tags and playlist metadata import will reuse. These are defined once in shared domain modules and documented here — later sub-versions reference them instead of re-specifying.

**1. Source priority** — `SOURCE_PRIORITY = {"service_import": 0, "playlist_assignment": 1, "manual": 2}`. The user's direct opinion always wins. Used by preferences, tags, and playlist metadata import. Lives in `src/domain/entities/sourced_metadata.py` with a `should_override(existing_source, new_source) -> bool` pure function.

**2. Append-only event log** — Every user-facing metadata entity (preferences, tags) has a companion `_events` table that logs every change with the source timestamp. Events are never updated or deleted. The pattern: main table holds current state, events table holds the full timeline. The idempotency guard (same state + source → skip, no event) lives in the use case, not the domain entity.

**3. Source timestamp from caller** — `preferred_at` / `tagged_at` are NOT NULL with no server-side default. Callers must explicitly provide the timestamp (from the source service for imports, `datetime.now(UTC)` for manual actions). All timestamp columns are `DateTime(timezone=True)` — TIMESTAMP WITH TIME ZONE.

### v0.7.0: Preference System
**Goal**: Let users rate tracks as hmm, nah, yah, or star — so they can stop maintaining four separate Spotify playlists just to express how they feel about a track, and instead use one gesture that Mixd understands natively.

**Context**: The curator's triage flow requires four tiers of opinion (see User Workflow Context). Today these live as Spotify playlists, which means curation logic is locked inside Spotify and can't be queried, used in workflows, or owned independently. This sub-version ships the core gesture: the user can rate any track and see their ratings in the Library. It doesn't migrate existing data — that's v0.7.1.

**What this unlocks**:
- Rate any track from the Track Detail page or CLI — one click / one command
- Browse and filter the Library by preference state — "show me my starred tracks"
- See preference distribution on the Dashboard
- Foundation for everything that follows: migration (v0.7.1), tags (v0.7.2), workflows (v0.7.5)

**Persona**: Weekly Curator — "I want to stop managing four separate Spotify playlists just to express how I feel about a track. I want one gesture that Mixd understands natively."

**Manual test**: Open Track Detail → click the star icon → navigate to Library → filter by "star" → the track appears. Change it to "nah" → filter by "nah" → it moved.

**Key Design Decisions**:
- **TrackLike continues to exist** — it tracks per-service sync state (Spotify liked, Last.fm loved). `TrackPreference` is the unified Mixd-owned canonical opinion. They coexist because the user wants Mixd to own their preferences independently of what Spotify or Last.fm think.
- **Multi-user scoping** — `track_preferences` includes `user_id` (FK→users), matching `track_likes`. UNIQUE on `(user_id, track_id)` — each user's preferences are independent.
- **Source priority for conflict resolution**: `manual > playlist_assignment > service_import`. The user's direct opinion always wins over automated imports — if they manually nah a track, a playlist re-import can't silently flip it back. Encoded as `SOURCE_PRIORITY` domain constant. The `source` field is always server-derived, never client-writable.
- **Preference ordering**: `star > yah > hmm > nah`. `hmm` sits above `nah` (expressing interest is stronger than rejection) and below `yah` (you haven't committed yet). Encoded as `PREFERENCE_ORDER` domain constant — single source of truth for conflict resolution, sorting, and filtering.
- **Source timestamps preserved** — see "Temporal History is Part of the Data" above. `preferred_at` carries when the user formed this opinion. For manual preferences in v0.7.0, this is just "now." The migration in v0.7.1 will backfill from source service timestamps.
- **Non-destructive event log** — every preference change is logged in `track_preference_events`. The user wants to know both when they first yah'd a track AND when they later starred it, so promoting a track must not erase the earlier date.

- [x] **Preference Database Schema & Repository**

    **Story**: The user wants their preferences to persist across
    sessions, be filterable in the Library ("show me my starred
    tracks"), and queryable by date ("what did I star in October").
    They also want the full history of preference changes preserved —
    when they promote a track from yah to star, both dates should be
    recoverable. This requires two tables: one for current state and
    one for the event log.

    - `preferred_at` is a NOT NULL column sourced from the caller, not

### v0.7.1: Preference Sync from Likes
**Goal**: When the user imports likes from Spotify or Last.fm, automatically create preferences from them — so the Library populates with preferences as soon as they connect their accounts and import, with the original dates from the source service preserved.

**Context**: The preference system (v0.7.0) lets users rate tracks manually, but the user has thousands of existing likes on Spotify and Last.fm. They don't want to re-rate every track by hand. When they import likes (via the existing likes sync feature), those likes should automatically flow into preferences: Spotify likes become `yah`, Last.fm loves become `star`. The user starts fresh — imports their likes, preferences appear in the Library with the dates they originally liked them, and they can start building workflows immediately.

**What this unlocks**:
- Import likes from Spotify → preferences appear automatically with original `liked_at` dates
- "Show me tracks I liked in October 2024" works — dates come from the source, not the import
- The Library feels populated as soon as the user connects their accounts
- Every future likes import also creates/updates preferences — not a one-time thing

**Persona**: Weekly Curator — "When I import my Spotify likes, I want them to show up as preferences with the right dates. I don't want to re-rate thousands of tracks by hand."

**Manual test**: Import likes from Spotify → open Library → filter by "yah" → see Spotify liked tracks → pick a track you recently liked (so you remember the date) → compare `preferred_at` in Mixd against the date shown in Spotify's liked songs list. They should match.

**Key Design Decisions**:
- **Source timestamps preserved** — `preferred_at` = `TrackLike.liked_at` from the source service. A track liked in 2019 shows as liked in 2019. See "Temporal History is Part of the Data" above.
- **Last.fm love → `star`, Spotify like → `yah`** — Last.fm's "love" is a deliberate explicit gesture (separate button), while Spotify's "like" is the default positive signal.
- **Highest preference wins on conflict** — a track liked on both services gets the higher tier (`star > yah`). The user cared enough to love it on Last.fm; that's the stronger signal.
- **Separate use case, not inline in the import** — `ImportSpotifyLikesUseCase` stays focused on `TrackLike` records. `SyncPreferencesFromLikesUseCase` reads likes and creates preferences as a post-import step. This avoids tangling the 200+ line import hot path with a second write concern, and makes the sync independently testable and re-runnable.
- **Idempotent** — safe to re-run. The sync handles the full `TrackLike` set each time, creating/updating preferences only where needed.

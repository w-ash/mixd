# v1.2.x: Track Share Links

For strategic overview, see the [planning overview](README.md).

---

### v1.2.0: Public Track Share Links

**Goal**: Every track in mixd has a public URL that opens a service picker — anyone, no account required, can land on the page and play that track on Spotify, Apple Music, YouTube Music, or another service of their choice.

**Context**: Sharing music with friends is currently impossible from mixd. A user has to copy a Spotify URL out of band, which assumes the friend uses Spotify and removes mixd's role from the loop entirely. This milestone makes sharing a track as easy as sharing a Letterboxd film: copy a `mixd.app/t/{id}` link, send it, and the recipient lands on a picker page that respects whichever service they prefer to listen on. The page is also the casual-enthusiast entry point — their first contact with mixd is a friend's track link, so it must load fast, look beautiful, and gently signal "made by mixd" without a signup wall.

**What this unlocks**:
- Every track gets a stable public URL — no per-link creation step
- Service picker landing page (Spotify / Apple Music / YouTube Music / Tidal / etc.)
- Share button on every track row and detail page (Web Share API + clipboard fallback)
- Rich social-media preview cards (OG meta tags) for shared links — the actual growth lever
- Foundation for v1.2.1's saved-preference auto-redirect

**Persona**: Casual Enthusiast — the entry-point persona. A friend's track link is their first contact with mixd. Weekly Curator secondary — sharing individual finds with friends.

**Key Design Decisions**:
- **Every track is publicly viewable, no per-link creation table**: Tracks are facts about music, not user-owned content. There's no "create a share link" action — every track has a stable URL the moment it exists in the database. The UI just exposes a copy-link button. Simpler mental model, no `track_share_links` table, no analytics-gating decision to make in v1.2.
- **Service picker over auto-redirect**: The default landing experience is a picker, not a redirect to the sharer's preferred service. The viewer's preference matters more than the sharer's, and the picker page is the brand surface that converts casual viewers into users. v1.2.1 adds opt-in auto-redirect for repeat viewers.
- **Deep links where we have IDs, search-link fallbacks where we don't**: Spotify/Apple deep links use existing `connector_tracks` data. Services we don't connect to (YouTube Music, Tidal, Deezer, SoundCloud) get search-URL fallbacks (`https://music.youtube.com/search?q={artist}+{title}`). Search-link buttons are styled as secondary to signal "less precise."
- **Server-rendered for OG and SEO**: The picker page is server-rendered HTML with full OG meta tags (`og:title`, `og:description`, `og:image` = album art, `og:type=music.song`). Rich preview cards on Discord/Slack/iMessage/Twitter are the actual growth lever — this is non-negotiable.
- **Anti-enumeration via UUIDv7 IDs**: Track IDs are already UUIDv7 (per v0.6.0). The public URL is `/t/{uuid}` — non-guessable, not sequential. No new ID scheme needed. 404 on unknown ID rather than redirect — opaque IDs already prevent enumeration; a redirect would leak existence.
- **Inherits anti-scraping from v1.1.1**: Unauthenticated rate limits, AI crawler blocks, robots.txt rules already shipped in v1.1.1 cover track pages too. No new abuse-prevention work in v1.2.
- **No user-specific data in the response**: The public route exposes track metadata only — never preferences, play history, or tags. Public surface stays strictly factual.
- **Confidence tiers shown subtly in the picker UI** *(revised 2026-07-02)*: Service buttons resolved from externally-verified anchors (platform-confirmed ISRC, MusicBrainz-confirmed MBID) get a subtle checkmark; unverified fan-out results get a muted "best available" label; links whose anchors *disagree* (ISRC lookup vs MBID metadata vs duration) are not surfaced at all. Defends against social-engineering attacks where a malicious sharer crafts a link that looks correct but resolves to the wrong song on the viewer's preferred service.
- **Public resolution reads the share-time fan-out cache, never per-user mappings** *(revised 2026-07-02, [governance memo §2-S1](identity-governance-design-space.md))*: resolution derives from platform affordances — source platform ID → ISRC → per-platform lookups (Apple `filter[isrc]`, Tidal `filter[isrc]`, Deezer `/track/isrc:`) → MusicBrainz recording as tiebreak — cached as a derived, disposable `public_track_links` row (the Songlink data model). Never reads any user's `track_mappings`, so (a) RLS bypass hacks are unnecessary, (b) sharer-A's bad import cannot poison viewer-B's resolution — there is no user-writable shared state to corrupt, and (c) rows refresh on TTL/dead-link detection. **v1.1.0 is no longer a hard dependency**; when its verified-identity layer ships, it upgrades this cache's inputs. Cache-poisoning surface is bounded by anchor cross-checks + short CDN TTLs (the OSM stale-snapshot lesson).
- **Identity-repair prerequisite** *(added 2026-07-02)*: the D1 repairs ([v0.8.18: Identity Integrity](v0.8.18.md)) precede this milestone — an anchor pipeline built on today's corrupted confidence/method data inherits the garbage documented in the [design-space memo §1-FM1/FM2](identity-resolution-design-space.md).

#### Epics

- [ ] **Public Track Read Route (Unauthenticated)**

    **Story**: A user wants to share `mixd.app/t/{track_id}` with a friend who doesn't have a mixd account so the friend can land on the picker page without hitting a login wall or revealing any private data about the sharer.

    **Decisions**:
    - Track data is not user-scoped — no visibility check needed. Tracks exist as shared facts, not personal content.
    - Per-user preferences, play history, and tags are NEVER exposed via this route. Strict allowlist of fields in the response schema.
    - 404 on unknown ID rather than redirect — opaque UUIDv7 IDs already prevent enumeration.

    **Spec**:
    - `GET /api/v1/public/tracks/{track_id}` — JSON: `{title, artist, album, duration_ms, art_url, services: [{service: "spotify", play_url: "...", kind: "deep_link" | "search", confidence_tier: "verified" | "unverified"}]}`
    - **Source of truth: the share-time fan-out cache (`public_track_links`)** *(revised 2026-07-02, governance memo §2-S1)* — derived from platform affordances at share time, refreshed on TTL/dead-link detection; never reads per-user `track_mappings`. `verified` = anchor-confirmed deep link; anchor-conflicting links are not surfaced.
    - Server-rendered HTML route at `/t/{track_id}` (web layer) consumes this API
    - Inherits unauthenticated rate limits + AI crawler blocks from v1.1.1 (Anti-Scraping & Abuse Prevention)
    - Cache headers: `Cache-Control: public, max-age=300, s-maxage=3600` — track metadata is stable, CDN-cacheable; the fan-out row itself carries its own TTL so a corrected link propagates within the CDN window

    **Tests**:
    - (integration) `GET /api/v1/public/tracks/{id}` returns track metadata without auth
    - (integration) Unknown track ID returns 404 (not redirect, not 200 with empty body)
    - (integration) Response schema excludes any user-specific fields (no preferences, no play history, no tags)
    - (integration) Rate limits from v1.1.1 fire on this route
    - (integration) `services[*].confidence_tier` reflects anchor verification state; anchor-conflicting links don't appear in the response
    - (integration) Resolution result is identical regardless of which user shared the track (no per-user mapping leakage)

    Effort: S | Dependencies: v1.1.1 Unauthenticated Read Routes, Share-Time Resolution Cache (see revised Track-to-Service URL Resolution epic) | Status: Not Started

- [ ] **Track-to-Service URL Resolution**

    **Story**: A user wants the picker to show "Spotify" only when we have a precise Spotify match for the track, and graceful search-link fallbacks for services we can't deep-link to, so they're never sent to a 404 on Spotify or — worse — to the wrong song.

    **Decisions**:
    - **Resolution derives from platform affordances, never the sharer's mappings** *(revised 2026-07-02)*: input is the track's anchor tuple (ISRC, verified MBID, known platform IDs from globally-shared `connector_tracks`) feeding the share-time fan-out; per-user `track_mappings` are local overrides that never become shared truth (governance memo §2-S1 — the Plex/Navidrome discipline).
    - Two link kinds: `deep_link` (resolved via the fan-out cache to a service-specific URL) and `search` (URL-encoded `artist + title` query for services we can't resolve).
    - Apple Music URLs require a storefront region. Default to `us` initially; later let the viewer's IP geolocation choose (Post-Deploy revision candidate).
    - YouTube Music has no public API for direct ID lookup without OAuth. Use search-link fallback (`https://music.youtube.com/search?q=`).
    - Service buttons with no link available are hidden, not greyed out — clutter at scale, and absence is a clearer signal than disabled UI.
    - URL templates per service in a config map — keep it data, not switch statements, so adding Tidal/Deezer is a one-line config change.

    **Spec**:
    - New domain service `TrackShareResolver` in `src/domain/services/track_share.py` (pure function, no I/O — domain layer rule)
    - Input: `Track` entity + connector mappings; output: `list[ServiceLink]`
    - `ServiceLink = (service: ServiceName, url: str, kind: Literal["deep_link", "search"])`
    - `ServiceName = Literal["spotify", "apple_music", "youtube_music", "tidal", "deezer", "soundcloud"]` — domain enum, extensible
    - Initial services: Spotify (deep), Apple Music (deep, US storefront), YouTube Music (search), Tidal (search), Deezer (search), SoundCloud (search)

    **Tests**:
    - (unit) Track with Spotify connector resolves to a Spotify deep link of the form `https://open.spotify.com/track/{id}`
    - (unit) Track without an Apple connector either resolves to no Apple link or a search fallback, depending on config
    - (unit) Track with no connector mappings still produces all `search` fallbacks
    - (unit) Unicode artist/title properly URL-encoded in search links (e.g., Björk renders correctly)
    - (unit) Empty/whitespace artist or title omits the broken search link rather than producing a malformed query

    Effort: M | Dependencies: Public Track Read Route | Status: Not Started

- [ ] **Service Picker Landing Page**

    **Story**: A casual viewer following a friend's link wants a clear, beautiful page showing "play this track on..." with one tap per service so they can listen in under 10 seconds without thinking — and feel curious about mixd, not signup-walled.

    **Decisions**:
    - Server-rendered route — full HTML on first paint for OG crawlers (Discord, Slack, iMessage, Twitter), then hydrates for the picker UI. Preview crawlers don't run JS, so client-only would break the growth mechanism.
    - Album art prominent — visuals sell the share.
    - "Powered by mixd" footer with a "What is mixd?" link — soft pitch, no popup or signup wall.
    - Accessibility: `<a>` tags for service buttons (right-click "open in new tab" works), keyboard-navigable, screen-reader labels.
    - Mobile-first: most shared links are opened on phones (Discord/iMessage). Single-column layout, large tap targets.
    - Ordered by service popularity (Spotify, Apple Music, YouTube Music first), search-link buttons grouped below as secondary.

    **Spec**:
    - New route `/t/{track_id}` in `web/src/pages/`, server-rendered
    - OG meta tags rendered server-side: `og:title="{title} by {artist}"`, `og:description="Play on Spotify, Apple Music, YouTube Music..."`, `og:image="{album_art_url}"`, `og:type="music.song"`, `twitter:card="summary_large_image"`
    - Service buttons: brand-colored, official service logos (verify each service's brand-asset license), ordered by popularity
    - Search-link buttons styled as secondary (subtle outline, not filled) — visually signals "less precise"
    - Empty state if no services resolve at all: "We're working on this track's links — try searching on your favorite service" with a generic search box that builds search URLs client-side
    - Footer: "powered by mixd" + small "Sign up" CTA, no modal interrupts

    **Tests**:
    - (E2E) Open `/t/{id}` unauthenticated, click Spotify button, verify navigation to expected URL
    - (E2E) OG meta tags present in initial HTML payload (validates server rendering, not just client hydration)
    - (component) Service buttons render in priority order; search-link styling is visually distinct from deep-link styling
    - (E2E) Discord-style preview crawler (curl with `User-Agent: Discordbot`) gets the OG-tagged HTML
    - (E2E) Mobile viewport: tap targets meet 44×44pt minimum

    Effort: M | Dependencies: Track-to-Service URL Resolution | Status: Not Started

- [ ] **Share Button in App**

    **Story**: An authenticated user browsing their library wants a one-tap share button on every track row and detail page so they can copy a public link and paste it into Discord/iMessage in seconds.

    **Decisions**:
    - Web Share API on mobile (native share sheet), `navigator.clipboard` fallback on desktop with toast confirmation.
    - Same button placement on track rows and detail pages — consistent muscle memory.
    - No "preview before sending" — users can preview by clicking the link if they want.

    **Spec**:
    - Reusable `<TrackShareButton trackId={...} />` component
    - Uses `navigator.share()` if available, falls back to `navigator.clipboard.writeText()` with sonner toast: "Link copied"
    - Track row: icon button in the row's overflow/secondary action area
    - Track detail page: prominent button next to play actions

    **Tests**:
    - (component) Clicking calls `navigator.share` when available
    - (component) Falls back to clipboard write + toast when `navigator.share` is undefined (desktop)
    - (E2E) Share link from track detail page → paste into a fresh tab → lands on `/t/{id}` picker

    Effort: S | Dependencies: Service Picker Landing Page | Status: Not Started

### v1.2.1: Saved Service Preference

**Goal**: Repeat viewers of mixd track links can pick "always open in Spotify" once and have subsequent links auto-redirect — saved in a cookie if anonymous, in their account if logged in.

**Context**: After v1.2.0 ships, casual users will land on the picker page repeatedly (every link from every friend). The picker is great the first time and friction the tenth. This milestone adds an opt-in "remember my choice" flow that, once set, auto-redirects to the chosen service's link with a 1-second courtesy interstitial ("Opening in Spotify… [pick again]"). Anonymous viewers get a 1-year cookie; signed-in mixd users get a DB-backed preference that follows them across devices.

**What this unlocks**:
- Opt-in: pick service + check "always open in {service}" on the picker page
- Anonymous: 1-year cookie
- Authenticated: per-user `default_share_service` setting in mixd, syncing across devices
- Override link to fall back to picker for any single click (e.g., `/t/{id}?pick=1`)
- "Forget my preference" / "change my preference" controls in the interstitial and settings

**Persona**: Casual Enthusiast — turns the second-share friction into zero friction. Weekly Curator gets it via account preference; their links open exactly where they listen.

**Key Design Decisions**:
- **Opt-in only**: Default behavior on the picker page is unchanged. The "remember" checkbox is opt-in and prominent but not pre-checked. Forcing redirect-by-default would degrade the brand surface for first-time visitors.
- **Cookie before DB**: Anonymous users never need to sign up to get the convenience. Cookie is the canonical store; DB is the sync target for authenticated users.
- **One-way migration on sign-in**: When an anonymous user with a cookie preference signs into mixd, the cookie value migrates to the user's DB preference (overwrites if DB had a different value — most-recent-wins). This handles the natural "I picked Spotify on my friend's link, then made an account" path without surprising the user.
- **Always-visible escape hatch**: Every auto-redirect interstitial includes an "open picker instead" link that bypasses the preference for that single click. No preference is "trapped."
- **No preference == picker**: Removing the preference (clearing the cookie or unsetting the DB field) returns the user to the picker page. No fallback chain, no mystery state.
- **`?pick=1` query param forces picker**: Useful for sharer-side preview ("what does my friend see?") and for users who want to deliberately choose a different service for one link.

#### Epics

- [ ] **Anonymous Service Preference (Cookie)**

    **Story**: An anonymous viewer wants to set "always open in Spotify" once and have every future track link auto-redirect there, without making a mixd account.

    **Decisions**:
    - Cookie name: `mixd_default_service`. Value: `ServiceName` slug (`spotify`, `apple_music`, `youtube_music`, etc.).
    - Cookie attributes: `Secure; SameSite=Lax; Max-Age=31536000` (1 year), `HttpOnly=false` because the picker UI reads it client-side to render "remember my choice" pre-checked state.
    - Set client-side from the picker page's checkbox + service tap — no backend round-trip, no settings page (anonymous users don't have one).
    - GDPR: this is a functional/user-preference cookie, not tracking — no consent banner required, but include a one-line note on the picker page near the checkbox.

    **Spec**:
    - Cookie set client-side from the picker page on opt-in tap
    - Picker page checks for cookie on load — if present and matches a valid `ServiceName`, redirect after a 1-second interstitial
    - Interstitial: "Opening in {service}… [open picker instead]" with a manual escape-hatch link
    - "Forget my preference" link visible on every interstitial — clears the cookie and returns to picker
    - `?pick=1` query param bypasses the cookie for that single page load (no cookie clear)

    **Tests**:
    - (component) Picker page with no cookie renders the picker
    - (component) Picker page with valid cookie shows interstitial and redirects after ~1s
    - (component) Invalid cookie value (unknown service slug) is ignored and the cookie cleared
    - (component) `?pick=1` query param forces the picker even with a valid cookie present
    - (E2E) Set preference → reload → verify interstitial → verify redirect to expected service URL
    - (E2E) Click "Forget my preference" → reload → picker is back

    Effort: S | Dependencies: v1.2.0 Service Picker Landing Page | Status: Not Started

- [ ] **Authenticated Service Preference (DB)**

    **Story**: A signed-in mixd user wants their service preference to follow them across devices so the same link opens in their preferred service on phone and laptop without resetting.

    **Decisions**:
    - Stored as `default_share_service: ServiceName | None` on `user_profiles` (extending the table introduced in v1.1.1 User Profile Model). If v1.1.1 hasn't shipped at the time this is built, add to the existing `users` table directly and migrate later.
    - Authenticated viewer: DB preference takes priority over cookie. Picker page reads DB via the existing authenticated session.
    - Setting the preference also sets the cookie — keeps anonymous-mode intact if the user later signs out on the same device.
    - Settings UI: new toggle in Settings → Sharing & Privacy (introduced in v1.1.1): "When I tap a mixd track link, open it in [service ▾]" + "Forget my preference" button.

    **Spec**:
    - Migration: add `default_share_service` nullable column to `user_profiles` (or `users` if v1.1.1 not yet shipped)
    - API: `PUT /me/share-preference {service: ServiceName | null}`, `GET /me/share-preference`
    - Picker page: when authenticated, fetches DB preference on load; if null, falls back to cookie; if both null, shows picker
    - Sign-in handler: after successful auth, if cookie preference exists and DB value is null, copy cookie → DB

    **Tests**:
    - (integration) `PUT /me/share-preference` persists to DB
    - (integration) `PUT /me/share-preference {service: null}` clears the preference
    - (integration) `GET /me/share-preference` returns null for new users
    - (E2E) Sign in → set preference → sign out → sign in elsewhere → preference is restored
    - (E2E) Anonymous user with cookie preference signs up → DB preference matches cookie value
    - (E2E) Authenticated user clears DB preference → picker returns (cookie not consulted)

    Effort: S | Dependencies: Anonymous Service Preference (Cookie), v1.1.1 (User Profile Model) | Status: Not Started

### v1.1.0: Privacy Controls & Public Profiles
**Goal**: Per-content visibility controls and public-facing profile pages — the foundation for all social features.

**Context**: Before any social features go live, users need control over what's visible. Narada's social model is inspired by Letterboxd: personal utility first, social as an additive layer that makes the tool discoverable and spreadable. Public profiles and shareable playlists are the discovery mechanism — a curated playlist URL shared on social media is how strangers find Narada. This milestone establishes the visibility system and public-facing routes that everything else builds on. Anti-scraping and AI crawler protection are not just technical measures — they're the product promise that sharing on Narada is safe. The Data Sovereignty principle (see CLAUDE.md) guarantees that public content serves the user's intent to share, not a third party's intent to extract.

**What this unlocks**:
- Per-playlist visibility (private / followers / public)
- Public-facing profile pages at `/u/{username}`
- Unauthenticated access to public content (no account needed to view a shared playlist)
- Foundation for follows (v1.1.1), activity feed (v1.1.2), and sharing (v1.1.3)

**Persona**: Weekly Curator + Tinkerer — both need to control what's visible before social features go live.

**Key Design Decisions**:
- **Three visibility tiers**: private (only you), followers (people who follow you), public (anyone, even unauthenticated). Designed for growth.
- **Playlists + preferences are shareable**: These are the primary social artifacts. Tags are opt-in shareable. Play history is NEVER public — it's too intimate, and Letterboxd's "watch activity is public by default" is their most-complained-about design choice.
- **Public content drives discovery**: Unauthenticated read routes are essential. A curated playlist link shared on Discord or Twitter should render beautifully without requiring a Narada account to view.
- **Rate limiting for unauthenticated requests**: Prevents abuse without requiring auth. Essential for hosting cost management.

#### Privacy Foundation Epics

- [ ] **User Profile Model**
    - Effort: S
    - What: `UserProfile` entity extending the v1.0.0 user model with social fields
    - Why: Users need an identity beyond email/password for social interactions
    - Dependencies: v1.0.0 (User Authentication System)
    - Status: Not Started
    - Notes:
        - Fields: `username` (unique slug, URL-safe), `display_name`, `bio` (optional, max 280 chars), `avatar_url` (optional), `default_visibility` (enum: private/followers/public)
        - `username` assigned at signup or first social opt-in — immutable slug for profile URLs
        - Extends `users` table from v1.0.0 (add columns or separate `user_profiles` table)
        - API: `GET /u/{username}`, `PUT /me/profile`

- [ ] **Visibility System**
    - Effort: M
    - What: `Visibility` enum and per-content visibility controls across playlists, preferences, and tags
    - Why: Users must control what's shared before any social features exist
    - Dependencies: User Profile Model
    - Status: Not Started
    - Notes:
        - `Visibility`: `Literal["private", "followers", "public"]` — domain type alias
        - Per-playlist visibility: `visibility` column on `playlists` table, defaults to user's `default_visibility`
        - Preference sharing: per-user toggle — share loved/liked status or keep all preferences private. Dislike is always private (negative signals shouldn't be social).
        - Tag sharing: per-user toggle — opt-in (default off). When enabled, tags on shared playlists are visible.
        - Play history: always private. No toggle — this is a product decision, not a user choice. Prevents surveillance-y social dynamics.
        - Visibility enforcement: repository queries filter by visibility + viewer relationship. Public = everyone. Followers = authenticated + follows owner. Private = owner only.
        - Migration: all existing playlists default to `private` — users must opt in to sharing

- [ ] **Public Profile Page**
    - Effort: M
    - What: `/u/{username}` route rendering a public-facing profile with shared content
    - Why: The discovery mechanism — profile URLs are how people find curators
    - Dependencies: Visibility System
    - Status: Not Started
    - Notes:
        - Public view (unauthenticated): display name, bio, public playlists, public preference stats ("X loved tracks")
        - Authenticated view: public content + followers-only content (if viewer follows the user)
        - Owner view: everything including private content, with visibility badges
        - Empty state for new users: "Set up your profile to share playlists with friends"
        - Responsive design — profile URLs will be shared on mobile

- [ ] **Unauthenticated Read Routes**
    - Effort: M
    - What: Public API endpoints that serve content without requiring authentication
    - Why: Shareable links must work without a Narada account
    - Dependencies: Visibility System, Public Profile Page
    - Status: Not Started
    - Notes:
        - `GET /u/{username}` — public profile data
        - `GET /u/{username}/playlists` — public playlists list
        - `GET /playlists/{id}` — playlist detail (if visibility = public)
        - Rate limiting: stricter for unauthenticated requests (100 req/hour per IP vs 10,000 for authenticated)
        - SEO: server-rendered HTML with OG meta tags for profile and playlist pages
        - CORS: allow read-only cross-origin requests for embeds (v1.1.3)
        - Cache headers: public content can be CDN-cached (ETag + Cache-Control)

- [ ] **Anti-Scraping & Abuse Prevention**
    - Effort: S
    - What: Rate limiting, bot filtering, and content protection for public endpoints
    - Why: Public routes are essential for growth but also the attack surface — protect user data and hosting costs without breaking legitimate use
    - Dependencies: Unauthenticated Read Routes
    - Status: Not Started
    - Notes:
        - **Rate limiting** (FastAPI `slowapi` or custom middleware):
            - Unauthenticated: 100 req/hour per IP, 10 req/minute burst
            - Authenticated: 10,000 req/hour per user
            - Return `429 Too Many Requests` with `Retry-After` header
        - **robots.txt**: Allow Googlebot, social media preview fetchers (Twitterbot, facebookexternalhit, Slackbot, Discordbot). Disallow aggressive crawlers. `Crawl-delay: 10` for unrecognized bots.
        - **Pagination caps**: Public list endpoints capped at 50 items/page, max 20 pages deep (1000 items total). Prevents full-database enumeration.
        - **No bulk export via public API**: No `?limit=10000` or unpaginated list endpoints. Aggregate query limits (e.g., max 500 unique track lookups per hour per IP).
        - **User-agent filtering**: Block known scraper/headless browser signatures (empty UA, `python-requests` default, `curl` without referrer on HTML pages). Allowlist social preview bots.
        - **Honeypot links**: Hidden `<a>` in public pages (CSS `display:none`) pointing to a trap endpoint. Real users never click; bots following all links get flagged and rate-limited.
        - **Cache as protection**: CDN-cached public pages (via `Cache-Control: public, max-age=300`) reduce origin load from both legitimate traffic and scraping.
        - **AI agent protection** (2026-era threat): AI agents (GPT, Claude, Perplexity, etc.) crawl dozens of pages per query, extracting and aggregating structured data. They look like legitimate single-user traffic but can enumerate entire public datasets.
            - `ai.txt` / `llms.txt` directives (emerging 2025-2026 standard) — opt out of AI training crawlers
            - Block known AI crawler user-agents: `GPTBot`, `ChatGPT-User`, `Claude-Web`, `PerplexityBot`, `Bytespider`, `CCBot`, `Google-Extended`
            - Aggregate query limits: cap total unique entities fetchable per IP per hour (e.g., max 200 unique tracks/profiles), not just request count. Prevents "one request per second for 8 hours" full-database extraction.
            - `noai` / `noimageai` meta tags on public pages — signal to respectful AI crawlers
            - Consider `X-Robots-Tag: noai` HTTP header on API responses
        - **NOT doing** (overkill at hobby scale): CAPTCHA, JavaScript challenges, browser fingerprinting, commercial bot management (Cloudflare Bot Management, Akamai). Revisit if scraping becomes a real problem.

- [ ] **Privacy Settings UI**
    - Effort: S
    - What: Settings page for managing visibility defaults and sharing toggles
    - Why: Users need a clear, centralized place to control what's shared
    - Dependencies: Visibility System
    - Status: Not Started
    - Notes:
        - Settings page section: "Sharing & Privacy"
        - Controls: default visibility for new playlists (dropdown), preference sharing toggle, tag sharing toggle
        - Per-playlist visibility override on playlist detail page (dropdown in playlist header)
        - Preview: "This is how your profile looks to the public" link
        - Confirmation modal when changing from private to public: "X playlists will become visible"

### v1.1.1: Social Graph & Follows
**Goal**: One-way follow model so users can subscribe to each other's shared content.

**Context**: Follows are Letterboxd's social primitive — follow someone to see their shared content in your feed. One-way (no approval needed) keeps friction low and scales naturally from 5 friends to 500 strangers. The "followers" visibility tier from v1.1.0 becomes meaningful here — content marked "followers" is now visible to people who actually follow you.

**What this unlocks**:
- Follow/unfollow other users
- "Followers" visibility tier becomes functional
- User discovery based on shared taste
- Foundation for activity feed (v1.1.2)

**Persona**: Weekly Curator — "I want to see what my friends are loving this week"

**Key Design Decisions**:
- **No follow requests**: If content is set to "followers only", anyone who follows you sees it. Opt to share, accept followers will see it. Block (v1.1.3) is the safety valve.
- **Taste-based onboarding**: Suggest users to follow during signup based on overlapping preferences, not auto-follow everyone.
- **Taste-based discovery**: "People you might know" based on overlapping loved tracks or shared playlist content — organic discovery, not algorithmic recommendation.

#### Social Graph Epics

- [ ] **Follow Model**
    - Effort: S
    - What: `UserFollow` entity and database table for one-way follow relationships
    - Why: Core social primitive — everything else depends on knowing who follows whom
    - Dependencies: v1.1.0 (User Profile Model)
    - Status: Not Started
    - Notes:
        - `UserFollow(follower_id, followed_id, created_at)` — attrs frozen entity
        - DB table: `user_follows` with UNIQUE on `(follower_id, followed_id)`, FK to `users`
        - Unfollow = DELETE row (no soft delete)
        - Repository: `follow(follower_id, followed_id)`, `unfollow(follower_id, followed_id)`, `is_following(follower_id, followed_id)`, `get_followers(user_id, limit, offset)`, `get_following(user_id, limit, offset)`, `get_follower_count(user_id)`, `get_following_count(user_id)`
        - Index on `followed_id` for efficient "who follows me" queries

- [ ] **Follow API & UI**
    - Effort: M
    - What: Follow/unfollow endpoints, profile page integration, followers/following lists
    - Why: Users need to manage their social graph through the UI
    - Dependencies: Follow Model
    - Status: Not Started
    - Notes:
        - `POST /u/{username}/follow` — follow a user
        - `DELETE /u/{username}/follow` — unfollow
        - `GET /u/{username}/followers` — paginated follower list
        - `GET /u/{username}/following` — paginated following list
        - Profile page: follower/following counts, follow/unfollow button
        - Followers/following pages: user list with display name, bio preview, follow/unfollow action
        - Own profile: "X followers, following Y users" with links to lists

- [ ] **Followers-Only Visibility**
    - Effort: M
    - What: Implement the "followers" tier in the visibility system
    - Why: Makes the middle visibility tier functional — shared with people who opted in to your content
    - Dependencies: Follow Model, v1.1.0 (Visibility System)
    - Status: Not Started
    - Notes:
        - Repository query changes: when fetching playlists/preferences/tags, check `visibility` + follow relationship
        - `visibility = "public"` → visible to everyone
        - `visibility = "followers"` → visible to authenticated users WHERE `user_follows.follower_id = viewer AND user_follows.followed_id = owner`
        - `visibility = "private"` → visible to owner only
        - Auth middleware: inject `viewer_follows_owner` boolean into request context for visibility checks
        - Profile page: show followers-only content to authenticated followers, public-only to everyone else

- [ ] **User Discovery**
    - Effort: S
    - What: Browse and search for users, taste-based "people you might know" suggestions
    - Why: Users need to find people to follow beyond direct invites
    - Dependencies: Follow Model
    - Status: Not Started
    - Notes:
        - `/users` page: browse all users with public profiles, search by username/display name
        - "People you might know": users who love the same tracks you do, ranked by overlap count
        - Algorithm: `SELECT followed_id, COUNT(*) FROM track_preferences tp1 JOIN track_preferences tp2 ON tp1.track_id = tp2.track_id WHERE tp1.user_id = :me AND tp2.user_id != :me AND tp1.state = 'love' AND tp2.state = 'love' GROUP BY tp2.user_id ORDER BY count DESC`
        - Onboarding: suggest users to follow based on taste overlap during signup flow

### v1.1.2: Activity Feed & Social Context
**Goal**: See what the people you follow are doing with their music — the social payoff.

**Context**: This is the milestone where social becomes tangible. Two complementary features: a chronological activity feed (Letterboxd's home page equivalent) showing what followed users are doing, and social context on entity pages ("Alex loves this track" when you're viewing a track). The feed creates a reason to check Narada daily; the social context enriches every interaction with the library.

**What this unlocks**:
- Chronological feed of followed users' deliberate actions (loved a track, updated a playlist)
- Social context on track/playlist pages ("3 people you follow love this")
- Lightweight notification system for direct interactions
- The "Sunday ritual" now includes seeing what friends are into

**Persona**: Weekly Curator — the curation ritual gains a social dimension

**Key Design Decisions**:
- **Strictly chronological feed**: No algorithmic ranking. You see what your friends did, in order. This is a Letterboxd core principle that builds trust — users know exactly why they see what they see.
- **Deliberate actions only**: Preference changes, playlist updates, tag additions, workflow completions. NOT passive plays. This keeps the feed curated and prevents "Alex listened to 47 tracks today" noise.
- **Bulk-import throttling**: When a new user imports 1000+ likes, collapse into a single "imported N tracks" event. Prevents feed flooding (Letterboxd does this with backlog diary entries).
- **Social context > social feed**: Seeing "Alex loves this" on a track page you're already viewing is higher-value than a dedicated feed page. Both matter, but context is the killer feature.

#### Feed & Context Epics

- [ ] **Activity Event Model**
    - Effort: S
    - What: `ActivityEvent` entity and database table for recording shareable user actions
    - Why: Foundation for the feed — events must be stored before they can be displayed
    - Dependencies: v1.1.0 (Visibility System)
    - Status: Not Started
    - Notes:
        - `ActivityEvent(user_id, event_type, entity_type, entity_id, metadata, created_at, visibility, id)`
        - `event_type`: `Literal["preference_set", "playlist_updated", "tag_added", "workflow_completed", "bulk_import"]`
        - `entity_type`: `Literal["track", "playlist", "workflow"]`
        - `metadata`: JSONB — event-specific context (e.g., `{state: "love"}` for preference_set, `{tracks_added: 5}` for playlist_updated, `{count: 1523}` for bulk_import)
        - `visibility`: inherits from the content's visibility setting at time of emission
        - DB table: `activity_events`, indexed on `(user_id, created_at)` for feed queries
        - BRIN index on `created_at` for time-range queries
        - TTL: events older than 90 days can be pruned (configurable) — feed is recent activity, not archive

- [ ] **Activity Event Emission**
    - Effort: M
    - What: Use cases emit activity events when relevant shared actions occur
    - Why: Events must be created at the right moment with the right visibility
    - Dependencies: Activity Event Model
    - Status: Not Started
    - Notes:
        - `SetTrackPreferenceUseCase`: emit `preference_set` event when user loves/likes a track (not on dislike — negative signals are private)
        - `UpdatePlaylistUseCase` / workflow destination nodes: emit `playlist_updated` with track count delta
        - `TagTrackUseCase`: emit `tag_added` if user has tag sharing enabled
        - `ExecuteWorkflowRunUseCase`: emit `workflow_completed` with result summary
        - Bulk-import throttling: `ImportLikedTracksUseCase` and `SyncPlaylistMetadataUseCase` collapse into a single `bulk_import` event when >10 items are processed
        - Visibility check at emission: only emit if the content's visibility is "followers" or "public". Private content generates no events.
        - Lightweight: single INSERT per event, no fan-out. Feed query does the fan-out at read time.

- [ ] **Activity Feed API & UI**
    - Effort: M
    - What: Chronological feed page showing events from followed users
    - Why: The primary social surface — "what are my friends doing with their music?"
    - Dependencies: Activity Event Emission, v1.1.1 (Follow Model)
    - Status: Not Started
    - Notes:
        - `GET /feed` — paginated, chronological events from followed users. Query: `SELECT * FROM activity_events WHERE user_id IN (SELECT followed_id FROM user_follows WHERE follower_id = :me) AND visibility IN ('followers', 'public') ORDER BY created_at DESC`
        - Feed page in web UI: event cards with user avatar, action description, entity context (track art, playlist name)
        - Event card examples:
            - "Alex loved *Reckoner* by Radiohead" (preference_set)
            - "Beth added 5 tracks to *Chill Vibes*" (playlist_updated)
            - "Chris tagged *Idioteque* as mood:intense" (tag_added)
            - "Dana's *Weekly Obsessions* workflow updated 3 playlists" (workflow_completed)
        - Filter by event type (preferences, playlists, tags, workflows)
        - Empty state: "Follow some users to see their activity here" with link to user discovery
        - Add to sidebar navigation

- [ ] **Social Context on Entity Pages**
    - Effort: M
    - What: "Friends who love this" section on track/playlist detail pages
    - Why: The highest-value social feature — enriches objects you're already viewing with friends' perspectives
    - Dependencies: Activity Feed API & UI, v1.1.1 (Follow Model)
    - Status: Not Started
    - Notes:
        - Track detail page: "People you follow" section showing:
            - Who loves/likes this track (preference badges next to user names)
            - Tags applied by followed users (if they share tags)
            - "Also on X playlists by people you follow" links
        - Playlist detail page:
            - "Also curated by" — other users who have playlists containing similar tracks
            - Curator profile link and follow button
        - Query: batch-load preferences/tags for the current track from followed users
        - Subtle placement — below the fold or in a collapsible section. Social context enriches, doesn't dominate.
        - Privacy-aware: only show data from users whose visibility settings allow it

- [ ] **Notification System**
    - Effort: S
    - What: Lightweight in-app notifications for direct interactions
    - Why: Users need to know when someone follows them or interacts with their content
    - Dependencies: Activity Event Model, v1.1.1 (Follow Model)
    - Status: Not Started
    - Notes:
        - Notification types: "X started following you", "X loved a track on your playlist [name]"
        - NOT notifying on every action — only direct interactions with your content or profile
        - DB table: `notifications` (user_id, type, actor_id, entity_type, entity_id, read, created_at)
        - API: `GET /notifications` (paginated), `POST /notifications/mark-read`
        - UI: notification bell icon in header with unread count badge
        - No email or push notifications initially — in-app only
        - Notification preferences: toggle per notification type in settings

### v1.1.3: Sharing & Growth
**Goal**: Make playlists shareable beyond Narada — the viral growth mechanism.

**Context**: Public profiles and follows create a social experience within Narada. This milestone extends it outward: shareable playlist links with rich social media previews, embeddable widgets, and an invite system. When someone shares "check out my Chill Vibes playlist" on Twitter or Discord, the OG preview card sells Narada. This is the Letterboxd model: beautiful public pages drive organic discovery.

**What this unlocks**:
- Shareable playlist links with OG meta tags (social media preview cards)
- Embeddable playlist widgets for blogs and profiles
- Controlled user growth via invite system
- Safety features (block/mute) for the social graph
- RSS and public API for ecosystem growth

**Persona**: Weekly Curator + Casual Enthusiast — sharing a curated playlist should be as easy as sharing a Letterboxd list

**Key Design Decisions**:
- **OG tags are the growth engine**: When a public playlist URL is shared on social media, the preview card (playlist name, track count, curator name, cover art grid) is what makes someone click. This is the single most important growth mechanism.
- **Invite system controls hosting costs**: Until monetization exists, invite-only signup prevents runaway hosting costs. Instance owner controls invite rate.
- **Block/mute ship with social**: Safety features are not optional — they must ship with or shortly after the social graph. Block prevents a user from following you; mute hides their activity from your feed.

#### Sharing & Growth Epics

- [ ] **Shareable Playlist Links**
    - Effort: M
    - What: Public playlist pages with rich OG meta tags for social media preview cards
    - Why: The primary viral growth mechanism — beautiful preview cards drive clicks from social media
    - Dependencies: v1.1.0 (Unauthenticated Read Routes, Public Profile Page)
    - Status: Not Started
    - Notes:
        - Public playlist URL: `/u/{username}/playlists/{slug}` — human-readable, shareable
        - OG meta tags: `og:title` (playlist name), `og:description` (track count + curator name), `og:image` (cover art grid — 2x2 album art from first 4 tracks), `og:type` (music.playlist)
        - Twitter card: `twitter:card=summary_large_image` for maximum visual impact
        - Responsive public page: track list, curator info (avatar + display name + bio excerpt), "Join Narada" CTA
        - Track rows: title, artist, album — no play counts (private data)
        - "Listen on Spotify/Apple Music" deep links per track (uses existing connector track identifiers)
        - Server-side rendered for SEO and social media crawlers

- [ ] **Playlist Embed Widget**
    - Effort: S
    - What: Lightweight iframe embed code for blogs and social profiles
    - Why: Extends reach beyond social media links — curators can embed playlists in their personal sites
    - Dependencies: Shareable Playlist Links
    - Status: Not Started
    - Notes:
        - Embed endpoint: `/embed/playlists/{id}` — minimal standalone page (no navigation, no header)
        - Shows: playlist name, curator name, first 5-10 tracks, "View full playlist on Narada" link
        - Compact design: 300x400px default, responsive
        - "Copy embed code" button on playlist detail page (for playlist owner)
        - CORS headers: allow cross-origin iframe embedding
        - Lightweight: no JS framework, minimal CSS, fast load

- [ ] **User Invite System**
    - Effort: S
    - What: Invite links for controlled user growth
    - Why: Controls hosting costs and signup rate until monetization exists
    - Dependencies: v1.0.0 (User Authentication System)
    - Status: Not Started
    - Notes:
        - Invite model: `UserInvite(inviter_id, code, max_uses, uses, expires_at, created_at)`
        - Admins can create invite links with configurable max uses and expiry
        - Signup page: invite code required (configurable — can be disabled for open registration)
        - "Invited by [username]" displayed on new user's profile (provenance)
        - Rate limiting: configurable max invites per user per month
        - Config: `REGISTRATION_MODE = "invite_only" | "open" | "closed"`

- [ ] **Block & Mute**
    - Effort: S
    - What: Block and mute controls for the social graph
    - Why: Safety valve — required for any social feature set
    - Dependencies: v1.1.1 (Follow Model)
    - Status: Not Started
    - Notes:
        - Block: prevents target from following you, seeing your followers-only content, or appearing in your social context. Mutual — also removes your follow of them if it exists.
        - Mute: hides target's activity from your feed without unfollowing. They don't know they're muted. You still see their content on entity pages.
        - DB tables: `user_blocks(blocker_id, blocked_id, created_at)`, `user_mutes(muter_id, muted_id, created_at)`
        - API: `POST /u/{username}/block`, `DELETE /u/{username}/block`, `POST /u/{username}/mute`, `DELETE /u/{username}/mute`
        - UI: block/mute options in profile page overflow menu
        - Settings: list of blocked/muted users with unblock/unmute actions

- [ ] **Public API & RSS**
    - Effort: S
    - What: RSS feeds for public activity and read-only API keys for third-party integrations
    - Why: Enables ecosystem growth — bloggers, other apps, and automation can consume Narada data
    - Dependencies: v1.1.2 (Activity Event Model), Shareable Playlist Links
    - Status: Not Started
    - Notes:
        - RSS feed: `/u/{username}/feed.rss` — public activity events (new playlists, loved tracks)
        - Read-only API keys: users can generate API keys in settings for third-party integrations
        - Public API: `GET /api/v1/public/u/{username}`, `GET /api/v1/public/playlists/{id}` — same data as public web pages, JSON format
        - Rate limiting: 60 req/min per API key (matches Discogs model)
        - Documentation: auto-generated OpenAPI spec for public endpoints

# v1.1.x: Social & Sharing

For strategic overview, see the [planning overview](README.md).

> **Sequencing revision (2026-07-02) — links-first, graph gated.** Product decision from the identity/governance research review: mixd's sharing is for friend circles, not an engagement network. Execution order becomes **v1.1.1 (privacy foundation) → v1.1.4 (share links) + v1.2.x (track links) → then v1.1.2/v1.1.3 (follows/feed) only when link-sharing demonstrates demand** — sub-version numbers are retained, only execution order changes. Share links deliver "share curation with friends" with no social graph at all (recipients need no account; possession of an unguessable link is the access control). When the graph does ship, it stays invite-scoped: registration remains `invite_only` by default (the v1.1.4 invite system with its "Invited by" provenance — the Lobsters accountability pattern — becomes a permanent trust primitive, not a cost-control stopgap). Structural anti-toxicity principle adopted: **no public vanity metrics** — follower counts, like counts, and play counts never render on public surfaces; there is nothing to farm, so gaming has no reward (see [identity-governance-design-space.md](identity-governance-design-space.md) §1–2).

> **Renumbering (2026-07-16)**: cross-user track identity re-homed from the former v1.0.3 as **v1.1.0** — the social-era opener — and the four social milestones shifted +1 (privacy → v1.1.1, graph → v1.1.2, feed → v1.1.3, sharing → v1.1.4). The links-first execution order above is unchanged (share links never hard-depended on cross-user identity).

### v1.1.0: Cross-User Track Identity

**Goal**: Build a global, append-only ledger of cross-user track-to-service mappings with confidence tiers so public surfaces (sharing, social, recommendations) can resolve tracks reliably without trusting any single user's local state — and recover automatically when bad mappings land.

**Context**: Today's matching pipeline (Fellegi-Sunter scoring with ID > MBID > ISRC > Title+Artist hierarchy in `src/domain/matching/`) is sophisticated but per-user: each user's `track_mappings` are RLS-isolated to their own `user_id`. That works for personal libraries but breaks every cross-user surface mixd is heading toward — public track sharing (v1.2.x), shareable playlist links (v1.1.4), social context on entity pages (v1.1.3), and any future recommendation system. Worse, it leaves the door open to two adversarial scenarios: (1) **gaming** — a malicious sharer crafts a link that resolves to the wrong song on the viewer's preferred service, and (2) **drift** — a streaming service rotates a track ID (takedown, re-upload, label change) and every user's stale mapping silently points at the wrong song. The fix is a layer above the existing matcher that treats each match decision as an *observation* rather than a fact, weights observations by contributor trust, and re-validates against external truth (MusicBrainz / ISRC) on a schedule. This is not a rewrite — `connector_tracks` is already global, ISRCs are already extracted, MusicBrainz is already wired. v1.1.0 is the abstraction layer that makes the existing data shareable across users safely.

**What this unlocks**:
- Public share-link resolvers (v1.2.x) read canonical mappings without RLS bypass hacks
- v1.1.x social features can show cross-user track context safely
- "Verified" / "best available" UI hints in the v1.2.0 service picker
- Audit trail: every mapping change traceable to its source observation
- Self-healing: monthly re-validation flips bad mappings as new evidence arrives
- Foundation for any future mixd-to-MusicBrainz contribution (give back to the commons)

**Persona**: All three. Weekly Curator — reliable mappings for shared playlists. Tinkerer — full data sovereignty + audit trail of every mapping decision. Casual Enthusiast — share-link resolutions stay correct over time without anyone noticing the machinery.

> **Research revision (2026-07-02)** — the identity-resolution and governance research passes ([design-space memo](identity-resolution-design-space.md), [governance memo](identity-governance-design-space.md)) validated this feature's core principle (external APIs as arbiter, users as observers; append-only supersession) and contradicted parts of its machinery at mixd's scale. These revisions govern implementation; where a Key Design Decision below conflicts, this block wins:
> - **Shared tiers collapse to `verified` / `unverified`.** Only externally-verified identities (MusicBrainz-confirmed recording MBID with 301-merge handling, platform-confirmed ISRC, platform-asserted successors) are readable by public/cross-user surfaces. The `strong` tier (≥3 trust-weighted observations) is removed: under open registration it is a three-sock-account quorum (governance memo §2-S3); MusicBrainz's own quorum rule inverts into an attack accelerator at small N.
> - **TrustScore is parked** (epic below marked Deferred). Activity-earned trust is farmable (Wikipedia extended-confirmed gaming record). Revisit trigger: a feature genuinely requires sharing *unanchored* identities AND invite-tree identity cost (v1.1.4 invite provenance) exists to gate it.
> - **ObservationLedger narrows** to provenance + a candidate queue for the verification worker — not a voting substrate. This also removes the queue-rot risk: the queue's consumer is a worker, not a human.
> - **Re-validation mechanics**: MusicBrainz explicitly discourages API polling; trickle at 1 req/s works to ~100k mappings, CC0 dumps/canonical data beyond. "Batch ISRC lookups" here means sequential-under-rate-limit — no batch endpoint exists.
> - **Backfill prerequisite**: the D1 confidence repairs ([v0.8.18 Identity Integrity](v0.8.18.md)) and the supersession substrate ([v0.10.2's Mapping Supersession milestone](v0.10.x.md#v0102-mapping-supersession--resolution-event-log)) must land first — stored confidences currently include bump-to-100 inflation that a backfill would launder into tiers.
> - **Last.fm-sourced MBIDs never produce `verified`** without a MusicBrainz recording-level cross-check (they are frequently Track MBIDs, and merge-stale).
> - **Sequencing note**: v1.2.0 no longer hard-depends on this feature — public share links resolve via the share-time fan-out cache (governance memo §2-S1); this feature upgrades that cache's inputs when it ships.

**Key Design Decisions**:
- **External APIs are arbiter, users are observers**: MusicBrainz / ISRC lookups produce `verified` tier evidence that outweighs any crowd-sourced count of user observations. Crowd consensus is the fallback when external truth is unavailable, not the primary signal. This is the gameability defense — to corrupt a `verified` identity, an attacker would need to corrupt MusicBrainz itself.
- **Append-only observation ledger, never destructive deletes**: Retracted observations are flagged (`retracted_at`), not removed. Bad mappings are *superseded* (linked to their replacement via `superseded_by_id`), not deleted. This is the self-healing defense — every change is reversible, every state is reachable from the audit trail.
- **Confidence tiers, not a single boolean**: `verified` (ISRC/MBID confirmed by MusicBrainz), `strong` (≥3 distinct trust-weighted observations agree), `weak` (1–2 observations, no external verification), `disputed` (contradicting observations not yet resolved). The picker UI surfaces the tier subtly so casual viewers can make their own trust decision when verification is incomplete.
- **TrustScore is computed, not user-controlled**: Account age + successful-observation history + disputed-observation penalty, computed by a background worker. Range `[0.0, 10.0]`. New users get `1.0`; can't bias the system on day one. This dilutes burner-account vote-stuffing.
- **No write API for canonical mappings**: Users contribute *observations*; they never directly write `ConnectorIdentity` rows. The aggregator job (or the same-transaction observation handler for new evidence) recomputes tier and winner. Even admin tools work by emitting high-trust observations, not by mutating identities.
- **Privacy by aggregation**: Public resolvers return aggregate counts only, never contributor identity. The `evidence_payload` JSONB is queryable internally for moderation but not exposed publicly.
- **GDPR-aware**: When a user deletes their account, their observations are retracted (not deleted) and the affected identities recompute. The canonical state stays correct because other users' observations remain.

#### Cross-User Identity Epics

- [ ] **ConnectorIdentity Entity & Repository**

    **Story**: The system wants to store a single, globally-readable assertion that "mixd `track_id={uuid}` ↔ Spotify `track_id=abc123`" with a confidence tier so any non-user-scoped surface (public share links, social previews, recommendation prep) can resolve a track to its external service URLs without violating row-level security or trusting any single user.

    **Decisions**:
    - One row per `(track_id, connector_name, connector_track_id)` triple — the identity is the relationship, not a property of one side.
    - **Globally readable** (no RLS) so public surfaces can resolve without bypass hacks.
    - `superseded_by_id` self-FK for soft-replacement when a `disputed` identity is resolved — keeps history intact.
    - `confidence_tier` is a derived field, recomputed when observations change. Persisting it (vs computing on read) is the right trade because reads dominate writes.

    **Spec**:
    - New domain entity at `src/domain/entities/connector_identity.py` (frozen attrs, slots)
    - Fields: `id: UUID`, `track_id: UUID` (FK→tracks), `connector_name: ConnectorName`, `connector_track_id: str` (FK→connector_tracks), `confidence_tier: Literal["verified", "strong", "weak", "disputed"]`, `total_trust_weight: Decimal`, `external_verification_at: datetime | None`, `superseded_by_id: UUID | None`, `created_at`, `updated_at`
    - DB table `connector_identities` — **no `user_id` column**, no RLS policy
    - Unique on `(track_id, connector_name, connector_track_id)`
    - Index on `(track_id, confidence_tier)` for the public-resolver query
    - Repository protocol in `src/domain/repositories/`, SQLAlchemy impl in `src/infrastructure/persistence/repositories/`, wired into UoW

    **Tests**:
    - (unit) Tier ordering: `verified > strong > weak > disputed`
    - (integration) Two users observing the same `(track_id, connector_track_id)` produce one row, not two
    - (integration) Public resolver reads identity rows without any user session context (RLS bypass not required)

    Effort: M | Dependencies: None | Status: Not Started

- [ ] **ObservationLedger Entity & Repository**

    **Story**: When a user imports a track or a system process verifies a mapping, the system wants to record that decision as an immutable piece of evidence — so we can recompute the canonical truth as new evidence arrives, audit any mapping change, and recover from bad data without losing history.

    **Decisions**:
    - Append-only — `retracted_at` instead of DELETE — because deletes destroy the audit trail and break post-hoc dispute resolution.
    - One observation per `(connector_identity_id, user_id, source_kind)` per user import. A user re-importing the same track doesn't multiply their voice.
    - System sources (`musicbrainz_lookup`, `spotify_api_isrc`) have `user_id = NULL` and `trust_weight_at_observation` set high enough to outweigh plausible user-vote totals.
    - `evidence_payload` is JSONB so we can carry source-specific context (the raw ISRC, the API response timestamp, the matcher's confidence score) without schema churn.

    **Spec**:
    - New domain entity at `src/domain/entities/observation.py`
    - Fields: `id: UUID`, `connector_identity_id: UUID` (FK), `source_kind: ObservationSource`, `user_id: UUID | None` (FK→users, nullable for system sources), `trust_weight_at_observation: Decimal`, `evidence_payload: Mapping[str, JsonValue]`, `observed_at: datetime`, `retracted_at: datetime | None`, `retraction_reason: str | None`
    - `ObservationSource = Literal["user_import", "manual_override", "musicbrainz_lookup", "spotify_api_isrc", "merge_result", "admin_override"]`
    - DB table `observations` — globally readable; index on `(connector_identity_id, retracted_at)`, BRIN index on `observed_at` for time-range queries
    - Unique on `(connector_identity_id, user_id, source_kind)` where `retracted_at IS NULL` — partial unique index

    **Tests**:
    - (integration) Append-only enforcement: repository has no `delete` method
    - (integration) Re-importing a track creates one observation, not duplicates
    - (integration) System source observations can have null `user_id` and aren't constrained by the user-uniqueness rule when source_kind differs

    Effort: M | Dependencies: ConnectorIdentity Entity & Repository | Status: Not Started

- [ ] **TrustScore on Users**

    **Story**: A new burner account shouldn't have the same vote weight as a 2-year-old account with thousands of validated observations — so the contribution of each observation to a `ConnectorIdentity`'s `total_trust_weight` is multiplied by the contributor's current `TrustScore`, computed on a schedule from validated history.

    **Decisions**:
    - Computed by background worker, not real-time. Real-time would create a write storm on every observation.
    - Range `[0.0, 10.0]`. New users start at `1.0`; can't manipulate from day one. Banned users at `0.0`. Aged contributors with strong history cap at `10.0`.
    - Stored on `users` table (scalar field), not in a separate `user_trust` table. Single column, no join overhead on observation writes.
    - Frozen at observation time (`trust_weight_at_observation` on the `observations` row). Recomputing all historical observations when a user's score changes would be quadratic; freezing is the right trade.

    **Spec**:
    - Migration: add `trust_score: Decimal NOT NULL DEFAULT 1.0` to `users`
    - New use case `RecomputeTrustScoresUseCase` in `src/application/use_cases/`
    - Inputs to scoring: account age (months capped at 24), count of validated observations (those that became part of `verified` or `strong` identities), count of retracted observations, count of observations on `disputed` identities not in the winning side, observation diversity (distinct days with activity)
    - Background worker schedule: weekly recomputation. Can be triggered ad-hoc by admin.
    - Helper at `src/domain/services/trust.py`: `compute_trust_score(metrics: TrustMetrics) -> Decimal` — pure function, no I/O

    **Tests**:
    - (unit) `compute_trust_score` boundary cases: brand-new account → 1.0, old + zero observations → 1.0, old + many validated → high but capped at 10.0, account with many retractions → low
    - (integration) Recomputation is idempotent
    - (integration) Score changes don't retroactively alter past observation weights

    Effort: M | Dependencies: ObservationLedger Entity & Repository | Status: Deferred (2026-07-02 — see Research revision block; revisit only with an unanchored-sharing need + invite-tree identity cost)

- [ ] **Backfill Existing Mappings into Observations**

    **Story**: Users have existing `track_mappings` rows from their imports — those mappings represent real evidence we shouldn't throw away when v1.1.0 ships. The system wants to seed `ConnectorIdentity` and `ObservationLedger` from existing mapping rows so the new abstraction layer is populated on day one rather than empty.

    **Decisions**:
    - Single-shot migration script (not an ongoing process). After this runs, new observations come from imports going forward.
    - Initial tier from `match_method`: `isrc` or `mbid` → `verified` (after MusicBrainz cross-check), `direct` → `strong`, `artist_title` → `weak`. Tier may upgrade later as more users observe.
    - Each existing `track_mappings` row produces exactly one `user_import` observation with `trust_weight_at_observation = 1.0` (we don't have historical trust scores; default).
    - Existing `track_mappings` rows are NOT deleted — they remain as the per-user "which connector identities have I matched" index. New writes during the v1.1.0 transition still write both the mapping and the observation; later versions can drop the dual-write.

    **Spec**:
    - Alembic data migration (separate from the schema migration) at `alembic/versions/0XX_backfill_observations.py`
    - Idempotent: re-running produces no duplicates (uses the partial unique index on `observations`)
    - Logs progress at INFO level — backfill of millions of mappings should be observable
    - Runs in batches of 10,000 rows with explicit transaction commits to avoid lock pressure on large libraries

    **Tests**:
    - (integration) Backfilling N existing mappings produces N observations and ≤N identities (with deduplication where multiple users observed the same identity)
    - (integration) Re-running the backfill is a no-op
    - (integration) Tier assignment matches the source `match_method`

    Effort: M | Dependencies: ConnectorIdentity Entity & Repository, ObservationLedger Entity & Repository | Status: Not Started

- [ ] **Tier Re-computation & Conflict Resolution Worker**

    **Story**: When a new observation arrives, the system wants to immediately recompute the affected identity's tier (cheap aggregation). When a `disputed` identity exists, a periodic worker tries to resolve it via MusicBrainz lookup so the system self-heals without admin intervention.

    **Decisions**:
    - Tier re-computation is synchronous on observation write (cheap, single-row update). Conflict resolution is async (requires external API call).
    - Resolution outcomes: `disputed` may become `verified` (when MusicBrainz returns an authoritative answer) or stay `disputed` indefinitely (no external evidence available; falls back to the highest-trust-weighted observation).
    - Losing identities are not deleted — `superseded_by_id` is set, preserving the audit trail. Future queries should follow the supersession chain.

    **Spec**:
    - Use case `RecomputeIdentityTierUseCase` — given an identity, count distinct unretracted observations, sum trust weights, check for contradictions, set tier
    - Use case `ResolveDisputedIdentitiesUseCase` — runs as background worker, picks up `disputed` identities, calls MusicBrainz, applies resolution
    - Worker registered alongside existing background tasks (likely the v0.8.0 scheduler if shipped, or a standalone async loop)
    - Rate-limited to MusicBrainz's 1 req/sec policy
    - Resolution algorithm follows the existing `merge_mappings_to_track` CTE pattern in `src/infrastructure/persistence/repositories/track/core.py`

    **Tests**:
    - (unit) Tier transitions: weak + 2 more observations → strong; strong + ISRC verification → verified; strong + contradicting evidence → disputed
    - (integration) Disputed identity resolved by MusicBrainz lookup transitions to verified; loser gets `superseded_by_id` set
    - (integration) Resolution is idempotent — re-running on a resolved identity is a no-op

    Effort: M | Dependencies: TrustScore on Users | Status: Not Started

- [ ] **MusicBrainz Re-validation Worker**

    **Story**: Streaming services occasionally rotate track IDs (takedowns, re-uploads, label changes) which silently breaks our cached `verified` mappings — the system wants to re-validate `verified` identities monthly so stale mappings get demoted before they cause user-visible failures.

    **Decisions**:
    - Monthly cadence — Odesli's experience shows 24–48h lag is normal; monthly is conservatively safe.
    - Re-validation only runs against `verified` tier (the only tier that needs upholding by MB). Other tiers are already revisited whenever new observations arrive.
    - On contradiction: demote `verified → disputed`, do NOT immediately pick a new winner. Let the conflict-resolution worker handle it on its next cycle. This keeps re-validation simple.
    - Spotify-side direct ISRC query (`/v1/tracks/{id}` returns ISRC) is cheaper than MusicBrainz round-trips when we're checking a Spotify-side identity. Use the cheapest available source per connector.

    **Spec**:
    - Use case `RevalidateVerifiedIdentitiesUseCase` running on a monthly cron via the v0.8.0 scheduler
    - Rate-limited and batched — 100 identities per worker run, configurable
    - Reuses `src/infrastructure/connectors/musicbrainz/matching_provider.py` for batch ISRC lookups
    - Emits a `musicbrainz_lookup` observation on success (refreshes the verification timestamp), or demotes to `disputed` on contradiction

    **Tests**:
    - (integration) Verified identity with consistent MB result: `external_verification_at` updated, tier stays `verified`
    - (integration) Verified identity with contradictory MB result: tier becomes `disputed`, contradicting observation logged
    - (integration) MB lookup failure (network error, rate limit) doesn't change tier — retried next cycle

    Effort: M | Dependencies: Tier Re-computation & Conflict Resolution Worker | Status: Not Started

- [ ] **Public Resolver Use Case**

    **Story**: A public, unauthenticated request for `/api/v1/public/tracks/{track_id}` wants the canonical list of service-resolved URLs for that track — drawn from `ConnectorIdentity` (not any user's mappings), grouped by service, with the highest-tier identity per service winning.

    **Decisions**:
    - Read-only query — never writes. Anti-abuse handled at the HTTP layer (v1.1.1's rate limits).
    - Per-connector winner selection: `verified > strong > weak > disputed`. If the highest-tier identity per connector exists, use it; otherwise show next-highest. `disputed` identities are not surfaced publicly (would expose internal state).
    - Response includes `confidence_tier` per service so v1.2.0's picker UI can show the verified checkmark or "best available" label.
    - Cache headers: `Cache-Control: public, max-age=300, s-maxage=3600` plus `ETag`. CDN can serve.

    **Spec**:
    - Use case at `src/application/use_cases/resolve_public_track_identities.py`
    - Input: `track_id: UUID`
    - Output: `PublicTrackIdentities = list[ResolvedService]` where `ResolvedService = (connector_name, connector_track_id, confidence_tier)`
    - Plugs into v1.2.0's Public Track Read Route (replaces the placeholder "read mappings" stub)
    - Plugs into v1.1.4's Shareable Playlist Links to resolve each track row

    **Tests**:
    - (integration) Track with verified Spotify + weak Apple identities returns both, tier-labeled
    - (integration) Track with disputed-only identities returns empty (don't surface unresolved disputes publicly)
    - (integration) Response is reproducible — same inputs always produce same outputs (caching invariant)

    Effort: S | Dependencies: ConnectorIdentity Entity & Repository | Status: Not Started

- [ ] **Admin Audit Endpoints**

    **Story**: A site operator wants to see "who said what when" for any track's service mapping — for moderation, for debugging surprising resolutions, and for responding to abuse reports.

    **Decisions**:
    - Admin-only routes (existing auth scope). Not exposed publicly.
    - Show observation history, contributor user IDs, evidence payloads, retraction reasons.
    - Allow forced supersession and forced retraction with `admin_override` `source_kind` and a required reason. These actions are themselves observations — the admin's intervention is auditable.

    **Spec**:
    - `GET /admin/tracks/{track_id}/identities` — full list with observation counts per identity
    - `GET /admin/identities/{identity_id}/observations` — full observation history including retracted
    - `POST /admin/identities/{identity_id}/supersede {winner_id, reason}` — emits `admin_override` observation, sets `superseded_by_id`
    - `POST /admin/observations/{observation_id}/retract {reason}` — sets `retracted_at`, recomputes parent identity tier

    **Tests**:
    - (integration) Non-admin user gets 403
    - (integration) Admin retraction triggers tier recomputation
    - (integration) Admin supersession creates the audit observation and links the chain

    Effort: S | Dependencies: Tier Re-computation & Conflict Resolution Worker | Status: Not Started

### v1.1.1: Privacy Controls & Public Profiles
**Goal**: Per-content visibility controls and public-facing profile pages — the foundation for all social features.

**Context**: Before any social features go live, users need control over what's visible. Mixd's social model is inspired by Letterboxd: personal utility first, social as an additive layer that makes the tool discoverable and spreadable. Public profiles and shareable playlists are the discovery mechanism — a curated playlist URL shared on social media is how strangers find Mixd. This milestone establishes the visibility system and public-facing routes that everything else builds on. Anti-scraping and AI crawler protection are not just technical measures — they're the product promise that sharing on Mixd is safe. The Data Sovereignty principle (see CLAUDE.md) guarantees that public content serves the user's intent to share, not a third party's intent to extract.

**What this unlocks**:
- Per-playlist visibility (private / followers / public)
- Public-facing profile pages at `/u/{username}`
- Unauthenticated access to public content (no account needed to view a shared playlist)
- Foundation for follows (v1.1.2), activity feed (v1.1.3), and sharing (v1.1.4)

**Persona**: Weekly Curator + Tinkerer — both need to control what's visible before social features go live.

**Key Design Decisions**:
- **Three visibility tiers**: private (only you), followers (people who follow you), public (anyone, even unauthenticated). Designed for growth.
- **Playlists + preferences are shareable**: These are the primary social artifacts. Tags are opt-in shareable. Play history is NEVER public — it's too intimate, and Letterboxd's "watch activity is public by default" is their most-complained-about design choice.
- **Public content drives discovery**: Unauthenticated read routes are essential. A curated playlist link shared on Discord or Twitter should render beautifully without requiring a Mixd account to view.
- **Rate limiting for unauthenticated requests**: Prevents abuse without requiring auth. Essential for hosting cost management.

- [ ] **Anti-Scraping & Abuse Prevention**
    - Effort: S

    - Dependencies: Unauthenticated Read Routes
    - Status: Not Started

        - **robots.txt**: Allow Googlebot, social media preview fetchers (Twitterbot, facebookexternalhit, Slackbot, Discordbot). Disallow aggressive crawlers. `Crawl-delay: 10` for unrecognized bots.
        - **Pagination caps**: Public list endpoints capped at 50 items/page, max 20 pages deep (1000 items total). Prevents full-database enumeration.
        - **No bulk export via public API**: No `?limit=10000` or unpaginated list endpoints. Aggregate query limits (e.g., max 500 unique track lookups per hour per IP).
        - **User-agent filtering**: Block known scraper/headless browser signatures (empty UA, `python-requests` default, `curl` without referrer on HTML pages). Allowlist social preview bots.
        - **Honeypot links**: Hidden `<a>` in public pages (CSS `display:none`) pointing to a trap endpoint. Real users never click; bots following all links get flagged and rate-limited.
        - **Cache as protection**: CDN-cached public pages (via `Cache-Control: public, max-age=300`) reduce origin load from both legitimate traffic and scraping.
        - **AI agent protection** (2026-era threat): AI agents (GPT, Claude, Perplexity, etc.) crawl dozens of pages per query, extracting and aggregating structured data. They look like legitimate single-user traffic but can enumerate entire public datasets.
            - `ai.txt` / `llms.txt` directives (emerging 2025-2026 standard) — opt out of AI training crawlers
            - Block known AI crawler user-agents: `GPTBot`, `ChatGPT-User`, `Claude-Web`, `PerplexityBot`, `Bytespider`, `CCBot`, `Google-Extended`
            - Aggregate query limits: cap total unique entities fetchable per IP per hour (e.g., max 200 unique tracks/profiles), not just request count. Prevents "one request per second for 8 hours" full-database extraction.
            - `noai` / `noimageai` meta tags on public pages — signal to respectful AI crawlers
            - Consider `X-Robots-Tag: noai` HTTP header on API responses
        - **NOT doing** (overkill at hobby scale): CAPTCHA, JavaScript challenges, browser fingerprinting, commercial bot management (Cloudflare Bot Management, Akamai). Revisit if scraping becomes a real problem.

    - Dependencies: Visibility System
    - Status: Not Started
    - Notes:
        - Settings page section: "Sharing & Privacy"

### v1.1.2: Social Graph & Follows
**Goal**: One-way follow model so users can subscribe to each other's shared content.

**Context**: Follows are Letterboxd's social primitive — follow someone to see their shared content in your feed. One-way (no approval needed) keeps friction low and scales naturally from 5 friends to 500 strangers. The "followers" visibility tier from v1.1.1 becomes meaningful here — content marked "followers" is now visible to people who actually follow you.

**What this unlocks**:
- Follow/unfollow other users
- "Followers" visibility tier becomes functional
- User discovery based on shared taste
- Foundation for activity feed (v1.1.3)

**Persona**: Weekly Curator — "I want to see what my friends are loving this week"

**Key Design Decisions**:
- **No follow requests**: If content is set to "followers only", anyone who follows you sees it. Opt to share, accept followers will see it. Block (v1.1.4) is the safety valve.
- **Taste-based onboarding**: Suggest users to follow during signup based on overlapping preferences, not auto-follow everyone.
- **Taste-based discovery**: "People you might know" based on overlapping loved tracks or shared playlist content — organic discovery, not algorithmic recommendation.

### v1.1.3: Activity Feed & Social Context
**Goal**: See what the people you follow are doing with their music — the social payoff.

**Context**: This is the milestone where social becomes tangible. Two complementary features: a chronological activity feed (Letterboxd's home page equivalent) showing what followed users are doing, and social context on entity pages ("Alex loves this track" when you're viewing a track). The feed creates a reason to check Mixd daily; the social context enriches every interaction with the library.

**What this unlocks**:
- Chronological feed of followed users' deliberate actions (loved a track, updated a playlist)
- Social context on track/playlist pages ("3 people you follow love this")
- Lightweight notification system for direct interactions
- The "Sunday ritual" now includes seeing what friends are into

**Persona**: Weekly Curator — the curation ritual gains a social dimension

**Key Design Decisions**:
- **Strictly chronological feed**: No algorithmic ranking. You see what your friends did, in order. This is a Letterboxd core principle that builds trust — users know exactly why they see what they see.
- **Deliberate actions only**: Preference changes, playlist updates, tag additions, workflow completions. NOT passive plays. This keeps the feed curated and prevents "Alex listened to 47 tracks today" noise.
- **Bulk-import throttling**: When a new user imports 1000+ likes, collapse into a single "imported N tracks" event. Prevents feed flooding (Letterboxd does this with backlog diary entries).
- **Social context > social feed**: Seeing "Alex loves this" on a track page you're already viewing is higher-value than a dedicated feed page. Both matter, but context is the killer feature.

#### Feed & Context Epics

- [ ] **Activity Event Model**
    - Effort: S
    - What: `ActivityEvent` entity and database table for recording shareable user actions
    - Why: Foundation for the feed — events must be stored before they can be displayed
    - Dependencies: v1.1.1 (Visibility System)
    - Status: Not Started
    - Notes:
        - `ActivityEvent(user_id, event_type, entity_type, entity_id, metadata, created_at, visibility, id)`
        - `event_type`: `Literal["preference_set", "playlist_updated", "tag_added", "workflow_completed", "bulk_import"]`
        - `entity_type`: `Literal["track", "playlist", "workflow"]`
        - `metadata`: JSONB — event-specific context (e.g., `{state: "love"}` for preference_set, `{tracks_added: 5}` for playlist_updated, `{count: 1523}` for bulk_import)
        - `visibility`: inherits from the content's visibility setting at time of emission
        - DB table: `activity_events`, indexed on `(user_id, created_at)` for feed queries
        - BRIN index on `created_at` for time-range queries
        - TTL: events older than 90 days can be pruned (configurable) — feed is recent activity, not archive

- [ ] **Activity Event Emission**
    - Effort: M
    - What: Use cases emit activity events when relevant shared actions occur
    - Why: Events must be created at the right moment with the right visibility
    - Dependencies: Activity Event Model
    - Status: Not Started
    - Notes:
        - `SetTrackPreferenceUseCase`: emit `preference_set` event when user loves/likes a track (not on dislike — negative signals are private)
        - `UpdatePlaylistUseCase` / workflow destination nodes: emit `playlist_updated` with track count delta
        - `TagTrackUseCase`: emit `tag_added` if user has tag sharing enabled
        - `ExecuteWorkflowRunUseCase`: emit `workflow_completed` with result summary
        - Bulk-import throttling: `ImportLikedTracksUseCase` and `SyncPlaylistMetadataUseCase` collapse into a single `bulk_import` event when >10 items are processed
        - Visibility check at emission: only emit if the content's visibility is "followers" or "public". Private content generates no events.
        - Lightweight: single INSERT per event, no fan-out. Feed query does the fan-out at read time.

### v1.1.4: Sharing & Growth
**Goal**: Make playlists shareable beyond Mixd — the viral growth mechanism.

**Context**: Public profiles and follows create a social experience within Mixd. This milestone extends it outward: shareable playlist links with rich social media previews, embeddable widgets, and an invite system. When someone shares "check out my Chill Vibes playlist" on Twitter or Discord, the OG preview card sells Mixd. This is the Letterboxd model: beautiful public pages drive organic discovery.

**What this unlocks**:
- Shareable playlist links with OG meta tags (social media preview cards)
- Embeddable playlist widgets for blogs and profiles
- Controlled user growth via invite system
- Safety features (block/mute) for the social graph
- RSS and public API for ecosystem growth

**Persona**: Weekly Curator + Casual Enthusiast — sharing a curated playlist should be as easy as sharing a Letterboxd list

**Key Design Decisions**:
- **OG tags are the growth engine**: When a public playlist URL is shared on social media, the preview card (playlist name, track count, curator name, cover art grid) is what makes someone click. This is the single most important growth mechanism.
- **Invite system controls hosting costs**: Until monetization exists, invite-only signup prevents runaway hosting costs. Instance owner controls invite rate.
- **Block/mute ship with social**: Safety features are not optional — they must ship with or shortly after the social graph. Block prevents a user from following you; mute hides their activity from your feed.

### v0.8.7: Import/Sync Reliability (backend + CLI)
**Goal**: Make Spotify import and playlist sync *trustworthy* — re-import reconciles against the real remote, no position is ever silently lost, destructive syncs are gated, and every run leaves a durable record.

**Context**: Three real bugs traced to one root error — the code sourced "the external playlist's current state" from the canonical Mixd playlist itself (a self-join through the mapping): (1) silent track loss on import (unmatched/local tracks dropped), (2) push sync a total no-op (canonical diffed against itself), (3) the destructive-push guard dead (same self-diff → 0 removals). The fix is **one reconciliation primitive** (`PlaylistReconciliationEngine`): fetch the real remote fresh → diff against canonical at the **connector-identifier** level → safety-gate → atomic apply → record a per-link base. Built across Phases 1–5; ships as backend + CLI (the web UI is v0.8.8).

**What this unlocks**: Re-importing a playlist actually reconciles it (was a no-op once a snapshot was cached); unmatched/local tracks are preserved as first-class "unresolved" rows instead of vanishing; a destructive sync prompts for confirmation; a failed overnight sync is visible the next time the user opens Mixd; unresolved tracks can be repaired when a match later appears.

**Persona**: Curator (trusts the synced playlist is correct), Casual (import "just works" and re-runs safely).

- [x] **Reconciliation engine + entry identity + data model** (Phases 1–3)

    **Story**: The user wants imported playlists to be *complete and faithful* — right tracks, right order, nothing silently dropped — and sync to compare against the real external state, not a self-reference.

    **What shipped**: `PlaylistEntry.id` (`eq=False`, value-equality stays track-based) + `track: Track | None` + `ConnectorTrackRef` for unresolved positions; `SyncPlan`/`build_sync_plan` (connector-identifier multiset diff, ordered `is_noop`); `PlaylistSyncBase` per-link base; migration `030` (nullable `track_id`, `connector_track_id` FK, `unresolved_metadata`, CHECK `track_id IS NOT NULL OR unresolved_metadata IS NOT NULL`, partial index `ix_playlist_tracks_unresolved`). Repository persists + reads unresolved entries with entry-id-first membership matching.

    Effort: L | Status: Completed (2026-06-20)

- [x] **Engine-routed sync/push + fail-loud + safety gate** (Phase 4)

    **What shipped**: `PlaylistReconciliationEngine.preview`/`apply` for pull/push; one shared `connector_push` module (the workflow destination's identical self-diff bug fixed for free); fail-loud `execute_connector_operations` (partial push → `ConnectorSyncError` → link `ERROR`, never silent `SYNCED`); destructive guard always against freshly-fetched remote; multiset counts so duplicates can't dilute the removal ratio; resolved-only PUSH target so the gate matches what the executor performs.

    Effort: L | Status: Completed (2026-06-20)

- [x] **Import on the engine + durable audit** (Phase 5 / Slice 1)

    **Story**: Re-importing a playlist whose canonical has diverged must reconcile it (not silently no-op), and a failed/partial import must *look* failed.

    **What shipped**: `import_connector_playlist_as_canonical` rewired — first-imports keep the CREATE path, re-imports route through `engine.apply(PULL)` (the `has_fresh_cache` short-circuit deleted; bug #2 dead). Import + sync return an `OperationResult` so the SSE seam records real status/counts on the `OperationRun` audit row; per-playlist failures recorded via `append_run_issue`. Bespoke sync route migrated onto `launch_sse_operation` (audit row + `run_id` + 429 cap) with a synchronous **409** pre-flight for destructive unconfirmed syncs.

    Effort: M | Status: Completed (2026-06-21)

- [x] **Confirm-token, repair, operation-awareness, direction model** (Phase 5 / Slice 2)

    **What shipped**: secret-less staleness **confirm-token** (preview mints it; a stale token re-prompts via 409 with a fresh one); `RepairUnresolvedEntriesUseCase` + `POST /playlists/{id}/repair` (DB-only, idempotent re-resolution against existing mappings — consumes, never creates them); **operation-awareness** via the existing `/operation-runs` surface (`?status=running` + `operation_id` on rows for SSE re-attach; migration `031`) rather than a parallel `/operations/*` API; direction model reconciled — **PULL is the new default** (entity + command + API request) and one `direction_label` vocabulary ("Spotify → Mixd (replaces Mixd)").

    Effort: L | Status: Completed (2026-06-21)

- [x] **CLI parity** (Phase 5 / Slice 3)

    **What shipped**: `import-spotify`/`sync`/`sync-preview`/`repair` over the same use cases. Unified on the existing `--source {spotify,mixd}` vocabulary (the leaky `--direction-override {push,pull}` raw-enum removed); `import-spotify` always pulls (the incoherent `--source mixd` import path removed); `sync` gained a two-step destructive-confirm UX (preview + `--confirm` hint); new `mixd playlist repair` command.

    Effort: M | Status: Completed (2026-06-21)

### v0.8.8: Import/Sync Web UI
**Goal**: Bring the reconciliation flow to the web so the user can *trust* it — see what a sync/re-import will change before committing, be safely stopped before a destructive overwrite, watch a long-running operation and reconnect after a reload (and learn when an overnight run failed), repair tracks that couldn't be matched, and read sync direction one consistent way.

**Context**: v0.8.7 shipped the engine, REST API, and CLI; the web is the last leg and carries the hardest UX of the epic. Three audit findings reshape the original three-story sketch:
1. **The "silent overwrite" is worse than a bypass.** `PlaylistDetail.tsx:759` sends `confirmed: true` — but the real `SyncLinkRequest` field is `confirm_token: str | None`, so `confirmed` is an *unrecognized field the server drops*. The shipped staleness-token gate is **entirely unused client-side**. The fix is to *wire up* the `confirm_token` 409 round-trip, not to stop hard-coding a value.
2. **There is no import-preview endpoint.** The only pre-commit diff is the **sync** preview on the link side. "Preview-first import" must be scoped to that (see the import story).
3. **`OperationsProvider` is absent** and `useActiveRuns` is workflow-scoped (`/workflows/active-runs`). Generalize the proven `WorkflowExecutionContext` + `useWorkflowSSE` pattern to a new provider over `GET /operation-runs?status=running`.

The backend contracts (confirm-token, `/operation-runs?status=running`, `direction_label`, `/repair`, `Last-Event-ID` SSE replay) are in place. Full UX research — comparison matrix, principles (P#), anti-patterns, borrow-kit (BK-#) — lives in **[v0.8.8-ux-research.md](v0.8.8-ux-research.md)**.

**What this unlocks**: See exactly what a sync/re-import will change before committing; be stopped (not silently allowed) when a sync would wipe most of a playlist; watch import/sync progress, reconnect after reload, and *see when an overnight run failed*; repair "couldn't match" tracks from the playlist page; one direction vocabulary everywhere, leading with what gets overwritten.

**Persona**: Curator (trusts the synced playlist is correct), Casual (import "just works"), Tinkerer (reads direction + diffs precisely).

#### UX Principles & References

Every UX story below cites the research doc. Non-negotiables distilled from the exemplars (full set + sources in [v0.8.8-ux-research.md](v0.8.8-ux-research.md)):
- **Count the consequence, don't noun it** — "47 of 50 tracks", never "items". **The verb is the button** — "Remove 47 tracks", never "Confirm"/"Sync". *(P1, P2)*
- **Friction matches the backend's signal — nothing more.** The engine already decides destructiveness (`safety_flagged`): flagged → one clear confirm; not flagged → just do it. No extra tiers, no type-to-confirm, no confirm on safe actions. *(P3)*
- **The SSE stream is a view over durable server state**, never the source of truth — replay backlog on reconnect, then append live; show an explicit **"Reconnecting…"** state. **Honest progress, never a lying spinner.** *(P4, P5)*
- **Absence = healthy** (global indicator silent at zero) · **never silently drop** (unresolved is first-class) · **co-locate the remedy with the failure** · **direction leads with what gets overwritten**. *(P6–P9)* · **Collapse the unchanged behind a count.** *(P10)*
- **Anti-patterns to design against**: the silent overwrite · the lying spinner · the generic confirm · the silent drop · the directionless sync · the fire-and-forget · the orphaned job · the invisible failure · the unparseable diff · repair-by-reimport.
- **Stack constraints**: borrow within `radix-ui` / shadcn / `sonner`; status is always **icon + color + text** (never color alone); **Motion is not a dependency** — keep motion CSS-first; use `ResponsiveDialog` and `ConfirmationDialog` (default-focus Cancel).
- **Simplicity guardrails (the spine of this plan):** mirror the backend's binary signals — don't build tiers on `safety_flagged`, a confidence ramp on binary resolved/unresolved, or ahead/behind counts on a one-way *replace*. **Never confirm a non-destructive action** (a first import only adds). Deliberately deferred as over-engineering: type-to-confirm, undo-snapshot, per-track Fix Match, the "Reconnecting…" chip — see [v0.8.8-ux-research.md](v0.8.8-ux-research.md) § *Simplicity guardrails*.

- [ ] **Global operations provider + sidebar re-attach + failure surfacing**

    **Story**: The user kicks off an import/sync, navigates away or reloads, and wants to see it's still running, reconnect to its progress, and — crucially — *learn that last night's sync failed* even though they weren't watching.

    **Decisions**:
    - Build a new `OperationsProvider` (React Context, no Redux/Zustand) on the proven `WorkflowExecutionContext` split: a domain context + an SSE-liveness context wrapping `useWorkflowSSE`. *Absent today.*
    - New `useActiveOperations` hook over `GET /operation-runs?status=running` (the existing `useActiveRuns` is `/workflows/active-runs`-scoped; reuse its adaptive-poll shape: 5s active / 25s idle, `refetchIntervalInBackground`). Each row carries `id` (run_id) + `operation_id` (nullable SSE key).
    - Re-attach = poll → on a fresh in-flight run, `adopt(operation_id)` → snapshot-seed (`GET /operation-runs/{run_id}`) → live SSE (`GET /operations/{operation_id}/progress`), backlog-replayed via `Last-Event-ID`/`evt_N`. *(BK-13, P4)*
    - **Absence = healthy**: sidebar badge hidden at zero; shows `N operation(s) running` when active and links to the **existing** `/operation-runs` surface (no new flyout to build). *(Chrome downloads, Linear; BK-13, P6)*
    - **Surface failures after the fact**: on terminal `final_status = failed`, fire a Sonner toast with a `Retry failed only`-style affordance; distinguish *cancelled/superseded* from *failed* via `final_status` (no distinct `cancelled` SSE event exists — cancellation arrives as `error`). *(BK-14, P8)*

    **Spec**: `OperationsProvider` mounted above the router; `useActiveOperations`; sidebar badge linking to `/operation-runs`; reuse `OperationProgress` (+ `useOperationProgress`) and `RunStatusBadge`. *(Optional polish, not required: a "Reconnecting…" chip — `useWorkflowSSE` already recovers via snapshot poll, so it's cosmetic.)*
    - **States**: empty (badge hidden) · loading · running (badge + live progress) · partial/error (toast + run row) · success (toast).

    **Tests**: Vitest + RTL + MSW (poll → adopt → SSE; failure toast). Playwright E2E: start op → unmount/reload → re-attach; simulated terminal failure surfaces a toast.

    **Guards**: the orphaned job · the invisible failure · the lying spinner.

    Effort: M | Dependencies: v0.8.7 (✅) | Status: Not Started

- [ ] **Destructive-sync confirm round-trip (real 409 `confirm_token`)**

    **Story**: The user runs a sync and, when it would remove most of a playlist, must be *stopped with the exact numbers in front of them* — not silently allowed through.

    **Decisions**:
    - Wire the **real 409 round-trip** through the existing `SyncConfirmationDialog` (extendable — it already shows +add/−remove/unchanged + `safety_flagged`). First POST omits `confirm_token`; on `409 CONFIRMATION_REQUIRED`, read `details.{removals,total,remaining,confirm_token}`, show the destructive dialog, re-POST echoing `details.confirm_token`. The token is a **staleness hash** (changes if the remote snapshot/diff moves) — a stale token re-prompts with a fresh one; handle the re-409. *(replaces the ignored `confirmed: true` field)*
    - **Verb-locked, count-bearing copy**: title `Remove tracks` · body `This sync will remove 47 of 50 tracks from "Roadtrip Mix".` · button `Remove 47 tracks` · success toast `47 tracks removed`. *(Vercel, macOS; BK-5, BK-6, P1, P2)*
    - **One gate, driven by the backend — no extra tiers.** The engine already sets `safety_flagged` when a sync is destructive: flagged → show the one confirm modal above; not flagged → sync directly (success toast only). **Deliberately not building**: type-to-confirm (BK-8 — right for irreversible repo/account deletion, an over-hoop for a recoverable playlist sync) and undo-toast/snapshot (BK-20 — a real undo needs a prior-state snapshot, a separate backend feature; a cosmetic one would lie). *(P3)*
    - **Warning callout only when `safety_flagged`** (asymmetric left-accent per design system): `This removes most of the playlist — double-check before continuing.` *(BK-7)*. Modal behavior: caller owns `open`, `loading` disables both buttons, **never self-dismiss on confirm** so a sync error stays visible *(BK-9)*.
    - **(backend, deferred from v0.8.7 review) Consolidate the sync gate's double remote fetch.** A web sync currently fetches the remote twice — the route's synchronous `_ensure_sync_confirmed` preview (so the 409 is reachable before backgrounding) and again in the background `engine.apply` (which runs with `confirmed=True`). When rebuilding this flow, thread the pre-fetched remote/plan from the synchronous gate into the background apply so it fetches once; this also closes the negligible gate→apply TOCTOU. Touches `PreviewPlaylistSyncUseCase` / `engine.apply` / `engine.preview` / the sync route. Not a correctness fix (gating is already correct) — an efficiency + single-gate cleanup. See handoff §6c.7.

    **Spec**: extend `SyncConfirmationDialog` for the 409 round-trip + stale-token re-prompt; single confirm modal gated on `safety_flagged`; verb-locked microcopy; warning callout; live progress via `useOperationProgress` → `OperationProgress`.
    - **States**: no-op (`Nothing to sync`) · loading · not-flagged (syncs directly) · flagged (confirm modal) · stale-token re-prompt (rare) · sync-error (modal stays up) · success (toast).

    **Tests**: Vitest + RTL + MSW (not-flagged → syncs directly; flagged → 409 → confirm → re-POST; stale-token re-409). Playwright E2E: destructive sync blocked then confirmed.

    **Guards**: the silent overwrite · the generic confirm · confirm fatigue.

    Effort: M | Dependencies: v0.8.7 (✅) | Status: Not Started

- [ ] **Connector import flow (additive — no preview hoop)**

    **Story**: The user picks Spotify playlists to import and wants them to land with honest progress — without being made to "confirm" something that can only *add* to their library.

    **Decisions**:
    - **A first import is purely additive (CREATE path) — so no preview/confirm step.** Pick in `ConnectorPlaylistPickerDialog` → import → watch progress. The picker already shows track counts + `ImportStatusPill`; already-imported playlists are marked and steered to "sync" instead — their reconciliation *can* be destructive, and that path is Story 2's gated flow. **This dissolves the missing-`import/preview`-endpoint problem: first imports don't need one.** *(Lightroom pre-skip; BK-4)*
    - **Remove the "Force re-fetch" toggle** — it is *still fully wired* (schema → route → use case → engine) and surfaced in `ImportPlaylistsConfirmDialog.tsx`; removal is a real cross-layer task. Keep the picker's header "Refresh" button (legitimate browse-cache bypass).
    - **Honest progress**: `Importing "Roadtrip Mix" (2 of 4)`, `Matching tracks — 312 of 1,204` (the `sub_progress` event already carries `resolved`/`unresolved`); terminal line `12 imported · 3 unresolved` via `ImportPlaylistResultRow`. *(LogRocket; BK-11, BK-12, P5)*

    **Spec**: pick → import → SSE progress → result; reuse picker + `OperationProgress` + result rows; remove force toggle (coordinate API cleanup); surface unresolved count for the repair story. (A diff-style preview lives only where it's *earned* — the destructive *sync* confirm, Story 2 — not on additive imports.)
    - **States**: empty (no playlists) · loading (fetch) · importing (progress) · partial/error · success.

    **Tests**: Vitest + RTL + MSW (import → progress → result; already-imported marked/steered to sync; force toggle gone). Playwright E2E: import → progress → result.

    **Guards**: the fire-and-forget · the lying spinner · *over-confirming a safe action*.

    Effort: S–M | Dependencies: v0.8.7 (✅) | Status: Not Started

- [ ] **Unresolved-tracks list + bulk repair**

    **Story**: When an imported track couldn't be matched, the user wants to *see* it (not have it vanish), understand it's unresolved, and fix it in one action when a match becomes available.

    **Decisions**:
    - **Never silently drop**: render unresolved entries as first-class rows with one clear status (icon + label — unresolved is *binary* in our model, so no confidence ramp) and a header roll-up badge `Repair unresolved (N)`. *(Roon "Unidentified"; BK-16, P7)*
    - **Bulk repair now**: `POST /playlists/{id}/repair` is **bulk-only** (no body; returns `{ repaired, still_unresolved }`). The v0.8.8 action is the `Repair unresolved (N)` button → count toast (`Repaired 4 · 2 still unresolved`).
    - **Per-track "Fix match"** (Picard/Plex-grade in-place re-identification, confidence colors) is a **future** enhancement requiring per-track repair endpoints — file to `unscheduled.md`, don't over-promise here. *(BK-17)*

    **Spec**: unresolved rows on `PlaylistDetail.tsx`; header badge with count; bulk repair button + result toast; idempotent (button hidden/disabled at zero).
    - **States**: none-unresolved (badge hidden / `All tracks matched`) · has-unresolved (badge + rows) · repairing (pending) · partial (`N repaired · M still unresolved`) · all-repaired (toast + badge clears).

    **Tests**: Vitest + RTL + MSW (badge count; repair → partial/all-clear toast; empty state). Playwright E2E: import with unresolved → repair → count drops.

    **Guards**: the silent drop · repair-by-reimport.

    Effort: S–M | Dependencies: v0.8.7 (✅) | Status: Not Started

- [ ] **Unified `SyncDirectionIndicator` / `DirectionChooser` + Orval regen**

    **Story**: The user sees one consistent way of expressing sync direction everywhere (import flow, link dialog, sync panel), always leading with what gets overwritten.

    **Decisions**:
    - One shared pair of components driven by the API's `direction_label` (already exists, exactly two strings: `Spotify → Mixd (replaces Mixd)` / `Mixd → Spotify (replaces Spotify)`). **Extract from the three bespoke direction UIs** today (`LinkPlaylistDialog` Select, `SyncConfirmationDialog` button pair, `ImportPlaylistsConfirmDialog` radio) — this *removes* duplication, it doesn't add surface.
    - **Verb = direction, overwrite up front**: lead with the "(replaces …)" side; plain `Nothing to sync` empty state. *(GitHub Desktop, FreeFileSync; BK-18, P9)* — **not doing** ↑↓ ahead/behind counts (BK-19): that's git's two-way-merge metaphor; our sync is one-way *replace*, where it would mislead.
    - **Orval regen** after the client changes (`pnpm --prefix web sync-api`) — picks up the `confirm_token` request field, `direction_label`, `/repair`, `/operation-runs?status=running`. (No API schema changes are required by any of these stories.)

    **Spec**: `SyncDirectionIndicator` (read-only label) + `DirectionChooser` (push/pull select); both consume `direction_label`; used in import flow + link dialog + sync panel; Orval regenerated.
    - **States**: pull-selected · push-selected · read-only indicator · up-to-date / nothing-to-sync.

    **Tests**: (component) chooser + indicator render the unified vocabulary for both directions and the empty state.

    **Guards**: the directionless sync.

    Effort: S | Dependencies: v0.8.7 (✅) | Status: Not Started

#### Backend follow-ups (deferred from v0.8.7 review)

- [ ] **(backend) Per-item atomicity for batch import**

    **Story**: When a batch import of several playlists has one playlist fail, no other playlist should be left half-applied, and a playlist reported "failed" must not have actually been written.

    **Decisions**:
    - Discovered in the v0.8.7 review (deferred — a naive fix is unsafe; see below). `upsert_canonical_playlist` delegates to `CreateCanonicalPlaylistUseCase` / `UpdateCanonicalPlaylistUseCase`, which each `uow.commit()` per item, so the import batch's outer `async with uow:` is **not** atomic. A failure *after* an inner commit (engine `_record_base`, or the post-loop `create_links_batch`) can leave an orphan canonical (committed without its link) or report an item "failed" whose canonical actually updated.
    - **Do NOT just add `uow.rollback()` in the per-item failure handler** — the first-import links sit *pending* in the session (added by `create_links_batch` before the re-import loop, committed only at the final commit), so a mid-batch rollback would discard them.
    - Restructure to **per-item atomic commits** (each item commits canonical + link + base together, on its own transaction) so partial-success semantics hold without cross-item bleed. Decide whether `upsert_canonical_playlist` should stop owning the commit (thread `commit=False`) and the batch loop own it per item.
    - Severity is low / self-healing today: an orphan canonical gets its link on the next import (no link → treated as a new id → upsert finds the canonical → UPDATE → link created), and the base snapshot isn't read for planning yet.

    **Tests**: (unit) a batch where item B fails after item A's inner commit leaves A committed and B fully rolled back; (integration) `create_links_batch` failure leaves no orphan canonical.

    Effort: M | Dependencies: v0.8.7 (✅) | Status: Not Started

- [ ] **(backend) Diff-engine duplicate-track removal on push**

    **Story**: Pushing a Mixd playlist to a connector whose remote holds the *same* track at two or more positions must remove the right number of copies, not mis-target them.

    **Decisions**:
    - CONFIRMED in the v0.8.7 review (pre-existing — `diff_engine.py` is not in the v0.8.7 diff; the new `external_as_playlist` exposes it by resolving duplicate remote ids to one shared `Track` instance, as the processing service also does). `calculate_remove_operations` / the add path key positions by `{id(track): idx}` (Python object identity), so duplicate same-instance tracks collapse to one position. Repro: `calculate_playlist_diff([X, X] → [Y])` emits **two REMOVE ops both at position 1** plus ADD Y (should remove positions 0 and 1). Removing *one* of two duplicates is fine; removing ≥2 is wrong.
    - Root fix is in the hardened diff engine (covered by `test_playlist_update_preservation_bugs_v2.py`), so do it as a focused change with that safety net — make remove/add position-aware (consume positions / match by `track.id` with multiplicity) like the move logic already is, rather than keying on `id(track)`. A symptom-patch that makes `external_as_playlist` copy duplicate tracks would be inconsistent (the processing service still shares instances) and wouldn't fix the canonical-side.
    - Only affects the PUSH path (the API execution strategy); the canonical write path now goes through `reconcile_entries_from` + the repository's membership matching, not `calculate_remove`.

    **Tests**: (unit, domain) `calculate_playlist_diff` for `[X,X]→[Y]` and `[X,Y,X]→[Y]` emits removes at the correct distinct positions; keep the v2 preservation test green.

    Effort: M | Dependencies: v0.8.7 (✅) | Status: Not Started

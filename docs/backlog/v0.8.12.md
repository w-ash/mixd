### v0.8.12: The Fable Sweep — Structural Hardening
**Goal**: Spend the Claude Fable 5 window on a relentless, file-by-file structural audit of the whole codebase, then execute the approved cuts — less duplication, no oversized modules, no dead code — for a tighter, more maintainable, more performant footprint, with **zero user-visible behavior change** except clearly net-positive UX/perf wins.

**Context**: The codebase is large but *litter-free* — 0 TODO/FIXME/HACK, 0 `# type: ignore` (basedpyright disables them globally), no commented-out code (ruff `ERA`). So the debt is **structural, not litter**: 1,000+ line modules, five parallel connector trees with 3–5× duplicated files, 60 CRUD use-cases mirrored one-to-one by 18 route modules, and tooling that has *deferred* enforcement (ruff `PLR` complexity rules switched off "with rationale"; basedpyright `reportUnknown*`/type-arg checks downgraded to `warning`). This is the regime Claude Fable 5 leads — its edge over Opus 4.8 grows with task length and breadth. The full why-Fable-why-now basis + sources live once in [fable-sweep/README.md](fable-sweep/README.md).

*Already in place*: `vulture` + a curated `vulture_whitelist.py` (empirical dead-code pass), import-linter layer enforcement, ~3,066 backend + ~692 frontend tests as the regression net, and the **clean-breaks / no-shims / immutable-domain** conventions that make aggressive refactors safe. This version adds no capability — it *consumes* that safety net to shrink the footprint under it.

**What this unlocks**: Faster, safer feature delivery for every persona downstream — fewer places for bugs to hide, less code to hold in mind, quicker onboarding for the next contributor, and a lint/type bar that **ratchets** so the debt can't silently regrow. User-facing payoff is indirect but real (reliability + velocity); the flagged UX/perf wins ship directly.

**Persona**: Maintainer (Ash) directly; Curator / Tinkerer / Casual indirectly via reliability + faster iteration. **Departs from the feature-story format on purpose** — there is no Given/When/Then slice; the "slice of user value" is a smaller, sturdier codebase, verified by a green suite rather than a manual test scenario.

- [x] **Story 1 — Relentless fan-out audit → findings hub + per-suggestion work orders** (Fable, in the window)

    **Story**: Fable fans out file-by-file across `src/` and `web/src`, building an understanding of what each module does and *why it serves a user*, and records every cleanup opportunity as a self-contained work order — so the user can approve them piecemeal and hand each to whatever model fits.

    **Decisions**:
    - **Hub-and-spoke, not one monolith plan** — because a single mega-refactor is unreviewable and unschedulable; discrete work orders let the user approve/skip/sequence and farm each to a right-sized agent (Fable for the wide refactors, Opus/Haiku for the mechanical ones).
    - **Investigation is the Fable deliverable; execution is farmed** — because Fable's edge is holding the whole codebase in one 1M-token context with a persistent-memory ledger, not per-file edit speed. Use it for the reasoning pass; spend cheaper models on the diffs.
    - **Fable may counter-propose** — if it sees a better batching/sequencing than the seeds below, it records the alternative in the hub before execution starts.
    - **Seeds are a starting point, not the census** — the audit finds its own targets; the survey below only warms the start.

    **Spec**:
    - Working dir `docs/backlog/fable-sweep/`: `README.md` = the **hub** (meta-task, method, guardrails, and the master index table — the persistent-memory ledger); `NN-<slug>.md` = one **spoke** per suggestion, matching `_TEMPLATE.md`.
    - Empirical dead-code pass: `uv run vulture` against the curated `vulture_whitelist.py`; every proposed removal cross-checked (framework false positives stay whitelisted, not deleted).
    - **Seed targets (2026-07-01 survey — seeds, not the full list):**
      - *Oversized modules to decompose:* `infrastructure/persistence/database/db_models.py` (1,516), `interface/cli/playlist_commands.py` (1,487) & `workflow_commands.py` (1,207), `.../repositories/track/connector.py` (1,333), `.../playlist/core.py` (1,155), `.../track/core.py` (1,099), `.../base_repo.py` (1,091); frontend `pages/PlaylistDetail.tsx` (944), `pages/Library.tsx` (747), `components/shared/ConnectorPlaylistPickerDialog.tsx` (677), `components/playlist/PlaylistTrackEditor.tsx` (582).
      - *Parallel-tree duplication vs the `_shared` base* under `infrastructure/connectors/{spotify,lastfm,musicbrainz,apple_music,listenbrainz}`: `error_classifier.py` ×5, `matching_provider.py` ×4, `conversions.py` ×3, the play importers/resolvers.
      - *CRUD boilerplate:* 60 `application/use_cases/` modules mirrored one-to-one by 18 `interface/api/routes/` modules (create/update/delete/list quartets for playlists, links, assignments, tags).
      - *Frontend:* leave biome-excluded `api/generated/**` + `components/ui/**` (shadcn) alone; target the hand-written pages/dialogs above and the parallel `pages/settings/*` screens.
    - Each spoke names its **suggested executor** (Fable / Opus / Haiku) + **Effort** so the user can dispatch without re-reading the code.

    **Tests**: none — investigation only, no code changes. Acceptance = the hub index enumerates every spoke with area/effort/ROI/risk/executor, and each spoke is self-contained enough to hand to a cold agent.

- [ ] **Story 2 — Approve & execute each work order behavior-preservingly** (per-spoke Fable/Opus/Haiku agents) — **→ rescheduled as [v0.8.13–v0.8.16](v0.8.13-0.8.17.md)** (2026-07-01); Decisions/Spec below stay binding for the waves

    **Story**: The user approves the spokes worth doing; each is executed by its suggested model as an isolated, test-green change — so the codebase shrinks without any user noticing, except where a spoke is explicitly flagged as a net-positive UX/perf win.

    **Decisions**:
    - **Behavior-preserving by default** — because this version must not become a stealth-refactor that regresses prod; any user-visible change is called out in its spoke and re-trues the relevant `docs/web-ui/01-user-flows.md` entry.
    - **Clean breaks, no shims** — because the standing convention is one import path per thing; a compat alias would defeat the point of the sweep.
    - **One spoke = one commit** — because small, revertible increments keep the regression net meaningful and let a bad idea back out cheaply.

    **Spec**:
    - Per spoke, the executing agent follows `_TEMPLATE.md`'s guardrail block verbatim: no shims/aliases/re-export layers; every call site updated; a `git grep` gate asserting the removed/renamed symbol is gone; inward-only layer flow preserved (domain stays `@define(frozen=True, slots=True)` + pure; the `interface/`→infra OAuth-shared-helpers exception stands); unrelated semantic debt flagged in the hub, not fixed inline.
    - Green at every increment: `uv run pytest` + `pnpm --prefix web test`. Reuse existing factories (`make_track`, `make_mock_uow`); add a characterization test *before* a risky decomposition where none guards it.
    - Net-positive UX/perf wins ship as their own flagged spokes with a before/after note.

    **Tests**: behavior-preserving refactors are guarded by the *existing* ~3,066 backend + ~692 frontend tests; each spoke adds a characterization test only where it decomposes previously-untested surface. **No test may be weakened to make a refactor pass.**

    Effort: XL (many spokes; sized per work order) | Dependencies: Story 1 (approved hub) | Status: Rescheduled → v0.8.13–v0.8.16 (2026-07-01)

- [ ] **Story 3 — Ratchet the lint/type bar + Fable code-reviews the sweep** — **→ rescheduled as [v0.8.17](v0.8.13-0.8.17.md#v0817-sweep-closeout--ratchet--review)** (2026-07-01), with the free basedpyright promotion pulled forward to v0.8.13; Decisions below stay binding

    **Story**: Once the big functions have shrunk and the duplication is gone, tighten the tooling so the debt can't creep back, and have Fable review the whole campaign end-to-end — so the gains are locked in and the sweep leaves the codebase provably healthier, not just momentarily tidy.

    **Decisions**:
    - **Ratchet the bar, don't just cut code** — because nothing structurally prevents regrowth otherwise; re-enabling the suppressed rules turns the cleanup into an enforced floor.
    - **Re-enable a rule only as the code clears it** — because flipping a rule on over still-failing code just adds noise; each rule flips in the same spoke that makes it pass (or a dedicated closeout spoke once its class is clear).
    - **Fable does the final review** — because the model that built the mental map is best placed to catch a missed call site or behavior drift across the whole diff.

    **Spec**:
    - Re-enable ruff `PLR0912/0915/0914/0913/0917/0911` + `PLR1702` in `pyproject.toml` as the oversized functions clear them; promote basedpyright `reportUnknownParameterType/VariableType/MemberType/ArgumentType/LambdaType`, `reportMissingTypeArgument`, `reportImplicitOverride`, `reportDeprecated` from `"warning"` to `"error"` — **no new `# noqa`/suppressions added to dodge them** (`# type: ignore` is already disabled globally).
    - Full matrix green on closeout: `uv run pytest -m ""`, `uv run basedpyright src/`, `uv run ruff check .`, `pnpm --prefix web check && pnpm --prefix web build`.
    - Fable code-reviews the aggregate diff (or per-batch): call-site completeness, behavior preservation, layer-boundary compliance (import-linter still green).
    - Ship as the `0.8.17` closeout bump once the sweep lands (each wave v0.8.13–v0.8.16 bumps at its own deploy): `pyproject.toml` → `uv sync` → `pnpm --prefix web sync-api`; flip the README rows to `🚀 Shipped`.

    **Tests**: the re-enabled ruff rules + promoted basedpyright errors are themselves the regression gate (they fail CI if the debt regrows); the import-linter contract stays green.

    Effort: M | Dependencies: Story 2 | Status: Rescheduled → v0.8.17 (2026-07-01)

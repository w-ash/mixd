### v0.8.12: Trustworthy Workflow Play-History Config
**Goal**: When the editor says a play-history workflow is valid, it actually produces tracks — and its date-window fields read the way they behave.

**Context**: v0.8.9 shipped play-history filters and caught one validation gap; the review found the same class of gap one door over, plus a field-naming trap that already shipped inverted labels once. Both live in the play-history config + validation subsystem, so they ship together rather than churning those files twice.

**What this unlocks**: The Tinkerer can trust the editor's green check instead of discovering an empty playlist at run time, and can read a date-window field label and get the filter they expect.

**Persona**: Tinkerer — builds custom play-history workflows and relies on the editor's validation feedback.

- [ ] **Validation that reflects what a workflow will actually produce**

    **Story**: A Tinkerer building a `filter.by_metric` over a play-history enricher wants the editor to warn them up front when the enricher isn't configured to emit that metric — not save clean and then return an empty playlist at run time.

    **Decisions**:
    - **Check the enricher's actual config, not its capability** — because `enricher.play_history` only emits the metrics named in its `metrics` config, so "this enricher *can* emit X" is not "this enricher *will*". A capability-based check certifies a workflow that produces nothing.
    - **Warn, don't block** — consistent with the existing enrichment-dependency warnings; the run still proceeds, but the editor flags the empty-result risk before the user is surprised by it.

    **Already in place (v0.8.9)**: the fixed-requirement path (`filter.by_first_played_date`) already inspects the upstream config via `_enricher_emits`; `DEFAULT_PLAY_HISTORY_METRICS` is the single source for the default metric set.

    **Spec**:
    - Make the metric-consumer branch (`filter.by_metric` / `sorter.by_metric`) in `validation.py` config-aware — reuse `_enricher_emits` instead of the capability-based `_enricher_metrics()` set.
    - Surface the `period_days`-is-inert footgun: warn when `period_days` is set without `period_plays` in the enricher's metrics — or make `period_days` actually window `total_plays` so it isn't silently ignored (decide during build).
    - Concrete trigger to reproduce: `filter.by_metric(metric_name="period_plays")` over a default `play_history` enricher.

    **Tests**:
    - (unit) `filter.by_metric(metric_name="period_plays")` over a default enricher → validation warns; with `period_plays` requested → no warning.
    - (unit) `period_days` set without `period_plays` → warns (or windows the count, per the build decision).

    Effort: S–M | Dependencies: v0.8.9 (✅ `_enricher_emits`, `DEFAULT_PLAY_HISTORY_METRICS`) | Status: Not Started

- [ ] **Day-window config keys that read the way they behave**

    **Story**: A contributor (or a user reading the editor) wants the play-history date-window field name to match its behavior the first time — without re-learning that "min" means "older" and "max" means "within".

    **Decisions**:
    - **Rename to self-documenting keys** (e.g. `played_within_days` / `not_played_in_days`) — because `min_days_back` / `max_days_back` encode age backwards and already shipped inverted editor labels once (v0.8.9, fixed at the `_date_range_fields` source). A name that states the behavior stops the labels drifting again.

    **Spec**:
    - Rename across `config_fields.py`, `_helpers.py` (`calculate_time_window`), `play_history.py`, and the web editor.
    - **Breaking**: needs a config migration for saved workflows, a rewrite of every seed JSON under `definitions/`, and a `mixd workflow seed-personal` re-run.

    **Tests**:
    - (unit) `calculate_time_window` maps the renamed keys to the same effective window as the old ones.
    - (integration/migration) a workflow saved on the old keys loads and runs unchanged after migration.

    Effort: M (breaking) | Dependencies: none | Status: Not Started

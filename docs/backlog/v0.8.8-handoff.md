## Why this exists (read this first, keep it in your head)

A user reclaimed their listening data and built playlists in Mixd. Now they want to **trust sync**.
Today they can't — they can't see what a sync will change before it runs, nothing stops a sync
that would wipe most of a playlist, a reload loses the progress of a long import, an overnight
failure is silent, and unmatched tracks just… vanish. Every story below is one of those trust
breaks. The personas:

- **Curator** — trusts the synced playlist is correct.
- **Casual** — import "just works" and re-runs safely.
- **Tinkerer** — reads direction + diffs precisely.

If you're ever unsure whether a piece of UI is worth building, ask: *does the backend expose a
decision the user must make, or am I decorating a binary?* That single question kills most
over-engineering in this epic.

## Landmines I hit so you don't (verified against code)

- **The silent-overwrite bug is now a compile error, by design.** The Step-0 regen replaced the
  ignored `confirmed?: boolean` with `confirm_token?: string | null`. Story 2 is about *wiring the
  real 409 round-trip*, not removing a hardcoded `true`.
- **Re-attach snapshot endpoint:** use `GET /operations/{operation_id}/snapshot` (live status+nodes).
  **Not** `GET /operation-runs/{run_id}` — that's the audit shape (counts/issues), no live progress.
  The proven recovery machinery (`useWorkflowSSE` → `useOperationSnapshot`) already polls the right one.
- **`OperationsProvider` mount:** *inside* `BrowserRouter`, wrapping `<Routes>` (`web/src/App.tsx:124`)
  — the failure toast needs `useNavigate`. The existing `WorkflowExecutionProvider` sits *above* the
  router (no router context); **don't copy that placement.**
- **There is no distinct `cancelled` SSE event** — cancellation arrives as `error`. Distinguish
  *cancelled/superseded* from *failed* via the terminal `final_status`, not by waiting for a cancel event.
- **Failure-toast dedup:** `useRef<Set<run_id>>` seeded on the *first* poll (so a pre-existing
  terminal failure doesn't retro-toast). Provider owns *failure* toasts; the Sync card owns *success*
  (shared ledger so neither double-fires).
- **The `force` re-fetch toggle removal is a real cross-layer task** (schema → route → use case →
  engine) — and **only the playlist-import `force`**, not the likes `force` (that's a different
  full-rescan semantic). Verify re-imports still fetch fresh remote without it before deleting.
- **Retry = server-reconstructed.** The toast fires after the user navigated away, so the client no
  longer holds the request. Rebuild the call from the run alone: failed ids from
  `OperationRun.issues[]`, re-invoke config from `request_params`, owner **from auth — never from
  stored data**. The endpoint re-invokes the *same* `run_import_connector_playlists_as_canonical`
  with the failed subset (DRY — no new orchestration).

## Stay simple (the spine of this plan)

Mirror the backend's binary signals — the UI should be exactly as simple as the data:

- `safety_flagged` is binary → **one** confirm modal when flagged, nothing when not. No tiers, no
  type-to-confirm, no confirm on additive actions.
- resolved/unresolved is binary → one clear "unresolved" treatment + a roll-up count. No confidence ramp.
- `direction_label` is two strings → lead with what gets overwritten ("… replaces …"). No ahead/behind
  counts (that's git's two-way-merge metaphor; ours is one-way replace).
- The SSE stream is a *view over durable server state* — replay/seed from snapshot, then append live;
  never treat the stream as the source of truth. Honest progress, never a lying spinner.

Deliberately deferred (don't build): type-to-confirm, undo-snapshot, per-track "Fix match", the
"Reconnecting…" chip. See the research doc's *Simplicity guardrails* for the reasoning.

## How you'll know it works

- **Per change:** `pnpm --prefix web test src/path/to/X.test.tsx` (Vitest + RTL + MSW).
- **Before commit:** `pnpm --prefix web test` · `pnpm --prefix web check` (Biome + tsc) ·

- **E2E:** Playwright specs for the two headline flows — start op → reload → re-attach, and an
  off-page terminal failure surfacing a toast.
- **Manual, end-to-end** (Docker up, `pnpm dev`): import 2+ playlists → reload mid-run → badge
  persists + re-attaches; force a per-playlist failure → navigate away → failure toast → **Retry
  failed only** re-imports just the failed one; sync that removes most tracks → 409 → `Remove N
  tracks` modal → confirm → `N tracks removed`; open a playlist with unresolved entries → repair →
  count drops; direction reads identically across import flow, link dialog, sync panel.

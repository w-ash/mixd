### v0.8.9: Manual Playlist Track Editing
**Goal**: Let the user edit a canonical playlist's contents directly — add, remove, and reorder tracks from its detail page.

**Context**: The design-debt review found web flows 3.4 (add) / 3.5 (reorder) / 3.6 (remove) specced but entirely unbuilt — no endpoint, no UI, no CLI — while the spec doc misleadingly marks the endpoints "Exists." Confirmed for build 2026-06-15. The DB was *already* built for this (incident `e1045cab` shipped `DBPlaylistTrack.id` + `sort_key` for identity-preserving reorder); the domain entity just never wired the id through. So this is enablement of latent infrastructure, not building from scratch. Full design-space and the incident-hardened path it must respect in [design-debt-findings.md](design-debt-findings.md) §4 (F6). Break into sub-versions (add → remove → reorder) if it warrants.

**What this unlocks**: Direct manual curation of a playlist's contents — a core Weekly-Curator behavior that currently has no surface (the page is read-only, steering users to links/workflows).

**Persona**: Curator (hands-on curation), Casual (basic playlist editing without learning workflows).

- [ ] **Playlist track add / remove / reorder via entry identity**

    **Story**: The user wants to add tracks to a playlist, remove individual or batch-selected tracks, and reorder by drag-and-drop or keyboard — directly from the detail page, not only via connector sync or workflow output.

    **Decisions**:
    - **Thread a stable entry id end-to-end.** Add `id: UUID = field(factory=uuid7, eq=False)` to `PlaylistEntry` (`domain/entities/playlist.py:40` currently has only `track`/`added_at`/`added_by`). **`eq=False` is load-bearing** — attrs equality must stay value-based so the playlist diff/dedup engine doesn't start distinguishing otherwise-identical entries.
    - **Duplicate policy decided here**: honor the spec (manual add allows duplicates; the DB is designed for it). This must **not** disturb the workflow-append dedupe filter (`_append_entries`, `update_canonical_playlist.py:447`), which is justified for its only consumer (weekly append-mode re-runs). Manual add and workflow append are different paths with different correct behavior.
    - **Reorder is full-list semantics** (client sends complete ordered `entry_ids`; repo renumbers `sort_key`); single-track move is an optional later optimization.
    - Independent of the `UpdateCanonicalPlaylistUseCase` mode-split (tracked separately); keep decoupled.

    **Spec**:
    - Domain: `PlaylistEntry.id`; mapper populates from `DBPlaylistTrack.id` (`mapper.py:114`).
    - Repository: in `playlist/core.py` consumption matching, match by `entry.id` first, fall back to `track_id` FIFO for id-less (newly-added) entries — preserving the incident-fix behavior for existing entries.
    - Schema: `id: UUID` on `PlaylistEntrySchema` + `to_playlist_entry`.
    - Three frozen-Command use cases: `AddPlaylistTracksUseCase` (`playlist_id`, `track_ids`, `position=None`), `RemovePlaylistEntriesUseCase` (`playlist_id`, `entry_ids`), `ReorderPlaylistEntriesUseCase` (`playlist_id`, ordered `entry_ids`).
    - Endpoints (corrected types): `POST /playlists/{id}/tracks`, `DELETE /playlists/{id}/tracks/{entry_id}` + batch `DELETE /playlists/{id}/tracks` (body `entry_ids: UUID[]`), `PATCH /playlists/{id}/tracks/reorder` (`entry_ids: UUID[]`). Handlers 5–10 lines.
    - Frontend: switch row keys from `entry.position` to `entry.id` (`PlaylistDetail.tsx:896,914` — latent reconciliation bug under optimistic reorder); Add-Tracks search modal, checkbox/X remove, drag + keyboard reorder with optimistic update + revert-on-failure.
    - CLI parity: `mixd playlist add-tracks` / `remove-tracks` / `reorder`.
    - Docs: flip flows 3.4/3.5/3.6 to Implemented; re-run `pnpm --prefix web sync-api`.

    **Tests**:
    - **(integration, write FIRST — characterization)** Pin current duplicate-FIFO consumption: save `[A,B,A]`, reorder to `[A,A,B]` via the existing replace path; lock record-id stability. Net: `tests/integration/test_playlist_update_preservation_bugs_v2.py` (record-id stability through reorder/duplicate/removal) — do not break it.
    - (integration) `PlaylistEntry.id` survives load→save→load; `eq=False` keeps two same-track entries value-equal.
    - (integration) add appends; batch-remove removes only the named entry_ids (removing one of two identical tracks leaves the other); full-list reorder renumbers `sort_key`, preserves `added_at`.
    - (integration) reorder/remove against a stale `entry_id` → 404, list refetches.
    - (component) optimistic reorder reverts on API failure; rows keyed by `entry.id`.

    Effort: L | Dependencies: None (entry-identity threading is self-contained) | Status: Not Started

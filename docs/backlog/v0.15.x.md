# v0.15.x: Sovereignty & Gate Readiness

For strategic overview, see the [planning overview](README.md).

## User Workflow Context

The last cycle before the doors open (v1.0.0). Two things must be true before other people's data lives on a hosted mixd: their data must be *leavable* — scheduled, complete, re-importable archives to storage they control ([PDR-001](../decisions/PDR-001-data-ownership-model.md): exit rights precede hosting others' data) — and the instance must be *joinable*: a signup flow that doesn't require the operator's hand-holding, per-user limits that survive a curious stranger, and setup docs that stand alone. Nothing here is a feature for the existing user; all of it is the promise the gate makes to the next one.

Persona view: the **Tinkerer** gets data sovereignty made tangible; the **Casual Enthusiast** gets ownership without self-hosting; the **Weekly Curator** gets disaster insurance for years of curation.

Cross-cutting contract (stated once — per-epic Specs reference it): every new use case this cycle must be classified in the chat-tool parity registry (`src/application/tools/registry.py`) — CI (`tests/unit/application/tools/test_registry_parity.py`) fails on any unclassified use case. Reads get a `ToolSpec` or an explicit classification; mutations additionally get two-phase confirmation wiring like the existing `*_write` dispatchers.

### v0.15.0: Data Sovereignty — Archive & Exit Rights

**Goal**: The user's listening data exists at all times in a form they can hold — scheduled full exports in documented open formats, to storage they control, with a verified round-trip import path.

**Context**: Deliberately sequenced immediately before the v1.0.0 gate: exit rights must be real *before* other people's data lands on hosted instances. This is the direction-neutral half of the open data-ownership decision ([PDR-001](../decisions/PDR-001-data-ownership-model.md)) — every candidate answer, including a future local-first pivot, requires this machinery; nothing here forecloses that decision. It is also the concrete mechanism behind the "Users own their data" core principle: ownership through control-and-exit (the credibility model behind Obsidian's "your notes are just files"), delivered on a server-authoritative architecture.

**What this unlocks**:
- mixd (hosted or self-hosted) can disappear without taking years of play history, likes, and playlists
- The hosted-instance ownership story for non-technical users: account + automatic archive to their own storage + a documented way to walk away
- Groundwork for instance migration ("leave to a friend's server") and any future federation

**Persona**: All three — the Tinkerer gets data sovereignty made tangible; the Casual Enthusiast gets ownership without self-hosting; the Weekly Curator gets disaster insurance for years of curation.

**Key Design Decisions**:
- **Scheduled, not on-demand only** — ownership that requires remembering a button isn't ownership; a stale archive is a broken promise.
- **Usable without mixd** — documented open formats (JSON/CSV per entity + one SQLite bundle); format docs ship with the feature.
- **Round-trip is the bar** — an archive that can't import into a fresh instance is a backup, not an exit right. Full cross-instance migration (ID remapping) can follow later; single-user fidelity is required now.

#### Data Sovereignty Epics

- [ ] **Continuous Personal Archive**

    **Story**: The user wants their listening data to exist in a form they can hold — files in documented open formats, on storage they control — so that mixd could disappear tomorrow without taking years of play history, likes, and playlists with it.

    **Decisions**:
    - **User-controlled destinations**: local download, and at least one push target (user's own S3/WebDAV/Dropbox) so hosted-instance users get the same guarantee as self-hosters.
    - Runs on the existing scheduler; cadence + destination in settings.
    - **The archive exports the observation ledger, not just canonical plays** — `connector_plays` + `play_sources` are the authoritative play record (v0.10.0); re-import restores the ledger and replays the projection (`mixd plays rebuild`), which makes the play-history round-trip exact *by construction* rather than by careful copying. An archive of only `track_plays` would be a backup, not an exit right — it couldn't survive a future merge-algorithm change.

    **Spec**:
    - Export bundle: all user-scoped tables (~28, incl. `connector_plays` + `play_sources`) as JSON/CSV + SQLite; manifest with schema version + export timestamp
    - Import path for plays: restore ledger → replay projection (v0.10.0 rebuild command), not row-copy of `track_plays`
    - Scheduled job (existing scheduler infra); settings UI for cadence + destination
    - `mixd archive export` / `mixd archive import` CLI equivalents

    **Tests**:
    - (integration) Export → import into a fresh instance round-trips plays/likes/playlists/tags with timestamps intact
    - (integration) Play round-trip goes through the projection: imported canonical plays are byte-identical to the source instance's (ids aside)
    - (integration) Archive of a user with 50k+ plays completes and is re-importable
    - (unit) Manifest schema-version mismatch fails import with an actionable message

    Effort: L | Dependencies: None | Status: Not Started

### v0.15.1: Onboarding & Multi-User Hardening

**Goal**: A stranger with an invite can sign up, connect their services, and import their history — without the operator in the loop and without endangering the instance.

**Context**: mixd's multi-user machinery (RLS, per-user OAuth, per-user schedules) exists and is exercised by one user. This milestone is the gap between "multi-tenant by architecture" and "multi-tenant in practice": the signup path, sensible per-user limits, and the abuse posture [PDR-002](../decisions/PDR-002-multi-user-scale-posture.md) documents. Sketch-level by design — anchor as direction, spec against reality when the cycle approaches.

**What this unlocks**: the v1.0.0 gate stops being aspirational — every checklist item it needs from the product side exists.

**Persona**: the *next* user — every decision here is judged by their first hour.

**Key Design Decisions**:
- **Invite-only by default** — registration stays gated (the v1.1.x invite-provenance system is the eventual trust primitive; a simple invite code suffices at gate time), because open registration before the social-era abuse machinery exists is PDR-002's documented trap.

#### Gate Readiness Epics

- [ ] **Onboarding & Multi-User Hardening**

    **Story**: The invited tester signs up, connects Spotify/Last.fm, imports their history, and schedules their syncs — entirely self-serve — while the operator sleeps.

    **Spec** (starting points, revisit when scheduled):
    - Signup flow polish: invite gate, first-run experience (connect prompts, import guidance)
    - Per-user quotas/limits review: import sizes, schedule counts, API-key validation rate limits
    - PDR-002 abuse-posture review actioned (BYPASSRLS decision, cache thundering-herd stance)
    - Self-serve connector setup docs (BYO keys/tokens per connector)
    - Ops runbook: adding/removing a user, quota bumps, incident basics

    **Tests**:
    - (manual — the milestone acceptance) a fresh account completes signup → connect → import → schedule with zero operator intervention

    Effort: M | Dependencies: v0.15.0 (archive — exit rights precede invitations) | Status: Not Started
